LLM locaux sous macOS : Ollama, LM Studio, llama.cpp, MLX et Apple Silicon

Envoyer chaque requête vers une API tierce devient rapidement fastidieux, surtout lorsque la moitié des demandes consistent en des requêtes du type « réécrivez ce paragraphe » ou « quel est le schéma JSON pour ceci ». Les LLM locaux ont résolu ce problème pour moi sur Apple Silicon plus rapidement que je ne l’aurais imaginé. Un modèle de 7 milliards de paramètres quantifié en 4 bits fonctionne sans problème sur un MacBook de 16 Go, et toute interaction se fait directement depuis le clavier.

La question qui reste donc en suspens est de savoir quel logiciel utiliser pour les faire fonctionner. Ollama, LM Studio, llama.cpp, MLX, ainsi que quelques autres outils, reposent tous sur des moteurs d’inférence similaires et les mêmes fichiers GGUF. Ils diffèrent quant au niveau de complexité nécessaire entre l’utilisateur et le modèle : d’un côté, il suffit de faire un double-clic et de taper ; de l’autre, il faut compiler à partir du code source avant de consulter la documentation.

En résumé : utilisez Ollama pour une API locale simple, LM Studio pour une interface graphique et un endpoint compatible OpenAI, llama.cpp lorsque vous avez besoin d’un contrôle direct sur l’inférence GGUF, et MLX si vous souhaitez des workflows Python natifs pour Apple Silicon.

Concepts clés concernant les LLM locaux sous macOS

  1. Inférence : exécution du modèle afin de générer du texte.
  2. Quantification (GGUF) : technique permettant de réduire la taille du modèle avec une perte de qualité minimale. Vous verrez des noms de fichiers tels que llama-3-8b-Q4_K_M.gguf. Le Q4 par « part », on entend une quantification à 4 bits, qui consomme bien moins de mémoire RAM que les poids à 16 bits complets.
  3. Apple Silicon (Metal) : Les puces de la série M d’Apple partagent la mémoire RAM entre le CPU et le GPU (Apple qualifie cela de « Unified Memory »). C’est pourquoi un MacBook disposant de 32 Go ou 64 Go peut charger des modèles qui nécessiteraient une GPU dédiée coûteuse sur un ordinateur de bureau.

Prérequis

  • Matériel : il vous faut un Mac équipé d’Apple Silicon (de la série M1 à M4). Les Macs Intel fonctionnent également, mais ils sont nettement plus lents.
  • RAM :
    • 8 GB : suffisant pour de petits modèles (Mistral 7B, Llama 3 8B).
    • 16 GB+ : idéal pour des modèles plus volumineux ainsi que pour le multitâche.
  • Espace disque : les modèles occupent beaucoup d’espace. Prévoyez environ 10 à 20 GB pour disposer d’une bibliothèque de base utile.

1. Ollama – L’option « qui fonctionne simplement »

Téléchargement : ollama.com

Pensez à Ollama comme au « Docker pour les LLM ». Il encapsule le llama.cpp Le moteur, intégré dans un paquet natif macOS, charge les modèles sur demande et redirige automatiquement le traitement vers Metal. Vous l’installez, exécutez une seule commande, et vous pouvez commencer à chatter. C’est la solution la plus simple si vous souhaitez simplement disposer d’un LLM fonctionnel ainsi que d’une interface HTTP vers laquelle pointer votre code.

Flux de travail exemple

# 1. Download and run Llama 3 (it auto-downloads if needed)
ollama run llama3

# 2. Use it in your code via the local API
curl http://localhost:11434/api/generate -d '{
  "model": "llama3",
  "prompt": "Explain quantum computing to a 5-year-old",
  "stream": false
}'

Avantages et inconvénients

✅ Avantages❌ Inconvénients
Mise en place la plus simple (glisser-déposer) .dmg)L’application principale est à source fermée.
Cli propreollama list, ollama pull)Un contrôle moins fin sur les paramètres de génération
Vaste bibliothèque de modèles prédéfinis

2. LM Studio – L’explorateur visuel

Téléchargement : lmstudio.ai

LM Studio représente l’option axée sur l’interface graphique. Il dispose d’un navigateur de type App Store permettant de rechercher directement sur HuggingFace, prend en charge le format MLX d’Apple ainsi que GGUF (qui peut être plus rapide sur certains Macs), et expose un serveur local compatible avec OpenAI. Par conséquent, le code client existant qui communique déjà avec OpenAI fonctionne généralement sans modification.

Flux de travail typique

Vous pouvez utiliser le SDK Python officiel, ou tout client compatible avec OpenAI orienté vers le serveur local :

# Using the official LM Studio SDK
from lmstudio import LMStudio

client = LMStudio()
response = client.complete(
    model="llama-3-8b",
    prompt="Write a haiku about debugging."
)
print(response.content)

Avantages et inconvénients

✅ Avantages❌ Inconvénients
Interface affinée et facile à utiliserLa interface graphique est à source fermée.
Support natif tant pour les modèles GGUF que pour les modèles MLXTéléchargement plus volumineux (~750 MB)

3. llama.cpp – Outil destiné aux utilisateurs avancés

Répertoire : github.com/ggml-org/llama.cpp

Il s’agit du moteur que presque tous les autres outils utilisent en couche intermédiaire. Si vous souhaitez obtenir des performances maximales, accéder aux dernières fonctionnalités dès leur sortie, ou intégrer un LLM dans votre propre application en C++, c’est ici que vous devez vous tourner. Il s’agit d’une solution basée sur le matériel brut et très légère, mais à ce prix, vous devez gérer vous-même tout le processus : les téléchargements, les formats, ainsi que des dizaines d’options en ligne de commande.

Flux de travail exemple

# 1. Install via Homebrew
brew install llama.cpp

# 2. Download a model manually (e.g., from HuggingFace)
huggingface-cli download TheBloke/Llama-3-8B-Instruct-GGUF --local-dir .

# 3. Run inference with full control
llama-cli -m llama-3-8b-instruct.Q4_K_M.gguf \
  -p "Write a python script to sort a list" \
  -n 512 \
  --temp 0.7 \
  --ctx-size 4096

Avantages et inconvénients

✅ Avantages❌ Inconvénients
Contrôle total sur chaque paramètreCourbe d’apprentissage très raide (uniquement en ligne de commande)
Licencié sous MIT (logiciel open source)Gestion manuelle des modèles
Très léger (<30 MB)

4. GPT4All – Priorité à la confidentialité et RAG

Téléchargement : gpt4all.io

GPT4All repose sur deux principes fondamentaux : la confidentialité et les documents. Sa fonction phare, LocalDocs, vous permet d’indiquer à l’application un dossier contenant des PDF, des notes ou du code, afin de discuter directement avec son contenu. Tout s’exécute en mode hors ligne, sans collecte de données de suivi. C’est le moyen le plus simple d’installer un système RAG fonctionnel sur votre machine, sans avoir à écrire le moindre code.

Avantages et inconvénients

✅ Avantages❌ Inconvénients
LocalDocs RAG fonctionne parfaitement sans aucune configuration supplémentaire.Uniquement en mode GUI (pas de mode sans interface)
Entièrement hors ligne et privéUtilisation des ressources plus élevée que celle d’Ollama

5. KoboldCPP – Pour les conteurs

Repo : github.com/LostRuins/koboldcpp

Une branche de llama.cpp Destiné à l’écriture créative et aux RPG de type plateau. Il fonctionne en tant qu’application web locale, équipée d’outils pour la génération de longs textes : « Informations sur le monde », mémoire des personnages, ainsi que des techniques de cohérence narrative visant à maintenir le modèle dans le bon cadre même lors de générations de plusieurs milliers de tokens. Ce produit s’adresse aux écrivains et aux personnes qui organisent des RPG basés sur du texte ; si vous préférez principalement discuter ou coder, l’interface utilisateur standard paraîtra trop restreinte.

Flux de travail exemple

# 1. Download the single binary
wget https://github.com/LostRuins/koboldcpp/releases/latest/download/koboldcpp-mac.zip

# 2. Run it (launches a web server)
./koboldcpp --model llama-3-8b.gguf --port 5001 --smartcontext

Avantages et inconvénients

✅ Avantages❌ Inconvénients
Outils puissants pour l’écriture créativeUI spécialisée (pas idéale pour le codage ou les conversations)
Exécutable unique en un seul fichier (pas d’installation requise)Licence AGPL (restreinte pour une utilisation commerciale)

Mention d’honneur : MLX-LM

Si vous êtes un développeur Python travaillant sur Apple Silicon, consultez MLX-LM de la part d’Apple. Il s’agit d’un framework optimisé pour les puces de la série M, et sur le matériel adéquat, c’est souvent le moyen le plus rapide d’exécuter un modèle localement. L’inconvénient est qu’il offre moins de contraintes que Ollama : davantage de code en Python et moins de règles de sécurité.

Que choisir concrètement

  • Commencez par Ollama si vous souhaitez simplement avoir quelque chose en fonctionnement dès aujourd’hui.
  • Optez pour LM Studio si vous préférez d’abord explorer les modèles de manière visuelle.
  • Sélectionnez llama.cpp lorsque vous avez besoin d’un contrôle total sur l’inférence.

Points clés

  1. Ollama représente la voie la plus rapide pour disposer d’un serveur de modèles local aux fins de développement.
  2. LM Studio est le plus performant lorsque l’on souhaite disposer d’une interface graphique desktop et d’un accès simple à l’exploration des modèles.
  3. llama.cpp offre le contrôle le plus direct sur les modèles GGUF ainsi que sur les paramètres de exécution de bas niveau.
  4. MLX représente la solution native d’Apple Silicon pour ceux qui maîtrisent l’écriture en Python et l’utilisation des formats de modèles MLX.
