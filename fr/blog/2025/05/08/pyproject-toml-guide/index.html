    pyproject.toml Guide : Emballage de Python, dépendances et configuration des outils

    Si vous avez déjà ouvert un projet Python et tenté de déterminer où se trouvent réellement les dépendances, les paramètres de compilation ainsi que les configurations des outils, vous connaissez bien cette frustration. setup.py, setup.cfg, requirements.txt, MANIFEST.in, ainsi que quelques fichiers dotfiles pour chaque outil de linting et de formatage — tous lisant depuis des emplacements différents.

    pyproject.toml il regroupe la majeure partie de ces éléments en un seul fichier.

    Résumé

    pyproject.toml est d’environ le package.json pour Python. Un seul fichier contient les métadonnées de votre projet, ses dépendances et les paramètres des outils. Que vous utilisiez .venv, pyenv, ou uv, en plaçant tout ici, on facilite la configuration ainsi que la collaboration.

    Qu’est-ce que pyproject.toml? PEP 518 (2016) a introduit le [build-system] une table permettant aux outils de construction de déclarer leurs exigences de manière standardisée. PEP 621 (2020) a ajouté le [project] table contenant les métadonnées du paquet principal : nom, version, dépendances, et ce genre de choses.

    Aujourd’hui, la plupart des outils Python (Black, isort, pytest, Ruff, mypy) lisent leurs paramètres de configuration depuis [tool.*] les sections de ce fichier.

    Pourquoi c’est important

    Moins de fichiers à gérer

    Un projet hérité typique devait gérer en même temps setup.py, setup.cfg, requirements.txt, MANIFEST.in, ainsi qu’un groupe de fichiers dotfiles (.flake8, .coveragerc, et similaires). La plupart d’entre eux se regroupent en un seul endroit.

    Construis indépendants du backend

    Lorsque vous exécutez pip install ., pip lit pyproject.toml et installe tous les outils de compilation nécessaires à votre projet : setuptools, flit, hatchling, choisissez celui que vous préférez. Vous n’êtes plus contraint d’utiliser uniquement setuptools.

    Une seule source pour la configuration des outils

    Les analyseurs de code, les formatteurs, les exécutants de tests et les vérificateurs de types savent où chercher. Votre IDE, votre pipeline CI et vos collègues consultent le même fichier.

    Anatomie d’un pyproject.toml

    Un fichier typique comporte trois sections principales :

    # 1. Build system - tells pip/build how to package your project
[build-system]
requires = ["hatchling"]
build-backend = "hatchling.build"

# 2. Project metadata and dependencies
[project]
name = "awesome-app"
version = "0.1.0"
description = "Short demo of pyproject.toml"
readme = "README.md"
requires-python = ">=3.12"
dependencies = [
  "fastapi>=0.111",
  "uvicorn[standard]>=0.30",
]

# Expose CLI commands
[project.scripts]
awesome-cli = "awesome_app.cli:main"

# Optional dependencies (e.g., for development)
[project.optional-dependencies]
dev = ["pytest", "ruff", "mypy"]

# 3. Tool configuration
[tool.ruff]
line-length = 100
target-version = "py312"

[tool.pytest.ini_options]
addopts = "-ra -q"
testpaths = ["tests"]

    Décortiquage [build-system]

    Obligatoire si vous souhaitez empaqueter ou distribuer votre projet. Il indique à pip et aux outils de compilation (comme python -m build) quel backend utiliser. [project]

    Vos métadonnées de paquet. C’est ici que se trouvent les dépendances au lieu de requirements.txt.

    • dependencies: les exigences en termes de temps d’exécution pour votre paquet. optional-dependencies: groupes de dépendances supplémentairesdev, test, docs).
    • scripts: crée des commandes exécutables. Dans l’exemple ci-dessus, l’installation du paquet vous fournit une awesome-cli commande qui exécute le main fonction dans awesome_app/cli.py.

    [tool.*]

    Configuration pour tout outil qui la prend en charge. Chaque outil dispose de son propre espace de noms : [tool.pytest.ini_options], [tool.mypy], [tool.ruff], et le reste.

    Remplace-t-il requirements.txt?

    Pour les workflows modernes, oui. Des outils tels que Poésie, PDM, Hachure, et UV stocker les dépendances directement dans le [project] section et générer des fichiers de verrouillage pour assurer la reproductibilité.

    Vous souhaitez probablement encore requirements.txt if :

    • Vous travaillez avec un système de déploiement obsolète qui l’exige.
    • Vous disposez d’un script CI qui n’a pas encore été mis à jour.

    La plupart des outils modernes permettent d’exporter l’un des vôtres pyproject.toml lorsque c’est nécessaire :

    uv export > requirements.txt

    Sélection d’un backend de compilation

    Le choix du backend de compilation fait partie des décisions les plus déroutantes. Voici une comparaison rapide :

    Backend Idéal pour Avantages Cons
    Nourrisson Standard moderne Rapide, extensible, prend en charge les plugins Support plus récent avec moins de prise en charge des anciennes versions
    Flit Paquets simples Extrêmement simple, aucune configuration requise Pas adapté aux builds complexes (extensions C)
    Setuptools Ancien et complexe Prend en charge tout (extensions C, etc.). Plus lent, nécessite plus de configuration
    Poésie Utilisateurs de Poetry Intégré à l’écosystème de Poetry Verrouillé dans le flux de travail Poetry

    Pour les nouveaux projets en Python pur, Hatchling constitue un choix par défaut raisonnable. C’est ce que uv init écrit pour vous.

    Migrer un projet existant

    Si vous disposez d’un projet Python ancien, voici comment le moderniser :

    1. Ajouter [build-system]. Si vous n’êtes pas sûr du backend à utiliser, commencez par requires = ["setuptools>=61", "wheel"]2. Déplacer les métadonnées vers [project]. Transférer le nom, la version et les dépendances depuis setup.py ou setup.cfg3. Convertir les dépendances de développement. Les placer dans [project.optional-dependencies].dev.
    2. Configurer les outils. Ajouter [tool.*] Sections pour Black, pytest, mypy, etc.
    3. Déterminer ce qu’il convient de faire avec requirements.txt. Soit supprimez-le, soit générez-le à partir de votre fichier de verrouillage pour les systèmes anciens qui en ont besoin.

    Après la migration, vous pouvez généralement le supprimer. setup.py, setup.cfg, ainsi que la plupart de ces fichiers de configuration au format dotfile.

    Quelques fonctionnalités utiles

    Points d’entrée CLI

    Au lieu de l’ancien console_scripts bloquer dans setup.py, utiliser [project.scripts]:

    [project.scripts]
my-tool = "my_package.main:run"

    Quand quelqu’un installe votre paquet, il peut saisir my-tool dans leur terminal.

    Espaces de travail (monorepos)

    Des outils tels que uv et hatch prise en charge des espaces de travail, qui vous permettent de gérer plusieurs paquets au sein d’un seul répertoire :

    [tool.uv.workspace]
members = ["packages/*"]

    Vous pouvez développer plusieurs paquets interdépendants et les installer tous dans un même environnement virtuel aux fins de test.

    Flux de travail typiques avec uv

    Lancement d’un nouveau projet

    uv init my_app          # creates folder with pyproject.toml and .venv
cd my_app
uv add requests fastapi # adds to [project.dependencies] and installs
uv run pytest           # runs tests in the venv

    Exécution des scripts

    Vous pouvez soit définir des scripts dans pyproject.toml (en utilisant un exécuteur de tâches comme poe ou hatch) ou simplement appeler uv run:

    uv run python main.py

    Conseils pratiques

    1. Ne fixez pas de versions précises dans les bibliothèques. Utilisez plutôt des plages comme requests>=2.30 Ainsi, votre bibliothèque ne entrera pas en conflit avec les autres éléments installés dans l’environnement de l’utilisateur.
    2. Fixez les versions dans les applications. Utilisez un fichier de verrouillage (uv.lock, poetry.lock) afin d’obtenir des builds reproductibles.
    3. Groupez les dépendances de développement. Conservez les dépendances liées aux tests, au linting et à la documentation dans des groupes optionnels distincts.dev, test, docs).
    4. Ne pas afficher toutes les options en même temps pyproject.toml. Respectez les paramètres par défaut au niveau du projet et laissez les fichiers de configuration spécifiques à l’outil gérer le reste en cas de complexité. pyproject.toml Il s’agit principalement d’éliminer les petits goulots d’étranglement quotidiens. Vous n’avez plus besoin de chercher quel fichier est responsable de la compilation. La configuration du linter se trouve à côté des dépendances. Pip et votre IDE sont enfin d’accord sur ce qui est installé. Choisissez un backend de compilation et placez vos dépendances à l’intérieur [project], et permettez à vos outils de lire depuis une seule source. Si vous commencez à zéro, uv init vous permet d’aller très loin dans votre travail avec une seule commande.

    Points clés

    1. pyproject.toml C’est le fichier de configuration central des projets Python modernes.
    2. Le PEP 518 définit les exigences relatives au système de construction ; le PEP 621 définit les métadonnées du projet.
    3. Conservez la configuration des outils dans pyproject.toml Lorsque l’outil le prend en charge, mais ne forcez pas des configurations non liées dans un seul fichier.
    4. Les fichiers verrouillés et les métadonnées de construction résolvent des problèmes différents : des installations reproductibles par rapport à l’identité du paquet.
