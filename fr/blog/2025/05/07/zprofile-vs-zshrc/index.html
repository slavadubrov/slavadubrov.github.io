Fichiers de démarrage de Zsh: ~/.zprofile vs ~/.zshrc sur macOS et Linux

Si votre terminal semble lent, ou si vos variables d’environnement ne sont pas chargées là où vous l’attendiez, c’est probablement dû à l’ordre de démarrage de Zsh.

La séparation entre ~/.zprofile et ~/.zshrc c’est l’une des causes les plus fréquentes de confusion lorsqu’on passe à Zsh, en particulier sous macOS, où les paramètres par défaut se comportent différemment de ceux de Linux.

TL;DR

~/.zprofile Ce fichier sert à la configuration de l’environnement. Il est exécuté une fois par connexion, ce qui, sous macOS, signifie une fois par onglet de terminal. Placez votre PATH, EDITOR, ainsi que les gestionnaires de versions tels que fnm ou pyenv Voilà.

~/.zshrc Il s’agit d’une configuration interactive. Elle s’exécute chaque fois que vous lancez un nouveau shell. Y placer les alias, les thèmes de prompt et les raccourcis clavier.

Le flux de démarrage du shell

Pour savoir où placer les éléments, il faut connaître le moment où les fichiers sont chargés. Zsh dispose d’une hiérarchie spécifique.

Shell de connexion vs. shell interactif

Un shell de connexion est le premier shell que vous obtenez après l’authentification. Sur macOS, chaque nouvelle fenêtre ou onglet de terminal est par défaut un shell de connexion. Sur Linux, l’ouverture d’un terminal démarre généralement un shell interactif non de connexion, ce qui est à l’origine de la plupart des confusions interplateformes.

Un shell interactif est tout environnement dans lequel vous pouvez taper des commandes.

Voici ce qui se passe réellement lorsque vous ouvrez un terminal sur macOS :

L’ordre de chargement

  1. ~/.zshenv (Optionnel). S’exécute pour chaque shell, y compris les scripts non interactifs. Ne placez pas de résultats ni de logique lourde ici — cela peut endommager les scripts qui chargent votre configuration. Utilisez-le uniquement pour les variables d’environnement qui doivent exister partout, ce qui est rare pour la plupart des utilisateurs.
  2. ~/.zprofile. S’exécute uniquement pour les shells de connexion. Votre phase de configuration.
  3. ~/.zshrc. S’exécute dans les shells interactifs. Votre phase de personnalisation.
  4. ~/.zlogin (opcinnel). S’exécute à la toute fin du démarrage d’un shell de connexion.

Où va-t-il ? ~/.zprofile: la couche d’environnement

C’est ici que vous configurez les éléments que tous les autres processus héritent — chemins, variables, gestionnaires de versions de langage.

Ce qui doit figurer ici :

  • PATH Modifications.
  • Variables d’environnement : EDITOR, LANG, GOPATH, JAVA_HOME.
  • Initialisation d’outil affectant l’environnement : pyenv, rbenv, fnm, cargoElles n’ont besoin d’être calculées qu’une seule fois. Si vous les mettez dans .zshrc, elles sont recalculées chaque fois que vous ouvrez un sous-shell ou exécutez un script, ce qui consomme du temps et peut entraîner des entrées dupliquées dans votre PATH.
# ~/.zprofile

# 1. Set up your PATH
# Local bin first so your tools override system ones
export PATH="$HOME/.local/bin:/opt/homebrew/bin:$PATH"

# 2. Global variables
export EDITOR="nvim"
export VISUAL="nvim"
export LANG="en_US.UTF-8"

# 3. Version managers (the heavy ones)
# Doing this here keeps shell startup fast
eval "$(fnm env --use-on-cd)"
eval "$(pyenv init -)"

~/.zshrc: la couche interactive

C’est ici que vous personnalisez la console dans laquelle vous tapez réellement.

Ce qui doit y figurer :

  • Aliases : alias g='git'.
  • Votre prompt : Starship, Powerlevel10k, Pure.
  • Complétions : compinit.
  • Liaisons de raccourci : bindkey.
  • Options de shell : setopt autocd, setopt histignorealldupsElles n’ont d’importance que lorsque c’est un humain qui est au clavier. Un script exécuté en arrière-plan n’a pas besoin de votre invite de commande ni de vos alias Git.
# ~/.zshrc

# 1. Prompt
autoload -Uz promptinit && promptinit
prompt pure

# 2. Aliases
alias ll='ls -lah'
alias g='git'
alias gs='git status'

# 3. Shell options
setopt autocd              # cd by typing the directory name
setopt histignorealldups   # skip duplicate history entries
setopt share_history       # share history across tabs

# 4. Completions
autoload -Uz compinit && compinit

Pourquoi ne pas tout mettre dans un seul fichier ?

Vous vous demandez peut-être : si .zprofile Il s’exécute en premier, pourquoi ne pas y définir vos alias et sauter cette étape ? .zshrc entièrement ?

Il s’agit essentiellement de la question de l’héritage par rapport à la redéfinition.

Les variables d’environnement sont héritées

Lorsque vous export EDITOR="vim" Dans un shell parent, chaque processus enfant l’hérite : sous-shells, scripts, programmes. Vous le définissez une seule fois et il se propage le long de l’arborescence, c’est pourquoi .zprofile est-il l’endroit approprié pour export### Les alias et les fonctions ne le font pas

Les alias tels que alias g='git' Les fonctions shell sont locales au shell actuel. Elles ne sont pas transmises aux shells enfants.

Si vous définissez un alias dans .zprofile, il existe dans votre shell de connexion de niveau supérieur. Dès que vous tapez zsh pour lancer un sous-shell ou exécuter un script, cet alias n’existe plus. Afin que les alias soient disponibles partout, il faut les redéfinir dans chaque nouveau shell interactif. C’est précisément ce qui .zshrc est destiné à.

Les scripts n’ont pas besoin de fonctionnalités humaines

Lorsque vous exécutez un script shell (./deploy.sh), il lance un nouveau shell non interactif. Il n’a pas besoin de votre invite de commande, il n’a pas besoin de vos alias Git, et il ne veut certainement pas attendre oh-my-zsh Pour finaliser le chargement. Conserver la configuration interactive en .zshrc Cela signifie que les scripts s’exécutent rapidement et de manière propre, sans que vos personnalisations personnelles ne se retrouvent infiltrées.

Pièges courants

Mise en œuvre nvm ou pyenv dans .zshrc

Vous ouvrez une nouvelle fenêtre de terminal et il faut deux ou trois secondes avant de pouvoir taper quoi que ce soit. Les gestionnaires de versions présentent généralement une phase d’initialisation lourde, et .zshrc s’exécute à chaque fois. Déplacez-les vers ~/.zprofile et le retard disparaît.

Une croissance PATH

Votre $PATH on se retrouve avec les mêmes dossiers listés cinq fois. La cause est presque toujours export PATH="$HOME/bin:$PATH" en attente dans .zshrc: à chaque rechargement (source ~/.zshrc) ou la sous-shell ajoute à nouveau le chemin. Déplacer PATH definitions vers ~/.zprofile.

Rechargement après modification

Changements apportés à ~/.zprofile n’est pas applicable à votre shell actuel, car .zprofile n’est lu qu’au moment de la connexion. Vous pouvez soit fermer cette fenêtre et en ouvrir une nouvelle (l’option la plus simple), soit exécuter source ~/.zprofile manuellement. .zshrc, simplement :

source ~/.zshrc

Une configuration fonctionnelle sous macOS et Linux

Si vous utilisez différents ordinateurs (MacOS au travail, Linux à la maison, ou l’inverse), vous rencontrerez un problème. Les terminaux Linux démarrent souvent en tant que shells sans connexion, ce qui signifie qu’ils sautent ~/.zprofile entièrement.

La solution de contournement habituelle consiste à utiliser une source .zprofile de .zshrc lorsqu’il n’a pas encore été chargé :

# ~/.zshrc

# On Linux/non-login shells, make sure the environment is set
if [[ -o interactive && ! -o login ]]; then
    [[ -f ~/.zprofile ]] && source ~/.zprofile
fi

# ... rest of your interactive config

Résumé

FichierObjectifExemples
~/.zshenvVariables d’environnement critiquesZDOTDIR (uniquement pour les utilisateurs avancés)
~/.zprofileMise en place de l’environnementPATH, EDITOR, eval "$(pyenv init -)"
~/.zshrcConfiguration interactivealias, prompt, bindkey, compinit

Deux règles couvrent la majeure partie de ce cas. Les variables sont héritées le long de l’arbre de processus, donc export doit figurer dans .zprofile. Les alias et les fonctions ne sont pas hérités, donc ils doivent être placés dans .zshrc et doivent être redéfinis pour chaque nouvelle shell interactive. Si les nouvelles onglets semblent lentes, la cause est presque toujours une charge importante pyenv init ou nvm.sh en attente dans .zshrc au lieu de .zprofile---

Principaux enseignements

  1. Utiliser .zprofile pour la configuration du shell de connexion, comme l’initialisation de PATH dans le Terminal macOS.
  2. Utiliser .zshrc pour le comportement de la shell interactive, tel que les alias, la configuration du prompt, les complétions et les options de shell.
  3. Ne dupliquez pas les modifications du PATH dans les deux fichiers ; cette duplication rend difficile le débogage de l’ordre de démarrage.
  4. En cas de doute, placez le démarrage de l’environnement à .zprofile et des fonctionnalités interactives avantageuses dans .zshrc.
