Conception pilotée par le domaine pour les agents IA : contextes bornés, outils et règles métier

Résumé

La conception pilotée par le domaine (DDD) fournit aux équipes d’agents IA un vocabulaire commun ainsi que des interfaces claires entre les sous-systèmes. L’utiliser lorsque vos requêtes s’éloignent des objectifs métier et que vos règles sont dispersées dans divers modèles.

Pourquoi la conception pilotée par le domaine est importante pour les agents IA

La plupart des projets d’agents échouent non pas à cause d’un code de mauvaise qualité, mais parce que ceux qui rédigent les instructions et ceux qui sont véritablement responsables du processus métier ne parviennent pas à s’accorder sur la signification des éléments concernés. La conformité exige une « vérification des politiques », mais se heurte à des résultats… process_data() méthode. Personne ne sait à quoi elle sert, ce qui entraîne des évolutions aléatoires des exigences et un durcissement progressif du système.

Le DDD résout ce problème en plaçant le domaine métier au cœur. Pas le schéma de base de données, ni le modèle de prompt, mais bien le processus réel du monde réel que vous essayez de modéliser. Les effets concrets sont les suivants :

  • Langage commun. Le service produit, les équipes opérationnelles et l’équipe d’ingénierie utilisent tous les mêmes termes. Lorsque la conformité mentionne une « demande de remboursement », c’est bien ce terme qui apparaît dans votre code, vos prompts et votre documentation.
  • Portée ciblée. Vous développez uniquement ce qui est essentiel : les flux de travail principaux et les règles dont quelqu’un est réellement responsable. Moins de code de liaison sujet aux pannes lors des changements d’exigences.
  • Adaptabilité. Lorsque les politiques changent, vous mettez à jour une partie bien définie plutôt que de devoir fouiller dans un monolithe.

Cela est particulièrement important dans des domaines où les règles évoluent fréquemment : la finance, la santé, les opérations régulées. Le DDD vous donne une chance réelle de rester à la hauteur.

Blocs de construction stratégiques

DDD est un ensemble d’outils basés sur des patrons plutôt que simplement une idée unique. Il est généralement divisé en deux parties :

  1. Conception stratégique : les aspects liés à la « vue d’ensemble ». Il s’agit de définir les limites, les équipes et les mécanismes de communication entre systèmes. Cela est essentiel pour les systèmes multi-agents.
  2. Conception tactique : les patrons au niveau du code (Entités, Agrégats) qui permettent de maintenir une logique interne propre chez vos agents.

Les concepts présentés ci-dessous sont ceux que vous utiliserez réellement au quotidien.

Langage universel

Le vocabulaire commun présent partout : réunions, documentation, instructions, noms de méthodes. Il n’existe pas de couche de traduction entre le « langage métier » et le « langage de code ».

Si les exigences réglementaires mentionnent « vérification de politique », votre méthode doit run_policy_check(), non process_data(). Si les médecins demandent « admettre le patient », vous écrivez admit_patient(), non add_user().

class PatientRegistry:
    def admit_patient(self, patient_id: str) -> None:
        """Admit a patient to the registry - term used by medical staff."""
        ...

Lorsque le langage utilisé dans le code correspond au langage parlé en salle, les modifications des exigences apparaissent sous forme de renommages évidents en un seul endroit. Vous cessez alors de débattre de ce process_data devait être fait.

Contextes bornés

Les systèmes de grande envergure nécessitent des limites explicites. Pourquoi ? Parce que le même terme peut avoir des significations différentes selon les différents domaines métier.

Prenons l’exemple du « produit » dans le e-commerce. Dans le contexte de l’Inventaire, un produit correspond à un élément de catalogue doté de codes SKU et de quantités en stock. Dans le contexte de la Facturation, il s’agit d’une ligne de facture incluant des règles de tarification et des calculs fiscaux. Enfin, dans la Gestion des commandes, il représente une quantité à livrer ainsi qu’une promesse de livraison.

Les contextes bornés permettent à chaque sous-domaine d’avoir sa propre définition sans conflit. Des couches de traduction ou des interfaces les relient lorsqu’ils doivent communiquer entre eux.

Cela permet de maintenir chaque modèle de taille réduite et d’éviter la création d’un objet « produit » géant qui devrait répondre aux besoins de trois équipes en même temps. Task avec l’ID 123 Il s’agit de la même tâche même si l’on modifie sa description, son statut ou sa date d’échéance. Deux entités sont égales si elles possèdent le même identifiant, indépendamment de leurs attributs.

from pydantic import BaseModel

class SupportTicket(BaseModel):
    ticket_id: str  # This is the identity
    customer: str
    issue: str
    status: str = "OPEN"

    def close(self) -> None:
        if self.status != "OPEN":
            raise ValueError("Ticket already closed")
        self.status = "CLOSED"

Les objets de valeur n’ont pas d’identité. Ils sont définis entièrement par leurs attributs. Deux TimeSlot Les objets ayant les mêmes horaires de début et de fin sont interchangeables. Les objets de valeur sont immuables ; au lieu de modifier un objet existant, il convient de créer un nouveau.

Utilisez des entités pour représenter les éléments ayant un cycle de vie.Order, User, AgentSession). Utilisez des objets de valeur pour les descriptions et les mesures.EmailAddress, Priority, Location).

from datetime import date
from pydantic import BaseModel, Field

class Task(BaseModel):
    id: str
    description: str

Le code externe ne touche jamais le tasks liste directement. Il appelle toujours add_task(). C’est ce qui garantit que la règle « pas d’ID dupliqués » ne peut être violée. Lorsque vous enregistrez des données dans une base de données, vous enregistrez généralement l’ensemble de l’agrégat d’un seul coup.

Répertoires

Les répertoires masquent la couche de persistance. Du point de vue du domaine, vous effectuez simplement une appel save(plan) et get(plan_id). Le fait que ces appels aboutissent finalement à Postgres ou Redis relève du problème de quelqu’un d’autre.

Cela présente deux avantages : les tests peuvent utiliser un répertoire en mémoire au lieu de simuler des appels à la base de données. De plus, lorsque l’on remplace finalement SQLite par une solution plus lourde, les règles métier restent inchangées.

from abc import ABC, abstractmethod
from pydantic import BaseModel, Field

class Task(BaseModel):
    id: str
    description: str
    completed: bool = False

class Plan(BaseModel):
    id: str
    tasks: list[Task] = Field(default_factory=list)

class PlanRepository(ABC):
    """Domain layer defines the interface."""
    @abstractmethod
    def save(self, plan: Plan) -> None:
        ...

    @abstractmethod
    def get(self, plan_id: str) -> Plan | None:
        ...

class InMemoryPlanRepository(PlanRepository):
    """Infrastructure layer provides the implementation."""
    def __init__(self) -> None:
        self.storage: dict[str, Plan] = {}

    def save(self, plan: Plan) -> None:
        self.storage[plan.id] = plan

    def get(self, plan_id: str) -> Plan | None:
        return self.storage.get(plan_id)

Votre code de domaine ne connaît que PlanRepository (l’interface). La couche d’infrastructure intègre l’implémentation réelle.

Événements de domaine

Les événements de domaine permettent de capturer les événements importants survenus au sein de votre système. Leur nommage est au passé composé (OrderPlaced, TaskCompleted, PaymentFailed) car ils décrivent des faits, et non des commandes. Les événements rendent explicites les effets secondaires implicites. Au lieu qu’un module appelle directement un autre lorsqu’un événement se produit, le domaine déclenche cet événement. D’autres parties du système s’abonnent à lui et y réagissent de manière indépendante.

from datetime import datetime
from pydantic import BaseModel

class TaskCompleted(BaseModel):
    task_id: str
    completed_at: datetime

Lorsqu’une tâche est terminée, vous émettez TaskCompleted. Un service de notification peut écouter cet événement et envoyer un e-mail. Un service de génération de rapports peut l’enregistrer à des fins d’analyse. Le point crucial : l’agrégateur de tâches n’a pas besoin de connaître les e-mails ou les analyses ; il se contente d’annoncer ce qui s’est produit.

C’est ainsi que la communication inter-contextuelle reste découplée. Cela convient également naturellement aux systèmes multi-agents, puisque ces derniers réagissent déjà les uns aux autres via des événements.

Traduire le DDD en architectures d’agents

Les véritables systèmes d’agents disposent de flux de travail à plusieurs étapes, des sorties de modèles de langage grand public qui sont erronées dans une certaine proportion de cas, ainsi que des exigences qui changent chaque trimestre. Les patterns du DDD s’avèrent particulièrement adaptés à ces problèmes.

Les contextes bornés deviennent des agents ou des compétences

Chaque agent (ou capacité majeure) représente un contexte borné. Un orchestrateur de recherche peut coordonner trois agents spécialisés :

  • Agent des tendances — collecte les données de marché en utilisant sa propre terminologie et ses outils
  • Agent de conformité — effectue des vérifications de politique en s’appuyant sur la terminologie réglementaire
  • Agent des coûts — estime les dépenses en appliquant des règles spécifiques au domaine financier

Chacun dispose de son propre modèle, de sa propre terminologie ainsi que de ses propres invariants. Ils communiquent via des interfaces ou des événements bien définis.

Même dans un système à un seul agent, il est possible de définir des contextes internes. Un module de planification et un module d’exécution, chacun disposant de son propre modèle de domaine.

Les prompts respectent le langage universellement utilisé

Utilisez des termes propres au domaine dans les prompts du système, les descriptions des outils et les signatures des fonctions. Si les experts en conformité emploient l’expression « vérification de politique », cette phrase exacte doit figurer dans vos prompts ainsi que dans votre code. L’avantage est simple : lorsque le suivi d’un agent montre run_policy_check, l’équipe de conformité peut le lire sans recourir à un traducteur.

Les états deviennent des entités explicites

Les LLM sont souvent sans état, mais les agents réels suivent de nombreux états : sessions de conversation, objectifs, résultats intermédiaires, sorties d’outils. Modélisez-les sous forme d’entités ou d’objets de valeur : ConversationSession entité dotée d’une identifiant et d’un historique de messages Task entité représentant des unités de travail ToolOutput Objet de valeur pour des résultats immuables

Une fois que ces éléments sont des objets explicites, il est possible d’y associer des règles de validation et des règles métier. A Task Une entité peut refuser d’être finalisée tant que ses dépendances ne sont pas terminées, sans que cette règle doive être répétée dans trois modèles de prompt différents.

Les agrégats expriment les plans des agents

A Plan La racine d’agrégation gère la liste des tâches et applique toutes les contraintes importantes pour l’entreprise. Lorsqu’un LLM propose d’ajouter 50 tâches alors que votre politique autorise seulement 10, la racine d’agrégation refuse ces tâches supplémentaires. De même, si le modèle suggère des tâches redondantes, la racine les rejette également. Le modèle peut être très enthousiaste ; le domaine, lui, reste maîtrisé.

Les événements du domaine pilotent l’orchestration

Les agents génèrent des événements tels que ResearchCompleted, ThresholdExceeded, ou PolicyViolationDetected. D’autres agents ou services s’abonnent et réagissent en conséquence. Rien n’est préconfiguré, ce qui permet d’ajouter un nouvel écouteur (ou un nouvel agent) à moindre coût.

Les règles métier encadrent les actions de l’IA

Les sorties des modèles LLM passent par des services de domaine ou des méthodes d’entité plutôt que directement dans la base de données. Si un modèle propose un remboursement dépassant les limites définies par la politique, votre RefundRequest Il le valide puis le rejette. Un LLM peut improviser ; ce sont les règles métier qui ont le dernier mot.

La couche anti-corruption (ACL)

Un LLM est probabiliste et se trompe parfois de manières surprenantes. Votre modèle de domaine doit rester déterministe. Les deux ne peuvent pas coexister directement.

C’est là que intervient la couche anti-corruption (ACL).

L’ACL se situe entre le modèle et le domaine d’application. Il transforme la sortie brute du LLM en types stricts attendus par ce domaine.

  1. Ingérer du texte brut ou du JSON provenant du LLM.
  2. Valider la structure et les types à l’aide de modèles Pydantic.
  3. Sanitiser les valeurs (pas de prix négatifs, pas de transactions datées à l’avance, etc.).
  4. Transformer les DTO (objets de transfert de données) en entités du domaine.

Si la validation échoue, l’ACL rejette les données et renvoie souvent l’erreur à l’LLM afin qu’il puisse tenter à nouveau. Le principe est simple : seules les données validées atteignent jamais votre logique métier principale.

Exemple : un assistant de tâches modélisé avec DDD

Nous allons construire un assistant personnel de tâches capable de gérer des demandes telles que « Rappelle-moi d’acheter du lait demain » ou « Qu’y a-t-il sur ma liste de tâches ? ». Cette démarche applique progressivement les éléments DDD mentionnés ci-dessus.

1. Cartographier les contextes

Commencez par décomposer le problème en sous-domaines :

  • Gestion des tâches — gestion des éléments de liste et des rappels (domaine principal)
  • Planification — événements calendaires et réunions
  • Notifications — envoi d’alertes et d’e-mails

Nous nous concentrerons d’abord sur la Gestion des tâches. Les autres peuvent évoluer en tant que contextes bornés distincts ou en agents complémentaires.

2. Utiliser le même langage

Choisissez le vocabulaire en fonction des personnes qui gèrent réellement le processus (ou faites appel au bon sens pour une application personnelle) : « tâche », « échéance », « rappel », « priorité ». Ensuite, utilisez ces termes exacts dans les modèles de prompts, les noms de méthodes et les étiquettes d’interface. Il n’existe pas de traduction distincte destinée au « monde professionnel ».

3. Capturer les entités, les objets de valeur et les événements

Modélisez maintenant les concepts fondamentaux :

  • Entité : Task avec identitéid) ainsi que l’état mutablecompleted)
  • Objet de valeur : Priority enum (immuable, défini par sa valeur)
  • Événement de domaine : TaskCompletedEvent pour signaler la fin du travail
from datetime import datetime, date, timezone
from enum import Enum
from pydantic import BaseModel, Field

class Priority(Enum):
    """Value object: priority is defined by its value alone."""
    LOW = 1
    NORMAL = 2
    HIGH = 3

class TaskCompletedEvent(BaseModel):
    """Domain event: announces a task was completed."""
    task_id: str
    time: datetime

class Task(BaseModel):
    """Entity: identity persists even as attributes change."""
    id: str
    description: str
    created_at: datetime = Field(default_factory=lambda: datetime.now(timezone.utc))
    due_date: date | None = None
    priority: Priority = Priority.NORMAL
    completed: bool = False

    def mark_completed(self) -> TaskCompletedEvent:
        """Business rule: can't complete an already-completed task."""
        if self.completed:
            raise ValueError("Task is already completed.")
        self.completed = True
        return TaskCompletedEvent(task_id=self.id, time=datetime.now(timezone.utc))

La règle métier (il est impossible de finaliser une tâche déjà terminée) est définie dans la méthode d’entité, et non dans un modèle de prompt. TaskList c’est notre racine globale. Elle contient plusieurs Task Les entités et applique des règles de cohérence entre elles. Toutes les modifications passent par les méthodes du racine.

from datetime import date
from pydantic import BaseModel, Field

class Task(BaseModel):
    id: str
    description: str
    due_date: date | None = None
    completed: bool = False

class TaskList(BaseModel):
    """Aggregate root: enforces invariants across all tasks."""
    owner: str
    tasks: list[Task] = Field(default_factory=list)

    def add_task(self, task: Task) -> None:
        """Business rule: no duplicate tasks on the same day."""
        if any(
            existing.description == task.description
            and existing.due_date == task.due_date
            for existing in self.tasks
        ):
            raise ValueError("A similar task on that date already exists.")
        self.tasks.append(task)

    def get_pending(self) -> list[Task]:
        """Query helper: find tasks that aren't done yet."""
        return [task for task in self.tasks if not task.completed]


TaskList.model_rebuild()  # Resolve forward references for Pydantic.

Le code externe ne touche jamais tasks directement. Il passe toujours par ce chemin. add_task() ou une autre méthode de racine, qui est ce qui garantit le respect de la règle « pas de doublons ».

5. Encapsuler la persistance dans un répertoire de stockage

Le répertoire de stockage abstrait l’opération d’enregistrement des données. La couche de domaine ne sait pas si les tâches sont stockées en mémoire ou dans Postgres.

from pydantic import BaseModel, Field

class Task(BaseModel):
    id: str
    description: str
    completed: bool = False

class TaskList(BaseModel):
    owner: str
    tasks: list[Task] = Field(default_factory=list)

TaskList.model_rebuild()  # Resolve forward references for Pydantic.

class TaskRepository:
    """Abstracts task storage - in-memory implementation for simplicity."""

    def __init__(self) -> None:
        self._data: dict[str, TaskList] = {}

    def get_task_list(self, owner: str) -> TaskList:
        """Retrieve a user's task list, or create a new empty one."""
        return self._data.get(owner, TaskList(owner=owner))

    def save_task_list(self, task_list: TaskList) -> None:
        """Persist changes to the task list."""
        self._data[task_list.owner] = task_list

En environnement de production, on le remplacerait par une implémentation basée sur une base de données (en utilisant SQLAlchemy ou directement vers Postgres), sans avoir à modifier le code du domaine.

6. Exécuter le flux

Lorsqu’un utilisateur envoie une demande, le flux se déroule comme suit :

# User says: "Remind me to buy milk tomorrow"
# (In reality, an LLM would parse this into structured data)
user_input = "Remind me to buy milk tomorrow"
intent = "add_task"

# Initialize repository
repo = TaskRepository()

if intent == "add_task":
    # 1. Load the user's task list
    task_list = repo.get_task_list(owner="User123")

    # 2. Create a new task entity
    task = Task(
        id=str(uuid4()),
        description="buy milk",
        due_date=date.today() + timedelta(days=1),
    )

    # 3. Domain layer enforces business rules
    try:
        task_list.add_task(task)
        repo.save_task_list(task_list)
        print(f"Task '{task.description}' added for {task.due_date}.")
    except Exception as exc:
        print(f"Sorry, I couldn't add that task: {exc}")

Les couches restent distinctes :

  • Couche LLM : elle analyse le langage naturel pour obtenir des données structurées (intention + paramètres)
  • Couche de domaine : elle applique les règles métier via des méthodes d’entité
  • Couche de répertoire : elle gère la persistance sans interférer avec la logique de domaine

L’LLM peut être créatif dans son analyse, mais c’est le domaine qui définit ce qui est cohérent. Si l’LLM tente d’ajouter une tâche déjà existante, la racine agrégée la rejette. Il n’est pas nécessaire d’inclure de clause spéciale dans votre prompt pour ce cas.

Outils pour mettre le modèle en œuvre

Le DDD ne nécessite pas de frameworks particuliers. Cependant, quelques outils facilitent l’implémentation, surtout pour les agents IA.

FastAPI

FastAPI Il s’aligne parfaitement sur les couches du DDD. Utilisez des routeurs pour séparer les contextes bornés./tasks, /schedule), des modèles Pydantic pour la validation des requêtes et des réponses, ainsi que l’injection de dépendances afin de connecter les repositories.

Structurez votre projet en couches :

project/
├── domain/          # Pure business logic (entities, aggregates, value objects)
├── application/     # Use cases and command handlers
├── infrastructure/  # Repositories, databases, external APIs
└── interface/       # FastAPI routers and HTTP contracts

Cette stratification (parfois appelée « architecture en oignon ») empêche les modifications de se propager dans l’ensemble de votre base de code. Changer de base de données signifie devoir intervenir directement sur infrastructure/ et rien d’autre. Modifier l’interface signifie intervenir sur interface/ et rien d’autre.

Pydantic et Pydantic AI

Pydantic Il impose des invariants et valide les données en temps de exécution. Utilisez-le pour les entités, les objets de valeur, et surtout pour valider les sorties des modèles de langage grand public.

Pydantic AI pousse cela encore plus loin : il impose que les réponses des LLM correspondent aux schémas de votre domaine. Définissez un AddTaskCommand Avec les champs requis, Pydantic AI valide la sortie JSON du modèle avant même que votre code ne la traite.

Formateur c’est une autre option possible ici. Elle modifie les clients OpenAI (et d’autres) afin qu’ils retournent directement des modèles Pydantic, ce qui constitue une méthode légère pour mettre en œuvre une couche de lutte contre la corruption.

Bibliothèques d’aide DDD

  • DDDesign — fournit des classes de base pour les entités, les repositories et les objets de valeur construits à l’aide de Pydantic**
  • Polyvalent — un framework complet pour DDD, CQRS et l’event sourcing, idéal si vous recherchez des fonctionnalités prêtes à l’emploi dès le départ

La plupart des développeurs Python évitent ces outils au profit d’classes classiques combinées à Pydantic, mais ils méritent d’être explorés pour les projets de grande envergure.

Outils orientés événements

Pour les événements de domaine, envisagez :émetteur clignotant** — dispatcheur d’événements léger en mémoire**

  • Redis-Py Pub/Sub ou RabbitMQ — pour les événements distribués entre services ou agents**
  • Modèles d’événements asyncio — si vous utilisez déjà l’approche asynchrone

Les événements sont essentiels pour l’orchestration multi-agents. Un agent émet ResearchCompleted; d’autres s’abonnent et réagissent. Aucun agent n’a besoin de savoir qui écoute.

Frameworks d’agents

LangChain, LangGraph, Haystack, Kernel sémantique, LlamaIndex, AutoGen, Google ADK, smolagents, et CrewAI Tous fournissent une structure pour les flux de travail des agents. Utilisez-les dans votre couche d’application ou d’infrastructure, puis enveloppez-les dans des interfaces gérées par votre couche de domaine. Le remplacement de frameworks devient ainsi un changement maîtrisé.

Tests

Un avantage concret du DDD : la couche de domaine peut être testée sans que l’ensemble du stack ne soit en exécution.

  • PyTest pour les tests unitaires sur les entités et les agrégats
  • Des répertoires fictifs (en mémoire) pour les tests d’intégration
  • Des stubs de LLM qui renvoient des sorties prédéterminées

Votre code métier ne doit jamais nécessiter un LLM en temps réel pour exécuter ses tests. Le LLM n’est qu’un détail d’implémentation. Les tests servent à valider les règles métier.

Liste de vérification pour commencer

Une séquence d’opérations pratique lors du lancement d’un nouveau projet d’agent :

  1. Interroger les experts du domaine. Élaborer le langage commun à tous. Le consigner par écrit.
  2. Cartographier les contextes délimités. Dessiner les sous-domaines et indiquer où ils doivent communiquer entre eux. Commencer par un contexte central.
  3. Modéliser les entités et les objets de valeur. Quels éléments possèdent une identité ? Quels éléments ne sont que des valeurs ? Intégrer des invariants dans leurs méthodes.
  4. Définir les racines d’agrégat. Regrouper les entités liées sous une racine qui applique les règles de cohérence.
  5. Créer des interfaces de répertoire. Ne pas implémenter encore le stockage. Se contenter de les définir. save() et get(). Le domaine reste ignorant de l’emplacement des données.
  6. Émettre des événements de domaine. Pour les changements significatifs (commande passée, tâche terminée), générer des événements. Connecter les écouteurs ultérieurement selon les besoins.
  7. Encadrer les sorties des LLM dans des schémas. Utiliser des modèles Pydantic pour imposer des contrats. Le texte libre ne doit pas s’infiltrer dans votre domaine.
  8. Ajouter une orchestration. Construire des services d’application qui coordonnent les agents via des commandes ou des événements structurés.

La règle vraiment importante : commencer par le domaine, et non par la pile technologique. Comprendre d’abord le problème métier. Le modéliser explicitement. Ensuite, intégrer les outils d’IA pour servir ce modèle.

Points clés

  1. DDD aide les systèmes d’agents car ces derniers ont besoin de limites métier explicites, et non simplement de listes d’outils.
  2. Les contextes bornés empêchent un agent polyvalent d’accumuler des responsabilités non liées entre elles.
  3. Un langage universel est essentiel, car les prompts, les outils, les schémas et les évaluations doivent utiliser les mêmes termes de domaine.
  4. L’orchestration des agents devient plus simple lorsque chaque agent correspond à une capacité métier bien définie.
