Qu’est-ce qu’un agent AI runtime ?

Un AI agent runtime constitue la couche d’infrastructure qui permet à un agent utilisant des outils de rester actif, isolé, observable et capable de reprendre son fonctionnement une fois l’appel au modèle terminé. Il gère l’état de la session, l’exécution des outils, checkpoints, les vérifications de politique, les secrets, les traces, les limites de coût ainsi que la structure de déploiement. Le modèle choisit l’action suivante ; le runtime détermine où cette action sera exécutée, si elle est autorisée, comment elle sera enregistrée, et comment la séquence d’exécution pourra reprendre en cas d’échec.

Les cinq primitives runtime dont tout agent AI à exécution prolongée a besoin

Anthropic’s Échelle des agents gérés Ce document présente un vocabulaire utile pour les cinq responsabilités associées à runtime. Le harness fait avancer l’agent étape par étape ; les enregistrements de session consignent ses actions ; c’est dans le sandbox que les commandes sont exécutées ; le checkpoint correspond au contenu que le prochain processus lit pour reprendre le travail ; quant aux traces, il s’agit des données que l’on consulte des jours plus tard afin de comprendre ce qui a mal tourné. Une implémentation peut fusionner certains composants, mais les responsabilités ainsi que les limites de défaillance doivent néanmoins disposer de noms bien définis.

Session. Un journal d’écriture seule qui enregistre tout ce qui s’est produit : les appels au modèle, tool calls, les résultats, les erreurs, les approbations. La récupération est wake(sessionId) → getSession(id) → resume from last event. Dans LangGraph, il s’agit d’un thread_id plus un pointeur de vérification Postgres (voir Persistance de LangGraph). OpenAI Agents SDK intègre dix backends de session prédéfinis, y compris SQLiteSession, RedisSession, SQLAlchemySession, MongoDBSession, et EncryptedSession (voir le Documentation des sessions).

Harness. La boucle d’orchestration. Elle appelle le modèle, analyse tool calls, les exécute, écrit les résultats en retour dans la session, et applique les règles de tentative. Anthropic l’exprime sans détour :

« Chaque composant au sein d’un harness encode une hypothèse concernant ce que le modèle ne peut pas faire de manière autonome. »

L’équipe Codex d’OpenAI qualifie cette discipline de harness engineering : la rédaction de logiciels exige toujours de la discipline, mais une part plus importante de celle-ci est désormais consacrée aux infrastructures de soutien plutôt qu’au code lui-même. LangGraph’s CompiledStateGraph, les agents profonds create_deep_agent, Claude Code lui-même constitue également un outil de ce type.

Sandbox. L’environnement d’exécution isolé dans lequel les commandes sont réellement exécutées. Les SDK d’OpenAI sandbox concepts page trace la limite de manière nette :

« L’ runtime externe conserve toujours la gestion des approbations, du traçage, des transferts ainsi que de la comptabilisation des reprises. La session sandbox gère les commandes, les modifications de fichiers et l’isolation de l’environnement. »

Sandboxes diffèrent selon la durée de leur existence et les données qu’ils conservent entre plusieurs exécutions. La forme la plus simple est celle des objets éphémères frais : on en crée un uniquement pour une tâche donnée, on le détruit à la fin de cette tâche, ce qui entraîne des coûts liés au démarrage froid à chaque exécution. Les sandboxes en pause persistants restent actifs entre les exécutions dans un état de pause ; le système de fichiers ainsi qu’un snapshot de la mémoire subsistent, permettant ainsi une reprise en quelques dizaines de millisecondes au lieu d’un démarrage complet. La méthode snapshot ou fork va encore plus loin : chaque tâche crée une copie sur écriture à partir d’une image parente déjà équipée des dépendances nécessaires et dont le cache est préchauffé, de sorte que les opérations coûteuses de configuration s’effectuent une seule fois et que N tâches peuvent partager cette image de base. Les stratégies sandboxes par arbre de travail attribuent à chaque tâche son propre espace de travail et sa propre pile d’outils d’observabilité : journaux, métriques et traces séparés. Cela permet de déboguer l’exécution d’un agent sans que cela n’affecte les autres agents. Les chiffres concrets concernant le démarrage froid et la persistance pour chaque fournisseur sont indiqués dans le tableau plus bas dans cette section.

Checkpoint. État reprendable. Celui de LangGraph PostgresSaver écrit un StateSnapshot à chaque frontière de super-étape, grâce à des écritures par tâche checkpoint_writes Ainsi, les sorties des nœuds réussis ne sont pas recalculées en cas d’échec d’un nœud frère. Le snapshot est un dictionnaire JSON-serialisable.v, ts, id, channel_values, channel_versions, versions_seen, pending_sends) documenté dans le langgraph-checkpoint-postgres page PyPI et le LangGraph checkpoints de référence.

Traçage. C’est l’interface de lecture et de débogage. Chaque appel de modèle, tool call, ainsi que chaque étape d’un sous-agent deviennent une entité représentant des données temporelles, des entrées, des sorties, des comptages de tokens et des coûts associés. Lorsqu’un exécution de six heures échoue, c’est le traçage qui permet de comprendre ce qui a mal tourné, car les sorties du terminal de cette exécution ont déjà disparu depuis longtemps. Celui de OpenTelemetry Conventions sémantiques des GenAI standardiser les noms des attributs (quel modèle, quel fournisseur, combien de tokens, quelle conversation, quel flux de travail), afin que la même trace s’affiche de manière propre dans Tempo, Jaeger, Honeycomb, ou LangSmith sans avoir à réinstrumenter le système.

Modes de défaillance des agents runtime en production AI

Le runtime gère les tentatives de réexécution, restaure le travail antérieur, isole les espaces de travail et applique les contraintes budgétaires. Les cinq primitives mentionnées ci‑dessus contrôlent le cycle de vie d’une exécution, tandis que les mécanismes de politique et de gestion des secrets interviennent à différents niveaux. Lorsque les exécutions s’étendent sur plusieurs nœuds de traitement ou fenêtres de contexte, les échecs entraînent des problèmes liés à l’état, des effets secondaires redondants, du sandbox et au non‑respect des budgets.

Les pannes se répartissent en quatre catégories :

  • Défaillances liées à la qualité de la sortie : l’agent proclame sa victoire avant que le travail ne soit réellement achevé, oublie ce qu’il a fait suite à un réinitialisation de la fenêtre de contexte, ou se fie à son propre évaluation pour livrer une sortie défectueuse.
  • Défaillances de contrôle des coûts : l’agent reste bloqué dans une boucle de tentative, ou épuise son budget de tokens ou d’appels d’outils sans produire de résultat utile.
  • Défaillances liées à l’état et aux plantages : les espaces de travail se détériorent parce qu’une exécution modifie des fichiers appartenant à une autre exécution, tool calls est déclenché plus d’une fois en raison des tentatives de répétition, ou le travail est perdu lorsque le processus cesse de fonctionner entre deux événements.
  • Défaillances de la fenêtre de contexte : le modèle effectue un résumé et s’arrête prématurément, pensant manquer d’espace, même lorsque la fenêtre dispose encore de marge disponible.

Le cycle de vie d’un exécution saine

Un exécution bien comportée est ennuyeuse : il s’agit d’une chaîne de petites étapes réparables, chacune enregistrant son résultat dans un stockage durable avant que la suivante ne commence, plutôt que d’une seule requête massive qui doit réussir du début à la fin. C’est en divisant l’exécution en telles étapes que celle-ci peut survivre aux pannes : lorsqu’un problème survient, seule l’étape en cours est perdue, et le prochain travailleur reprend à partir de la dernière étape terminée au lieu de tout recommencer.

  1. Démarrez soit depuis une session fraîche, soit depuis une session reprise. Lors de la reprise, montez l’espace de travail à partir de son dernier état connu, et lisez tous les fichiers de progression laissés par la tentative précédente (PROGRESS.md, feature-list.json), puis charger le dernier checkpoint depuis la base de données. C’est à ce stade que le harness transmet à l’agent tout ce que l’agent précédent avait en mémoire avant sa mort.
  2. Planifier avant de lancer tout tool calls. Définir clairement ce qu’on entend par « tâche terminée », le temps alloué à son exécution, les outils dont l’agent peut disposer, ainsi que les conditions qui doivent entraîner une interruption anticipée. Ces valeurs de planification deviennent des vérifications runtime ; sans elles, l’exécution ne dispose d’aucun critère de contrôle.
  3. Exécuter un tool call à la fois. La couche de politique décide si l’appel est autorisé. Le harness le lance, enregistre le résultat et inscrit un événement dans le journal de session. Une étape, un événement. Une panne survenant entre deux événements est réparable, car c’est le journal, et non la mémoire de l’agent, qui constitue la source de vérité.
  4. Faire un Checkpoint aux limites des super-étapes, ou après chaque événement dans un harness plus simple. Persister l’état du graphe, la différence d’espace de travail, ainsi que les références à tous les artefacts générés. C’est ce checkpoint que lit l’étape 1 lors de la reprise. Si le checkpoint fait défaut ou est obsolète, la récupération se réduit à la lecture complète du journal de session depuis le début, ce qui est beaucoup plus lent.
  5. Évaluer les artefacts lorsque l’agent estime avoir terminé : exécuter des tests, faire appel à un revueur avec un contexte frais, valider le schéma et effectuer des vérifications dans le navigateur. Si la vérification réussit, l’exécution se termine avec succès. En cas d’échec, l’exécution reprend à partir du dernier état propre checkpoint, le message d’erreur étant ajouté au contexte, afin de tenter à nouveau.

Aucune étape de cette liste ne nécessite que l’agent mémorise quoi que ce soit entre deux exécutions. L’état est conservé dans la session ainsi que dans le checkpoint, et l’agent le lit à nouveau à chaque reprise.

Les outils générant des effets secondaires doivent disposer d’une propriété d’idempotence. Tout outil présentant des effets secondaires nécessite une clé d’idempotence dérivée de l’ID de session et de l’ID d’appel de l’outil, qui doit être stockée avant que l’effet secondaire ne se produise. send_email(session_id, tool_call_id, message_hash). create_pr(session_id, tool_call_id, branch_name). charge_customer(session_id, tool_call_id, invoice_id). L’exécution au moins une fois est la valeur par défaut dans les files d’attente et les moteurs de workflows. Si la répétition d’un tool call peut causer des dommages réels, l’outil n’est pas prêt à être utilisé avec des agents.

L’évaluation doit intégrer des preuves provenant d’un contexte externe à celui de production. Un évaluateur indépendant permet de réduire le biais lié au contexte partagé, tandis que les tests, les outils de vérification de syntaxe, les contrôles dans le navigateur et la validation du schéma fournissent des preuves déterministes. La vérification peut retourner pass, fail, ou needs_human. Pour les agents de traitement de code, l’examinateur peut être une autre session de modèle dotée d’outils en lecture seule. Pour les agents chargés du traitement des données et de la génération de rapports, il convient de combiner une validation déterministe avec un modèle d’examinateur qui nécessite néanmoins une évaluation humaine.

Onze modèles de déploiement d’agents AI et les facteurs qui les distinguent

Une fois les cinq primitives nommées, la question qui se pose est de déterminer quelle forme de déploiement les exécutera. Par « forme », j’entends un agencement de ces primitives : l’emplacement où se trouve harness, l’endroit où le statut est persisté, ainsi que le type de sandbox qui gère les tâches. Il ne s’agit pas simplement d’un choix imposé par un seul fournisseur. Le graphique ci‑dessous indique sur quel segment de l’axe du nombre d’exécutions chaque forme s’épanouit le mieux. Le texte qui suit explique ce qui permet de faire le choix entre elles.

Si vous ne devez en lire qu’un seul parmi ces onze, choisissez le schéma 2 : file d’attente + travailleur + checkpoint DB. Il s’agit de la configuration par défaut que je recommande pour la plupart des équipes, du schéma utilisé dans le dépôt de référence, ainsi que du modèle de base autour duquel se déclinent la plupart des autres architectures : file d’attente → travailleur → état persistant, avec substitution du fournisseur de données sandbox, du responsable harness ou du moteur de gestion d’état. En commençant par le schéma 2, vous pourrez parcourir plus rapidement les autres configurations.

Le graphique compare les formes en fonction de leur longueur d’exécution. La matrice ci‑dessous les compare en fonction de leur propriétaire : l’endroit où chacune des cinq primitives se trouve physiquement. Les cellules vertes indiquent que la forme fournit elle‑même la primitive ; les cellules grises indiquent que vous devez la connecter manuellement.

1. SDK au sein d’un serveur d’application (synchrone, par requête)

La forme originale. L’agent SDK s’exécute à l’intérieur d’un gestionnaire de requêtes. Il convient aux tâches prenant moins de 30 secondes, aux démonstrations ainsi qu’aux outils internes. En revanche, il est inadapté pour tout scénario où un client HTTP pourrait se déconnecter. Le délai de timeout HTTP de Cloud Run Sa durée maximale est de 60 minutes, et toute panne au niveau du tier web met fin à l’exécution. Le SDK correspond au harness ; le processus web agit également en tant que sandbox, et l’état est généralement stocké en mémoire de processus, sauf si vous le transférez explicitement ailleurs. Ne l’utilisez pas pour des tâches d’une durée supérieure à quelques heures.

2. File d’attente + travailleur + checkpoint de base de données

La configuration par défaut que je recommande pour la plupart des équipes, ainsi que la version en environnement de production utilisée dans market-analyst-agent: un travailleur Python équipé d’un checkpointer PostgreSQL, Streams Redis (ou) RabbitMQ) Pour la file d’attente entrante, ainsi qu’un sidecar MCP dédié aux outils. Cela convient aux exécutions allant de 10 minutes à plusieurs heures, grâce à des étapes idempotentes. Le lanceur local permet de contourner la file d’attente pour un développement synchronique, mais celle-ci fait partie intégrante du schéma de production dès lors que l’on a besoin de soumissions asynchrones et de mécanismes de contrôle de charge.

L’application reçoit une demande, crée une ligne de session, envoie un job à exécution et renvoie un identifiant d’exécution. Le travailleur récupère le job, exécute le harness, enregistre les checkpoints, diffuse des mises à jour sur l’état et stocke les artefacts au fur et à mesure. PostgreSQL reste opérationnel, les travailleurs ne sont que des ressources interchangeables, et la profondeur de la file d’attente assure une contrainte de contrepression. Le calcul Spot/Préemptible fonctionne tant que le point de contrôle termine d’écrire sur le disque avant de signaler un succès.

Dans cette architecture, l’agent d’exécution est le harness. Son conteneur ainsi que l’espace de travail par thread définissent une frontière d’exécution, mais le code non fiable nécessite néanmoins un sandbox ou une VM renforcée. Postgres gère l’état des sessions et de checkpoint. Les traces sont acheminées via OpenTelemetry vers n’importe quel stack d’observabilité que vous utilisez.

3. Moteur de flux de travail durable (style Temporal)

Le code d’orchestration des agents s’exécute au sein d’un Workflow Temporal ; les appels aux modèles ainsi que tool calls sont traités en tant qu’Activités. L’état du workflow est stocké dans un journal d’historique d’événements hébergé par Cassandra, MySQL ou Postgres, ce qui permet une reprise propre de cet état lors de nouvelles déploiements. La version en prévisualisation publique Intégration Temporal × OpenAI Agents SDK envoie des OpenAIAgentsPlugin et un activity_as_tool aide, ainsi que le description détaillée de agentic et sandboxes décrivait le fait de faire fork d’un agent en cours d’exécution vers un fournisseur sandbox différent au milieu d’une conversation. Les workflows inactifs consomment aucune ressource de calcul. Les limites sont réelles : les agents de type streaming et vocaux ne sont pas pris en charge dans cette intégration actuelle, et LocalShellTool et ComputerTool Ils sont désactivés car ils ne correspondent pas à un modèle distribué.

Utilisez cette forme lorsque l’exécution comporte des points d’attente réels : approbations humaines, appels externes, périodes de sommeil prolongées, tentatives de réessai soumises à des règles métier, fenêtres de déploiement. Une approbation humaine se transforme en un sommeil durable qui ne consomme aucune ressource de calcul, contrairement à une boucle de sondage.

Le code du flux de travail correspond au harness. Le sandbox se trouve généralement en dehors de Temporal et est appelé depuis les activités. L’état de la session ainsi que celui du checkpoint sont intégrés au journal d’historique des événements de Temporal, tandis que la visibilité des traces provient de l’interface utilisateur de Temporal ainsi que des plages de temps OpenTelemetry associées à chaque activité.

5. Agents gérés par Anthropic (hébergés harness)

Lancé en bêta public le 8 avril 2026, en arrière-plan du managed-agents-2026-04-01 en-tête beta. Claude facture les agents gérés aux taux de tokens standards majorés de 0,08 $ par heure de session. La facturation s’effectue au niveau du millisecondes et n’est appliquée que tant que l’état de la session est « en cours ». Le temps d’inactivité est gratuit ; la session runtime « remplace le modèle de facturation basé sur les heures de conteneur pour l’exécution de code lors de l’utilisation des agents gérés Claude ». Vous bénéficiez d’une session hébergée, harness, sandbox, ainsi que d’un proxy MCP protégé par un coffre-fort. C’est la séparation entre le cerveau et les mains qui est à l’origine de… wake(sessionId) Achats : le harness peut être réinitialisé sur un nouveau nœud de travail sans perte d’état. Faites attention à la structure des coûts ; une boucle de tentative infinie sur un facturation par heure de session entraîne des dépenses plus élevées que celle d’un tarif par token.

Lisez les mises en garde. La remise pour le traitement par lots API ne s’applique pas (« Les sessions sont à état et interactives. Il n’existe pas de mode par lots. »). Managed Agents n’est pas disponible via AWS Bedrock ou Google Vertex AI. La Multi-agent coordination ainsi que l’auto-évaluation restent encore en phase de prévisualisation dans le cadre de la recherche. Le risque de verrouillage est élevé : on sacrifie la liberté harness d’exécuter manuellement le cycle d’itération au profit d’une solution prête à l’emploi.

Anthropic prend en charge les cinq primitives suivantes : session, harness, sandbox, checkpoint et trace. Vous lui transmettez le runtime et vous obtenez les résultats correspondants.

6. Déploiement d’agents profonds LangChain (géré en mode ouvert harness)

deepagents deploy packages a deepagents.toml dans un déploiement LangSmith offrant une exécution durable, une gestion fine de la mémoire, du multi-tenancy, human-in-the-loop, des capacités d’observabilité, une exécution de code en environnement sandboxé, ainsi que des exécutions planifiées. Les modes de déploiement cloud, hybride et auto-hébergé sont pris en charge. Les fournisseurs Sandbox (LangSmith Sandboxes, Daytona, Modal, Runloop ou personnalisés) peuvent être changés via une seule valeur de configuration. L’état est stocké dans un système de fichiers virtuel doté de backends interchangeables ; la mémoire est limitée au niveau de l’utilisateur, de l’assistant ou des deux. Le risque de verrouillage est inférieur à celui des agents gérés : le harness est licencié sous licence MIT, et les instructions utilisent des outils open source. AGENTS.md standard, et les agents sont exposés via MCP, A2A, ainsi que le Agent Protocol. Voir LangChain’s runtime – déploiement en production de deep agents rapport de synthèse.

Les cinq primitives sont hébergées par défaut, mais chacune peut être remplacée via des paramètres de configuration. Le sandbox est stocké derrière une valeur de configuration spécifique. La session ainsi que le checkpoint fonctionnent sur un système de fichiers virtuel doté de backends interchangeables. Les traces sont envoyées à LangSmith.

7. Service ou job Google Cloud Run

Cloud Run dispose de deux modes runtime distincts, et le choix du mode dépend de la manière dont l’agent est invoqué. Les services sont liés à HTTP et passent en état zéro entre les requêtes ; le harness s’exécute en tant que gestionnaire de requêtes qui cesse son fonctionnement une fois l’exécution terminée. Les jobs, quant à eux, se poursuivent jusqu’à leur fin sans point d’entrée HTTP ; le harness fonctionne comme un travailleur ponctuel qui quitte le processus dès que la tâche est achevée. Ces deux approches peuvent héberger le harness, mais aucune ne conserve d’état entre différentes exécutions. Les sessions ainsi que le checkpoints doivent être stockés dans Postgres, Spanner ou un système de stockage externe similaire.

Les limites intrinsèques diffèrent considérablement entre les deux approches. Délai d’attente dépassé pour la demande de service Cloud Run: Valeur par défaut : 300 s, valeur maximale : 3 600 s (60 min). WebSockets obtenir le même délai d’expiration. Les jobs Cloud Run: Par défaut, 10 minutes par tâche, avec une durée maximale de 168 h (7 jours) ; pour les tâches utilisant GPUs, la durée maximale est de 1 heure. Les services sont réduits à zéro sauf si vous activez le mode CPU permanent ; les jobs ne disposent pas de protocole HTTP et ne sont pas auto-échelonnés.

Utilisez un service pour les exécutions synchrones d’une durée maximale de 60 minutes. Préférez plutôt une tâche pour des travaux ponctuels plus longs ou asynchrones. Les Cloud Run Jobs permettent de maintenir une tâche active pendant plusieurs jours, mais ils ne garantissent pas une reprise fiable après des déploiements, des mises à jour de version ou le remplacement des travailleurs. Au-delà de 7 jours, évitez d’utiliser Cloud Run.

Cloud Run héberge le harness. L’état des sessions et de checkpoint est stocké dans Postgres, Spanner ou un autre système de stockage externe, et les traces peuvent être transmises via Cloud Logging et OpenTelemetry. Le conteneur de service constitue un environnement d’exécution ; il convient d’ajouter un sandbox distinct lorsque l’agent exécute du code non fiable.

8. AWS Lambda (pourquoi c’est l’outil inadapté)

Délai maximal d’exécution d’une fonction Lambda Cela représente 900 s (15 minutes), ce qui est très difficile. API Passerelle Il ajoute en outre une limite supplémentaire de 29 secondes. Un agent harness dont la durée d’exécution utile minimale se situe dans la fourchette « minutes à heures » ne peut pas fonctionner correctement sur Lambda sans un stockage d’état externe ainsi qu’une stratégie de réinvocation permettant de reconstruire entièrement la structure de la file d’attente et des travailleurs à partir de zéro. Utilisez Lambda pour les tool calls individuels, tels que les téléchargements de fichiers ou les mises à jour sur S3, qui sont déclenchés par un orchestrateur ayant une durée de fonctionnement plus longue. Ne placez pas l’orchestrateur lui-même sur Lambda.

Au maximum, Lambda peut contenir un seul tool call au sein de sa limite de 15 minutes. Le harness, c’est‑à‑dire la session, ainsi que les checkpoint et les sandbox, doivent tous être stockés ailleurs.

9. Tâche AWS ECS / Fargate par exécution

Il n’existe pas de limite maximale documentée pour les tâches runtime (contrairement à Lambda). Selon Quotas de limitation de Fargate, Les lancements sont soumis à une limitation de débit : une vague initiale de 100 opérations, suivie d’un rechargement à un rythme de 20 opérations par seconde, les requêtes sur demande et celles en mode spot étant gérées respectivement au sein de budgets distincts de 20 opérations/seconde. Quotas de service ECS: Les services utilisant la découverte de services AWS Cloud Map sont limités à 1 000 tâches par service ; le plafond théorique est de 5 000 instances EC2 par cluster. Fargate exige awsvpc En mode par tâche, chaque tâche dispose de sa propre interface réseau ainsi que d’une IP privée, ce qui constitue la configuration idéale lorsque l’on a besoin d’un accès aux données au sein d’une VPC. Fargate Spot est disponible, mais il faut prévoir un budget pour faire face aux interruptions de service. La durabilité des données relève de votre responsabilité : aucune fonction de réexécution similaire à celle de Temporal n’est intégrée.

Fargate héberge les harness et attribue à chaque exécution sa propre tâche. Cela permet de séparer les espaces de travail des identifiants des tâches, mais cela ne constitue pas en soi une protection complète sandbox contre du code malveillant. La gestion des sessions, checkpoint, ainsi que le suivi des traçages, sont gérés par des services externes tels que RDS ou DynamoDB, ainsi que par CloudWatch/X-Ray.

10. Job Kubernetes ou espace de noms par session

Excellente solution lorsque l’on travaille déjà avec ce système. Kubernetes et on souhaite une gestion sandbox par session accompagnée de contrôles à l’échelle du cluster. Cela est problématique lorsque l’on a besoin d’un démarrage en quelques sous-secondes, car le téléchargement de l’image de conteneur et l’initialisation du pod prennent trop de temps lors d’un démarrage à froid. Le schéma retenu consiste en un Job pour chaque exécution d’agent, avec activeDeadlineSeconds, une PersistentVolumeClaim pour l’espace de travail, ainsi qu’un sidecar destiné au serveur MCP. C’est à vous de mettre en place mécanismes de récupération en cas de panne. Adopter Kubernetes uniquement pour héberger des agents entraîne des coûts importants en termes de surcharge de configuration et de charge opérationnelle. Ce n’est rentable que si vous utilisez déjà K8s pour d’autres raisons.

Kubernetes héberge l’harness ainsi que l’environnement d’exécution par exécution, généralement au sein d’un Job et parfois dans un espace de noms dédié. Une forte isolation dépend néanmoins de la classe runtime, des politiques réseau, de la sécurité des pods, ainsi que des limites imposées par le conteneur ou la VM sous-jacente. Les sessions et l’état checkpoint sont stockés dans une base de données externe ou via un PersistentVolumeClaim.

Observabilité résistante tout au long de l’exécution

Les gestionneurs de requêtes de courte durée sont faciles à déboguer : en cas d’échec, il suffit de lire la réponse ainsi que le journal en temps réel. Les agents à exécution prolongée ne disposent pas de ce luxe. Lorsqu’une exécution de six heures échoue, l’événement pertinent s’est déjà produit il y a cinq heures, les sorties du terminal en direct ont disparu, et le processus qui les générait a été remplacé. Personne ne pourra reconstituer l’exécution à partir de sa mémoire. Il faut donc se baser sur des artefacts durables qui ont été enregistrés tant que l’exécution était encore active.

Les stacks de production couvrent généralement quatre types d’artefacts, regroupés en deux catégories. Deux d’entre eux sont consultés après la fin de l’exécution, aux fins d’analyse post-mortem et de rejeu : un journal des événements consultable pour chaque étape, ainsi que des traces OpenTelemetry indiquant où se sont écoulés le temps et les tokens. Les deux autres sont consultés pendant l’exécution, afin de suivre ce qui se passe en temps réel : une vue en continu de ce que l’agent génère dans l’espace de travail, et une stack d’observabilité par worktree que l’agent lui-même peut interroger tant qu’il est encore en cours d’exécution.

Journal d’événements structuré (à consulter après l’exécution)

Chaque appel de modèle, tool call, résultat, erreur ou validation est enregistré dans un stockage persistant, identifié par l’ID de session et une horodatage. Une fois l’exécution terminée, il est possible de le consulter comme une table de base de données ordinaire. Addy Osmani fixe clairement les standards en la matière. Agents à exécution prolongée: « Si l’on ne parvient pas à reconstituer les actions effectuées par l’agent au cours des 24 dernières heures à partir d’un stockage fiable, on a affaire à un script shell en exécution continue qui se contente d’appeler un LLM, et non à un agent en exécution continue. »

OpenTelemetry Traçages de l’IA générative (à lire après l’exécution)

Le même type de données étape par étape, mais émises sous forme de spans en utilisant les attributs standards provenant du gen_ai.* conventions sémantiques: Nom du modèle, fournisseur, nombre de tokens d’entrée et de sortie, identifiant de la conversation, nom du flux de travail (état de développement à partir de v1.36.0). Les champs spécifiques au fournisseur se trouvent dans des sous-espaces.anthropic.*, openai.*) désactivé par clé gen_ai.provider.nameLa raison d’utiliser cette norme réside dans la portabilité : le même suivi s’affiche correctement dans Tempo, Jaeger, Honeycomb ou LangSmith, sans nécessiter de réinstrumentation du code à chaque fois que l’on change de backend.

Chronologie des appels d’outils ainsi que les différences d’espace de travail (affichées pendant l’exécution)

Le moyen le plus rapide de savoir ce qu’un agent est en train de faire à l’instant présent consiste à suivre les éléments qu’il génère dans l’espace de travail, et non à parcourir un journal de session. Anthropic’s Harness Primitifs pour les agents Claude à exécution prolongée Le guide de démarrage rapide fournit deux hooks à cet effet : watch -n 5 'git log --oneline -8' affiche les derniers commits effectués par l’agent, et watch -n 5 'find screenshots -name "*.png" | tail -5' Affiche les captures d’écran les plus récentes qu’il a prises. Deux panneaux de terminal mis à jour toutes les cinq secondes suffisent pour déterminer si une exécution progresse réellement ou reste en boucle.

Une pile d’observabilité auto-hébergée minimale

Pour un cas similaire market-analyst-agent:

Le schéma de bundle de débogage

Lorsqu’un exécution échoue, le worker doit abandonner /workspaces/${THREAD_ID}/_debug/ contenant les artefacts que vous demanderiez lors d’un post-mortem :

  • session.jsonl: export complet du journal des événements provenant de PostgresSaver (checkpointer.list({"thread_id": ...})).
  • last_state.json: StateSnapshot.values à partir du dernier super‑étape réussi. trace.json: Les spans exportés en OTLP pour l’exécution.
  • tool_calls.csv: (ts, tool, input_hash, latency_ms, status, error).
  • workspace.tar.zst: le répertoire d’espace de travail plus git diff contrairement à la branche de démarrage. screenshots/*.png: Ce que l’agent a observé.
  • PROGRESS.md, feature-list.json, ainsi que tous les autres fichiers de progression générés par l’agent. env.txt: balises d’image, version du modèle, SHA de la commit harness.

Ce paquet fournit à un agent humain ou à un agent d’audit des preuves suffisantes pour reconstituer la cause de l’échec. L’expression « L’agent est bloqué » est trop vague. Un rapport illustratif, en revanche, est concret : session s_123 A consacré 71 % de ses tokens à répéter trois commandes successivement. npm install Échec.

Choisir la forme adéquate : un guide de décision

La majeure partie des comparaisons présentées ci-dessus se résume à un petit nombre de décisions à prendre.

Commencer par la longueur de série

Utilisez la longueur de série comme premier filtre :

  • Moins de 30 secondes, opération idempotente : cycle de vie de la requête SDK sur un serveur d’application.
  • De 30 s à 60 min, pas de récupération en cas de panne nécessaire : file d’attente + travailleur + base de données checkpoint.
  • De 60 min à 24 h : même file d’attente + travailleur, ou un Cloud Run Job pour des tâches ponctuelles. Utilisez un moteur de workflows durables si vous avez également besoin de gestion des versions et de réexécution.
  • Plus de 24 h, doit survivre aux déploiements : moteur de workflows durables (de type Temporal). Les Cloud Run Jobs peuvent gérer des tâches longues jusqu’à leur limite de tâches, mais ils ne prennent pas en charge de sémantique de réexécution.
  • Boucles d’entraînement par apprentissage par renforcement sur plusieurs jours : K8s Job + volume de stockage + Temporal.

Après ce filtre grossier, il convient d’évaluer les effets secondaires, les mécanismes de récupération, la possibilité de rejouer les opérations, l’isolation des composants, l’emplacement des données, ainsi que l’équipe chargée de l’exploitation du système.

Adaptation de la plateforme en fonction du cas d’usage

La matrice est dense car aucune seule cellule verte ne détermine l’architecture. Une couverture étendue des charges de travail est utile, mais elle ne reflète pas la résidence des données, les sémiotiques de rejouement, la dépendance vis-à-vis du fournisseur, le niveau de maturité opérationnelle, ni le coût lié au déplacement ultérieur de l’état.

Deep Agents Deploy couvre tous les types de charges de travail identifiés dans la matrice, ce qui en fait une solution adaptée lorsque l’on doit gérer des exécutions de courte durée, des tâches s’étendant sur plusieurs heures, des agents de codage, des agents de recherche ainsi que des jobs planifiés. Toutefois, cette polyvalence se traduit par un historique de déploiement en production plus limité par rapport à une architecture basée sur une file d’attente, des workers et Postgres. Considérez les cellules vertes comme des affirmations de capacités à valider, puis comparez les contraintes opérationnelles que la matrice ne peut pas prendre en compte.

Les agents gérés d’Anthropic s’adaptent soit entièrement à votre charge de travail, soit pas du tout. Ce produit présente trois contraintes strictes : il ne peut être utilisé que en mode hébergé, uniquement avec Claude, et les sessions ne peuvent durer moins de 24 heures. Si votre charge de travail répond à ces trois critères – par exemple, un agent de codage interne fonctionnant en séquences de 2 à 6 heures et que vous préférez ne pas gérer vous-même un harness – alors les agents gérés constituent une solution idéale, permettant de supprimer une grande partie des tâches liées à la plateforme pour votre équipe. En revanche, si l’une de ces contraintes n’est pas respectée, que ce soit en raison du besoin d’un modèle autre que Claude, d’une implémentation auto-hébergée ou de sessions de 48 heures, les agents gérés ne conviennent pas. Aucune modification des paramètres de configuration ne peut y remédier.

Il est essentiel de modéliser le coût avant de prendre une décision définitive, et non après. Faites cela avant de valider la solution, car migrer vers des agents gérés ultérieurement revient à effectuer un re-platforming, et non simplement à modifier des paramètres de configuration.

Hébergement harness contre infrastructure propriétaire harness

La différence réside ici sur le plan opérationnel, et non sur l’identité de celui qui a écrit le code harness. Une solution hébergée signifie que le fournisseur exécute la boucle harness au sein de ses infrastructures, tandis que vous appelez une API. Une solution propriétaire signifie que vous exécutez cette boucle sur vos propres infrastructures, même si le code harness lui-même provient d’un fournisseur.

LangChain apparaît des deux côtés de cette ligne, ce qui prête à confusion. Ils proposent LangGraph, une bibliothèque sous licence MIT que l’on héberge soi-même (et dont on est propriétaire), ainsi que Deep Agents Deploy, un produit géré qui exécute des Deep Agents. harness sur le déploiement de LangSmith en mode cloud par défaut (hébergé). Même entreprise, deux modèles opérationnels distincts. C’est vous qui choisissez le modèle, et non le fournisseur. (Deep Agents Deploy dispose également d’un mode auto-hébergé destiné aux équipes qui le souhaitent) harness

Choisissez une solution harness hébergée lorsque son support de modèles, ses limites de données, son comportement de récupération ainsi que ses points d’extension correspondent déjà à vos besoins. Optez plutôt pour une harness propriétaire lorsque ces contraintes représentent des exigences que vous prévoyez de modifier ultérieurement. La migration entre les deux approches modifie l’état du système, l’observabilité et les limites d’exécution ; il est donc essentiel de tester le chemin de sortie avant que des données en production ne dépendent de cette solution.

Environnement d’exécution hébergé sandbox versus environnement d’exécution propriétaire

Choisissez un sandbox hébergé lorsque l’isolation offerte par le fournisseur, les fonctionnalités de pause/reprise, ainsi que les spécificités de fork correspondent au modèle de risques et au budget alloué pour le démarrage du système. Docker ou Fargate conviennent aux charges de travail internes fiables nécessitant un accès à une VPC ou une stricte résidence des données, mais un conteneur standard ne constitue pas une barrière suffisante contre du code malveillant. Ajoutez gVisor, Kata, une microVM, ou un autre runtime renforcé lorsque l’agent installe des paquets non fiables ou exécute des programmes générés.

Stockeurs d’état : Git, bases de données et systèmes de stockage d’objets côte à côte

Les agents à exécution prolongée disposent généralement de trois stockeurs d’état fonctionnant simultanément, et non d’un seul, chacun gérant un type d’état distinct. Git stocke l’état de l’espace de travail : le code, les documents ainsi que les fichiers de progression que l’agent modifie. Chaque commit fournit à l’harness un point de récupération fiable et permet à la session suivante d’examiner un historique compact. La base de données checkpoint conserve l’état graphique de l’harness : ce qui a été décidé, quels nœuds ont été exécutés, quels résultats sont revenus, ainsi que ce qui doit être exécuté ensuite. C’est ce mécanisme qui permet au prochain processus de reprendre l’exécution là où elle s’était arrêtée. Le stockeur d’artefacts, quant à lui, contient les grands résultats finaux tels que des PDF, des fichiers Parquet ou des captures d’écran. Ces éléments ne doivent pas être stockés dans Git ni dans la base de données checkpoint.

Quand utiliser Git comme état

Utilisez Git lorsque le volume de travail est de nature codique (édits sur plusieurs fichiers, refactorisations, génération d’applications) ou suffisamment documentaire pour que l’historique des fichiers soit pertinent. Le schéma est simple : créez une branche de exécution, effectuez un commit initialisateur, puis effectuez des commits aux points clés : après la configuration, après chaque fonctionnalité, après que les tests ont réussi, et après le nettoyage final. Stockez l’SHA du dernier commit de l’espace de travail à côté de la ligne checkpoint. Lors de la reprise, le prochain travailleur fait checkout de la branche et lit git log --oneline -8, examine git status ainsi que la dernière différence, puis lit PROGRESS.md ou tout autre fichier de transfert généré par la session précédente.

Cela fait de Git une surface de récupération pour l’artefact en cours d’édition, et non un remplaçant pour la base de données checkpoint. Git peut répondre à deux questions : qu’est-ce qui a changé, et quelle version a réussi les tests. Il ne peut pas indiquer au harness quel nœud du graphe doit être exécuté en suivant, quel tool call attend encore une validation, ou quel essai a déjà utilisé sa clé d’idempotence. Le harness d’Anthropic utilise des commits d’initialisation ainsi que des commits par fonctionnalité comme source de vérité pour la récupération de l’espace de travail ; le modèle lit git log --oneline -8 afin de récupérer l’état. Omettre Git lorsque le produit du travail se résume à une seule réponse conversationnelle ; la surcharge n’en vaut pas la peine.

Quand utiliser les points de contrôle de base de données

Utiliser PostgresSaverUn mécanisme de sauvegarde de type checkpointing lorsque l’agent possède une structure en graphe composée de plusieurs nœuds dont l’état intermédiaire est crucial (planificateur → chercheur → rédacteur → vérificateur). Le dépôt de référence utilise cette approche précisément pour cette raison. Il ne faut pas stocker les artefacts du espace de travail à l’échelle de téraoctets dans checkpoint ; ceux-ci doivent être placés dans un système de stockage d’objets.

Quand utiliser un stockage d’artefacts (S3 / GCS)

Recourir au stockage d’objets dans trois situations précises. Tout d’abord, lorsque la taille du résultat dépasse quelques centaines de Ko. Checkpoint Les bases de données ne sont pas conçues pour stocker de gros fichiers binaires, et tenter de les y conserver rend à la fois la base de données et le format checkpoint peu pratiques à gérer. Ensuite, lorsque des consommateurs finaux tels que des outils d’analyse de données, des clients ou d’autres services ont besoin d’artefacts accessibles via une URL qu’ils peuvent récupérer directement sans passer par l’agent. Enfin, lorsque la durée de conservation de l’état de l’exécution diffère de celle de la version finale du résultat. Il est possible de supprimer les journaux de session après 30 jours tout en conservant le rapport final pendant des années. Structurer le layout en fonction de (thread_id, checkpoint_id, artifact_name) Ainsi, vous pouvez toujours reconstituer quel exécution a généré tel artefact.

Quand mettre en place des étapes d’approbation humaine

Intégrez des portes de contrôle lorsque le tool call est destructeur et irréversible (écritures dans une base de données, mouvements financiers, envoi de communications externes), lorsque le tool call sort de la zone d’influence de l’agent (déploiements en production, publications destinées aux clients), ou encore lorsque les régulateurs exigent une révision. Une liste de contrôle pratique pour la production

Avant que l’agent à exécution prolongée ne soit déployé, répondez à ces questions en utilisant des termes concrets relatifs à l’infrastructure.

  1. Quel stockage gère les événements de session et checkpoints ?
  2. Que se passe-t-il si le travailleur s’arrête en cours de tool call ?
  3. Un exécution peut-elle corrompre l’espace de travail d’une autre exécution ?
  4. Quelles actions nécessitent une approbation ?
  5. Le modèle ou sandbox peut-il lire des identifiants bruts ?
  6. Quel tool calls peut être réessayé en toute sécurité ?
  7. Où est appliquée la limite de coût par exécution ?
  8. Quelle vérification du contexte frais détermine que l’exécution est terminée ?
  9. Où sont stockés les résultats finaux une fois que sandbox a disparu ?
  10. Pouvons-nous expliquer une exécution échouée demain sans la relancer ?

Si la réponse à l’une de ces questions est « le prompt indique à l’agent de faire preuve de prudence », le système n’a pas encore été déployé. Il s’agit toujours d’une version de démonstration.

Principaux enseignements

  1. Les agents à exécution prolongée ont besoin d’un runtime, et non simplement d’un délai de timeout HTTP plus élevé. Le runtime permet de conserver la session, l’espace de travail, les tool results, les checkpoints, les traces, les budgets, les approbations ainsi que les identifiants en dehors de la mémoire du modèle.

  2. Les composants principaux sont la session, le harness, le sandbox, le checkpoint et la trace. Les vérifications de politique ainsi que la gestion des secrets s’appliquent à l’ensemble de ces éléments.

  3. La durée d’exécution constitue le premier filtre grossier pour le déploiement. Les effets secondaires, la réexécution, l’isolation, l’emplacement des données et la responsabilité opérationnelle complètent le processus de décision.

  4. Une architecture basée sur une file d’attente, des workers et une base de données checkpoint représente une solution par défaut fiable. Un service harness hébergé est adapté lorsque ses contraintes relatives au modèle, aux données, à la récupération et aux extensions correspondent déjà aux besoins du travail à effectuer.

  5. Les outils générant des effets secondaires nécessitent des clés d’idempotence. Dans les files d’attente et les moteurs de workflow, les tentatives de réexécution sont monnaie courante. Sans mécanisme de déduplication, une telle tentative peut envoyer le même e-mail, créer la même demande de modification, ou facturer deux fois le même client.

  6. L’observabilité doit survivre à l’exécution par les workers. En cas d’exécution échouée sur six heures, les artefacts utiles sont le journal des événements, la dernière checkpoint, les traces, le chronogramme des appels aux outils, les différences entre les espaces de travail, les captures d’écran ainsi que les métadonnées de l’environnement.

  7. Les hypothèses sous-jacentes au Runtime évoluent avec les modifications apportées aux modèles. Il convient de rétester les paramètres de réinitialisation du contexte, les schémas d’évaluation, les interfaces des outils et les règles de budget lors du changement de modèle ou de plateforme d’exécution.

  8. Les secrets doivent rester en dehors du harness ainsi que hors du contexte du modèle. L’agent obtient uniquement les capacités offertes par les outils, et non des identifiants bruts.
