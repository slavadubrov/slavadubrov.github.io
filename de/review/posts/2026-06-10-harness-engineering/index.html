    Harness engineering for AI agents: designing the loop around the model

    Part 6 of the Engineering the Agentic Stack series

    In a synthetic checkout run, a coding agent edits src/checkout.py, runs the nearest unit test, receives exit code 0, and reports success. The browser checkout still fails.

    The run stopped because its acceptance rule treated the unit test as enough evidence. That rule lives in the harness, the program around the LLM that builds each model request, executes or rejects tool calls, and decides when the evidence is sufficient. The code that makes that final decision is the acceptance gate. For this task, add the browser test to the gate and keep the loop open when the test fails.

    TL;DR: Start with one model, a few narrow tools, and the browser checkout test from the opening. Add retries after traces show transient call failures. Add a progress handoff when resumed sessions repeat work. When you test either change, keep the model version, tasks, grader, and total budget fixed. Remove the component when it does not improve the measured result.

    The diagram follows one proposed checkout edit. The harness supplies the task and files, checks the proposed edit_file arguments and permissions, runs the edit, executes the checkout tests, and either asks the model to continue or accepts the task.

    What the harness owns

    OpenAI's Codex loop walkthrough describes the basic cycle. The harness assembles a prompt, asks the model for the next action, executes a tool call, appends the result, and asks again until the system accepts the result or hands control back to the user.

    That cycle spans four parts that framework documentation often blurs:

    Term Job Coding-agent example
    Model Proposes text, a tool call, or a final answer Suggests an edit to checkout.py
    Reasoning loop Chooses the next move from the available context Inspect, edit, test, inspect again
    Harness Supplies task context, rejects disallowed calls, records results, and checks completion Allows edits under src/ and requires browser checkout
    Runtime Keeps the process alive and isolated Worker, sandbox, session store, queue

    Frameworks split these responsibilities differently, so diagnose the code that should respond to the failure. A poor plan may need better instructions or model reasoning. If edit_file targets a path outside src/, the harness should reject it. If the sandbox process dies before the edit runs, the runtime should restart the worker or report the crash.

    OpenAI's own harness-engineering case study describes a team that made one application instance bootable per worktree, wired browser automation into the agent environment, and exposed logs, metrics, and traces. A task such as "keep these four user journeys under two seconds" became testable because the agent could run the application and query the same signals an engineer would inspect. The case study is product-specific. The task became checkable only after the application and its performance signals were available inside the agent environment.

    Follow one action through the loop

    In the checkout task, suppose the model asks to replace calculate_discount in src/checkout.py. Several things happen before that edit counts as progress:

    1. The context builder supplies the task, repository instructions, relevant files, prior tool results, and the current plan.
    2. The model proposes an edit_file call with a path and replacement text.
    3. The tool boundary, the harness code between proposal and execution, validates the arguments, checks the path against the allowed scope, and asks for approval if the operation needs it.
    4. The runtime applies the edit in the sandbox and returns a structured result.
    5. The harness runs the checks attached to the task. Here that means checkout unit tests followed by an end-to-end browser test.
    6. A failed check becomes new context for the next model turn. A passing result is recorded as evidence for completion.

    At step 2, no file has changed. The harness can reject ../../secrets.env, require approval for a destructive command, or stop a run that has exhausted its budget. After the tests run, the harness reads their exit codes itself. The model does not get to mark its own edit as passing.

    The trace should show the proposed path and replacement text, the permission decision, the files that changed, and the results of the unit and browser tests. A final done message without those records does not prove that checkout works.

    Decide where each rule is enforced

    Enforce the missing browser test with normal program logic. The harness runs the command, reads its exit code, and refuses to end the run while the command fails. A prompt can remind the model to run the test, but the reminder cannot prevent the model from declaring success.

    Other rules fit different layers:

    Put the rule in Good fit Example
    Prompt or skill Search order, coding conventions, and plan format Read AGENTS.md before editing checkout code
    Tool boundary Argument validation, allowed paths, approvals, and tool access Permit writes only under src/
    Deterministic code Budgets, timeouts, retries, test exit codes, and release gates Keep the run open while browser checkout fails
    Separate evaluator Visual review or criteria that need human-like judgment Compare the checkout page with a written rubric

    The tool boundary needs more than a natural-language description. For a call such as create_test_order, the harness needs a contract with:

    • validated arguments, so malformed input is rejected before execution
    • a structured result such as { "order_id": "123", "created": true }, so later checks do not have to parse free-form text
    • an effect category that records whether the call only fetches information or changes a file, database record, or external service, and whether repeating the call is safe. This label tells the harness whether an automatic retry could duplicate work: it can retry get_order_status, but it must not blindly retry create_test_order because the first call may already have created the order
    • a timeout and retry policy, so a lost response does not trigger an unlimited sequence of calls
    • a permission rule that says what approval is required. Reading order status may run automatically, while creating an order may require confirmation

    The natural-language description is text shown to the model. It might say, "Create a test order for checkout verification." That sentence helps the model decide when to propose create_test_order. It does not authorize the call. Before execution, the MCP client, which is part of the harness in this example, validates the arguments and applies its own rules. It checks whether the server is trusted, whether creating an order requires approval, and whether a failed call may be retried.

    An MCP server publishes tool descriptions and optional behavior annotations to the client. A faulty or malicious server could describe a state-changing tool as harmless. If the client accepted that claim automatically, it might run or retry create_test_order without approval and create a duplicate. The MCP specification therefore requires clients to treat tool annotations as untrusted unless the server itself is trusted. Trust is a client-side setting: the operator decides which servers may supply metadata used for permission or retry decisions.

    Now suppose create_test_order creates the order, but its HTTP response is lost. The harness sees a timeout and cannot tell whether the server completed the request. Repeating the call may create a second order. Reads such as get_order_status can usually be retried. A creation call needs protection such as an idempotency key: the client attaches a unique request identifier, and the service returns the first result instead of creating another order when it sees that identifier again. Without this protection, the harness should check whether the order exists or ask for a human decision before another attempt. AWS documents this pattern in its idempotent API guidance.

    A successful create_test_order response proves only that the tool returned data. It does not prove that the edited checkout works. The harness should validate the response schema and run the test suite. If the acceptance rule includes visual hierarchy, a separate evaluator can compare a browser screenshot with a written rubric. Check that evaluator's decisions against human reviews before using it as a completion gate.

    Long runs need a handoff

    Now let the checkout change grow into a two-hour migration. The first session changes six files, starts a local payment sandbox, finds one failing browser test, and reaches its context limit. A conversation summary helps the next model turn, but it cannot restart the sandbox or prove which files are currently modified.

    The next session has to recover three things:

    What must be recovered What it includes How it can fail
    Conversation history Messages, tool calls, and returned results Old details crowd out the current task
    Working environment Files, processes, browser state, and created orders The transcript says a service is running after it died
    Task progress Plan, completed checks, pending approval, next action The next session repeats finished work

    Compaction replaces older messages with a shorter summary so the current session can continue. It does not restart a dead process or tell the next worker which files now differ. A progress handoff records what the next session needs: the current branch, changed files, last test command and output, and the next unresolved step. If the old conversation contains stale assumptions, the harness can start a fresh worker with that handoff and the current workspace.

    A 10-minute documentation edit may need none of these mechanisms. A migration that crosses sessions may need a handoff even when the conversation still fits in the context window, because the next worker must reconstruct the workspace and task status.

    Anthropic's experiments with long-running coding agents used git history and a progress file between sessions, while its later harness-design report separates compaction from a fresh-context handoff and reports the additional orchestration, token use, and wall time required by handoffs.

    Use the trace to separate different failures

    Across ten synthetic runs of the checkout task, six call the order API and receive a 503 before an order is created. In two runs, the order exists but the response disappears. The remaining two update the code successfully and skip the required browser test.

    One retry policy cannot repair all ten runs. The first six need bounded retries. The next two need idempotency or a status lookup before another request, while the last two need an acceptance gate tied to the missing browser test.

    A chat transcript records what the model saw, but it cannot prove whether the service created an order before the response disappeared. For each order API call, the trace should record the arguments, the local permission or approval decision, whether the server ran the call, whether a status lookup found the order, and whether the browser test later passed. That sequence distinguishes a lost response from a request that never reached the service.

    Repeated symptom Small change to try What to measure
    A transient read error ends the run Bounded retry with backoff Recovery rate, extra calls, wall time
    A resumed session repeats work Structured progress handoff Time to resume and actions repeated
    The agent finishes with a broken checkout Browser acceptance check Broken checkouts accepted as complete
    Visual defects survive passing tests Fresh evaluator with a written rubric Visual defects found, correct pages rejected, review time
    The agent edits outside its scope Narrower tool permission Out-of-scope calls blocked and manual overrides granted

    Before adding a component, name the repeated failure it should reduce and the number you will track. A progress file earns its cost only if it cuts resume time or repeated actions. A fresh reviewer should catch defects that tests and self-review miss. Remove planners, critics, or retries that do not move the chosen number.

    Measure one change at a time

    An ablation tests that expected effect by changing or removing one harness component while the rest of the experiment stays fixed. For example, does editor linting help this model on this task suite?

    Use the following protocol:

    1. Freeze the model version, task instances, environment, grader, and prompts outside the component under test.
    2. Give both variants the same total token, time, and dollar budget.
    3. Choose the number of trials or stopping rule before running the comparison.
    4. Run the same task instances in both variants. Because model outputs vary, repeat each task several times.
    5. Report the mean together with the spread or confidence interval.
    6. Count every started trial, including timeouts, policy stops, harness crashes, and evaluator failures.

    Success rate alone can hide an expensive component. At minimum, track broken tasks accepted as complete, cost and wall time per completed task, tool errors, duplicate orders, review minutes, and manual permission overrides. Choose the metric that carries the real cost for the product. A two-point rise in completed tasks is a bad trade if it doubles the review queue.

    A progress handoff gives the running example a concrete A/B test. Both variants use the same model, tasks, grader, and total budget. The handoff is the only switch, and the measured outcome is repeated work after the checkout migration resumes.

    The SWE-agent paper fixes GPT-4 Turbo on the 300-task SWE-bench Lite split and reports 18.0% resolved with its full interface, compared with 11.0% for the shell-only agent. The paper also changed individual interface features:

    Interface change Resolved
    Full SWE-agent interface 18.0%
    Editor without linting 15.0%
    Full file instead of a 100-line viewer 12.7%
    Full observation history instead of the last five 15.0%

    These numbers belong to that model, benchmark, and $4 per-task budget. The without linting, full file, and full history rows are the useful one-feature tests: each changed an interface feature while the model and evaluation setup stayed fixed.

    LangChain published a broader fixed-model comparison for deepagents-cli. It reports a Terminal-Bench 2.0 increase from 52.8% to 66.5% with gpt-5.2-codex fixed while its team changed the system prompt, tools, and middleware. The post bundles several changes and omits a confidence interval, fixed-total-budget comparison, and per-change ablation table. The combined package helped in that published setup, but the result does not identify which change caused the gain.

    Anthropic reports that an evaluator checked 27 criteria during one sprint on a long-running application. After the team switched to a later model, some tasks no longer improved enough to justify the extra evaluator calls. The team removed one harness component at a time and checked whether those criteria still passed, including components that had helped the previous model.

    Try the method in the companion lab

    The harness-demo project is a small, deterministic exercise with four configurations and 12 synthetic tasks covering flaky tools, lost progress, optimistic self-review, and ambiguous completion.

    The lab uses a deterministic simulation instead of an LLM call. Its token counts and outcomes are synthetic, so each failure remains easy to connect to the component that handles it. The lab's pass rate says nothing about live-agent performance. Measuring that requires sampled tasks, real model calls, and a product grader.

    Run it from the project directory:

    make check
make run
make failures

    make run prints the comparison matrix. make failures shows why each configuration missed a task. The full configuration passes the synthetic suite and spends extra simulated tokens, so the output puts cost beside pass rate.

    The failure checks are plain Python:

    if task.flaky_tool and not config.retry_policy:
    failures.append("flaky tool call was not retried")

if task.needs_progress_file and not config.progress_handoff:
    failures.append("next session lost task state")

if task.needs_fresh_evaluator and not config.evaluator:
    failures.append("self-review missed an implementation gap")

    Start with one loop and one acceptance check

    My first version of a coding-agent harness has one capable model, repository instructions, a few narrow tools, a sandbox, and the browser checkout test. I record tool calls, results, costs, and the final test in one trace so the first useful failures are visible without reconstructing them from terminal logs and chat transcripts.

    Retries come after traces show transient tool errors. Add a progress handoff when sessions repeat work or lose the next step. Add a separate evaluator when the tests pass but browser screenshots repeatedly violate the written UI rubric. For each addition, record who maintains it, how many tokens or seconds it adds, and the test that would justify removing it after a model upgrade.

    Six months later, someone who sees progress_handoff=True should be able to find the failed traces that justified it and the regression cases that still keep it there.
