    Harness Engineering for AI Agents: Designing the Loop Around the Model

    Part 6 of the Engineering the Agentic Stack series

    Part 5 covered the runtime around long-running agents: sessions, sandboxes, checkpoints, traces, and deployment shapes. This post goes one layer inward. The runtime keeps the run alive. The harness decides how each step happens.

    A harness is the loop around the model: the system prompt, tool surface, state files, retry policy, subagents, evaluator, acceptance checks, and trace hooks. The model still matters. But once the model is good enough to use tools for hours, the harness often decides whether the run converges or just produces confident motion.

    TL;DR: Harness engineering is assumption engineering. Every component you add says something specific about what the model cannot do reliably on its own. Progress files assume the model loses state across sessions. Fresh evaluators assume self-review is biased. Retry policies assume tool failures are not reasoning failures. Fail-closed checks assume ambiguous "done" will otherwise become done. Treat those assumptions as hypotheses, benchmark them, and remove the ones that stop paying rent.

    The supporting demo lives in work/demo/2026-06-10-harness-engineering/. Run it with:

    uv run demo.py

    It is deliberately offline. No model key, no Docker, no GPU. The point is to make the ablation method inspectable.

    What harness engineering is

    OpenAI's Codex team used the phrase in their field report, Harness engineering: leveraging Codex in an agent-first world. The interesting part was not "we prompted harder." It was that the team changed the environment until Codex could inspect the application, query local logs and traces, drive browser flows, read repo-local instructions, open pull requests, request agent reviews, and keep looping until checks passed.

    That is harness work. It is software around the model that makes the task legible and the feedback enforceable.

    Anthropic's Effective harnesses for long-running agents makes the same point from the long-context side. Agents can work for many hours, but long work happens across discrete sessions. A new session starts without the previous session's in-memory state. Compaction helps, but it is not the same as giving the next worker a clean, structured handoff.

    The practical definition I use:

    A harness is the executable contract between the model and the work.

    That contract has two jobs.

    First, it feeds the model the right shape of context before it acts: repo maps, skills, task lists, policies, examples, and tool affordances.

    Second, it turns the model's actions into observable, testable steps after it acts: tool results, traces, checks, diffs, screenshots, reviews, retries, and acceptance decisions.

    If you only do the first half, you have a prompt pack. If you only do the second half, you have a test runner that arrives too late. A useful harness does both.

    Four places to put behavior

    Most harness design arguments reduce to one question: where should this behavior live?

    System prompt

    Use the prompt for policies the model must consider while deciding what to do. Examples: decomposition style, coding conventions, when to ask for help, what "done" means, and where to look first.

    Prompts are cheap to change but weak to enforce. Do not put a hard security boundary, cost cap, or data-loss prevention rule only in the prompt. If violation has a real consequence, the harness needs code.

    Tool

    Use tools when the model needs a stable operation with a narrow interface. A good tool reduces search and ambiguity. It also gives the harness a place to validate arguments, attach approval gates, log structured events, and return errors that are useful to the model.

    This is the tool-design argument from Part 3 in harness form. The tool boundary is an ergonomics choice and a control surface.

    Scaffolding code

    Use scaffolding code for deterministic work: retries, idempotency, checkpoint writes, file watching, lint commands, test selection, trace collection, cost limits, and acceptance gates.

    The model can decide to run tests. The harness can make sure tests run before a PR opens. That distinction matters.

    Subagent

    Use a subagent when the task needs a different context, role, or permission set. Anthropic's Harness design for long-running application development is the cleanest current example: planner, generator, and evaluator agents, with the evaluator catching functionality gaps the generator missed.

    Subagents are not free. They add latency, cost, orchestration complexity, and more traces to read. Use them when isolation helps: fresh-context review, read-only audits, parallel research, or adversarial checks. If the subagent is just a slower function call, make it a tool.

    Context management is a harness decision

    Long-running agents fail in boring ways. They forget what was already done. They overfit to stale assumptions. They spend half the run rebuilding a mental map that should have been written down. They decide a task is done because the context window feels full.

    There are three common responses.

    Compaction summarizes older context so the same session can keep going. It is useful when the thread still has coherent direction and the agent needs continuity.

    Progress handoff writes external artifacts that the next session can read: PROGRESS.md, a feature list, a failing-test log, a plan with checked boxes, or a small state file. It is useful when the run may outlast a context window or worker.

    Context reset starts a fresh agent with a structured handoff. It is useful when the current thread has accumulated bad assumptions or when the model starts wrapping up too early.

    The mistake is treating these as model-independent best practices. They are harness hypotheses. On one model, compaction may be enough. On another, reset plus handoff may be load-bearing. On a new model, the reset may become dead weight.

    That is why harnesses need ablation.

    Measure the harness, not the vibes

    The strongest evidence for harness engineering is fixed-model measurement.

    OpenAI's SWE-bench Verified report warns that benchmark scores can reflect model plus scaffold, not model alone. Their GPT-4 example on SWE-bench Lite varied from 2.7% with an early RAG-based scaffold to 28.3% with CodeR.

    LangChain published a more direct harness-only example in Improving Deep Agents with harness engineering: they kept gpt-5.2-codex fixed and changed the harness around deepagents-cli, moving Terminal-Bench 2.0 from 52.8% to 66.5%.

    Those numbers are not a universal conversion table. They are a warning. If you change the prompt, tools, filesystem, retries, evaluator, and trace loop at the same time, you do not know which part helped.

    The minimum useful ablation loop is:

    1. Pick a task suite that represents real work.
    2. Keep the model fixed.
    3. Keep the scoring rule fixed.
    4. Switch one harness component at a time.
    5. Track success, cost, wall time, retries, escaped defects, and human review minutes.
    6. Remove components that no longer improve the metric you care about.

    This is unglamorous. It is also how you keep a harness from fossilizing around last quarter's model failures.

    A tiny ablation demo

    The demo in work/demo/2026-06-10-harness-engineering/ uses a deterministic fake agent. Each task has one or more failure modes: flaky tools, lost state, stale context, self-review optimism, or ambiguous acceptance.

    The core of the runner is intentionally small:

    def run_task(task: Task, config: HarnessConfig) -> RunResult:
    attempts = 1
    tokens = 850 + task.difficulty * 260
    failures: list[str] = []

    if task.flaky_tool and not config.retry_policy:
        failures.append("flaky tool call was not retried")
    if task.needs_progress_file and not config.progress_handoff:
        failures.append("next session lost task state")
    if task.needs_fresh_evaluator and not config.evaluator:
        failures.append("self-review missed an implementation gap")
    if task.ambiguous_done and not config.fail_closed_acceptance:
        failures.append("ambiguous acceptance defaulted to pass")

    Run it from the demo directory:

    uv run demo.py

    Expected shape:

    Harness ablation matrix
config             passed  success  avg_attempts  avg_tokens
bare-loop            1/12        8%          1.00        1738
retry-only           2/12       17%          1.33        1798
handoff-and-retry    5/12       42%          1.33        1890
full-harness        12/12      100%          1.33        2151

    The full harness costs more simulated tokens. That is the point. Some harness components are worth their overhead because they prevent escaped failures. Some are not. A real harness needs the same measurement against real traces.

    What to study in open harnesses

    You do not need to copy a whole agent framework to learn from it. Read harnesses for the assumptions they encode.

    Codex CLI is a local coding agent from OpenAI. The repository is useful because it shows a terminal-first harness with sandboxing, local execution, approvals, MCP support, and repo-level instructions in a production tool surface: openai/codex.

    Deep Agents is LangChain's opinionated harness on top of LangGraph. The docs call out planning, context compression, a virtual filesystem, MCP tools, shell execution, subagents, streaming, memory, permissions, and human-in-the-loop controls: Deep Agents overview.

    OpenHands is worth reading as a larger software-agent system. It exposes an SDK, CLI, local GUI, and cloud version, which makes the same harness ideas visible at different product boundaries: OpenHands/OpenHands.

    mini-SWE-agent is the opposite end of the spectrum: a small Python harness that still performs surprisingly well on SWE-bench Verified. It is useful because it strips away enough machinery that the remaining loop is easy to reason about: SWE-agent/mini-swe-agent.

    Terminal-Bench and Harbor are useful because they make the evaluation harness explicit. Terminal-Bench tasks include instructions, containers, tests, and oracle solutions; the harness connects agents to a sandboxed terminal and runs verification. That is close to the shape production teams need for their own regression suites: Terminal-Bench.

    Read each project with one question in mind: what does this harness believe the model cannot safely or reliably do by itself?

    The ablation habit

    A harness has to shrink too.

    It is tempting to add a progress file after one bad run, add a critic after one overconfident answer, add a retry after one flaky tool call, add a policy prompt after one bad shell command, and never remove any of it. Six months later the model is better, the task mix has changed, and half the harness is ritual.

    Anthropic makes this point directly in the March 2026 harness-design post: when a new model lands, re-examine the harness and strip away pieces that are no longer load-bearing.

    That is the discipline. The harness is not a shrine to model weaknesses. It is a working control system. When the model changes, measure again.

    Key Takeaways

    1. Harness engineering is the design of the loop around the model: prompts, tools, state, subagents, evaluators, retries, checks, traces, and approvals.
    2. Put behavior in the weakest place that can enforce it. Guidance can live in prompts. Hard boundaries belong in tools, scaffolding code, sandboxes, or policy gates.
    3. Treat every harness component as a hypothesis about model limitations.
    4. Compaction, progress handoff, and context reset solve different long-running-agent problems. Do not assume one is always best.
    5. Benchmark harness changes with the model and task suite held fixed.
    6. Remove components that stop improving success, cost, review time, or escaped-defect rate.
    7. Study open harnesses for their assumptions before copying their APIs.
