Schema-gestütztes Reasoning bei vLLM: Strukturierte Ausgaben mit xgrammar und Pydantic

Eine Wiederholungsloop ist im Grunde ein Eingeständnis – sie signalisiert: „Ich erwartete, dass das Modell gültiges JSON zurückgibt, was jedoch nicht der Fall war, daher versuche ich es erneut.“ Die meisten Agenten-Code-Beispiele für große Sprachmodelle, an denen ich gearbeitet habe, setzen stark auf Wiederholungsversuche, wodurch das JSON dennoch häufig fehlerhaft bleibt – so sehr, dass man oft Alarme dafür einrichtet.

Durch das Schema-gestützte Reasoning (SGR) wird dieses Wiederholungsverhalten umgangen, indem das Schema auf Token-Ebene durchgesetzt wird. Man definiert die Struktur des Reasonings als Pydantic-Schema, und der Inferenzmotor filtert bereits vor dem Zufallsauswählen aller Token solche aus, die gegen dieses Schema verstoßen würden. Dadurch sind die Ausgaben von vornherein gültig – und nicht erst durch wiederholte Versuche.

Kurzfassung: SGR nutzt eingeschränkte Dekodierung, um die Ausgabe eines LLM an ein von Ihnen kontrolliertes Pydantic-Schema zu binden. In Kombination mit dem xgrammar-Backend von vLLM ergeben sich stets gültige JSON-Daten bei nahezu vernachlässigbarem Latenzaufwand.

Was ist schema-gestütztes Reasoning?

Schema-gestütztes Reasoning ist eine Technik, die Rinat Abdullin im Jahr 2024 verfasst. Anstatt dem Modell die freie Vollendung von Texten zu ermöglichen – was zu inkonsistentem oder mehrdeutigem Ergebnis führen kann – wird ihm ein strenger Template zur Verfügung gestellt, der definiert:

– welche Schritte es durchlaufen muss, damit es die Analyse nicht überspringen kann, – die Reihenfolge dieser Schritte, sodass die Abwägungen von den Daten bis zur Entscheidung verlaufen, – wo es seine Aufmerksamkeit richten soll, damit die Analyse dort stattfindet, wo sie entscheidend ist.

Man kann sich das als eine kognitive Checkliste vorstellen, der das Modell folgen muss.

Warum SGR wichtig ist

Wenn Sie ein Schema mit Feldern wie definiert churn_analysis, margin_math, und max_discount_percentDas Modell muss die Felder in der richtigen Reihenfolge ausfüllen. Es kann nicht direkt zur Rabattentscheidung übergehen, ohne zuvor die Analyse durchzuführen.

Dadurch ergeben sich folgende Vorteile:

  • reproduzierbares Denkverhalten bei wiederholten Ausführungen
  • prüfbare Ergebnisse, bei denen jeder Schritt überprüft werden kann
  • Zwischenwerte, anhand derer man ein Testdatensatz nutzen kann, um die Leistung zu bewerten
  • kleinere Modelle, die praktikabel werden, da das Schema vorschreibt, was das Modell sonst selbst lernen müsste
  • eine Steigerung der Genauigkeit um 5–10 % häufig gemeldet bei Produktivlasten

Kausale Gedankenkette: besseres Reasoning, dasselbe Strukturproblem

Let's think step by step:
1. First, I'll analyze the customer's churn risk...
2. Then I'll calculate the margin...
3. Therefore, I recommend a 15% discount.

CoT verbessert die Genauigkeit des Reasonings, verschlechtert jedoch die Struktur. Das Ergebnis sind unvorhersehbare Texte, die nahezu unmöglich zuverlässig analysiert werden können. In der Regel ist daher ein zweiter Aufruf des LLM erforderlich, um die strukturierten Daten extrahieren zu können.

SGR: strukturierte Kette des Denkens

SGR beibehält die Intuition von CoT, dass intermediäres Reasoning die Genauigkeit erhöht. Es formalisiert lediglich die einzelnen Schritte.

class PricingLogic(BaseModel):
    # 1. Data Analysis (must complete before decision)
    churn_analysis: str = Field(..., description="Analyze churn_probability")
    financial_analysis: str = Field(..., description="Analyze cart_value and margin")

    # 2. Math Enforcement (explicit calculation)
    margin_math: str = Field(..., description="Calculate: 'Cart $X * Y% = $Z'")

    # 3. Decision Constraint (bounded by prior analysis)
    max_discount_percent: float = Field(..., description="Max allowed discount")

    # 4. Final Output
    offer_code: str
    customer_message: str

Das Modell kann keinen Ausgabewert liefern. max_discount_percent bis churn_analysis, financial_analysis, und margin_math sie werden mit Inhalten befüllt. Das Schema stellt die Reihenfolge der Abwägungen sicher.

SGR-Muster

SGR verfügt über drei Kernmuster, aus denen größere Workflows zusammengesetzt werden.

1. Kaskade: sequenzielle Schritte der Abwägung

Die Kaskade legt eine bestimmte Reihenfolge bei der Abwägung fest. Jedes Feld muss vor dem nächsten ausgefüllt werden.

from pydantic import BaseModel
from typing import Literal, Annotated
from annotated_types import Ge, Le

class CandidateEvaluation(BaseModel):
    """Evaluate a job candidate with enforced reasoning order."""

    # Step 1: Summarize (forces context awareness)
    brief_candidate_summary: str

    # Step 2: Rate (bounded integer)
    rate_skill_match: Annotated[int, Ge(1), Le(10)]

    # Step 3: Decide (constrained choices)
    final_recommendation: Literal["hire", "reject", "hold"]

Gute Anwendungsbereiche: Kandidatenbewertung, Dokumentklassifizierung, Konformitätsanalyse, medizinische Diagnose. Das Modell muss schreiben brief_candidate_summary Bevor es bewerten kann, muss es zunächst bewerten, und bevor es empfehlen kann, muss es zuvor bewertet haben. Es gibt keinen Abkürzungsweg.

2. Routing: ein semantisches Switch-Satzkonstrukt

Das Routing veranlasst das Modell, sich für einen Pfad aus einer Reihe von Optionen zu entscheiden, wobei dies mithilfe von Union Typen.

from pydantic import BaseModel
from typing import Literal, Union

class FeatureLookup(BaseModel):
    """Route to database lookup."""
    rationale: str
    tool_name: Literal["fetch_user_features"] = "fetch_user_features"
    user_id: str

class GeneralResponse(BaseModel):
    """Standard response for non-pricing queries."""
    tool_name: Literal["respond"] = "respond"
    content: str

class RouterSchema(BaseModel):
    """The model must pick exactly ONE branch."""
    action: Union[FeatureLookup, GeneralResponse]

Gute Anwendungsbereiche sind: Intent-Klassifizierung, Tool-Wahl, Support-Triage sowie Mehr-Agenten-Verteilung.

Der Literal Discriminator (tool_name) veranlasst das Modell, einen einzigen Zweig auszuwählen und nur die Felder auszufüllen, die dieser Zweig benötigt.

3. Zyklus: wiederholtes Reasoning mit Listen

Der Zyklus zwingt das Modell dazu, mehrere Elemente zu erzeugen, wobei Grenzen für die Anzahl festgelegt sind.

from pydantic import BaseModel
from typing import List, Literal, Annotated
from annotated_types import MinLen, MaxLen

class RiskFactor(BaseModel):
    explanation: str
    severity: Literal["low", "medium", "high"]

class RiskAssessment(BaseModel):
    """Generate 2-4 risk factors."""
    factors: Annotated[List[RiskFactor], MinLen(2), MaxLen(4)]

Gute Anwendungsgebiete sind: Risikobewertung, Problemanalyse, parallele Aufrufe von Tools sowie mehrstufige Planung. MinLen und MaxLen Die Grenzen erzwingen mindestens 2 und höchstens 4 Elemente. In Kombination mit Routing dient dies dazu, einen fest definierten Satz an Tool-Aufrufen zu versenden.

SGR effektiv einsetzen: eingeschränkte Dekodierung

Die oben genannten Muster sind lediglich Pydantic-Schemata. Was sie bindend macht, ist die eingeschränkte Dekodierung (auch als Strukturierte Ausgabe bezeichnet).

Durch die eingeschränkte Dekodierung wird der Token-Generierungsprozess verändert. Anstatt dem Modell freie Auswahl aus seinem Wortschatz zu ermöglichen, wendet das Engine eine Grammatikmaske an, die Tokens blockiert, die gegen das Schema verstoßen würden. Dies findet im Inferenzengine statt und nicht im Anwendungscode.

[!TIP] SGR benötigt keine „Reasoning-Modelle“ wie o1 oder DeepSeek-R1. Es funktioniert einwandfrei mit auf Anweisungen abgestimmten Modellen und insbesondere hervorragend mit aus solchen Reasoning-Modellen abgeleiteten Modellen.

Warum sich dieser Artikel auf vLLM und xgrammar konzentriert

Einige Gründe:

  • vLLM ist der am weitesten verbreitete Open-Source-Inferenzmotor für LLMs, wodurch die hier entwickelten Lösungen problemlos portiert werden können.
  • xgrammar ist in C++ implementiert und verursacht eine vernachlässigbare Verzögerung.
  • Die API von vLLM ist mit OpenAI kompatibel, wodurch die Migration von Cloud-Anbietern kostengünstig bleibt.
  • xgrammar kann komplexe, verschachtelte Schemata, Unionen sowie rekursive Strukturen verarbeiten.

Im nächsten Abschnitt wird erläutert, wie xgrammar tatsächlich eine Schema-Verifizierung auf Token-Ebene durchführt.

Wie xgrammar Schemata durchsetzt

Dieser Teil ist es wert, genau verstanden zu werden, da er die Art und Weise beeinflusst, wie Sie SGR-Arbeitsabläufe debuggen und optimieren.

Wo die Maskierung stattfindet

xgrammar modifiziert die Ausgabelogits nach dem Forward-Pass des Modells und vor der Stichprobenziehung. Es ändert das Modell selbst nicht; stattdessen filtert es aus, welche Token ausgewählt werden dürfen.

Ein Standard-Inferenzzyklus sieht wie folgt aus:

1. Input tokens → GPU Forward Pass → Logits (probability scores for all ~128K tokens)
2. Logits → Sampling (temperature, top-p, etc.) → Next Token
3. Repeat until done

xgrammar ruts zwischen Schritt 1 und Schritt 2 durch:

1. Input tokens → GPU Forward Pass → Raw Logits
2. Raw Logits → xgrammar Logits Processor → Masked Logits
3. Masked Logits → Sampling → Next Token (guaranteed valid)
4. Repeat until done

Das Modell berechnet weiterhin seine vollständige Wahrscheinlichkeitsverteilung auf der GPU. Anschließend wird xgrammar auf der CPU ausgeführt, wobei vor dem Sampling ein Bitmasken angewendet wird, um diese Logits zu filtern. Für ungültige Token werden ihre Logits auf einen bestimmten Wert gesetzt. -∞, wodurch ihre Wahrscheinlichkeit nach der Softmax-Berechnung exakt 0 wird.

Zwei Phasen

xgrammar teilt die Arbeit in die Kompilierungszeit und die Laufzeit auf, was seine Geschwindigkeit ausmacht.

Phase 1: Grammatik-Kompilierung, einmal pro Schema

# This happens once per schema
tokenizer_info = xgr.TokenizerInfo.from_huggingface(tokenizer)
grammar_compiler = xgr.GrammarCompiler(tokenizer_info)
compiled_grammar = grammar_compiler.compile_json_schema(schema_json)

Durante der Kompilierung wandelt xgrammar:

  1. das JSON Schema in eine kontextfreie Grammatik um.
  2. einen Pushdown-Automaten (PDA) auf, der ein Zustandsautomat mit einem Stack ist, um damit verschachtelte Strukturen verarbeiten zu können. {"a": {"b": {"c": ...}}}.
  3. Es werden im Voraus die Token ermittelt, die an jeder Grammatikposition gültig sind. Das Ergebnis ist der „adaptive Token-Masken-Cache“.
  4. Die Token werden als „kontextunabhängig“ (kann im Cache gespeichert werden) oder „kontextabhängig“ (muss zur Laufzeit anhand des Stack-Zustands überprüft werden) kategorisiert.

[!HINWEIS] Etwa 99 % der Token erweisen sich als kontextunabhängig und werden somit im Cache abgelegt.Das XGrammar-Paper). Die meisten Gültigkeitsprüfungen zur Laufzeit bestehen lediglich aus Cache-Aufrufen, weshalb xgrammar so schnell ist.

Phase 2: Erstellung der Laufzeit-Maske für jedes Token

In jedem Generierungsschritt: GrammarMatcher verfolgt die aktuelle Position in der Grammatik. 2. Es ruft das im Voraus berechnete Maskenwert für kontextunabhängige Token ab. 3. Es führt den PDA aus, um die verbleibenden kontextabhängigen Token zu überprüfen. 4. Es kombiniert diese zu einem endgültigen Bitmuster und wendet es auf die Logits an.

Warum Pushdown-Automaten und nicht reguläre Ausdrücke?

Wegen der Verkettung. Ein regulärer Ausdruck (eine endliche Zustandsmaschine) kann Strukturen wie folgende nicht zuverlässig abbilden:

{ "user": { "profile": { "settings": { "theme": "dark" } } } }

Der schwierige Teil sind die schließenden Klammern. }}}: Man muss sich merken, wie viele Elemente man geöffnet hat. Ein Pushdown-Automat verfügt über einen Stack, der dies trackt, wodurch er beliebige Nestungstiefen bewältigen kann. Genau deshalb kann xgrammar Union-Typen, verschachtelte Objekte sowie rekursive Schemata durchsetzen, wo herkömmliche regex-basierte Ansätze versagen.

Ein konkretes Beispiel: Erstellung eines Float-Felds

Wenn das Modell generiert… "max_discount_percent":, xgrammar erkennt aus dem Schema, dass ein float kommt als Nächstes. Die Maske:

  • erlaubt (Wahrscheinlichkeit unverändert): 0, 1, 2, …, 9, ., -
  • Blöcke (Wahrscheinlichkeit auf 0 gesetzt): ", {, [, true, false, nullsowie der Rest des Wortschatzes von über 128 Kilozeichen.

Der Forward-Pass könnte der Zeichenfolge eine hohe Wahrscheinlichkeit zugewiesen haben. "fifteen". Nach der Maskierung durch xgrammar hat dieser Token eine Wahrscheinlichkeit von 0. Das Modell muss daher Ziffern ausgeben.

Warum „nahezu null Overhead“

Drei Gründe:

  1. Parallelverarbeitung. Die Berechnung der Maskierung auf dem CPU überschneidet sich mit dem nächsten Vorwärtslauf auf der GPU. Während die GPU die Logits für Token N+1 berechnet, berechnet der CPU die Maskierung für Token N.
  2. Caching. Der größte Teil der Validierungsarbeiten erfolgt bereits zur Kompilierzeit. Im Laufzeitbetrieb handelt es sich hauptsächlich um Cacheabfragen.
  3. C++-Implementierung. Der häufig genutzte Codepfad ist in C++, nicht in Python, und die Maskierung wird direkt auf den Logits angewendet.

In Benchmarks zeigt xgrammar einen vernachlässigbaren Overhead, und strukturierte Generierung kann gelegentlich schneller sein als unbeschränkte Generierung, da das eingeschränkte Vokabular das Sampling erschwert.

Schritt 1: Definition der Schemata

# sgr/models/schemas.py
from pydantic import BaseModel, Field
from typing import Literal, Union


# --- Phase 1: Routing (Union for branching) ---
class FeatureLookup(BaseModel):
    """Route to DB lookup if pricing context is needed."""
    rationale: str
    tool_name: Literal["fetch_user_features"] = "fetch_user_features"
    user_id: str


class GeneralResponse(BaseModel):
    """Standard response for non-pricing queries."""
    tool_name: Literal["respond"] = "respond"
    content: str


class RouterSchema(BaseModel):
    action: Union[FeatureLookup, GeneralResponse]


# --- Phase 2: Pricing Logic (Cascade for sequential reasoning) ---
class PricingLogic(BaseModel):
    """
    Strict reasoning topology for dynamic pricing.
    Fields are ordered to enforce the analysis→decision flow.
    """
    # 1. Data Analysis (Reflection)
    churn_analysis: str = Field(...,
        description="Analyze churn_probability (High > 0.7).")
    financial_analysis: str = Field(...,
        description="Analyze cart_value and profit_margin.")

    # 2. Hard Math Enforcement
    margin_math: str = Field(...,
        description="Calculate absolute profit: 'Cart $200 * 0.20 Margin = $40'.")

    # 3. The Decision Constraint
    max_discount_percent: float = Field(...,
        description="Max allowed discount %. NEVER exceed margin.")

    # 4. Final Output
    offer_code: str = Field(..., description="Generated code (e.g. SAVE20).")
    customer_message: str = Field(..., description="The final polite offer text.")

Schritt 2: Ein LLM-Client, der xgrammar aktiviert

# sgr/utils/llm_client.py
from openai import OpenAI
from pydantic import BaseModel
from typing import TypeVar
import json

T = TypeVar("T", bound=BaseModel)


class LLMClient:
    """Wrapper for vLLM with xgrammar-enforced structured generation."""

    def __init__(self, base_url: str = "http://localhost:8000/v1"):
        self.client = OpenAI(base_url=base_url, api_key="EMPTY")
        self.model = self._get_available_model()

    def _get_available_model(self) -> str:
        """Auto-detect the model running on vLLM server."""
        try:
            models = self.client.models.list()
            if models.data:
                return models.data[0].id
        except Exception:
            pass
        return "Qwen/Qwen2.5-7B-Instruct"

    def run_sgr(self, messages: list[dict], schema_class: type[T]) -> T:
        """Run inference with Schema-Guided Response constraints.

        Uses vLLM's guided_json with xgrammar backend to enforce
        strict schema constraints at the token generation level.
        """
        schema_dict = schema_class.model_json_schema()

        # Enhance system message with schema for model guidance
        enhanced_messages = messages.copy()
        if enhanced_messages and enhanced_messages[0]["role"] == "system":
            schema_json = json.dumps(schema_dict, indent=2)
            enhanced_messages[0] = {
                "role": "system",
                "content": (
                    enhanced_messages[0]["content"]
                    + f"\n\nRespond with JSON matching this schema:\n{schema_json}"
                ),
            }

        # vLLM's guided_json with the xgrammar backend
        completion = self.client.chat.completions.create(
            model=self.model,
            messages=enhanced_messages,
            temperature=0.1,  # Low temp for deterministic reasoning
            extra_body={
                "guided_json": schema_dict,  # Pydantic schema as dict
                "guided_decoding_backend": "xgrammar",  # Hardware-enforced
            },
        )

        raw_response = completion.choices[0].message.content
        return schema_class.model_validate_json(raw_response)

[!HINWEIS] > guided_json nimmt ein JSON Schema-Diktum entgegen. Mit guided_decoding_backend: "xgrammar", das LLM kann ausschließlich Token erzeugen, die ein gültiges JSON bilden, das mit Ihrem Schema übereinstimmt.

Schritt 3: Den Agenten orchestrieren

# sgr/agent.py
from .models.schemas import PricingLogic, RouterSchema
from .prompts.routing import build_routing_prompt
from .prompts.pricing import build_pricing_context_prompt, ASSISTANT_FETCH_MESSAGE
from .store.hybrid_store import HybridFeatureStore
from .utils.llm_client import LLMClient


def pricing_agent(user_query: str, user_id: str) -> str:
    """Process a pricing query with three-phase SGR workflow."""

    llm = LLMClient()
    feature_store = HybridFeatureStore()

    # Build conversation history
    history = [
        {"role": "system", "content": build_routing_prompt(user_id)},
        {"role": "user", "content": user_query},
    ]

    # --- Phase 1: Routing (Uses RouterSchema) ---
    print(f"🤖 Processing: '{user_query}' for {user_id}")
    decision = llm.run_sgr(history, RouterSchema)
    print(f"📍 Routing decision: {decision.action.tool_name}")

    if decision.action.tool_name == "respond":
        return decision.action.content

    # --- Phase 2: Context Retrieval ---
    if decision.action.tool_name == "fetch_user_features":
        print(f"🔍 Fetching features for {user_id}...")
        context = feature_store.get_user_context(user_id)

        if not context:
            return "Error: User profile not found."

        print(f"   [Data] LTV: ${context.get('user_ltv')} | "
              f"Margin: {context.get('cart_profit_margin', 0) * 100}%")

        # Inject context into conversation
        history.append({"role": "assistant", "content": ASSISTANT_FETCH_MESSAGE})
        history.append({
            "role": "user",
            "content": build_pricing_context_prompt(
                churn_prob=context.get("churn_probability", 0.5),
                cart_val=context.get("current_cart_value", 100),
                margin=context.get("cart_profit_margin", 0.2),
                user_ltv=context.get("user_ltv", 0),
            ),
        })

        # --- Phase 3: SGR Logic Execution (Uses PricingLogic) ---
        print("🧠 Calculating Offer (Schema Enforced)...")
        offer = llm.run_sgr(history, PricingLogic)

        # Audit log — the SGR benefit: explicit reasoning traces
        print(f"   [Audit] Math: {offer.margin_math}")
        print(f"   [Audit] Max Allowed: {offer.max_discount_percent}%")

        return offer.customer_message

    return "I'm sorry, I couldn't process your request."


if __name__ == "__main__":
    response = pricing_agent("I want a discount or I'm leaving!", "user_102")
    print(f"\n💬 Final Reply: {response}")

Empfehlungen

Schema-Design

  1. Ordnen Sie die Felder nach dem logischen Ablauf. Analysefelder kommen vor Entscheidungsfeldern.
  2. Schreiben Sie beschreibende Texte. Field Beschreibungen. Sie lenken die Aufmerksamkeit des Modells genauso stark wie der Feldname.
  3. Einschränkung durch Literal und Annotated. Verwenden Sie Literal["a", "b"] für Enums und Annotated[int, Ge(1), Le(10)] für die Grenzen.
  4. Halten Sie die Schemata fokussiert. Ein Schema pro Überlegungsphase und anschließende Kombination durch mehrere Aufrufe.

vLLM-Konfiguration

  1. Verwenden Sie eine niedrige Temperatur (0,1–0,3) für deterministisches Denken.
  2. Überlassen Sie xgrammar die Strukturverwaltung. Gegenübergeben Sie keine Formatierungsanweisungen im Prompt, um Konflikte zu vermeiden.
  3. Achten Sie auf den Tokenverbrauch. SGR verwendet in der Regel weniger Tokens als CoT, da es keine ausführlichen Beschreibungen gibt.

Produktionsaspekte

  1. Versionieren Sie Ihre Schemata genauso wie APIs.
  2. Auch bei SGR müssen Netzwerk- und Serverfehler weiterhin sanft abgefangen werden.
  3. Protokollieren Sie die Rohausgaben von SGR zur Einhaltung von Vorgaben und zum Debugging.
  4. Testen Sie mit Randfällen, damit die Struktur an den Grenzen weiterhin funktioniert.

Fazit

SGR ist das Mittel, das es ermöglicht, von „funktioniert in einer Demo“ zu „funktioniert in der Produktion“ überzugehen. Sie definieren die Überlegungsstruktur in Pydantic, lassen xgrammar diese bei der Dekodierung durchsetzen – das Ergebnis sind:

  • immer gültige Ausgaben, ohne Wiederholungszyklen oder Parsing-Fehler
  • prüfbare Daten auf Feldebene
  • Nutzung kleinerer Modelle, da diese nicht mehr selbst die Formatierung sicherstellen müssen
  • geringere Laufkosten, da weniger Tokens, weniger Wiederholungen sowie kleinere Modelle benötigt werden

The sgr-Discount-Manager Die Demo verbindet jedes Codebeispiel aus diesem Beitrag mit einem echten vLLM-Server. Klonen Sie die Projektstruktur und beginnen Sie damit, die Schemata an Ihren eigenen Arbeitsablauf anzupassen.

Wichtige Erkenntnisse

  1. Das schema-gestützte Reasoning macht die Topologie der Abwägungen explizit, anstatt darauf zu hoffen, dass das Modell schriftliche Anweisungen befolgt.
  2. Durch eingeschränkte Dekodierung wird eine ungültige JSON-Struktur zur Generierungzeit vermieden, was sauberer ist als das Nachträgliche Validieren und Wiederholen.
  3. Stellen Sie Analysefelder vor Entscheidungsfelder, wenn das Schema eine vorherige Abwägung vor der Ausgabe erzwingen soll.
  4. Verwenden Sie SGR, wenn der nachgeschaltete Code von einer bestimmten Struktur abhängt – nicht dann, wenn das Endergebnis in Form freier Prosa vorliegt.
