Stable Diffusion unter macOS: Zeichnen mit DiffusionBee, ComfyUI, A1111 und Fooocus

Die lokale Ausführung der Image-Engine Models unter macOS ist im Wesentlichen eine Frage der gewählten Werkzeuge. Draw Things, DiffusionBee, ComfyUI, AUTOMATIC1111 und Fooocus überschneiden sich in ihren Funktionalitäten, weisen jedoch Unterschiede hinsichtlich der Einrichtung, der Unterstützung für Model, der Workflow-Steuerung, verfügbarer Erweiterungen sowie der Automatisierungsfähigkeiten auf.

Eine App zum Drag-and-Drop ermöglicht es Ihnen, schnell mit der Arbeit zu beginnen; ein Knotengraph hingegen erfordert mehr Lernzeit, bietet aber einen umfassenderen Einblick in Pipeline. Auf Apple Silicon beeinflussen die Anwendung, die Model-Familie, die Genauigkeit, die Bildgröße sowie Workflow maßgeblich, ob eine Generierung überhaupt möglich ist. Wählen Sie zunächst die für Ihre Aufgabe geeignete Benutzeroberfläche aus und prüfen Sie anschließend die genauen Model-Einstellungen.

Zusammenfassung. Wählen Sie bei Aufgaben, die bereits durch eine natives Anwendung abgedeckt werden, zunächst Draw Things oder DiffusionBee. Verwenden Sie ComfyUI, wenn der Graph selbst überprüft, geteilt oder automatisiert werden muss. Wählen Sie AUTOMATIC1111 ausschließlich für einen Workflow, der auf seine Benutzeroberfläche oder Erweiterungen angewiesen ist. Betrachten Sie Fooocus als einen stabilen Workflow aus der SDXL-Ära – nicht als Weg zur aktuellen Unterstützung von Model, da das offizielle Projekt nur eine begrenzte Langzeitunterstützung bietet.

Eine Model-Datei ist kein portabler Workflow

Mehrere Anwendungen können geladen werden. .safetensors Dateien allein reichen aus, doch eine entsprechende Erweiterung garantiert noch keine Kompatibilität. Ein vollständiger Pipeline erfordert möglicherweise zudem eine spezielle Architektur, einen Textencoder, eine VAE, einen Scheduler, ControlNet, LoRA, Tokenizer sowie eine an die Anwendung angepasste Konfiguration.

Erstellen Sie eine Aufzeichnung der Release-Einheit, die das Bild erzeugt hat:

  • Anwendung und Version
  • Model-Repositorium, Revision sowie Lizenz
  • Jeder Model-Komponente und -Adapter
  • Prompt, negative Prompt-, Seed-Werte, Sampler, Schritte, Leitprinzipien sowie Dimensionen
  • Graph oder Workflow-Datei, sofern die Anwendung eine solche bereitstellt

Dieser Datensatz ist wichtiger als die Beliebtheit der Anwendung. Derselbe Startwert kann nach einer Änderung der Backend, Model, des Samplers oder der Präzision dennoch ein anderes Ergebnis liefern.

Die vereinigte Speicherressource ist gemeinsam nutzbar – jedoch nicht unbegrenzt.

Apple Silicon ermöglicht es den CPU und GPU, auf denselben Speicherpool zuzugreifen. Dadurch entfällt eine eigene Kopiergrenze, doch macOS sowie alle laufenden Anwendungen konkurrieren um die gleiche Kapazität.

Bei der Bildgenerierung umfasst die Spitzenbelastung nicht nur Checkpoint Weights:

peak memory ≈ model components
            + intermediate activations
            + attention and runtime workspace
            + loaded adapters and control models
            + application and OS memory

Die Auflösung, die Batch-Größe, die Model-Architektur, die Präzision, das Upscaling, ControlNet sowie die Frage, ob die Komponenten weiterhin geladen bleiben, beeinflussen alle das Ergebnis. Kaufen oder auswählen Sie keine Maschine aufgrund einer allgemeinen Tabelle mit Angaben wie „8/16/32 GB“. Testen Sie den gewünschten Workflow und beobachten Sie den Speicherdruck. Eine Generierung, die ausschließlich durch kontinuierliches Ausnutzen des Swaps funktioniert, stellt keinen geeigneten Betriebspunkt dar.

Dinge zeichnen: nativer Interaktion und Skripting

Dinge zeichnen Es handelt sich um eine auf der Apple-Plattform laufende Anwendung zur lokalen Bildgenerierung. Die derzeitige Dokumentation behandelt Models, LoRAs, ControlNet, textbasierte Inversion, Core ML sowie versioniertes JavaScript-Skripting.

Wählen Sie diese Option, wenn die Aufgabe innerhalb einer nativen Anwendung ausgeführt werden soll, aber dennoch mehr Kontrolle erfordert als eine Prompt-Box. Bevor Sie sich darauf festlegen, importieren Sie die exakten Model-Komponenten und erstellen Sie ein repräsentatives Beispiel für eine Bearbeitung, einen Adapter oder eine Steuerung Workflow. Unter „nativ“ versteht man hier die Schnittstelle sowie die Implementierung; dies garantiert jedoch nicht die Unterstützung oder Geschwindigkeit in jeder neuen Architektur.

DiffusionBee: verpackte macOS Workflow-Anwendung

DiffusionBee Die Desktop-Anwendung integriert die Herunterladung von Model-Paketen sowie gängige Erstellungsaufgaben. In der Projektdokumentation werden Unterstützung für Verfahren wie Image-to-Image-, Inpainting-, Outpainting-, ControlNet-, LoRA-, SDXL-Verarbeitung sowie ausgewählte Flux-Funktionen aufgelistet.

Wählen Sie diese Option, wenn die Installation sowie eine integrierte Benutzeroberfläche wichtiger sind als die Exportierung eines Pipeline-Graphen. Prüfen Sie vor dem Herunterladen einer großen Model stets die neueste Version sowie die genaue Architektur – schließlich bedeutet ein Funktionslabel wie „Flux-Unterstützung“ nicht automatisch, dass alle abgeleiteten Komponenten, Quantization, oder Hilfskomponenten ebenfalls geladen werden.

ComfyUI: Der Workflow ist ein Artefakt

ComfyUI Es wird die Generierung als Knotengraph darstellt. Ein Workflow kann unabhängig von seinem Ausgabeergebnis als JSON gespeichert werden, wodurch der Pipeline überprüfbar und versionierbar ist. Zudem ermöglicht seine Executor-Struktur es, Graphabschnitte, deren Eingaben sich nicht geändert haben, erneut zu berechnen, zu vermeiden.

Die aktuelle ComfyUI Desktop-Version unterstützt Apple Silicon und verwaltet ihre eigene Umgebung – obwohl die offizielle macOS-Dokumentation sie weiterhin als Beta-Version bezeichnet. Eine Installation über Homebrew ist verfügbar:

brew install comfyui

Wählen Sie ComfyUI, wenn ein Pipeline über Branches, wiederverwendbare Komponenten, mehrere Models sowie Automatisierungsfunktionen verfügt. Bewahren Sie dabei die JSON, die Anwendungsversion, die Versionen der Core-Node‑Komponenten, die Revisionen der benutzerdefinierten Node‑Komponenten sowie das Model-Manifest zusammen.

Custom Nodes sind ausführbare Abhängigkeiten und keineswegs harmlose Vorlagen. Überprüfen Sie deren Quellcode sowie die installierten Skripte, sichern Sie bestimmte Revisionen ab und isolieren Sie die Umgebung. Ein Graph von einem unbekannten Ersteller kann auf Code sowie Models verweisen, den Sie noch nicht überprüft haben.

AUTOMATIC1111: Kompatibilität mit einer etablierten Web-Oberfläche

AUTOMATIC1111 Stable Diffusion WebUI Es stellt die Erstellungs-Einstellungen über eine Browser-Oberfläche zur Verfügung und unterstützt Skripte, Erweiterungen sowie ein API. Wählen Sie diese Option, wenn bereits vorhandene Tutorials, Automatisierungen oder Erweiterungen Teil der Anforderungen sind.

Der offizielle Leitfaden für Apple Silicon beschreibt Ausnahmen sowie Leistungsbeschränkungen, darunter eine schwache Trainingsleistung. Stellen Sie vor der Einführung einer auf Erweiterungen ausgerichteten Konfiguration sicher, dass die erforderlichen Funktionen unter der Ziel-MacOS- sowie den PyTorch-Versionen funktionieren.

Erweiterungen führen Code innerhalb der Anwendungsumgebung aus. Fixieren Sie sie, überprüfen Sie Updates und halten Sie den Dienst auf Loopback laufen – es sei denn, der Remote-Zugriff ist beabsichtigt und geschützt.

Fooocus: eine begrenzte Alternative aus der SDXL-Ära

Fooocus Er verbirgt absichtlich viele technische Entscheidungen hinter einer eigenwilligen Prompt-getriebenen Workflow. Das kann nützlich sein, wenn die Standardeinstellungen zum jeweiligen Aufgabenprofil passen.

Der aktuelle README des Projekts besagt, dass Fooocus auf SDXL basiert und nur eine eingeschränkte Langzeitunterstützung mit reinen Fehlerbehebungen bietet. Zudem wird angegeben, dass der Mac nicht umfassend getestet wurde, wobei die Installation auf diesem Betriebssystem als inoffiziell bezeichnet wird. Dadurch stellt Fooocus eine begrenzte Option für bereits vorhandene Workflow dar und ist somit nicht der Standardweg für neue Architekturen oder langfristig genutzte macOS-Infrastrukturen.

Ausführen eines repräsentativen Akzeptanztests

Verwenden Sie eine kleine Akzeptanzmenge, bevor Sie sich auf ein Tool festlegen:

  1. Erstellen Sie ein Basistext-zu-Bild-Ergebnis unter Verwendung einer gespeicherten Seed-Wertung sowie der entsprechenden Einstellungen.
  2. Führen Sie die anspruchsvollste erforderliche Operation aus: Inpainting, ControlNet, LoRA, Upscaling oder ein mehrstufiges Verarbeitungsgraphen-Modell.
  3. Starten Sie die Anwendung erneut und generieren Sie das Workflow anhand der gespeicherten Ergebnisse.
  4. Messen Sie den Cold Start, die Generierungszeit, den maximalen Speicherbedarf sowie die Ausgabemessungen.
  5. Verschieben Sie den Workflow auf ein sauberes Benutzerkonto oder eine andere Maschine und listen Sie alle fehlenden Abhängigkeiten auf.
  6. Überprüfen Sie die Bindung an das lokale Netzwerk, die Downloads, die Analyseeinstellungen, die Lizenzen sowie die Herkunft des Model.

Vergleichen Sie keine Werkzeuge mit unterschiedlichen Checkpoints, Auflösungen, Schrittzählungen oder Präzisionswerten und bezeichnen Sie das Ergebnis nicht als Runtime Benchmark.

Eine praktische Standardlösung

Für exploratorische Arbeiten sollte man mit einer nativen Anwendung sowie der von ihr dokumentierten Model beginnen. Erst wenn die Workflow an sich von Wert wird – also dann, wenn sie überprüft, wiederholt, automatisiert oder an Dritte weitergegeben werden muss – wechselt man zu ComfyUI. Wenn eine bestehende Abhängigkeit ihre spezifischen Funktionen nützlich macht, bleiben AUTOMATIC1111 oder Fooocus weiterhin geeignet.
