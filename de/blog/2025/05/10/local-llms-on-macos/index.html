Lokale LLMs unter macOS: Ollama, LM Studio, llama.cpp, MLX und Apple Silicon

Es wird langweilig, jeden Prompt an eine Drittanbieter-API zu senden – besonders dann, wenn die Hälfte der Anfragen darin besteht, einen Absatz umzuschreiben oder nach dem JSON-Schema zu fragen. Lokale LLMs haben dieses Problem auf Apple Silicon schneller gelöst, als ich erwartet hatte. Ein 7B-Modell in 4-Bit-Quantisierung läuft problemlos auf einem 16-GB-MacBook, wobei der gesamte Datenfluss direkt über die Tastatur abgewickelt wird.

Die offene Frage bleibt daher: Welche App soll zum Ausführen genutzt werden? Ollama, LM Studio, llama.cpp, MLX sowie einige weitere Tools nutzen alle ähnliche Inferenzengines sowie dieselben GGUF-Dateien. Der Unterschied liegt darin, wie viel Aufwand zwischen dem Benutzer und dem Modell entsteht: Einerseits reicht ein Doppelklick und das Eingeben von Text aus; andererseits muss man erst aus Quellen kompilieren und anschließend die entsprechenden Dokumentationen durchlesen.

Zusammenfassung: Verwenden Sie Ollama für eine einfache lokale API, LM Studio für eine GUI sowie einen mit OpenAI kompatiblen Endpunkt, llama.cpp, wenn Sie direkten Zugriff auf die GGUF-Inferenz benötigen, und MLX, wenn Sie Python-Arbeitsabläufe für Apple Silicon nutzen möchten.

Wesentliche Konzepte für lokale LLMs unter macOS

  1. Inferenz: Ausführung des Modells zur Erzeugung von Text.
  2. Quantisierung (GGUF): Eine Technik zur Verkleinerung der Modellgröße bei minimalem Qualitätsverlust. Sie werden Dateinamen wie diese sehen: llama-3-8b-Q4_K_M.gguf. Der Q4 „Part“ bezeichnet eine 4-Bit-Quantisierung, die deutlich weniger RAM verbraucht als vollständige 16-Bit-Gewichte.
  3. Apple Silicon (Metal): Die M-Serie-Chips von Apple teilen sich den RAM zwischen CPU und GPU (Apple bezeichnet dies als „Unified Memory“). Deshalb kann ein MacBook mit 32 GB oder 64 GB Modelle laden, für die auf einem PC eine teure dedizierte GPU erforderlich wäre.

Voraussetzungen

  • Hardware: Ein Mac mit Apple Silicon (von M1 bis M4) ist die richtige Wahl. Intel-Macs funktionieren zwar auch, sind aber deutlich langsamer.
  • RAM:
    • 8GB: ausreichend für kleine Modelle (Mistral 7B, Llama 3 8B).
    • 16GB+: ideal für größere Modelle sowie das Multitasking.
  • Speicherplatz: Die Modelle sind sehr datenintensiv. Planen Sie etwa 10–20 GB ein, um eine nützliche Ausgangsbibliothek zu erhalten.

1. Ollama – Die „einfach funktionierende“ Option

Herunterladen: ollama.com

Denken Sie an Ollama als den „Docker für LLMs“. Es umhüllt die llama.cpp Der Engine in einem nativen macOS-Paket lädt Modelle auf Abruf herunter und leitet die Verarbeitung automatisch an Metal weiter. Sie installieren ihn, führen einen Befehl aus – und schon können Sie chatten. Dies ist der einfachste Weg, wenn Sie lediglich einen funktionsfähigen LLM sowie eine HTTP-Schnittstelle benötigen, auf die Ihr Code verweisen kann.

Beispiel-Arbeitsablauf

# 1. Download and run Llama 3 (it auto-downloads if needed)
ollama run llama3

# 2. Use it in your code via the local API
curl http://localhost:11434/api/generate -d '{
  "model": "llama3",
  "prompt": "Explain quantum computing to a 5-year-old",
  "stream": false
}'

Vorteile und Nachteile

✅ Vorteile❌ Nachteile
Einfachste Einrichtung (Drag-and-Drop) .dmg)Die Kernanwendung ist Closed-Source.
Saubere CLI (ollama list, ollama pull)Geringere Feinabstimmungsmöglichkeiten bei den Generierungsparametern
Große Bibliothek vorkonfigurierter Modelle

2. LM Studio – Der visuelle Explorer

Herunterladen: lmstudio.ai

LM Studio ist die auf eine grafische Benutzeroberfläche ausgerichtete Alternative. Es verfügt über einen browserbasierten Suchmechanismus im Stil eines App-Stores, mit dem direkt in HuggingFace gesucht werden kann, unterstützt neben GGUF auch Apple’s MLX-Format – welches auf einigen Macs schneller arbeiten kann – und stellt außerdem einen mit OpenAI kompatiblen lokalen Server bereit. Daher funktioniert der bereits vorhandene Client-Code, der mit OpenAI kommuniziert, in den meisten Fällen weiterhin einwandfrei.

Beispiel-Arbeitsablauf

Sie können das offizielle Python SDK verwenden oder jeden mit OpenAI kompatiblen Client, der auf den lokalen Server verweist:

# Using the official LM Studio SDK
from lmstudio import LMStudio

client = LMStudio()
response = client.complete(
    model="llama-3-8b",
    prompt="Write a haiku about debugging."
)
print(response.content)

Vorteile und Nachteile

✅ Vorteile❌ Nachteile
Geeignetes, benutzerfreundliches InterfaceDie GUI ist quelloffen.
Native Unterstützung sowohl für GGUF- als auch für MLX-ModelleGrößere Downloadgröße (~750 MB)
Integrierter RAG (Chatten mit Ihren PDF-Dateien)

3. llama.cpp – Das Werkzeug für fortgeschrittene Nutzer

Repo: github.com/ggml-org/llama.cpp

Dies ist der Engine, den nahezu jedes andere Tool verwendet. Wenn Sie maximale Leistung, die neuesten Funktionen sobald sie verfügbar sind, oder die Einbettung eines LLM in Ihre eigene C++-Anwendung wünschen, dann ist dies die Quelle. Er arbeitet direkt auf dem Betriebssystem und ist leichtgewichtig – doch der Preis dafür ist, dass Sie alles selbst verwalten müssen: Downloads, Formate sowie Dutzende von CLI-Flaggen.

Beispiel-Arbeitsablauf

# 1. Install via Homebrew
brew install llama.cpp

# 2. Download a model manually (e.g., from HuggingFace)
huggingface-cli download TheBloke/Llama-3-8B-Instruct-GGUF --local-dir .

# 3. Run inference with full control
llama-cli -m llama-3-8b-instruct.Q4_K_M.gguf \
  -p "Write a python script to sort a list" \
  -n 512 \
  --temp 0.7 \
  --ctx-size 4096

Vorteile und Nachteile

✅ Vorteile❌ Nachteile
Vollständige Kontrolle über jeden ParameterStarke Lernkurve (nur über die CLI).
MIT-lizenziert (Open Source)Manuelle Modellverwaltung
Sehr leichtgewichtig (<30 MB)

4. GPT4All – Datenschutzfirst und RAG

Herunterladen: gpt4all.io

GPT4All basiert auf zwei Kernkonzepten: Datenschutz und Dokumente. Seine herausragende Funktion, LocalDocs, ermöglicht es Ihnen, die Anwendung auf einen Ordner mit PDFs, Notizen oder Code zu verweisen und direkt mit dem Inhalt zu chatten. Alle Vorgänge finden vollständig offline statt, ohne dass Telemetriedaten gesammelt werden. Es handelt sich dabei um die einfachste Methode, eine funktionsfähige RAG-Lösung auf Ihrem Rechner einzurichten – ohne dass Sie Code schreiben müssen.

Vorteile und Nachteile

✅ Vorteile❌ Nachteile
LocalDocs RAG funktioniert ohne weitere Einstellungen bereits hervorragend.Nur GUI-Version (kein Headless-Modus)
Vollständig offline und privatHöherer Ressourcenverbrauch im Vergleich zu Ollama
Plattformunabhängig (Mac, Windows, Linux)

5. KoboldCPP – Für Erzähler

Repo: github.com/LostRuins/koboldcpp

Ein Fork von llama.cpp geeignet für kreative Schreibarbeit sowie Rollenspiele im Tischspiel-Stil. Es läuft als lokale Webanwendung und bietet Werkzeuge zur Erstellung langer Texte: „Weltinformationen“, Charaktergedächtnis sowie Strategien zur Aufrechterhaltung der Story-Konsistenz, die dazu dienen, das Modell über Tausende von Tokens hinweg auf Kurs zu halten. Die Zielgruppe bestehen aus Schriftstellern und Personen, die textbasierte Rollenspiele betreiben; wer hauptsächlich Chatten oder Programmieren möchte, wird mit der regulären Benutzeroberfläche unzufrieden sein.

Beispiel-Arbeitsablauf

# 1. Download the single binary
wget https://github.com/LostRuins/koboldcpp/releases/latest/download/koboldcpp-mac.zip

# 2. Run it (launches a web server)
./koboldcpp --model llama-3-8b.gguf --port 5001 --smartcontext

Vorteile und Nachteile

✅ Vorteile❌ Nachteile
Starke Werkzeuge für kreatives SchreibenNischenorientierte Benutzeroberfläche (nicht ideal für Programmieren/Chatten)
Einzeldateien-Exekutabel (keine Installation erforderlich)AGPL-Lizenz (einschränkend für kommerzielle Nutzung)

Nennung zur Anerkennung: MLX-LM

Wenn Sie als Python-Entwickler auf Apple Silicon arbeiten, schauen Sie sich bitte an MLX-LM von Apple. Es handelt sich um ein Framework, das speziell für die M-Serie-Chips optimiert ist, und auf der richtigen Hardware stellt es oft die schnellste Methode dar, ein Modell lokal auszuführen. Der Nachteil besteht darin, dass es im Vergleich zu Ollama weniger benutzerfreundlich ist: mehr Python-Code und weniger Einschränkungen.

Was man tatsächlich wählen sollte

  • Beginnen Sie mit Ollama, wenn Sie einfach nur etwas sofort zum Laufen bringen möchten.
  • Wählen Sie LM Studio, wenn Sie die Modelle lieber zunächst visuell durchsuchen möchten.
  • Wählen Sie llama.cpp, wenn Sie eine vollständige Kontrolle über den Inferenzprozess benötigen.

Wesentliche Erkenntnisse

  1. Ollama ist der schnellste Weg zu einem lokalen Modellserver für die Entwicklung.
  2. LM Studio ist am leistungsstärksten, wenn man eine Desktop-Oberfläche sowie eine einfache Erkundung von Modellen benötigt.
  3. llama.cpp bietet die direkteste Kontrolle über GGUF-Modelle sowie über Flags auf niedrigerer Ebene während der Ausführung.
  4. MLX ist der nativ unterstützte Weg für Apple Silicon, wenn man sich beim Schreiben von Python sowie bei der Verwendung der MLX-Modellformate wohlfühlt.
