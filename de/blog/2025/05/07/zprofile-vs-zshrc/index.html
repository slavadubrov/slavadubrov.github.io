    Zsh-Startdateien: ~/.zprofile gegenüber ~/.zshrc auf macOS und Linux

    Falls Ihre Terminal-Interaktion langsam wirkt oder die Umgebungsvariablen nicht dort geladen werden, wo Sie es erwarten, liegt das wahrscheinlich am Startablauf von Zsh.

    Die Trennung zwischen ~/.zprofile und ~/.zshrc es gehört zu den häufigsten Quellen für Verwirrung, wenn man auf Zsh wechselt – insbesondere unter macOS, wo sich die Standardeinstellungen von denen unter Linux unterscheiden.

    TL;DR

    ~/.zprofile Es dient der Einrichtung der Umgebung. Es wird bei jedem Anmelden ausgeführt, was unter macOS bedeutet, dass es pro Terminal-Tab einmal abläuft. Legen Sie Ihren PATH, EDITORsowie Versionsverwaltungswerkzeuge wie fnm oder pyenv da.

    ~/.zshrc Es dient der interaktiven Konfiguration. Es wird jedes Mal ausgeführt, wenn Sie eine neue Shell starten. Legen Sie dort Aliase, Prompt-Themen sowie Tastenkombinationen ab.

    Der Startvorgang der Shell

    Um zu wissen, wo Elemente platziert werden sollen, muss man zunächst verstehen, wann Dateien geladen werden. Zsh verfolgt dabei eine spezifische Hierarchie.

    Login-Shell versus interaktive Shell

    Eine Login-Shell ist die erste Shell, die nach der Authentifizierung zur Verfügung steht. Unter macOS ist jede neue Terminalleiste oder -fenster standardmäßig eine Login-Shell. Unter Linux startet das Öffnen eines Terminals in der Regel hingegen eine nicht-Login-interaktive Shell – aus diesem Grund entsteht oft Verwirrung bei der Nutzung auf verschiedenen Plattformen.

    Eine interaktive Shell ist jede Umgebung, in der man Befehle eingeben kann.

    Die Ladereihenfolge

    1. ~/.zshenv (optionell). Wird für jede Shell ausgeführt, einschließlich nicht-interaktiver Skripte. Legen Sie hier keine Ausgaben oder aufwändige Logik ab – dies kann Skripte, die Ihre Konfiguration einbinden, beschädigen. Verwenden Sie es nur für Umgebungsvariablen, die überall vorhanden sein müssen, was bei den meisten Nutzern eher selten der Fall ist.
    2. ~/.zprofile. Läuft ausschließlich für Login-Shells – Ihre Konfigurationsphase.
    3. ~/.zshrc. Laufzeit für interaktive Shells. Ihre Anpassungsphase.
    4. ~/.zlogin (optionell). Wird ganz am Ende des Starts einer Anmelde-SHELL ausgeführt.

    Was kommt wohin? ~/.zprofile: Die Umgebungsstufe

    Hier richten Sie die Elemente ein, die von allen anderen Prozessen geerbt werden – Pfade, Variablen sowie Versionierungstools für die Programmiersprache.

    Was hier hinein gehört:

    • PATH Modifikationen. – Umgebungsvariablen: EDITOR, LANG, GOPATH, JAVA_HOME.
    • Tool-Initialisierung, die die Umgebung beeinflusst: pyenv, rbenv, fnm, cargo. Diese müssen nur einmal berechnet werden. Wenn Sie sie einfügen... .zshrc, sie werden jedes Mal neu berechnet, wenn Sie eine Untershell öffnen oder ein Skript ausführen – was sowohl Zeit verschwendet als auch zu doppelten Einträgen in Ihrer PATH.
    # ~/.zprofile

# 1. Set up your PATH
# Local bin first so your tools override system ones
export PATH="$HOME/.local/bin:/opt/homebrew/bin:$PATH"

# 2. Global variables
export EDITOR="nvim"
export VISUAL="nvim"
export LANG="en_US.UTF-8"

# 3. Version managers (the heavy ones)
# Doing this here keeps shell startup fast
eval "$(fnm env --use-on-cd)"
eval "$(pyenv init -)"

    ~/.zshrc: Die interaktive Schicht

    Hier richten Sie die Shell an, in die Sie tatsächlich eingeben.

    • Aliase: alias g='git'.
    • Ihr Prompt: Starship, Powerlevel10k, Pure.
    • Vervollständigungen: compinit.
    • Tastenkombinationen: bindkey.
    • Shell-Optionen: setopt autocd, setopt histignorealldupsDiese spielen nur dann eine Rolle, wenn ein Mensch am Keyboard sitzt. Ein im Hintergrund laufender Script benötigt weder Ihre Eingabeaufforderung noch Ihre Git-Alias-Einstellungen.
    # ~/.zshrc

# 1. Prompt
autoload -Uz promptinit && promptinit
prompt pure

# 2. Aliases
alias ll='ls -lah'
alias g='git'
alias gs='git status'

# 3. Shell options
setopt autocd              # cd by typing the directory name
setopt histignorealldups   # skip duplicate history entries
setopt share_history       # share history across tabs

# 4. Completions
autoload -Uz compinit && compinit

    Warum nicht alles in einer Datei unterbringen?

    Vielleicht fragen Sie sich: Wenn … .zprofile läuft zuerst – warum legen Sie Ihre Aliase nicht dort ab und überspringen den Schritt? .zshrc vollständig?

    Es kommt auf Vererbung gegenüber Neudefinition an.

    Umgebungsvariablen werden vererbt

    Wenn Sie export EDITOR="vim" In einer Elternshell erbt jeder Kindprozess diese Einstellung: Unter-Shells, Skripte und Programme. Sie legen sie einmal fest, und sie wird sich durch den gesamten Prozessbaum fortpflanzen – genau deshalb .zprofile ist der richtige Ort für export### Aliasen und Funktionen tun das nicht

    Aliasen wie alias g='git' Shell-Funktionen sind auf die aktuelle Shell beschränkt. Sie werden nicht an Kind-Shells weitergegeben.

    Wenn Sie einen Alias definieren .zprofile, es existiert in Ihrer obersten Anmelde-SHELL. Sobald Sie eingeben zsh Um einen Unter-Shell zu starten oder ein Skript auszuführen, existiert dieser Alias nicht mehr. Wenn Aliase überall verfügbar sein sollen, müssen Sie sie in jedem neuen interaktiven Shell erneut definieren. Genau das ist der Fall. .zshrc ist für.

    Skripte benötigen keine menschlichen Eigenschaften

    Wenn Sie ein Shell-Skript ausführen./deploy.sh), es startet eine neue, nicht-interaktive Shell. Sie benötigt weder Ihre Eingabeaufforderung noch Ihre Git-Alias-Einstellungen und will auf keinen Fall warten. oh-my-zsh um das Laden abzuschließen. Die interaktive Konfiguration wird beibehalten. .zshrc Das bedeutet, dass die Skripte schnell und sauber ausgeführt werden, ohne dass persönliche Anpassungen durchsickern.

    Häufige Fallstricke

    Implementierung nvm oder pyenv in .zshrc

    Man öffnet ein neues Terminal-Fenster, und es vergehen zwei bis drei Sekunden, bevor man etwas eingeben kann. Versionstreckverwaltungstools weisen in der Regel eine aufwendige Initialisierungsphase auf. .zshrc läuft jedes Mal. Verschieben Sie sie dorthin. ~/.zprofile und die Verzögerung verschwindet.

    Ein wachsendes PATH

    Ihre $PATH endet damit, dass dieselben Verzeichnisse fünfmal aufgelistet werden. Die Ursache liegt fast immer darin. export PATH="$HOME/bin:$PATH" in Position befindlich .zshrc: bei jeder Neuladungsource ~/.zshrc) oder die Untershell fügt den Pfad erneut hinzu. Verschieben PATH Definitionen zu ~/.zprofile.

    Neuladen nach Änderungen

    Änderungen an ~/.zprofile gilt nicht für Ihre aktuelle Shell, da .zprofile wird nur beim Anmelden ausgelesen. Sie können entweder das Tab-Fenster schließen und ein neues öffnen (die einfachste Option) oder ausführen source ~/.zprofile manuell. .zshrc, einfach nur:

    source ~/.zshrc

    Eine Konfiguration, die auf macOS und Linux funktioniert

    Falls Sie zwischen verschiedenen Maschinen wechseln (macOS bei der Arbeit, Linux zu Hause – oder umgekehrt), stoßen Sie auf ein Problem. Linux-Terminals starten häufig als nicht-eingebundene Shells, was bedeutet, dass sie bestimmte Schritte überspringen. ~/.zprofile vollständig.

    Die übliche Workaround-Lösung besteht darin, die Quelle zu verwenden. .zprofile von .zshrc wenn es noch nicht geladen wurde:

    # ~/.zshrc

# On Linux/non-login shells, make sure the environment is set
if [[ -o interactive && ! -o login ]]; then
    [[ -f ~/.zprofile ]] && source ~/.zprofile
fi

# ... rest of your interactive config

    Zusammenfassung

    Datei Zweck Beispiele
    ~/.zshenv Kritische Umgebungsvariablen ZDOTDIR (Nur für fortgeschrittene Benutzer)
    ~/.zprofile Umgebungs-Einrichtung PATH, EDITOR, eval "$(pyenv init -)"
    ~/.zshrc Interaktive Konfiguration alias, prompt, bindkey, compinit

    Zwei Regeln decken den größten Teil davon ab. Variablen werden entlang des Prozessbaums weitergegeben, sodass export gehört hierher .zprofile. Aliasen und Funktionen erben nicht, daher gehören sie in .zshrc und müssen für jede neue interaktive Shell neu definiert werden. Wenn neue Tabfenster langsam erscheinen, liegt das Problem fast immer an einer übermäßig großen Last. pyenv init oder nvm.sh in Verwendung befindlich .zshrc anstelle von .zprofile.

    Wichtige Erkenntnisse

    1. Verwenden Sie .zprofile für die Einrichtung einer Login-Shell, wie z. B. die Initialisierung von PATH im macOS Terminal.
    2. Verwenden Sie .zshrc für ein interaktives Shell-Verhalten wie Aliase, Prompt-Konfiguration, Autocompletion und Shell-Optionen.
    3. Duplizieren Sie die PATH-Einträge nicht in beiden Dateien; eine Duplikation erschwert das Debuggen der Startreihenfolge erheblich.
    4. Wenn Sie unsicher sind, veranlassen Sie den Initialisierungsprozess der Umgebung. .zprofile sowie interaktive Bequemlichkeiten in .zshrc.
