    pyproject.toml Leitfaden: Python-Paketierung, Abhängigkeiten und Konfiguration von Tools

    Falls Sie bereits einmal ein Python-Projekt geöffnet haben und versucht haben herauszufinden, wo sich die Abhängigkeiten, die Kompilierungs-Einstellungen sowie die Tool-Konfigurationen tatsächlich befinden, kennen Sie das Problem nur allzu gut. setup.py, setup.cfg, requirements.txt, MANIFEST.insowie einige Dotfiles für jeden Linter und Formatter – wobei alle von unterschiedlichen Orten gelesen werden.

    pyproject.toml erfasst den größten Teil davon in einer einzigen Datei.

    pyproject.toml ist ungefähr der package.json für Python. Eine Datei enthält die Metadaten Ihres Projekts, die Abhängigkeiten sowie die Einstellungen der Tools. Egal, ob Sie .venv, pyenv, oder uv, alles hier zu platzieren erleichtert die Einrichtung und die Zusammenarbeit.

    Was ist pyproject.toml? PEP 518 (2016) führte es ein. [build-system] Tabelle, damit Build-Tools ihre Anforderungen auf standardisierte Weise deklarieren können. PEP 621 (2020) fügte hinzu [project] Tabelle für die Metadaten des Kernpakets: Name, Version, Abhängigkeiten und ähnliches.

    Heutzutage liest die überwiegende Mehrheit der Python-Tools (Black, isort, pytest, Ruff, mypy) ihre Konfiguration aus [tool.*] Abschnitte in dieser Datei.

    Warum das wichtig ist

    Weniger Dateien, die verfolgt werden müssen

    Ein typisches Legacy-Projekt musste mehrere Aufgaben gleichzeitig bewältigen. setup.py, setup.cfg, requirements.txt, MANIFEST.in, sowie eine Gruppe von Dotfiles.flake8, .coveragerc, und ähnliche). Die meisten davon kollabieren an einem Ort zusammen.

    Backend-unabhängige Builds

    Wenn Sie es ausführen pip install ., pip liest ein pyproject.toml und installiert alle Build-Tools, die Ihr Projekt benötigt: setuptools, flit, hatchling – Sie können selbst wählen. Sie sind nicht länger an setuptools gebunden.

    Eine einzige Stelle für die Tool-Konfiguration

    Lint-Tools, Formatierer, Testlaufwerke sowie Typüberprüfer wissen alle, dass sie hier nachsehen müssen. Ihre IDE, Ihr CI-Pipeline sowie Ihre Kollegen lesen aus derselben Datei.

    Anatomie eines pyproject.toml

    Ein typisches Datei enthält drei Hauptabschnitte:

    # 1. Build system - tells pip/build how to package your project
[build-system]
requires = ["hatchling"]
build-backend = "hatchling.build"

# 2. Project metadata and dependencies
[project]
name = "awesome-app"
version = "0.1.0"
description = "Short demo of pyproject.toml"
readme = "README.md"
requires-python = ">=3.12"
dependencies = [
  "fastapi>=0.111",
  "uvicorn[standard]>=0.30",
]

# Expose CLI commands
[project.scripts]
awesome-cli = "awesome_app.cli:main"

# Optional dependencies (e.g., for development)
[project.optional-dependencies]
dev = ["pytest", "ruff", "mypy"]

# 3. Tool configuration
[tool.ruff]
line-length = 100
target-version = "py312"

[tool.pytest.ini_options]
addopts = "-ra -q"
testpaths = ["tests"]

    Aufschlüsselung

    [build-system]

    Erforderlich, wenn Sie Ihr Projekt verpacken oder distribuieren möchten. Gibt pip sowie Build-Tools (wie python -m build) Welchen Backend-Service soll verwendet werden. [project]

    Ihre Paketmetadaten. Hier befinden sich die Abhängigkeiten anstelle von requirements.txt.

    • dependencies: Die Laufzeitanforderungen für Ihr Paket. optional-dependencies: Gruppen zusätzlicher Abhängigkeitendev, test, docs).
    • scripts: Erstellt ausführbare Befehle. Im obigen Beispiel liefert die Installation des Pakets Ihnen einen awesome-cli Kommando, das ausgeführt wird main Funktion in awesome_app/cli.py.

    [tool.*]

    Konfiguration für jedes Tool, das sie unterstützt. Jedes Tool erhält seinen eigenen Namespace: [tool.pytest.ini_options], [tool.mypy], [tool.ruff], und der Rest.

    Ersetzt es das? requirements.txt?

    Für moderne Workflows, ja. Werkzeuge wie Poesie, PDM, Hatchung, und UV die Abhängigkeiten direkt im [project] Sektion erstellen und Lockfiles zur Wiederherstellbarkeit generieren.

    Sie möchten dies vermutlich weiterhin tun. requirements.txt if:

    • Sie arbeiten mit einem veralteten Deployment-System, das diese Voraussetzung erfordert.
    • Sie verfügen über ein CI-Skript, das noch nicht aktualisiert wurde.

    Die meisten modernen Tools können eine solche Datei aus Ihren Daten exportieren. pyproject.toml bei Bedarf:

    uv export > requirements.txt

    Auswahl eines Build-Backends

    Der Build-Backend ist eine der verwirrendsten Entscheidungen. Hier ist ein kurzer Vergleich:

    Backend Am besten geeignet für Vorteile
    Küken Moderne Standardversion Schnell, erweiterbar und unterstützt Plugins Jüngere Versionen mit geringerer Unterstützung für veraltete Technologien
    Flitzen Einfache Pakete Extrem einfach – keine Konfiguration erforderlich. Nicht für komplexe Builds mit C-Erweiterungen geeignet.
    Setuptools veraltet und komplex Unterstützt alles (C-Erweiterungen usw.). Langsamer, mehr Konfiguration nötig
    Poesie Poetry-Nutzer In das Ökosystem von Poetry integriert In den Poetry-Werkfluss eingebunden

    Für neue reine Python-Projekte ist Hatchling eine angemessene Standardwahl. Es handelt sich dabei um uv init schreibt für Sie.

    Migration eines bestehenden Projekts

    Falls Sie ein veraltetes Python-Projekt haben, erfahren Sie hier, wie Sie es modernisieren können:

    1. Fügen Sie hinzu [build-system]. Falls Sie sich nicht sicher sind, welchen Backend-Service Sie verwenden sollen, beginnen Sie mit requires = ["setuptools>=61", "wheel"]2. Verschieben Sie die Metadaten nach [project]. Übertragen Sie Namen, Versionen und Abhängigkeiten von setup.py oder setup.cfg3. Konvertieren Sie die Entwicklungszusammenhänge. Fügen Sie sie hinzu. [project.optional-dependencies].dev.
    2. Konfigurieren Sie die Tools. Fügen Sie hinzu [tool.*] Abschnitte für Black, pytest, mypy usw.
    3. Entscheiden, wie damit umgegangen werden soll. requirements.txt. Entweder entfernen Sie es oder erzeugen Sie es aus Ihrer Lockdatei für Legacy-Systeme, die es benötigen.

    Nach der Migration können Sie es in der Regel löschen. setup.py, setup.cfgsowie die meisten dieser Konfigurations-Dateien mit Punkt-Endung.

    Einige nützliche Funktionen

    Eingangspunkte der CLI

    Anstelle des alten console_scripts Blockierung setup.py, verwenden [project.scripts]:

    [project.scripts]
my-tool = "my_package.main:run"

    Wenn jemand Ihr Paket installiert, kann er Folgendes eingeben: my-tool in ihrer Terminal-Schnittstelle.

    Arbeitsräume (Monorepos)

    Werkzeuge wie uv und hatch Unterstützung für Workspaces, die es ermöglichen, mehrere Pakete in einem einzigen Repo zu verwalten:

    [tool.uv.workspace]
members = ["packages/*"]

    Man kann mehrere miteinander verknüpfte Pakete entwickeln und alle in einer einzigen virtuellen Umgebung zum Testen installieren.

    Typische Arbeitsabläufe mit uv

    Ein neues Projekt starten

    uv init my_app          # creates folder with pyproject.toml and .venv
cd my_app
uv add requests fastapi # adds to [project.dependencies] and installs
uv run pytest           # runs tests in the venv

    Ausführung von Skripten

    Man kann entweder Skripte definieren oder pyproject.toml (unter Verwendung eines Task-Runners wie poe oder hatch) oder rufen Sie einfach an uv run:

    uv run python main.py

    Praktische Tipps

    1. Pinnen Sie keine exakten Versionen in Bibliotheken. Verwenden Sie stattdessen Bereiche wie requests>=2.30 damit Ihre Bibliothek nicht mit anderen im Umfeld eines Benutzers installierten Komponenten in Konflikt gerät.
    2. Pinnen Sie die Versionen in den Anwendungen. Verwenden Sie dazu eine Lockdatei.uv.lock, poetry.lock) für reproduzierbare Builds.
    3. Gruppieren Sie die Entwicklungsabhängigkeiten. Behalten Sie die Abhängigkeiten für Tests, Linting und Dokumentation in separaten, optionalen Gruppen bei.dev, test, docs).
    4. Gießen Sie nicht alle Optionen aus. pyproject.toml. Befolgen Sie die projektweiten Standardwerte und überlassen Sie die verbleibenden Anpassungen den konfigurationsbezogenen Dateien der jeweiligen Tools, falls es unübersichtlich wird.

    Der eigentliche Vorteil liegt darin, dass pyproject.toml Es geht hauptsächlich darum, kleine, tägliche Hindernisse zu beseitigen. Man muss nicht mehr nach dem Suchen, welches Projektfile für die Kompilierung verantwortlich ist. Die Konfiguration des Linters befindet sich direkt neben den Abhängigkeiten. Pip und die IDE sind schließlich einig darüber, was installiert ist. Wählen Sie einen Build-Backend und platzieren Sie Ihre Abhängigkeiten darin. [project]und ermöglichen Sie es Ihren Tools, von einer einzigen Stelle aus zu lesen. Wenn Sie von vorne beginnen, uv init bringt Sie mit einem einzigen Befehl bereits weit voran.

    Wichtige Erkenntnisse

    1. pyproject.toml ist die zentrale Konfigurationsdatei für moderne Python-Projekte.
    2. PEP 518 definiert die Anforderungen an das Build-System; PEP 621 definiert die Projektmetadaten.
    3. Bewahren Sie die Konfiguration der Tools in pyproject.toml Wenn das Tool dies unterstützt – erzwingen Sie jedoch keine nicht verwandten Konfigurationen in eine einzige Datei.
    4. Verschlüsselung von Dateien und Build-Metadaten beheben unterschiedliche Probleme: reproduzierbare Installationen im Vergleich zur Paketidentität.
