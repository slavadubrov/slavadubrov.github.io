Was zuerst implementiert werden sollte

Löschen Sie zunächst die entsprechenden Berechtigungen. Wenn der Agent keine Schreibzugriffe auf GitHub benötigt, erteilen Sie ihm keinen Schreibtoken. Falls er lediglich Informationen zur Kalenderverfügbarkeit benötigt, gewähren Sie keinen vollständigen Zugriff auf den Posteingang. Das Prinzip des geringsten Privilegs ist effektiver als strenge Aufforderungen.

Fügen Sie anschließend eine Policy-Grenze vor jeden Tool-Aufruf ein. Überprüfen Sie den genauen Tool-Namen, die Argumente, das Zielressourcens, den Benutzer, die Umgebung sowie die Nebenwirkungen. Eine Benutzeranfrage mag harmlos erscheinen, während der generierte Shell-Befehl dies nicht unbedingt ist.

Fügen Sie Sandboxes für die Ausführung von Code, die Automatisierung von Browsern, den Zugriff auf Dateien sowie die Verarbeitung unzuverlässiger Dokumente hinzu. Eine Sandbox macht eine Aktion nicht automatisch korrekt, verringert aber den Schaden, der durch fehlerhafte Ergebnisse eines kompromittierten Tools oder ein fehlerhaftes Modell entstehen kann.

Verwenden Sie die Zustimmung eines Menschen für irreversible Aktionen. Genehmigen Sie nicht jeden einzelnen Schritt. Genehmigen Sie stattdessen nur Grenzbereiche: Produktionseinrichtungen, Datenlöschungen, E-Mails, Geldtransfers, Berechtigungsänderungen sowie regulatorisch relevanten Entscheidungen.

MCP-spezifische Risiken

MCP ist nützlich, da es den Zugriff auf Tools standardisiert. Gleichzeitig birgt es Risiken, da Beschreibungen der Tools, Schemata, Serveridentitäten, OAuth-Scope-Berechtigungen sowie die Ausgaben der Tools allesamt Teil des Entscheidungskontexts des Modells werden.

Für MCP würde ich bei Code Reviews folgende Regeln beibehalten:

  • Beschreibungen und Schemata der Tools vor der Freigabe überprüfen
  • bevorzugt enge Berechtigungen pro Server verwenden
  • unzuverlässige MCP-Server von sensiblen Tools trennen
  • nach der Installation Änderungen in den Tooldefinitionen überwachen
  • Toolausgaben als unzuverlässige Eingaben behandeln
  • jeden Server, jedes Tool, jeden Argument und jedes Ergebnis protokollieren

Schutzmaßnahmen reichen nicht aus

Schutzmaßnahmen können Eingaben und Ausgaben validieren. Sie lösen jedoch Probleme wie das Prinzip des geringsten Privilegs, begrenzte Berechtigungsbereiche, Sandbox-Umgebungen, Manipulationen von Tools oder Freigabepolitiken nicht. Behalten Sie sie zwar bei, platzieren Sie sie aber nach der Konzeption der Funktionalitäten und vor der für den Benutzer sichtbaren Ausgabe.
