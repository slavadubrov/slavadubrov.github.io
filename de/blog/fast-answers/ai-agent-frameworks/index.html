Wie man wählt

Beginnen wir mit dem Kontrollfluss.

Wenn der Agent klare Zustände, Wiederholungsversuche, Freigabeprozesse, Kontrollpunkte sowie die Möglichkeit zu fortgesetzten Ausführungen bietet, sollte LangGraph verwendet werden. Zunächst muss mehr Struktur definiert werden – doch genau diese Struktur bildet das System selbst. Dies ist die richtige Standardlösung, wenn die Ausführung länger andauern kann als eine HTTP-Anfrage oder wenn erklärt werden muss, warum der Agent bestimmte Aktionen ausgeführt hat.

Anschließend sollte die Passgenauigkeit zur Plattform geprüft werden.

Falls Ihr Technologiestack bereits OpenAI-Modelle verwendet und Sie eine kompakte Python-API wünschen, eignet sich das OpenAI Agents SDK. Hier erhalten Sie Agenten, Tools, Sicherheitsmechanismen, Sitzungen sowie Trace-Funktionen an einem Ort. Es versucht nicht, ein allgemeiner Graphen-Engine zu sein – was gerade zu seinem Reiz beiträgt.

Daraufhin sollte die Form der Daten betrachtet werden.

Wenn Ihr Agent hauptsächlich mit Dokumenten, Indizes, Such- und Abfragesystemen arbeitet, sollten Sie mit LlamaIndex beginnen. In den meisten Fällen ist dies besser, als zunächst eine eigene Suchschicht zu entwickeln und später ein Agentenframework darauf anzupassen. Die meisten Dokument-Agenten scheitern daran, dass Such- und Bewertungsprozesse unzureichend spezifiziert wurden – nicht weil der Algorithmus zu einfach strukturiert ist.

Verwenden Sie Rollen nur dann, wenn sie den tatsächlichen Arbeitsablauf widerspiegeln.

CrewAI ist nützlich, wenn im echten Prozess tatsächlich Rollen wie Forscher, Analyst, Reviewer, Autor oder Operator vorhanden sind. Er ist hingegen weniger sinnvoll, wenn man Rollen nur deshalb erfindet, um eine Multi-Agenten-Abstraktion nutzen zu können. Namen dienen nicht der Zustandsverwaltung.

Wann kein Agent-Framework verwendet werden sollte

Man sollte kein Agentenframework verwenden, wenn ein typisierter Workflow, ein Suchendpunkt oder ein Regelwerk das Problem bereits lösen kann. Agenten sind nützlich, wenn das System auf der Grundlage von Zwischenergebnissen entscheiden muss, welcher nächste Schritt ausgeführt werden soll. Für feste ETL-Prozesse, deterministische Freigabeverfahren, Abrechnungsflüsse oder alle anderen Szenarien, bei denen alle möglichen Abläufe im Voraus bekannt sind, eignen sie sich hingegen kaum.

Bauen Sie kein Multi-Agenten-System, bevor Sie bereits ein funktionierendes einzelnes Agent haben. Die meisten frühen Konzepte für Multi-Agentensysteme sind im Grunde nichts anderes als verteilte Prompt-Engineering-Methoden, die zusätzliche Latenz verursachen.

Verwechseln Sie die Observabilität eines Frameworks nicht mit der Produktbewertung. Traces zeigen Ihnen, was geschehen ist. Bewertungen hingegen geben an, ob es gut war.

Mein Standardpfad

  1. Erstellen Sie die erste Version als einzelnen Agenten mit eingeschränkten Tools.
  2. Fügen Sie Spuren sowie ein kleines Regressionssatzdatensatz hinzu, bevor Sie Speicher hinzufügen.
  3. Wechseln Sie zu LangGraph, sobald Zustandsübergänge integraler Bestandteil des Produkts werden.
  4. Verwenden Sie das OpenAI Agents SDK, wenn das Produkt native OpenAI-Funktionen aufweist und der Loop kompakt bleiben soll.
  5. Fügen Sie rollenbasierte Agenten nur dann hinzu, wenn die Verantwortlichkeiten tatsächlich voneinander getrennt werden können.
