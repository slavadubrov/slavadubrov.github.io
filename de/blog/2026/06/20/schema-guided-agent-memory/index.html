Lange laufende Agents greifen häufig auf veraltete Fakten zu, da herkömmliche Semantic Memory keine Regel besitzen, um festzulegen, welcher Wert aktuell ist.

Extrahieren Sie mögliche Speicherinhalte über eine Grenze mit strukturierter Ausgabe, speichern Sie Datensätze mit Tenant-Bereich, Gültigkeitsfenstern, Überlagerungsmöglichkeiten, Herkunftsinformationen sowie Schema-Versionierung und holen Sie anschließend den kleinsten aktuellen Datensatz über den Leseweg ab. Lange laufende Agents müssen Präferenzen, Status der Aufgaben, Kundendaten, getroffene Tools Entscheidungen, Konformitätsnotizen sowie frühere Fehler speichern. Dies funktioniert solange, bis sich eine der gespeicherten Informationen ändert.

Der Agent weist derzeit zwei Fristen für den Passausweis, zwei bevorzugte Formate oder zwei Entscheidungen bezüglich des Projekts auf. Semantic Search kann beide Versionen abrufen. Getrennte Extraktion von Model unabhängig vom Speichereigentum

Die Speicherung erfolgt über drei Schichten. Structured Output (SO) stellt die Form des Kandidatenobjekts sicher. Schema-gesteuerte Reasoning (SGR) kodiert die Schritte sowie die Reihenfolge, die Model einhalten muss, um zu diesem Kandidaten zu gelangen. Schema-gesteuerte Agent Memory (SGAM) verwaltet den Kandidaten als dauerhaften Zustand nach dem Aufruf von Model.

Ein Urteil, eine Route oder ein Plan verfallen in der Regel mit der aktuellen Anfrage. Ein späterer Ausführungsvorgang kann den entsprechenden Speicherinhalt bereits Tage später abrufen oder ihn zur Auswahl eines Tool Call nutzen. Diese längere Lebensdauer erfordert Speicherrichtlinien, die SGR nicht bereitstellt.

SGR beschränkt einen Aufruf von Model, indem es die Reasoning-Topologie dieses Aufrufs definiert. Bei einer Speicherung in das Gedächtnis kann das Schema Quellenbelege, normalisierte Subjekte sowie Attribute erfordern, außerdem einen Vergleich mit dem aktuellen Zustand – erst danach wird der vorgeschlagene Update durchgeführt. Pydantic oder JSON Schema beschreiben diesen Ablauf. Provider-eigene Structured Output-Mechanismen oder eine gesteuerte Dekodierung mittels Runtime wie XGrammar verhindern, dass der Model Felder überspringt oder eine andere Struktur zurückgibt.

Das Schema kann keine korrekte Schlussfolgerung garantieren. Es macht den erforderlichen Entscheidungsweg explizit und überprüfbar – einschließlich der Beweise sowie des Vergleichs, die zum Kandidaten geführt haben.

SGAM entscheidet darüber, was nach dem Bestehen dieses Objekts geschieht. Soll es gespeichert werden? Überlagert es eine ältere Eintragung? Welcher Mieter kann es einsehen? Ist es aktuell oder historisch? Welches Quellenereignis stützt es?

Schreib- und Lesepfade haben unterschiedliche Aufgaben

SGAM verfügt über einen Schreibweg und einen Leseweg. Nur der Schreibweg ändert den gespeicherten Zustand. Der Leseweg wählt die für die aktuelle Anfrage relevanten Datensätze aus.

Der Ingestion-Flow ist der Schreibpfad:

  1. Erfassen Sie eine unverarbeitete Episode aus Nachrichten, Tool Results, oder Geschäftsereignissen.
  2. Extrahieren Sie die eingegebenen Kandidaten mithilfe von Structured Output.
  3. Überprüfen Sie das Schema und lehnen Sie fehlerhafte Eingaben ab.
  4. Klären Sie Konflikte auf, schließen Sie veraltete Datensätze und bewahren Sie die Herkunftsnachweise auf.
  5. Speichern Sie den Eintrag im SGAM-Speicher.

Der Anfragenfluss ist der Leseweg:

  1. Beginnen Sie mit der Anfrage des Benutzers.
  2. Entscheiden Sie, ob für die Anfrage der aktuelle Zustand oder ein Zustand zu einem bestimmten Zeitpunkt erforderlich ist.
  3. Führen Sie eine Filterung nach Tenant, Speichertyp, Thema, Attribut sowie Gültigkeitsfenster durch.
  4. Fügen Sie eine Vektor- oder Graphenerweiterung nur dann hinzu, wenn eine Suche nach dem exakten Zustand nicht ausreicht.
  5. Erstellen Sie den kleinstmöglichen zitierten Kontext für den Model.

    row = db.execute(
        """

        from memory_facts
        where tenant_id = ?
          and subject = ?
          and attribute = ?
          and valid_to is null
        order by valid_from desc

        (fact.tenant_id, fact.subject, fact.attribute),

Hot-Pfad- und Hintergrundschreibvorgänge

Eine sofortige Extraktion lohnt sich, wenn die nächste Interaktion von dem neuen Speicherinhalt abhängt. Sollte der Benutzer angeben: „Merken Sie sich, dass ich kurze Antworten bevorzuge“, muss das System nicht erst auf eine tägliche Verarbeitungsschleife warten, um anders zu reagieren.

In den meisten Fällen ist es nicht notwendig, sofort etwas zu speichern. Man sollte die Rohdaten der Episode zusammen mit dem Tenant, Session, sowie den Metadaten des Tools aufbewahren und anschließend einen Hintergrundprozess damit beauftragen, später passende Ergebnisse auszuwählen. Durch die auf Wiederholungen basierende Konsolidierung puffert der Prozess schwache Signale und führt ein Fact erst dann ein, wenn ähnliche Belege erneut auftauchen oder der Benutzer dies bestätigt. Dadurch entsteht eine Verzögerung bei der Aktualisierung der Daten. Dies ist für Szenarien wie „Der Benutzer bittet häufig um CSV-Exporte“ akzeptabel, doch riskant, wenn „der Kunde die Lieferadresse geändert hat“.

Stellen Sie sicher, dass der Leseweg deterministisch ist. Setzen Sie zunächst die Scope- und Gültigkeitsregeln für den jeweiligen Tenant durch und wenden Sie den verschwommenen Retrieval erst dann an, wenn er nützliche Kontextinformationen liefern kann.

  1. Filtern nach Mieter, Speichertyp und Gültigkeitszeitraum.
  2. Abrufen des exakten, strukturierten Zustands vor den semantischen Nachbarn.
  3. Für unterstützende Belege, verwandte Entitäten und Beispiele Vektor- oder Graphenerweiterungen verwenden – nicht jedoch als Autorität für aktuelle Fakten.
  4. Den kleinstmöglichen zitierten Kontext zusammenstellen, der die Frage beantworten kann.

Betrachten Sie die Schema-Migration als Produktänderung, da sie beeinflusst, was der Agent abrufen, zitieren oder löschen kann. Zudem kann sie bestimmen, welche historischen Daten als aktuell gelten. Planen Sie die Migrationsskripte, die Rückfüllvorgänge, die Phasen des dualen Lesens sowie das Löschverhalten bereits im Rahmen derselben Version.

Wann SGAM die zusätzliche Komplexität rechtfertigt

Verwenden Sie SGAM, wenn sich Fakten im Laufe der Zeit ändern können:

  • Benutzerpräferenzen, die aktualisiert oder widerrufen werden können
  • Kundendaten oder Kontoinformationen mit Prüfanforderungen
  • Zustand von Aufgaben bei langlaufenden Assistenten
  • Speichermöglichkeiten für Coding-Agent-Projekte
  • multi-agent-gemeinsamer Zustand
  • Konformitätshinweise, bei denen die Herkunft entscheidend ist
  • Zeitbezogene Fragen wie „Was glaubten wir vor der Migration?“

SGAM ist übertrieben, wenn der Speicher nur vorübergehend zur Verfügung steht, die Exploration stattfindet oder die Neuberechnung günstig ist. Wenn das Agent lediglich einige Kontinuitätsschleifen benötigt, reichen ein Checkpoint sowie eine gekürzte Nachrichtenhistorie aus. Für die Qualitätskontrolle statischer Dokumente könnte allein RAG ausreichen. Falls das Anwendungsfeld so instabil ist, dass sich das Schema täglich ändert, wird eine typisierte Speicherlösung das Team nur verlangsamen.

Einschränkungen

SGAM ist meine Bezeichnung für ein Muster und nicht für ein Standard. Bestehende Projekte gehen bei der Aufgabenstellung unterschiedlich vor. LangGraph-Memory und LangMem Beschreiben Sie kurzfristige und langfristige Speicherstrukturen, Profile, Sammlungen, Schreibvorgänge auf Hot-Paths sowie Hintergrund-Prozesse zur Verwaltung von Speichermitteln. Zep Graphiti verwendet den Begriff temporales Kontextgraphen. Letta erhält editierbare Speicherblöcke bei, während Mem0 bietet eine verwaltete Speicherschicht. Microsoft GraphRAG, Die Property Graphs von LlamaIndex sowie die von Cognee werden die entsprechenden Teile des Problems als Wissensgraphen dargestellt.

Ein Benutzerprofil, ein Episodenprotokoll, ein Dokumentengraph sowie ein Agent-editierbarer Speicherblock lösen unterschiedliche Retrieval- und Aktualisierungsprobleme. Für den dauerhaften Speicher, der den aktuellen Zustand der Anwendung darstellt und daher Strukturen, Gültigkeitsprüfungen, Herkunftsinformationen, Konfliktbewältigung, Aufbewahrungsfristen sowie Migrationsmechanismen erfordert, reserviere ich SGAM.

Typisierte Speicherstrukturen können dennoch Fehler aufweisen. Ein Schema erleichtert die Überprüfung fehlerhafter Schreibvorgänge – es macht diese jedoch nicht zuverlässig. Es bleibt weiterhin notwendig, der Quelle Vertrauen zu schenken, sensible Daten durch Benutzerbestätigung zu sichern, Konfliktlösungsregeln anzuwenden, Daten zu löschen sowie ständig zu überwachen.

Die Migration von Schemata ist aufwendig. Sobald Daten im Speicher als Zustand gespeichert werden, sind Sie für das Versionieren, die Rückfüllung veralteter Datensätze sowie das Verhalten bei Löschvorgängen verantwortlich. Vermeiden Sie diese Aufwände – sonst überdauern alte Datensätze die Semantik oder die Speicherrichtlinien, nach denen sie ursprünglich erstellt wurden.
