Bewertung von AI Agents in der Produktion: Vom Traces bis zu Test-Suiten

Ein Chatbot liefert Ihnen eine einzige Antwort zur Bewertung. Ein Agent hingegen stellt Ihnen einen gesamten Entscheidungsbaum zur Verfügung: Pläne, Tool Calls, Wiederholungsversuche sowie den Zeitpunkt, an dem entschieden wurde, dass die Aufgabe abgeschlossen ist.

Differenzen dieser Art erfordern eine andere Bewertungsmethode. Eine Endlösung kann zwar korrekt erscheinen, selbst wenn der Agent ein erforderliches Werkzeug übersprungen hat, einen Aufruf 17 Mal wiederholt hat, ein Ergebnis falsch interpretiert hat oder einen Pfad verfolgt hat, der von den Produktionsrichtlinien ausgeschlossen ist. Die rein auf das Ergebnis ausgerichtete Bewertung verschleiert solche Fehler.

TL;DR: Agent Evals erfordern drei Ebenen: Ergebnismetriken, Trajektoriemetriken sowie Komponentenmetriken. Die Implementierung basiert auf folgendem Zyklus: Trace -> Benennung -> Clustering -> Doppelte Entfernung -> versionierte Dataset -> CI-Schleuse -> kontinuierliche Überwachung. Für die Reihenfolge der Tools, Argumente, Schleifen sowie Invarianten sollten deterministische Prüfungen verwendet werden. LLM Judges kommen nur dann zum Einsatz, wenn die Prüfung von Interpretationen abhängt; solche Judges sollten mithilfe von schema-gestütztem Reasoning (SGR)) geformt und vor dem Vertrauen darauf kalibriert werden, indem sie mit menschlichen Labels abgeglichen werden.

Warum Agent und Evals unterschiedlich sind

Herkömmliche LLM Evals bewerten in der Regel nur ein Eingab-Ausgabepaar: Relevanz, Treue, Korrektheit, Sicherheit sowie gegebenenfalls Stil. Agents fügen hingegen Planning, Tool Calls, Wiederholungsversuche sowie Abschlussprüfungen hinzu, wodurch jeder Schritt zu einem potenziellen Versagenspunkt wird.

Erhalten Sie eine Rückerstattung für Agent. Die Transkription kann trotz fehlerhafter Trace-Werte dennoch ein gutes Ergebnis liefern:

lookup_order -> issue_refund -> final_answer

Die Ausgabe Eval ist erfolgreich. Eine Bahn Eval sollte hingegen fehlschlagen, da verify_identity wurde noch nie zuvor ausgeführt. issue_refund. Bei der Nutzung von Werkzeugen durch Agents dienen die reinen Antwort-Evals-Tests lediglich als Smoke Tests: Sie erkennen nur totale Funktionsstörungen, übersehen jedoch alle anderen Probleme.

Es gibt ein weiteres Problem: Fehler kumulieren sich. Wenn ein 20-stufiger Workflow bei jedem Schritt zu 95 % zuverlässig ist, liegt seine Gesamterfolgsrate dennoch bei etwa 36 %:

0,9520≈0,360{,}95^{20} \approx 0{,}360,9520≈0,36

Daher kann der Agent bei isolierten Tests zwar ein solides Ergebnis liefern, bei den meisten vollständigen Ausführungen jedoch dennoch fehlschlagen. Der Fehler tritt in der Regel irgendwo in der Mitte auf, und seine Identifizierung erfordert eine Sicht auf die Komponentenebene – nicht lediglich einen erneuten Blick auf das Ergebnis.

Metriken nach Ausfallmodus auswählen

Die richtige Metrik hängt vom Ausfallmodus ab und nicht von Framework. Die sinnvolle Aufteilung umfasst drei Bereiche:

  1. Ergebnis Evals gibt an, ob die Aufgabe erfolgreich war.
  2. Bewegungsablauf Evals zeigt an, ob der Pfad gültig, effizient und mit den Richtlinien vereinbar war.
  3. Komponente Evals identifiziert, welche Werkzeug, Retriever, Unter-Agent-Komponente oder Entscheidungsschritt fehlerhaft war.

Jeder Scope kann vor der Veröffentlichung offline an festgelegten, wiederaufspielbaren Fallbeispielen ausgeführt werden oder online nach der Erstellung der Antwort an ausgewählten Produktionsszenarien Traces. Der untenstehende Abschnitt Guardrails erläutert diese Trennung im Detail. Für den offline Betrieb Evals können Goldens erforderlich sein. Beim online Betrieb Evals sollten Invarianten, Verteilungen sowie asynchrone Überprüfungen bevorzugt werden, die außerhalb des Request-Pfades liegen.

Richtigkeit von Tool-Aufrufen

Die Tool-Korrektheit vergleicht die aufgerufenen Tools mit den erwarteten Tools. Wählen Sie die Strenge bewusst aus:

  • Exakte Übereinstimmung: Die Sequenz muss exakt übereinstimmen. Verwenden Sie dies, wenn die Reihenfolge von Bedeutung ist, zum Beispiel lookup_order -> verify_identity -> issue_refund.
  • In-Ordnung-Match: Die erforderlichen Tools müssen in der korrekten relativen Reihenfolge auftreten, wobei zusätzliche, unbedenkliche Aufrufe zulässig sind.
  • Jeglicher-Ordnung-Match: Die erforderlichen Tools müssen vorhanden sein, doch ihre Reihenfolge darf variieren.

Ein kleiner lokaler Scorer reicht bereits aus, um mit der Arbeit zu beginnen:

def tool_correctness(called: list[str], expected: list[str], mode: str = "in_order") -> float:
    if not expected:
        return 1.0
    if mode == "exact":
        return float(called == expected)
    if mode == "any_order":
        return len(set(expected) & set(called)) / len(set(expected))

    rows = [[0] * (len(expected) + 1) for _ in range(len(called) + 1)]
    for i, tool in enumerate(called):
        for j, wanted in enumerate(expected):
            if tool == wanted:
                rows[i + 1][j + 1] = rows[i][j] + 1
            else:
                rows[i + 1][j + 1] = max(rows[i][j + 1], rows[i + 1][j])
    return rows[-1][-1] / len(expected)


called = ["lookup_order", "check_refund_policy", "issue_refund"]
expected = ["lookup_order", "verify_identity", "issue_refund"]

print(round(tool_correctness(called, expected, "exact"), 3))     # 0.0
print(round(tool_correctness(called, expected, "in_order"), 3))  # 0.667

Der in_order Der Wert entspricht dem Recall der längsten gemeinsamen Teilfolge: Es wird angegeben, welcher Anteil der erforderlichen Sequenz im richtigen Aufbau erhalten geblieben ist. Beachten Sie dabei, was dabei ausgeblendet wird. Unwichtige Aufrufe senken diesen Wert nicht, sodass ein Agent hier einen Wert von 1,0 erreichen kann, obwohl er doppelt so viele Aufrufe vornimmt wie nötig. Wenn zusätzliche Aufrufe Kosten verursachen oder den Zustand verändern, sollten Sie den Precision-Wert (Anzahl der korrekt abgewickelten Aufrufe im Verhältnis zur Gesamtanzahl der Aufrufe) gleichzeitig betrachten und beide Werte miteinander abgleichen. Der Recall zeigt die fehlenden Schritte auf, während der Precision die Fehlentscheidungen erkennbar macht.

Der Messwert für die Richtigkeit der Tools von DeepEval stellt dieselben Einstellmöglichkeiten über should_consider_ordering und should_exact_match.

Richtigkeit der Argumente

Die Verwendung des richtigen Tools mit falschen Argumenten ist oft schlimmer als die Verwendung des falschen Tools, da der Trace dann normal erscheint.

Bei einfachen Fällen wird JSON schema sowie die genauen Werte überprüft. Bei semantischen Fällen werden die erwarteten Argumente gespeichert und die Abweichungen bewertet:

{
    "trace_id": "tr_2417",
    "input": "Reschedule order A-100 for next Friday.",
    "expected_tools": ["lookup_order", "reschedule_delivery"],
    "expected_arguments": {
        "reschedule_delivery": {
            "order_id": "A-100",
            "date": "2026-06-19"
        }
    }
}

Ein Metrikwert, der nur den Tool-Namen berücksichtigt, kann dies nicht erfassen. 2026-06-17 wo die Richtlinie dies erfordert 2026-06-19. Der Dataset muss ebenfalls Argumente speichern.

Der mit Dataset verbundene Wert ist parameter-match: der Anteil an erwarteten (tool, key, value) Dreifacht die Anzahl der korrekten Antworten von Agent.

def argument_correctness(called_args: dict, expected_args: dict) -> float:
    total = matched = 0
    for tool, params in expected_args.items():
        for key, want in params.items():
            total += 1
            if called_args.get(tool, {}).get(key) == want:
                matched += 1
    return matched / total if total else 1.0

Genaue Gleichheit ist geeignet für IDs, Enums und Datumswerte. Für Freitexte und Fließkommazahlen hingegen ist sie unangemessen, da == Markiert eine richtige Antwort als falsch. Bewerten Sie diese Felder nach ihren eigenen Kriterien: eine normalisierte Zeichenkettenübereinstimmung, eine Datumsanalyse sowie eine numerische Toleranz. Die Metrik bleibt unverändert; nur der Vergleichsmechanismus pro Feld wechselt.

Effizienz, Schleifen und Sackgassen

Ein Agent, der die Aufgabe nach fünf überflüssigen Tool Calls abgeschlossen hat, weist weiterhin ein Planning-Problem auf und ist außerdem aufwändiger in der Ausführung.

Günstige Signale, mit denen Sie beginnen sollten:

  • Rate von redundanten Aufrufen: Dasselbe Tool Calls mit denselben Argumenten wird mehr als zweimal wiederholt.
  • Anomalien in der Trace-Struktur: plötzliche Anstiege bei der Tiefe, der Anzahl der Werkzeugaufrufe, der Token-Anzahl, den Latency-Werten oder den Kosten.
  • Konvergenz des Pfades: Wie nah liegt die Ausführung am kürzesten bekannten gültigen Pfad für die Aufgabe.
  • Richtigkeit der Beendigung: Ob der Agent vorzeitig gestoppt wurde, nach einem Erfolg weitergearbeitet hat oder ohne die erforderliche Zustandsänderung einen Erfolg gemeldet hat.
  • Befolgung des Plans: Wenn der Agent vor dem Handeln einen Plan erstellt, muss überprüft werden, ob der Trace diesem Plan folgt. Ein guter Plan, der ignoriert wird, sowie ein schlechter Plan, dem perfekt gefolgt wird, scheitern beide aus entgegengesetzten Gründen; der Unterschied zwischen dem Plan und dem Trace zeigt an, welcher Fall vorliegt.

Führen Sie diese Schritte immer dann aus, wenn möglich, vor einem Judge. Ein Loop-Detektor befindet sich einige Zeilen über dem Trace. Er benötigt keinen Model.

Aufgabenabschluss und Bewertung der Ergebnisse

Von Anfang bis Ende stellt sich die Frage: „Hat der Benutzer genau das erhalten, was er angefordert hat?“

Zwei Muster funktionieren am besten:

  • Bewertung der zielorientierten Aufgabenerfüllung ohne Referenzdaten: Das Ziel wird aus der Eingabe extrahiert und Judge ob der Trace Plus die endgültige Antwort hat das Problem gelöst. Dies funktioniert im Online-Betrieb, da der Produktivverkehr selten „goldene“ Ausgabewerte aufweist.
  • Einstufung des Umgebungszustands: Man vergleicht die endgültigen Datenbankzeilen, Dateien, Tickets, Buchungen oder Datensätze mit einem annotierten Zielzustand. Dies ist robuster als das Abgleichen von Transkripten, weil Agents Es ist möglich, gültige Pfade zu finden, die Sie nicht aufgeführt haben.

Die zweite Option ist vorteilhafter, wenn Sie sie selbst implementieren können. Der endgültige Zustand stellt den Vertrag dar. Die Transkription dient lediglich als Beweismittel.

Zwei Einschränkungen sorgen dafür, dass diese Bewertungsmethode realistisch bleibt – beide stammen von Benchmark, der die Einführung der Zustandsbewertung popularisiert hat. tau-bench kann einem Agent, der bei bestimmten „No-Op“-Aufgaben keine Ausführung vornimmt, trotzdem eine bestandene Bewertung erteilen, da der Ausgangszustand bereits das Ziel erfüllte. Zudem berichtete Anthropic über einen Lauf von Opus 4.5, bei dem eine Aufgabe im tau2-bench aufgrund eines Schlupflochs in der Strategie zwar als fehlgeschlagen eingestuft wurde, dieser Schlupflauf jedoch tatsächlich zum besseren Ergebnis für den Benutzer führte. Die Zustandsbewertung ist zwar überlegen gegenüber der Transkriptvergleichsmethode, doch der Zielzustand bleibt weiterhin eine Annotation – und Annotationen weisen Fehler auf. Prüfen Sie daher insbesondere die Fälle, die zu leicht bestehen, nicht nur diejenigen, die fehlschlagen.

Der Trace-zum-Eval-Flywheel

Vermeiden Sie Produktionsausfälle durch das Erkennen von Fehlern bereits vor dem Ausarbeiten weiterer Eval-Fälle.

Die Schleife:

  1. Erfassen Sie den vollständigen Trace.
  2. Kennzeichnen Sie, was fehlgeschlagen ist.
  3. Gruppieren Sie ähnliche Fehler.
  4. Behalten Sie pro Cluster ein repräsentatives „Gold-Exemplar“ bei.
  5. Führen Sie eine Versionskontrolle für den Dataset durch.
  6. Führen Sie die Überprüfung im CI-Umfeld aus.
  7. Halten Sie weiterhin die Bewertung der ausgewählten Produktionsdaten des Traces online.

Das begleitende Repository trace2evals führt den vollständigen Loop für einen fehlerhaften Support Agent aus. Dabei werden OpenTelemetry GenAI Spans erfasst, Fehler mithilfe deterministischer Regeln erkannt, Fälle in eine versionierte „goldene“ Dataset Struktur dedupliziert und jede dieser goldenen Versionen im CI erneut ausgeführt. Die standardmäßige Skriptlösung erfordert keinen API Schlüssel, sodass make demo führt den Prozess offline aus.

Fehleranalyse bei Mine-Fehlern

Hamel Husain und Shreya Shankar erklären eine Fehleranalyse Workflow für genau diesen Schritt; Hamels Leitfaden Er geht die Prozesse Schritt für Schritt durch. Die ersten beiden Schritte leihen ihre Bezeichnungen aus der qualitativen Forschung, doch die Methode selbst ist einfach: Man liest Traces, macht Notizen und benennt die Muster.

  1. Offener Kodieren: Lesen Sie 30 bis 50 echte Traces-Beispiele und fassen Sie frei formulierte Notizen darüber zusammen, was schiefgelaufen ist.
  2. Axialer Kodieren: Gruppieren Sie diese Notizen in 5 bis 6 benannte Kategorien von Fehlern.
  3. Markieren Sie alles entsprechend der Taxonomie.
  4. Erstellen Sie Metriken für die größten Gruppen.

Beginnen Sie nicht mit Labels wie reasoning_issue oder tool_problem. Sie sind zu vage, um getestet zu werden. Verwenden Sie Labels wie missing_identity_verification, date_argument_mismatch, retried_same_tool_after_429, oder stopped_before_database_update. Duplikate entfernen, bevor Sie in die Produktion überführen

Der Trace-Mining-Loop birgt ein Problem: Er fügt ständig alle fehlerhaften Trace hinzu. Dadurch entsteht ein Dataset, der groß, ressourcenintensiv und eingeschränkt ist. Er übernimmt nahezu identische Fälle aus dem März, während er die neue Form desselben Fehlers im Juni übersieht.

Gruppiere zunächst die Elemente. Wähle pro Cluster einen repräsentativen „Golden“ aus. Speichere die damit verbundenen Trace-IDs in den Metadaten, damit ein Prüfer später die Produktionsbelege überprüfen kann.

Falls ein Fehlercluster nach einer Korrektur erneut auftritt, bedeutet dies, dass der Fall der Regression nicht generalisiert werden konnte. Gruppieren Sie die Beispiele erneut und erweitern Sie den „Golden Set“, anstatt 15 zusätzliche Beispiele hinzuzufügen.

Versionierung des Dataset

Die Versionierung von Datasets erfolgt auf dieselbe Weise wie bei der Versionierung von Prompts und Code. Immer dann, wenn sich bedeutende Änderungen ergeben (Model, Prompt, Tool Schema, Judge Prompt oder im Verhalten der Anwendung), sollte man die gleiche Dataset Version vor und nach der Änderung ausführen.

Das CI-Gate sollte folgende Werte festlegen:

  • Dataset-Version
  • Anwendungsversion
  • Prompt-Version
  • Judge Model
  • Judge Prompt
  • Evaluator-Code-Version

Wenn eine dieser Änderungen vorgenommen wird, wird der Vergleich vor und nach unklar. A goldens-v3.json Bei kleinen Datenmengen funktioniert eine Datei im Git-System problemlos. Die tool-eigene Snapshots-Funktionalität in Langfuse, Phoenix, Braintrust oder LangSmith ist dann von Vorteil, wenn die Dataset-Arbeit kooperativ gestaltet wird.

Ein fehlerhafter Metrikwert muss zu einem fehlerhaften Build führen – andernfalls ist das Eval-Suite lediglich ein Dashboard, das niemand liest.

Der Test sollte den aktuellen Agent erneut mit dem Referenz-Eingang ausführen. Er darf nicht lediglich den alten, fehlgeschlagenen Trace erneut abspielen:

@pytest.mark.parametrize("golden", GOLDENS, ids=[item["id"] for item in GOLDENS])
def test_agent_regression(golden: dict) -> None:

    refired = set(flag_failures(fresh_trace)) & set(golden["failure_modes"])
    assert not refired, f"failure mode regressed: {sorted(refired)}"

    assert tool_correctness(
        called=[call["name"] for call in fresh_trace["tool_calls"]],
        expected=golden["expected_tools"],

    ) >= golden.get("tool_threshold", 1.0)

Diese Unterscheidung ist leicht falsch zu verstehen. Die Aufgabe des Dataset besteht darin, zu erkennen, ob die nächste Version von Agent denselben alten Fehler wiederholt, und nicht darin, den Fehler selbst zu archivieren.

Kalibrieren Sie den Judge, bevor Sie ihm vertrauen

LLM als Judge ist hilfreich. Gleichzeitig ist es jedoch auch einfach, sich selbst zu täuschen.

G-Eval Es wurde gezeigt, dass ein Judge, der zunächst explizite Bewertungsschritte festlegt und anschließend die Bewertungen durch Summierung der einzelnen Bewertungsebenen unter Berücksichtigung ihrer Token-Wahrscheinlichkeit (score=∑ip(si) si\text{score} = \sum_i p(s_i)\,s_iscore=∑i​p(si​)si​) bewertet, menschliche Bewertungen besser erfassen kann als die älteren automatischen Metriken, die er ersetzt hat. Der schrittweise Ansatz mit Wahrscheinlichkeitsgewichtung erfordert Judge-Logwahrscheinlichkeiten, die von einigen gehosteten Models-Systemen nicht bereitgestellt werden. Das umfassendere Ergebnis unterstützt weiterhin einen expliziten Bewertungsweg vorzuziehen, anstelle einer reinen Punktzahl.

MT-Bench Es zeigte sich, dass GPT-4 in etwa genauso oft den Präferenzen von Menschen zustimmte wie Menschen untereinander, was dazu beitrug, LLM als Maßstab für die Bewertung des Mainstreams zu nutzen. Spätere Studien machten Positionierungs-, Längen- sowie Selbstpräferenzverzerrungen sichtbar. Die Judge-Werte können außerdem schwanken, wenn sich die Prompt- oder Model-Version ändert.

JudgeBench Es wurden Antwortpaare erstellt, bei denen eine der Antworten aufgrund überprüfbarer Fakten, Reasoning, mathematischer Berechnungen sowie Codeaussagen objektiv falsch ist. Der GPT-4o erzielte auf dieser Testmenge einen Wert von 50,9 %, während der leistungsstärkste getestete Judge-Modell etwa 64 % erreichte. Zuverlässige, aber dennoch falsche Antworten stellen weiterhin ein schwieriges Problem für Model-basierte Judges-Systeme dar.

Betrachten Sie den Judge als Messinstrument: Kalibrieren Sie ihn anhand menschlicher Labels, bevor er Bewertungen vornimmt, und überprüfen Sie ihn erneut, sobald sich die Judge Model oder Prompt ändern.

Wenn ein Judge erforderlich ist, muss das Ergebnis strukturiert dargestellt werden. Schema-gestütztes Reasoning (SGR) ist das Standardmuster hier: Definieren Sie den Reasoning-Pfad von Judge als Pydantic-Schema, führen Sie es über Structured Outputs oder Constrained Decoding aus und zwingen Sie Model dazu, Felder wie diese auszugeben. evidence, passed_criteria, failed_criteria, failure_mode, und score.

Platzieren Sie die Beweisfelder vor der Bewertung. Anschließend wendet das Judge bei jeder Ausführung und in allen kompatiblen Model denselben vorgefertigten Bewertungsstufenablauf in derselben Feldreihenfolge an. Ein Prüfer kann dabei auf benannte Felder zurückgreifen, anstatt einen Absatz zu analysieren; CI kann einen stabilen JSON-Objekt vergleichen, und die Calibration-Menge kann anzeigen, welche Bewertungsstufe von der menschlichen Beschriftung abwich.

Es verändert außerdem die Kostenkurve. Wenn die Topologie von Reasoning explizit definiert ist und die Ausgabe eingeschränkt wird, reichen günstigere Models in der Regel aus, um routinemäßige Bewertungen durchzuführen. Sparen Sie sich die aufwändigeren Judge für Fälle mit Unstimmigkeiten, hoher Risikostufe oder Calibration-Ausführungen auf, anstatt sie bei jeder Trace zu nutzen.

Standard-Checkliste für die Judge-Hygiene:

  1. Wählen Sie bei Möglichkeit einen binären Erfolg/Ablehnung-Ansatz. Fünfstufige Skalen führen zu einer künstlichen Präzision.
  2. Beschriften Sie vor der Erstellung des endgültigen Bewertungsschemas 30 bis 50 Trajektorien manuell.
  3. Messen Sie die Judge-Übereinstimmung zwischen Mensch und System mithilfe von Cohens Kappa (einer auf Zufall korrigierten Übereinstimmungsrate, wodurch ein Judge, der stets „Erfolg“ ausweist, nahe Null liegt) oder direkt über TPR/TNR.
  4. Zerlegen Sie grobe Kriterien in spezifischere Unterkriterien. Die Frage „Hat das Agent die Identität vor der Rückerstattung Tool Call überprüft?“ ist aussagekräftiger als die allgemeine Frage „War die Trajektorie gut?“.
  5. Geben Sie das Urteil über ein SGR-Schema mit Belegen, fehlgeschlagenen Kriterien, Fehlerursachen sowie der erreichten Punktzahl wieder.
  6. Verwenden Sie bei Möglichkeit einen Judge aus einer anderen Model-Familie als der Generierer.
  7. Randomisieren Sie die Paarreihenfolge und bilden Sie anschließend den Durchschnitt aus beiden Richtungen.
  8. Bestrafen Sie im Bewertungsschema unzulässig lange Antworten – eine längere Antwort bedeutet nicht automatisch eine bessere.
  9. Fixieren Sie die Werte von Judge Model, Prompt, Dataset, dem Schema sowie der App-Version.
  10. Kalibrieren Sie erneut nach Änderungen in Model, Prompt, Tools, Richtlinien oder Schemata.

Bei besonders kritischen Bewertungen sollte anstelle eines großen Judge ein kleineres Gremium eingesetzt werden. PoLL Ich habe genau diese Konfiguration getestet: ein Panel aus kleineren Judges, die aus voneinander unabhängigen Model-Familien stammen, wobei ihre Bewertungen zu einem einzigen Gesamtscore zusammengefasst werden. Über einen Zeitraum von sechs Datasets zeigte das Panel eine bessere Treue gegenüber menschlichen Urteilen als ein einzelner GPT-4 Judge, vermied den Selbstpräferenzbias, den ein einzelner Judge bezüglich der Ergebnisse seiner eigenen Familie aufweist, und kostete dabei mehr als sieben Mal weniger. Lassen Sie menschliche Überprüfungen für Entscheidungen vornehmen, die finanzielle Aspekte, Zugriffsmöglichkeiten, Sicherheit oder Compliance betreffen.

Falls ein Judge bei Ihrer Aufgabe eine Kappa-Wert von 0,55 gegenüber Menschen aufweist, sollten Sie ihn nicht zur Blockierung von Deploys verwenden. Nutzen Sie ihn stattdessen zur Sortierung der Überprüfungsqueues. Liegt der Wert nahe bei 0,75 und die Kosten im Falle eines Fehlers sind moderat, ist es deutlich einfacher, eine CI-Gate-Prüfung zu rechtfertigen.

Guardrails-Blöcke inline einbinden und Evals anschließend überwachen

Menschen verwechseln diese Konzepte oft, weil beide Arten Werte erzeugen. Der Unterschied liegt in der Platzierung: entweder inline im Anfragenpfad, vor der Freigabe oder nach der Antwort.

Guardrails werden inline ausgeführt. Sie sind schnell, deterministisch und für den Benutzer sichtbar. Ein Guardrail kann einen Tool Call blockieren, personenbezogene Daten maskieren, Prompt Injection abweisen oder einen Neversuch erzwingen, bevor die Antwort Ihr System verlässt. Ein falsch positives Ergebnis stellt einen Produktionsfehler dar.

Offline Evals-Ausführungen vor der Veröffentlichung. Sie sind reproduzierbar. Sie stellen sicher, dass Prompts, Models, Tools, Retrieval-Systeme sowie Policies gegenüber einer festgelegten Dataset abgesichert sind.

Online Evals-Ausführungen finden nach der Antwort statt, in der Regel anhand von ausgewählten Datenmengen. Da sie sich nicht auf dem Latency-Pfad befinden, können sie langsamere LLM Judges-Verfahren nutzen. Ihre Aufgabe besteht darin, Abweichungen zu erkennen, neue Ausfallcluster zu identifizieren und die Ergebnisse an den nächsten offline laufenden Dataset weiterzuleiten.

Wenn die Platzierung falsch ist, hat das in jedem Fall negative Folgen:

  • Ein Judge im Anfragenpfad führt zu zusätzlichem Latency sowie einer neuen Quelle für Instabilität.
  • Ein auf asynchrone Bewertung verlagertes Guardrail ermöglicht es, Verstöße gegen Richtlinien den Endnutzern zugänglich zu machen.

In Systemen mit hohem Durchsatz sollte eine kleine Stichprobe mit einem leistungsstärkeren Judge bewertet werden, während für größere Stichproben günstigere Klassifikatoren eingesetzt werden können. Es sind Warnungen bei Auftreten von Clustern sowie bei Überschreitung der Konfidenzintervalle erforderlich – anstelle einer einzigen, störanfälligen Schätzung.

Die Hinweise zu den Kompromissen beschreiben lediglich die Quellen der Kosten, nicht deren genaue Natur. Die Preisseiten können sich ändern, und die Anbieter berücksichtigen unterschiedliche Faktoren wie Traces, Beobachtungen, Spans, Bewertungswerte, Nutzerzahlen, die Retention-Rate oder die verarbeiteten Daten. Überprüfen Sie daher vor dem Abschluss stets die aktuellen Preise.

Entscheidungshilfen:

  • Benötigt ein selbst gehostetes Tracing mit OTel-Portabilität: Beginnen Sie mit Phoenix oder Langfuse.
  • Benötigt eine CI-Schleuse, bei der Code im Vordergrund steht: Beginnen Sie mit DeepEval.
  • Schon auf LangGraph fokussiert: LangSmith ist sehr praktisch.
  • Möchte eine verwaltete Überprüfung von PR-Regressionen: Braintrust ist unschlagbar.
  • Sicherheitsfragen und Red-Team-Szenarien stehen im Vordergrund: Promptfoo ist das passende Werkzeug dafür.
  • Forschung zum Thema Sicherheit oder kontrollierte Benchmark-Arbeiten: Inspect AI eignet sich besser.

Die Wahl des Tools spielt nur eine untergeordnete Rolle. Wenn Produktionsfehler nicht zu Testfällen werden, zahlt man im Wesentlichen für die Speicherung von Trace.

Eine praktische Checkliste für die Einführung

Erstellen Sie zunächst die Beweismittel Pipeline, bevor Sie den Metriken-Stack erweitern. Beginnen Sie damit, zu entscheiden, woher die Beispiele stammen sollen.

  1. Sammeln Sie zunächst die historischen Ausführungen ein. Falls der Agent bereits vorhanden ist, holen Sie vor einer Änderung der Implementierung zunächst Traces, Support-Tickets, Fehlerberichte, negative Bewertungen Sessions, manuelle QA-Transkripte sowie Notizen aus dem internen Testbetrieb ab. Wenn der Agent noch nicht existiert, protokollieren Sie ab dem ersten Tag jede Prototypen- und manuelle Testausführung.

  2. Instrumentieren Sie die Trace-Struktur. Erfassen Sie Nachrichten, Tool Calls, Argumente, Ausgaben von Tools, Fehler, Token-Zählwerte, Latency, Kosten, Nutzerfeedback, die App-Version, Prompt-Version, Model-Version, Tool Schema-Version sowie den endgültigen Zustand der Umgebung. Verwenden Sie dazu OpenTelemetry Konventionen für Generative KI oder im OpenInference-Stil Spans, falls Sie Portabilität wünschen. Verwenden Sie Langfuse, LangSmith, Phoenix oder Braintrust, wenn Sie sofort eine Trace Benutzeroberfläche sowie Dataset Workflow benötigen.

  3. Reale Ausfälle in Beispielfälle umwandeln. Lesen Sie zuerst den Traces, bevor Sie diese mit einem Model zusammenfassen. Für jeden nützlichen Ausfall sollten Eingabedaten, die Quell-Trace-ID, der erwartete Zustand, die erwarteten Tool-Invarianten, der Ausfallmodus, die Schweregrad sowie eine Anmerkung des Prüfers gespeichert werden. Langfuse kann Dataset-Elemente wieder auf die Produktionsumgebung Traces verknüpfen; LangSmith kann aus den protokollierten Ausführungen Datasets erstellen. Bewahren Sie den Quelllink auf, damit der Fall weiterhin überprüfbar bleibt.

  4. Falls keine Historie vorliegt, werden Cold-Start-Fälle erzeugt. Man beauftragt einen LLM damit, Aufgaben auf der Grundlage von Produktanforderungen, Richtlinien, Tool Schemas, State Machines sowie unterstützender Makros zu entwerfen. Es sollen sowohl erfolgreiche Abläufe als auch Fälle erzeugt werden, bei denen „es nicht passieren sollte“: fehlerhafte Berechtigungen, fehlende Identitätsprüfungen, veraltete Tool Results-Werte, unklare Datumsangaben, Wiederholungsversuche nach Rate-Limits sowie Tool-Ausgaben, die im Widerspruch zur Anfrage des Benutzers stehen.

  5. Verlassen Sie sich nicht auf synthetische Fälle, bevor sie von einem Menschen überprüft wurden. Synthetische Beispiele sind nützlich für die Abdeckungsanalyse, aber nicht für die Überprüfung der Richtigkeit. Kennzeichnen Sie sie entsprechend. source: syntheticEs ist erforderlich, dass ein Prüfer das erwartete Ergebnis freigibt, gegebenenfalls einen bekannten Referenzpfad ausgeführt wird und vermieden wird, dieselbe Model-Familie sowohl zur Erstellung des Falls als auch zur Judge-Erzeugung des Ergebnisses zu verwenden.

  6. Entwickeln Sie einen kleinen, ausgewogenen Dataset. Dieser sollte Erfolgsfälle, Misserfolge, Ablehnungen, Randfälle, Fälle mit langen Dialogabläufen, fallempfindliche Szenarien sowie gültige alternative Handlungswege umfassen. Es darf sich dabei nicht um eine „exakte Kopie des ursprünglichen Transkripts“ handeln; vielmehr muss das Referenzobjekt das erforderliche Ergebnis, zulässige Invarianten sowie mögliche Fehlermodi enthalten.

  7. Fügen Sie zunächst deterministische Überprüfungen hinzu. Die erforderliche Reihenfolge der Tools – wobei die Reihenfolge selbst eine Richtlinie darstellt – umfasst die Einreichung der notwendigen Argumente, die Validierung des Schemas, die Vergleich der Endzustände, die Begrenzung von Schleifen, die Obergrenzen für Token und Latency sowie task-spezifische Invarianten, die alle vor der Ausführung jeglicher Judge erfolgen müssen.

  8. Fügen Sie einen SGR-förmigen Judge hinzu. Verwenden Sie ihn ausschließlich für den Teil, der einer Interpretation bedarf. Kalibrieren Sie ihn anhand menschlicher Etiketten. Falls er auf der Calibration-Sammlung keine Unterscheidung zwischen guten und schlechten Beispielen vornehmen kann, müssen Sie die Bewertungskriterien zunächst korrigieren, bevor Sie ihn in den CI-Prozess integrieren.

  9. Schließen Sie den Loop. Führen Sie das kleine Offline-Testpaket im CI aus, starten Sie das größere Paket vor der Veröffentlichung, bewerten Sie den ausgewählten Produktivverkehr online und führen Sie wiederkehrende Online-Fehlercluster zurück in den Offline-Betrieb Dataset.

Ihre erste Eval-Suite wird auf langweilige Weise Fehler aufweisen. Schicken Sie sie dennoch raus. Eine Suite, die täglich genutzt wird, ist einfacher zu reparieren als ein perfektes Designdokument, das niemals einen fehlerhaften Pull Request blockiert.
