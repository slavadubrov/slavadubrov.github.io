Was ist ein AI Agent Runtime?

Ein AI Agent Runtime stellt die Infrastruktursschicht dar, die dafür sorgt, dass ein toolnutzender Agent nach Beendigung des Model-Aufrufs weiterhin aktiv bleibt, isoliert ist, überwacht werden kann und seine Ausführung wieder aufnehmen kann. Sie verwaltet den Session-Zustand, die Tool-Ausführung, Checkpoints, Richtlinienprüfungen, Geheimnisse, Traces, Kostenlimits sowie die Deployment-Struktur. Der Model wählt die nächste Aktion aus; der Runtime entscheidet darüber, wo diese Aktion ausgeführt wird, ob sie erlaubt ist, wie sie protokolliert wird und wie die Ausführung nach einem Fehler wieder aufgenommen wird.

Die fünf Runtime-Primitiven, die jedes langlaufende AI Agent benötigt

Anthropics Skalierung von verwalteten Agents Der Dokumentationsteil liefert ein nützliches Vokabular für die fünf Runtime-Verantwortungsbereiche. Der Harness treibt den Agent Schritt für Schritt voran; der Session protokolliert dessen Handlungen; der Sandbox ist der Ort, an dem Befehle ausgeführt werden; der Checkpoint enthält die Informationen, die der nächste Prozess beim Wiederaufnehmen seiner Arbeit liest; der Trace dient dazu, Tage später zu verstehen, warum etwas schiefgelaufen ist. Eine Implementierung kann Komponenten zusammenführen, doch die jeweiligen Verantwortungsbereiche sowie die Grenzen bei Fehlern benötigen weiterhin klare Bezeichnungen.

Session. Ein ausschließlich zum Hinzufügen bestimmtes Protokoll über alles, was geschehen ist: Model-Aufrufe, Tool Calls, Ergebnisse, Fehler sowie Freigaben. Die Wiederherstellung erfolgt wake(sessionId) → getSession(id) → resume from last event. In LangGraph wird dies als ein thread_id plus ein Postgres Checkpointer (siehe Persistenz von LangGraph). Der OpenAI Agents SDK stellt zehn integrierte Session Backends zur Verfügung, darunter SQLiteSession, RedisSession, SQLAlchemySession, MongoDBSession, und EncryptedSession (siehe den Sessions Dokumentation).

Harness. Der Orchestration-Loop. Er ruft den Model auf, parset Tool Calls, führt diese aus, schreibt die Ergebnisse zurück in den Session und wendet Wiederholungsregeln an. Anthropic formuliert es unverblümt:

„Jedes Komponente in einem Harness kodiert eine Annahme darüber, was das Model nicht eigenständig bewältigen kann.“

Das Codex-Team von OpenAI bezeichnet diese Disziplin als Harness Engineering: Die Entwicklung von Software erfordert weiterhin Disziplin, doch heute fließt ein größerer Teil dieser Anstrengungen in die Infrastruktur und Rahmenstrukturen als in den Code selbst. LangGraphs CompiledStateGraph, Deep Agents’ create_deep_agent, Auch Claude Code selbst zählen in diesem Sinne zu den Harnesses.

Sandbox. Die isolierte Ausführungsumgebung, in der die Befehle tatsächlich ausgeführt werden. Der OpenAI Agents SDK Sandbox-Konzepte Die Seite zieht die Grenze sauber:

„Der äußere Runtime ist weiterhin für die Freigaben, Tracing, den Übertrag von Aufgaben sowie die Fortsetzung der Buchhaltung verantwortlich. Der Sandbox Session kümmert sich um Befehle, Dateiveränderungen und die Isolierung der Umgebung.“

Sandboxes unterscheiden sich dadurch, wie lange sie aktiv sind und was sie zwischen den Ausführungen behalten. Die einfachste Form ist frischer vorübergehender Zustand: Ein Agent wird nur für eine einzige Aufgabe gestartet, nach Abschluss der Aufgabe wieder zerstört, wodurch bei jeder Ausführung die Kosten für einen Neustart anfallen. Sandboxes persistente pausierte Zustände bleiben zwischen den Ausführungen im pausierten Zustand erhalten; das Dateisystem sowie ein Speichermanagement Snapshot bleiben bestehen, sodass das Wiederaufnehmen nach nur wenigen Sekunden stattfindet, anstatt ein vollständiger Neustart erforderlich zu sein. Snapshots oder Forks gehen noch einen Schritt weiter: Jede Aufgabe erstellt eine Kopie auf Schreiben basierend auf einem Elternelement, das bereits alle benötigten Abhängigkeiten installiert sowie den Cache vorgewärmt hat, wodurch die aufwendigen Initialisierungsarbeiten nur einmal durchgeführt werden müssen und mehrere Aufgaben dieselbe Basiskopie teilen können. Sandboxes Arbeitstree-basierte Ansätze stellen sicher, dass jede Aufgabe über ihren eigenen Arbeitsraum und seinen eigenen Observability Stack verfügt: getrennte Protokolle, Metriken sowie Traces. Dadurch kann man den Lauf einer bestimmten Agent ohne Einfluss auf andere Aufgaben debuggen. Konkrete Werte zu den Kosten für den Neustart sowie zur Persistenz je Anbieter sind in der Tabelle weiter unten in diesem Abschnitt aufgeführt.

Checkpoint. Wiederaufnahmefähiger Zustand. Bei LangGraphs PostgresSaver schreibt einen StateSnapshot an jeder Grenze eines Super-Schritts, durch schrittbezogene Schreibvorgänge pro Aufgabe checkpoint_writes Daher werden erfolgreiche Node-Ausgaben nicht erneut berechnet, wenn ein Pendant fehlschlägt. Der Snapshot ist ein JSON-serialisierbares Dictionary.v, ts, id, channel_values, channel_versions, versions_seen, pending_sends) dokumentiert in dem langgraph-checkpoint-postgres PyPI-Seite und der LangGraph Checkpoints-Referenz.

Trace. Die Wiedergabeschnittstelle sowie die Debugging-Oberfläche. Jeder Aufruf eines Model, Tool Call sowie jeder Unter-Agent-Schritt wird zu einem Span, der Zeitangaben, Eingaben, Ausgaben, Token-Zählwerte sowie Kosten enthält. Wenn eine sechsstündige Ausführung fehlschlägt, dient die Trace dazu, herauszufinden, was schiefgelaufen ist – schließlich sind die Terminalausgaben aus dieser Ausführung bereits längst verschwunden. Die von OpenTelemetry erzeugten semantische Konventionen von GenAI Standardisieren Sie die Namen der Attribute (welcher Model, welcher Anbieter, wie viele Tokens, welche Konversation, welcher Workflow), damit derselbe Trace einheitlich dargestellt wird. Geschwindigkeit, Jaeger, Honigwabenstruktur, oder LangSmith ohne erneute Instrumentierung.

Richtlinien und Geheimnisse bilden getrennte Runtime Grenzen

Zwei Grenzen durchqueren alle fünf Primitiven und lassen sich besser als getrennte Aufgaben betrachten. Es handelt sich dabei um die Runtime-Version des Sicherheitsarguments aus Teil 4.

Richtlinienmotor

Eine Berechtigungsprüfung wird vor jedem Tool Call durchgeführt, um zu entscheiden, ob dieser ausgeführt werden darf. In der Produktion kommen zwei gängige Ansätze zum Einsatz. Bei tiefen Agents-Architekturen kann jeder Unteragent angeben, welche Dateipfade er lesen oder schreiben darf, wobei das Middleware-System alles außerhalb dieser Angaben blockiert. Anthropic Managed Agents leitet jeden Tool Call über einen MCP-Proxy weiter, sodass der Proxy statt des Agent-Codes die Berechtigungen durchsetzt. Wenn ein sensibler Aufruf eine menschliche Freigabe erfordert, übernimmt dies bei LangGraphs interrupt() und die Deep Agents-Genehmigungs-Hook-Funktion stoppt das Graphen, bis eine Person „Ja“ sagt.

Fehlermodi bei Produktivumgebungen AI Agent Runtime

Der Runtime steuert Wiederholungsversuche, stellt vorherige Arbeit wieder her, isoliert Arbeitsräume und sorgt für die Einhaltung von Budgets. Die fünf oben genannten Primitiven überwachen den gesamten Laufzyklus, während Richtlinien sowie Mechanismen zur Verwaltung von Geheimnissen diese Prozesse überschneiden. Da Ausführungen über mehrere Worker sowie Context Windows verteilt sind, treten Fehler häufig im Zusammenhang mit Zustandsveränderungen, doppelten Nebeneffekten, Sandbox-Abweichungen sowie der Budgetkontrolle auf.

Die Ausfälle lassen sich in vier Gruppen einteilen:

  • Fehler bei der Qualität der Ausgabe: Der Agent erklärt vorzeitig einen Erfolg, bevor die Arbeit tatsächlich abgeschlossen ist, vergisst, was er in einem Kontextfenster-Reset getan hat, oder verlässt sich auf seine eigene Selbstbewertung und liefert eine fehlerhafte Ausgabe aus.
  • Fehler bei der Kostenkontrolle: Der Agent gerät in eine Wiederholungsloop, verbraucht ein Token oder das Budget für Toolaufrufe, ohne etwas Nützliches zu erzeugen.
  • Fehler im Zustandsmanagement und bei Abstürzen: Die Arbeitsumgebungen verändern sich, weil eine Ausführung Dateien berührt, die einer anderen Ausführung gehören, Tool Calls wird mehrfach ausgelöst, da Wiederholungsversuche diese Vorgänge erneut ausführen, oder die Arbeit geht verloren, wenn ein Worker zwischen den Ereignissen abstirbt.
  • Fehler im Kontextfenster: Der Model fasst zusammen und beendet die Ausführung frühzeitig, weil er annimmt, dass der Speicherplatz erschöpft sei, obwohl im Fenster noch genügend Platz vorhanden ist.

Die Tabelle ordnet jedem Fehler einer Abhilfemaßnahme zu, dem dazu verwendeten Runtime-Hook sowie dem Umfang der angeführten Belege. Da sich das verhaltensspezifische Verhalten von Model ändern kann, dient die Spalte mit den Belegen als Prompt, um die Annahme erneut zu überprüfen, anstatt als permanente Ja/Nein-Regel zu fungieren.

Der gesunde Lebenszyklus einer Laufanwendung

Ein ordnungsgemäß ablaufender Vorgang ist langweilig – es handelt sich dabei um eine Abfolge kleinerer, wiederherstellbarer Schritte, wobei jeder Schritt sein Ergebnis vor dem Start des nächsten in einer dauerhaften Speicherung hinterlegt, anstatt dass ein einziger großer Anfragenstrom von Anfang bis Ende erfolgreich sein muss. Genau durch die Aufteilung des Vorgangs in solche Schritte ist er widerstandsfähig gegen Ausfälle: Wenn etwas fehlschlägt, geht nur der gerade ausgeführte Schritt verloren, und der nächste Worker setzt seine Arbeit an dem letzten abgeschlossenen Schritt fort, anstatt von vorn anzufangen.

  1. Starten Sie entweder von einem neuen Session oder von einem wiederbelebten System aus. Beim Wiederaufnehmen wird der Arbeitsbereich aus seinem letzten bekannten Zustand montiert und alle Fortschrittsdateien, die vom vorherigen Versuch übrig geblieben sind, ausgelesen.PROGRESS.md, feature-list.json), und lädt den letzten Checkpoint aus der Datenbank. An dieser Stelle übergeben die Harness dem Agent alles, was der vorherige Worker noch im Speicher hatte, bevor er abstürzte.
  2. Planen Sie vor jedem Ausführungsvorgang eines Tool Calls sorgfältig. Legen Sie fest, wie „erledigt“ aussieht, wie viel Zeit für die Ausführung zur Verfügung steht, welche Tools die Agent aufrufen dürfen, und was eine vorzeitige Beendigung auslösen soll. Diese Planungswerte werden zu Runtime-Prüfungen; ohne sie hat die Ausführung keinen Anhaltspunkt für Korrekturen.
  3. Führen Sie jeweils nur einen Tool Call aus. Die Policy-Schicht entscheidet, ob der Aufruf gestattet wird. Der Harness führt ihn aus, speichert das Ergebnis und schreibt ein Ereignis ins Session-Log. Ein Schritt – ein Ereignis. Ein Absturz zwischen den Ereignissen ist behobar, da das Log und nicht der Speicher des Workers die einzige zuverlässige Quelle darstellt.
  4. Checkpoint an den Grenzen der Über-Schritte oder nach jedem Ereignis in einem einfacheren Harness. Speichern Sie den Zustand des Graphen, die Unterschiede im Arbeitsbereich sowie Verweise auf alle erzeugten Artefakte. Dieser Checkpoint wird vom Schritt 1 beim nächsten Neustart gelesen. Wenn der Checkpoint fehlt oder veraltet ist, reduziert sich die Wiederherstellung auf das Erneuten Abspielen des gesamten Session-Logs von vorn, was deutlich langsamer ist.
  5. Es wird eine Bewertung anhand der Artefakte durchgeführt, sobald der Agent den Abschluss des Vorgangs erkennt: Tests, Überprüfung durch einen Reviewer im sauberen Kontext, Validierung des Schemas sowie Browser-Prüfungen. Wenn alle Prüfungen erfolgreich sind, endet der Lauf erfolgreich. Sollte eine Prüfung fehlschlagen, setzt der Lauf von dem letzten korrekten Checkpoint aus und fügt die Fehlermeldung zum Kontext hinzu, um es erneut zu versuchen.

In dieser Liste gibt es keinen Schritt, der vom Agent verlangt, zwischen den Ausführungen etwas zu merken. Der Zustand wird im Session sowie im Checkpoint gespeichert, und der Agent lädt ihn bei jedem Neustart wieder ab.

Tools mit Nebeneffekten benötigen Idempotenz. Jeder Tool mit Nebeneffekten muss einen Idempotenz-Schlüssel besitzen, der aus dem Session-ID sowie dem Tool-Aufruf-ID abgeleitet wird und vor Auslösung des Nebeneffekts gespeichert wird. send_email(session_id, tool_call_id, message_hash). create_pr(session_id, tool_call_id, branch_name). charge_customer(session_id, tool_call_id, invoice_id). Die mindestens-einmalige Ausführung ist die Standard-Einstellung in Warteschlangen sowie Workflow-Engines. Wenn die Wiederholung eines Tool Call tatsächlich Schäden verursachen kann, ist das Tool noch nicht für Agents bereit.

Die Bewertung muss Belege aus Kontexten enthalten, die nicht zur Erstellung verwendet wurden. Ein frischer Evaluator verringert den Bias durch gemeinsame Kontexte, während Tests, Linter, Browserprüfungen sowie Schema-Validierungen deterministische Belege liefern. Die Prüfung kann dabei einen Rückgabewert liefern. pass, fail, oder needs_human. Für den Code Agents kann der Prüfer ein weiterer Model Session mit nur lesbaren Befugnissen sein. Bei Daten und Berichten Agents sollte man deterministische Validierungsverfahren mit einem Prüfer Model kombinieren, bei dem weiterhin eine menschliche Beurteilung erforderlich ist.

Beginnen mit der Lauflängenmethode

Verwenden Sie die Lauflänge als ersten Filter:

  • Unter 30 Sekunden, idempotent: Request-Lifecycle SDK auf einem App-Server.
  • 30 Sekunden bis 60 Minuten, keine Krash-Wiederherstellung erforderlich: Queue + Worker + Checkpoint-DB.
  • 60 Minuten bis 24 Stunden: Dieselbe Queue + Worker oder ein Cloud Run Job für einmalige Aufgaben. Verwenden Sie einen persistenten Workflow-Motor, falls auch Versionierung und Wiederausführung benötigt werden.
  • Länger als 24 Stunden – muss Deploys überstehen: Persistenter Workflow-Motor im Temporal-Stil. Cloud Run Jobs können lange Aufgaben bis zu ihrem Task-Limit speichern, bieten jedoch keine Wiederausführungsfunktionen.
  • Mehrtägige RL-Trainingszyklen: K8s Job + Volume + Temporal.

Nach diesem groben Filter sollten Nebeneffekte, Wiederherstellbarkeit, Wiedergabefunktionen, Isolierung, Datenlage sowie das Team, das das System betreiben wird, überprüft werden.

Plattformkonformität nach Anwendungsfall

Die Matrix ist dicht, da keine einzelne grüne Zelle über die Architektur entscheidet. Eine breite Abdeckung der Arbeitslasten ist zwar vorteilhaft, liefert jedoch keine Informationen zur Datenresidenz, zu den Replay-Semantiken, zur Abhängigkeit vom Anbieter, zur betrieblichen Reife oder zu den Kosten für spätere Zustandsverschiebungen.

Deep Agents Deploy umfasst alle Arten von Arbeitslasten, die in der Matrix erfasst sind, wodurch es eine geeignete Lösung darstellt, wenn eine Plattform sowohl kurze Ausführungen, mehrstündige Aufgaben, Code Agents, Forschungsarbeiten Agents als auch terminierte Jobs bewältigen muss. Aufgrund dieses breiten Anwendungsbereichs weist es jedoch eine kürzere Einsatzhistorie im Produktivumfeld im Vergleich zu einem Stack aus Queue, Worker und Postgres auf. Betrachten Sie die grün markierten Felder als Aussagen zu den verfügbaren Fähigkeiten, die überprüft werden sollten, und vergleichen Sie anschließend die betrieblichen Einschränkungen, die in der Matrix nicht dargestellt werden können.

Anthropic Managed Agents passt entweder vollständig zu Ihrer Arbeitslast oder überhaupt nicht. Das Produkt weist drei strenge Einschränkungen auf: Es ist ausschließlich hostbasiert, ausschließlich für Claude vorgesehen und erlaubt Laufzeiten von unter 24 Stunden pro Session. Falls Ihre Arbeitslast alle drei Kriterien erfüllt – beispielsweise interne Programmierarbeiten Agent, die in Intervallen von 2 bis 6 Stunden ausgeführt werden – und Sie keine eigene Harness betreiben möchten, dann ist Managed Agents eine hervorragende Lösung, da sie einen großen Teil der Plattformverwaltungsaufgaben von Ihrem Team abnimmt. Sollte eine der Einschränkungen nicht erfüllt werden, weil Sie einen nicht‑Claude‑Model benötigen, eine selbst gehostete Umgebung wünschen oder längere Laufzeiten von 48 Stunden benötigen, eignet sich Managed Agents nicht – keine Anpassungen der Konfiguration können dies ändern.

Es lohnt sich, die Kostenstruktur im Voraus zu modellieren, bevor man eine Entscheidung trifft – und nicht erst danach. Die Gebühr pro Session-Stunde beträgt 0,08 US-Dollar pro Stunde zusätzlich zu den üblichen Token-Kosten. Wenn ein einzelner Session kontinuierlich läuft, entspricht das etwa 58 US-Dollar pro Monat pro Session. Bei 100 Sessions, die ständig in Betrieb sind, liegen die Kosten vor Abzug der Token etwa bei 5.800 US-Dollar pro Monat. Multiplizieren Sie 0,08 mit der erwarteten Anzahl an gleichzeitigen Session-Stunden, addieren Sie diesen Betrag zu Ihrer Token-Rechnung und vergleichen Sie ihn mit den Kosten einer eigenen Queue- sowie Worker-Infrastruktur. Führen Sie diese Berechnungen unbedingt vor dem finalen Commit durch, denn ein späterer Wechsel von der verwalteten Agents-Lösung ist ein umfassender Re-Platforming-Prozess und keine einfache Konfigurationsanpassung.

Hosteter Harness im Vergleich zu eigenem Harness

Der Unterschied liegt hier in der Betriebsweise und nicht daran, wer den Code für Harness geschrieben hat. Hosted bedeutet, dass der Anbieter den Harness-Loop in seiner Infrastruktur ausführt und Sie eine API aufrufen. Owned bedeutet hingegen, dass Sie den Loop in Ihrer eigenen Infrastruktur betreiben – auch dann, wenn der Code für Harness ursprünglich von einem Anbieter stammt.

LangChain tritt auf beiden Seiten dieser Zeile auf, was viele Anwender verwirrt. Das Unternehmen stellt LangGraph bereit – eine unter der MIT-Lizenz stehende Bibliothek, die man selbst hosten kann – sowie Deep Agents Deploy, ein verwaltetes Produkt, das einen Deep Agents Harness auf LangSmith Deployment im Standard-Cloud-Modus ausführt. Es handelt sich um dasselbe Unternehmen mit zwei unterschiedlichen Betriebsmodellen Models. Man wählt das für sich passende Modell Model aus, nicht den Anbieter. (Deep Agents Deploy verfügt außerdem über einen selbst gehosteten Modus für Teams, die die Benutzerfreundlichkeit von Harness ohne Cloud-Komponente wünschen; dieser Modus fällt in die Kategorie der selbst gehosteten Lösungen.)

Wählen Sie einen gehosteten Harness, wenn seine Unterstützung für Model, seine Datengrenzen, sein Verhalten bei Fehlern sowie seine Erweiterungspunkte bereits passen. Wählen Sie hingegen einen eigenen Harness, wenn diese Einschränkungen eigentlich Anforderungen sind, die Sie in Zukunft ändern möchten. Der Wechsel zwischen diesen beiden Ansätzen verändert sowohl den Observability als auch die Ausführungsgrenzen, weshalb der Ablauf nach dem Umstieg unbedingt getestet werden muss, bevor darauf produktive Daten angewiesen sind.

Hosteter Sandbox im Vergleich zu einer eigenen Ausführungsumgebung

Wählen Sie einen gehosteten Sandbox aus, wenn die Isolierungsmechanismen des Anbieters, die Funktionen zum Pause/Resume sowie die Fork-Semantik mit den Anforderungen bezüglich der Bedrohung Model sowie des verfügbaren Startbudgets übereinstimmen. Docker oder Fargate eignen sich für vertrauenswürdige interne Workloads, die auf VPC-Zugriff oder strenge Datenerhaltung angewiesen sind, doch ein herkömmlicher Container stellt keine ausreichende Abschottung gegen feindlichen Code dar. Fügen Sie gVisor, Kata, eine MicroVM oder einen weiteren verschärften Runtime hinzu, wenn das Agent unvertrauenswürdige Pakete installiert oder generierte Programme ausführt.

Zustandsspeicher: Git, Datenbanken und Objektspeicher nebeneinander

Lange laufende Agents-Prozesse verfügen in der Regel über drei gleichzeitig laufende Zustandsspeicher statt nur einen, wobei jeder einen anderen Zustandstyp enthält. Git speichert den Arbeitsplatzzustand – also den Code, Dokumente sowie Fortschrittsdateien, die vom Agent geändert werden. Jeder Commit liefert dem Harness einen stabilen Wiederherstellungspunkt und ermöglicht es dem nächsten Session, eine kompakte Historie zur Überprüfung zu erhalten. Die Checkpoint-DB enthält den Graphenzustand des Harness: Was entschieden wurde, welche Knoten ausgeführt wurden, welche Ergebnisse zurückkamen und was als Nächstes ausgeführt werden soll. Genau das ermöglicht es dem nächsten Worker, die Ausführung dort fortzusetzen, wo sie unterbrochen wurde. Der Artefaktspeicher hingegen speichert die großen Endausgaben wie PDFs, Parquet-Dateien und Screenshote. Diese gehören weder in Git noch in die Checkpoint-DB.

Wann sollte git als Zustandsverwaltung verwendet werden?

Verwenden Sie Git, wenn die Arbeitslast kodenbasiert ist – also mehrere Dateien bearbeitet werden müssen, Refaktorisierungen durchgeführt oder Anwendungen generiert werden – oder wenn es sich um dokumentenbasierte Arbeiten handelt, bei denen die Versionsgeschichte der Dateien von Bedeutung ist. Das Vorgehen ist einfach: Erstellen Sie einen Ausführungs-Zweig, führen Sie einen Initialisierungs-Commit durch und fügen Sie anschließend Commits an sinnvollen Meilensteinen hinzu – nach der Einrichtung, nach jedem Feature, nachdem die Tests bestanden haben sowie nach der abschließenden Aufräumarbeit. Speichern Sie den SHA-Wert des neuesten Workspace-Commits neben der Zeile Checkpoint. Beim Wiederaufnehmen überprüft der nächste Mitarbeiter den Zweig und liest git log --oneline -8, inspiziert git status und der neueste Diff, anschließend wird er ausgelesen. PROGRESS.md oder irgendein Übertragungsfile, das der vorherige Session erstellt hat.

Das macht aus Git eine Wiederherstellungsoberfläche für das gerade bearbeitete Artefakt – nicht jedoch einen Ersatz für die Checkpoint-Datenbank. Git kann zwei Fragen beantworten: Was hat sich geändert und welche Version hat die Tests bestanden. Er kann dem Harness jedoch nicht mitteilen, welcher Graphenknoten als Nächstes ausgeführt werden soll, welcher Tool Call auf Freigabe wartet oder welche Wiederholung bereits ihre Idempotenzschlüssel verwendet hat. Anthropics Harness nutzt Initialisierungscommits zusammen mit Featurespezifischen Commits als Quelle der Wahrheit zur Wiederherstellung des Arbeitsbereichs; der Model liest git log --oneline -8 um den Zustand wiederherzustellen. Überspringen Sie Git, wenn das Ergebnis eine einzelne, konversationelle Antwort ist – der damit verbundene Aufwand lohnt sich in diesem Fall nicht.

Wann DB-Checkpointing eingesetzt werden sollte

Verwenden PostgresSaver-stiliges Checkpointing, wenn der Agent eine Graphenstruktur mit mehreren Knoten aufweist, deren Zwischenzustände von Bedeutung sind (Planner → Forscher → Autor → Prüfer). Das Referenz-Repo nutzt dieses Verfahren aus genau diesem Grund. Legen Sie keine Arbeitsplatzdaten im Terabyte-Bereich in den Checkpoint ab; diese sollten in Objektspeicherplattformen gespeichert werden.

Wann ein Artifact-Store (S3 / GCS) verwendet werden sollte

Wenden Sie Objektspeicher in drei Situationen an. Erstens, wenn die Ausgabe größer als einige hundert KB ist. Checkpoint Datenbanken sind nicht dafür konzipiert, große Dateiblöcke zu speichern, und ihr Versuch, diese dort aufzubewahren, erschwert sowohl die Datenbank selbst als auch den Checkpoint-Format. Zweitens benötigen nachgelagerte Nutzer wie BI-Tools, Kunden oder andere Dienste URL-basierte Artefakte, die sie eigenständig abrufen können, ohne über den Agent zugreifen zu müssen. Drittens weichen die Aufbewahrungsfristen für den Laufzustand von denen der Endprodukte ab. Eine praktische Überprülliste für die Produktion

Bevor ein langlaufender Agent veröffentlicht wird, beantworten Sie diese Fragen unter Verwendung konkreter Begriffe aus der Infrastruktur.

  1. Welcher Store ist für die Session-Ereignisse sowie die Checkpoints verantwortlich?
  2. Was geschieht, wenn der Worker mitten in einem Tool Call abbricht?
  3. Kann eine Ausführung den Arbeitsraum einer anderen Ausführung beschädigen?
  4. Welche Aktionen erfordern eine Genehmigung?
  5. Können der Model oder der Sandbox rohe Anmeldeinformationen einsehen?
  6. Welche Tool Calls können sicher wiederholt werden?
  7. Wo wird die Obergrenze für die Kosten pro Ausführung durchgesetzt?
  8. Welche Prüfung des frischen Kontexts entscheidet über „abgeschlossen“?
  9. Wo werden die endgültigen Ausgaben nach Beendigung des Sandbox gespeichert?
  10. Können wir einen fehlgeschlagenen Lauf morgen ohne Neuauflauf erklären?

Falls die Antwort auf eine dieser Fragen lautet: „Der Prompt weist den Agent an, vorsichtig zu sein“, dann ist das System noch nicht im Einsatz. Es handelt sich weiterhin um eine Demo-Version.

Wichtige Erkenntnisse

  1. Langlaufende Agents benötigen einen Runtime, und nicht lediglich eine höhere HTTP-Timerout-Dauer. Der Runtime sorgt dafür, dass der Session, das Arbeitsumfeld, Tool Results, Checkpoints, Traces, Budgets, Genehmigungen sowie Anmeldeinformationen außerhalb des Speichers des Model bleiben.
  2. Die Kernkomponenten sind Session, Harness, Sandbox, Checkpoint sowie Trace. Richtlinienüberprüfungen sowie das Verwaltungssystem für Geheimnisse betreffen alle diese Komponenten.
  3. Die Laufzeit ist der erste grobe Deployment-Filter. Weitere Faktoren wie Nebeneffekte, Wiederholungsverhalten, Isolation, Datenstandort sowie operative Verantwortlichkeiten tragen zur Entscheidungsfindung bei.
  4. Eine Kombination aus Queue, Worker und Checkpoint-DB stellt den standardmäßigen, gut abgesicherten Ansatz dar. Ein gehosteter Harness eignet sich dann, wenn seine Model-Anforderungen hinsichtlich Daten, Wiederherstellung und Erweiterbarkeit bereits mit der Arbeitslast übereinstimmen.
  5. Tools mit Nebeneffekten benötigen Idempotenzschlüssel. In Queues sowie Workflow-Engines sind Wiederholungsversuche normal. Ohne Deduplizierung kann ein solcher Versuch denselben E-Mail-Nachrichtenversand, die Erstellung desselben Pull Requests oder die doppelte Abrechnung bei demselben Kunden verursachen.
  6. Observability muss auch nach Abschluss der Ausführung durch den Worker erhalten bleiben. Bei einem fehlgeschlagenen sechsstündigen Lauf sind nützliche Artefakte das Ereignisprotokoll, der letzte Checkpoint-Wert, die Trace-Zeitlinie der Toolaufrufe, der Unterschied im Arbeitsumfeld, Screenshots sowie Metadaten zum Umfeld.
  7. Die Annahmen hinter Runtime veralten mit den Änderungen von Models. Wenn Sie Models oder andere Plattformen wechseln, müssen Kontexteinstellungen, Evaluator-Muster, Toolfunktionen sowie Budgetregeln erneut getestet werden.
  8. Geheime Daten sollten außerhalb des Harness sowie außerhalb des Model-Kontexts bleiben. Der Agent erhält Funktionalitäten der Tools, nicht rohe Zugangsdaten.

Dieser Runtime kann eine Ausführung am Laufen halten und wiederherstellbar machen, doch Langlebigkeit beweist nicht automatisch, dass die Arbeit korrekt ist.
