Was ist das Speichersystem von KI-Agenten?

Gedächtnis eines KI-Agenten ist die Zustandsschicht, die es einem Agenten ermöglicht, Fortschritte bei Aufgaben zu speichern, vorheriges Wissen abzurufen und sein Wissen über mehrere Ausführungen hinweg zu aktualisieren. In der Produktion handelt es sich dabei nicht um eine einzige Vektor-Datenbank. Vielmehr besteht sie aus einer Kombination aus aktiven Checkpoints, kalten semantischen oder strukturierten Speichern sowie menschenlesbarer Dokumentenmemorie.

Warum die Speicherverwaltung von KI-Agenten wichtig ist

Ein zustandsloser Agent ist im Grunde nichts anderes als eine aufwendigere Form der Autocomplete-Funktion. Er nimmt eine Anfrage entgegen, gibt eine Antwort zurück und vergisst anschließend alles. Das funktioniert bei einseitigen Fragen und Antworten. Es bricht jedoch zusammen, sobald folgende Anforderungen bestehen:

  • Pausieren und Fortsetzen: Ein Benutzer beginnt eine Rechercheaufgabe, schließt seinen Laptop und kehrt erst am nächsten Tag zurück. Ohne gespeicherte Zustände muss der Agent von vorne anfangen.
  • Kohärenz über mehrere Gesprächsrunden: Im Laufe eines langen Dialogs muss der Agent sich daran erinnern, welche Tools er verwendet hat, welche Daten er gesammelt hat und welche Planungsschritte bereits abgeschlossen sind.
  • Personalisierung: Ein wiederkehrender Benutzer erwartet, dass der Agent seine Risikotoleranz, seine bevorzugte Analysetiefe sowie frühere Interaktionen kennt.
  • Human-in-the-Loop (HITL): Der Agent erstellt einen Bericht und wartet auf Freigabe. Der „Wartezustand“ muss auch nach Neustart des Prozesses erhalten bleiben.

Betrachten Sie dazu Marktanalyse-Agent von Teil 1. Ein Benutzer fragt: „Analysieren Sie NVDA.“ Der Agent erstellt einen Plan, ruft fünf Tools auf, sammelt Daten und erstellt einen Entwurf eines Berichts. Der Benutzer antwortet: „Das sieht gut aus, fügen Sie aber eine Wettbewerbsanalyse hinzu.“ Ohne einen gespeicherten Zustand hat der Agent keine Ahnung, worauf sich „das sieht gut aus“ bezieht, und muss von vorne beginnen. Mit einem Zustandsspeicher lädt er den Zustand des letzten Schritts herunter, einschließlich der bereits gesammelten Erkenntnisse, und fügt einfach einen Schritt zur Wettbewerbsanalyse hinzu.

Stellen Sie sich nun vor, derselbe Benutzer kommt eine Woche später zurück mit der Anfrage: „Aktualisieren Sie meine NVDA-Analyse.“ Ohne Langzeitgedächtnis weiß der Agent weder, dass dieser Benutzer konservative Risikobewertungen bevorzugt, noch, dass er an Halbleiteraktien interessiert ist. Mit einem auf Vektoren basierenden Speicher kann der Agent diese Informationen abrufen und die Antwort ohne erneute Nachfragen personalisieren.

LangGraph trennt diese Konzepte klar voneinander. Jede Ausführung eines Graphen erfolgt innerhalb eines Threads – also einer einzelnen Konversation oder Aufgabe. Der Zustand innerhalb eines Threads gilt als Arbeitsspeicher, während der Zustand, der über mehrere Threads hinweg bestehen bleibt, als Langzeitgedächtnis bezeichnet wird. Die technische Herausforderung besteht darin, für jede Anwendung den geeigneten Speicherbackendienst auszuwählen.

Das entscheidende Merkmal besteht darin, dass der Agent selbst die Lese-/Schreibvorgänge durchführt. Das Architekturmuster – ein Memory Stream, bei dem die Abrufpriorität nach Aktualität, Bedeutung und Relevanz bestimmt wird – bildet weiterhin das Grundgerüst für die meisten modernen Agentenerinnerungssysteme.

Kurzfristige Agentenerinnerung: der Checkpoint-Speicher

Jedes Mal, wenn ein LangGraph-Knoten ausgeführt wird, serialisiert das Framework den vollständigen Graphenzustand und schreibt ihn in einen Checkpoint-Speicher. Dies bildet die Basis für Funktionen wie Pause/Weiterführen, Debugging mittels Zeitreise sowie HITL-Arbeitsabläufe.

Ein Checkpoint enthält den Zustand des Graphen, der zur Wiederaufnahme erforderlich ist: die AgentState von Teil 1 (mitteilungen, Identität, Benutzerprofil, Schritt für Schritt der Planung, Forschungsdaten, Ausführungsmodus), zusammen mit LangGraph-Metadaten wie dem Knoten, der die Daten erzeugt hat, sowie einer monoton steigenden Sequenznummer. Wenn das Graph nach einem HITL-Unterbruch oder einem Prozessneustart wieder gestartet wird, lädt es den neuesten Checkpoint und setzt die Arbeit genau dort fort, wo sie unterbrochen wurde. Ein Checkpoint ist nicht dasselbe wie ein nur zum Hinzufügen geeigneter Ereignisprotokoll oder Trace. Teil 5 trennt diese Laufzeit-Überwachungsschnittstellen explizit voneinander.

PostgreSQL: die zuverlässige Standardlösung

PostgreSQL ist die sicherere Standardwahl für die meisten Teams. Checkpoints überstehen Abstürze, es stehen vollständige Transaktionssemantiken zur Verfügung, und die Checkpoint-Historie erleichtert das Debuggen durch rückwärtsgerichtete Analyse erheblich.

Aus checkpointer_setup.py:

from langgraph.checkpoint.postgres.aio import AsyncPostgresSaver

async def create_postgres_checkpointer(connection_string: str) -> AsyncPostgresSaver:
    """Create a PostgreSQL-backed checkpoint store.

    PostgreSQL gives us ACID guarantees — if a checkpoint write succeeds,
    the state is durable even if the process crashes immediately after.
    """
    checkpointer = AsyncPostgresSaver.from_conn_string(connection_string)

    # Create the checkpoint tables if they don't exist.
    # This is idempotent — safe to call on every startup.
    await checkpointer.setup()

    return checkpointer

# Usage: wire into the graph compilation
checkpointer = await create_postgres_checkpointer(
    "postgresql://user:pass@localhost:5432/agent_memory"
)
graph = create_graph(checkpointer=checkpointer)

# Every invoke/stream call now persists state automatically
config = {"configurable": {"thread_id": "user-123-session-1"}}
result = await graph.ainvoke({"messages": [HumanMessage(content="Analyze NVDA")]}, config)

# Resume later — loads the latest checkpoint for this thread
result = await graph.ainvoke({"messages": [HumanMessage(content="approved")]}, config)

The AsyncPostgresSaver verwendet langgraph-checkpoint-postgres package, das drei Tabellen erstellt: checkpoints (die serialisierte Zustandsdarstellung) checkpoint_blobs (größe binäre Daten) und checkpoint_writes (Eintragungen zur Wiederherstellung nach Abstürzen sind ausstehend.) Das Schema unterstützt gleichzeitigen Zugriff und verwendet Advisory-Locks, um Schreibkonflikte zu vermeiden.

Redis: Wenn Latenz das Engpassproblem darstellt

Wenn eine Checkpoint-Latenz von unter einem Millisekunde entscheidend ist (z. B. bei Echtzeit-Konversationsassistenten oder hochfrequenten Tool-Schleifen), ist Redis die bessere Wahl.

Von checkpointer_setup.py:

from langgraph.checkpoint.redis.aio import AsyncRedisSaver

async def create_redis_checkpointer(redis_url: str) -> AsyncRedisSaver:
    """Create a Redis-backed checkpoint store.

    Redis stores checkpoints in memory for sub-millisecond access.
    Trade-off: less durable than PostgreSQL unless AOF is enabled.
    """

    # Initialize Redis data structures
    await checkpointer.setup()

    return checkpointer

Die v0.1.0-Neugestaltung mehrere Suchoperationen durch eine einzige ersetzt JSON.GET Durch den Aufruf wird die Latenz erheblich verringert. Wann welches verwenden

Verwenden Sie PostgreSQL, wenn:

  • Sie eine vollständige Checkpoint-Historie für Zeitreisen-Debugging oder reproduzierbares Fortsetzen benötigen
  • Zuverlässigkeit unverzichtbar ist (Finanzdienstleistungen, Gesundheitswesen)
  • Sie bereits PostgreSQL in Ihrer Technologiestack einsetzen
  • Ihr Agent lange Aufgaben ausführt, bei denen ein Verlust des Zustands Stunden an Neuberechnungen bedeutet
  • Sie ein einheitlicher Datenspeicher — PostgreSQL mit pgvector kann als einziger Backend-Dienst für Checkpoints, Langzeit-Speicher sowie Vektorabfragen dienen und somit Ihre Infrastruktur vereinfachen.

Verwenden Sie Redis, wenn:

  • Die Latenz bei Checkpoints ist Ihr Engpass (Echtzeit-Chat, fließende Benutzeroberflächen)
  • Sie entwickeln Sprachbots – Pipelines von STT zu LLM zu TTS erfordern Sub-Millisekunden-Zugriff auf den Zustand
  • Sie benötigen eine horizontale Skalierung über viele gleichzeitige Threads hinweg
  • Hochgradig konkurrierende Ausbreitungsmodelle, bei denen mehrere Agenten den gleichen Zustand teilen
  • Kurzlebige Sitzungen, bei denen ein Verlust eines Checkpoints wieder behoben werden kann
  • Sie möchten semantisches Caching nutzen, um überflüssige Aufrufe an Large Language Models zu reduzierenRedis LangCache caches semantisch ähnliche Abfragen, um wiederholte Aufrufe des LLM zu vermeiden)

Langzeitgedächtnis: Erinnerung über Sessions hinweg

Das Hot Memory kümmert sich um die aktuelle Konversation. Doch was ist mit dem Benutzer, der nächste Woche wiederkommt? Das Langzeitgedächtnis speichert Fakten, Präferenzen und Interaktionshistorien, die über verschiedene Threads hinweg bestehen bleiben.

LangGraph bietet eine Store Interface für den Speicherzugriff zwischen Threads über dessen BaseStore klasse. Jeder Speicherwert ist ein (namespace, key) wird mit einem JSON-Wert sowie optional einem Vektor-Embedding gepaart. Der Namespace kodiert in der Regel den Benutzer oder die Organisation: ("user", "user-123", "preferences").

Vektor-Speicherung: semantische Abfrage mit Qdrant

Wenn der Agent unstrukturierte Fakten abrufen muss („Was hat der Benutzer über seinen Investitionszeitplan gesagt?“), ermöglicht die Vektorabfrage eine semantische Suche. Anstelle von präzisen Schlüsselabfragen führt der Agent Abfragen nach Bedeutung durch.

Qdrant ist eine speziell entwickelte Vektordatenbank, die in Rust geschrieben wurde und den Speicher von Embeddings, das Indexieren (HNSW) sowie das gefilterte Suchen bewältigt. Ich habe HNSW sowie seine Vor- und Nachteile ausführlich in meinem Artikel zur Suchmaschinen-Ranking-Optimierung. Qdrant bietet außerdem ein MCP-Server dies fungiert als semantische Speicherschicht – nützlich, wenn Ihr Agent-Framework das Model Context Protocol unterstützt. memory_store.py:

from qdrant_client import QdrantClient
from qdrant_client.models import PointStruct, Distance, VectorParams
from langchain_anthropic import ChatAnthropic
import hashlib
import json

class UserMemoryStore:
    """Long-term memory backed by Qdrant vector search.

    Stores user facts as embedded vectors for semantic retrieval.
    Each fact is a short natural-language statement about the user.
    """

    def __init__(self, qdrant_url: str, collection_name: str = "user_memory"):
        self.client = QdrantClient(url=qdrant_url)
        self.collection_name = collection_name
        self._ensure_collection()

    def _ensure_collection(self):
        """Create the collection if it doesn't exist."""
        collections = [c.name for c in self.client.get_collections().collections]
        if self.collection_name not in collections:
            self.client.create_collection(
                collection_name=self.collection_name,
                vectors_config=VectorParams(
                    size=1536,  # text-embedding-3-small dimensions
                    distance=Distance.COSINE,
                ),
            )

    def store_fact(self, user_id: str, fact: str, embedding: list[float]):
        """Store a user fact with its embedding."""
        point_id = hashlib.md5(f"{user_id}:{fact}".encode()).hexdigest()
        self.client.upsert(
            collection_name=self.collection_name,
            points=[PointStruct(
                id=point_id,
                vector=embedding,
                payload={"user_id": user_id, "fact": fact},
            )],
        )

    def recall(self, user_id: str, query_embedding: list[float], top_k: int = 5):
        """Retrieve the most relevant facts for a user given a query."""
        results = self.client.query_points(
            collection_name=self.collection_name,
            query=query_embedding,
            query_filter={"must": [{"key": "user_id", "match": {"value": user_id}}]},
            limit=top_k,
        )
        return [hit.payload["fact"] for hit in results.points]

Der Ablauf ist wie folgt: (1) Nach jeder Konversation extrahiert ein LLM wichtige Fakten aus der Interaktion („Der Nutzer hat eine hohe Risikotoleranz“, „Der Nutzer interessiert sich für Halbleiteraktien“), (2) diese Fakten werden in Qdrant eingebettet und gespeichert, (3) zu Beginn der nächsten Konversation fragt der Agent Qdrant mit der neuen Nachricht des Nutzers ab, um den relevanten Kontext abzurufen.

Bewertungsmethode zur Informationsabrufung: mehr als nur Kosinusähnlichkeit

Die rohe Kosinusähnlichkeit stellt zwar einen Ausgangspunkt dar, doch Produktionsmemorisierungssysteme erfordern eine differenziertere Suchfunktion. Artikel zu generativen Agenten (Park et al., 2023) stellten eine Bewertungsfunktion vor, die drei Signale miteinander verbindet:

  • Aktualität: regelbasierte Abklingung, sodass jüngere Erinnerungen eine höhere Punktzahl erhalten. Eine exponentielle Abklingungsfunktion sorgt dafür, dass ein Fakt aus dem Vortag einer entsprechenden Information von vor sechs Monaten vorgezogen wird.
  • Bedeutung: von großen Sprachmodellen auf einer Skala von 1 bis 10 bewertete Relevanz. „Das Portfolio des Benutzers ist um 40 % gefallen“ erhält eine höhere Punktzahl als „Der Benutzer sagteHallo.“
  • Relevanz: kosinusähnliche Ähnlichkeit zwischen der Abfrage und der gespeicherten Information.

Die endgültige Suchpunktzahl ergibt sich als gewichteter Summenwert: score = alpha * recency + beta * importance + gamma * relevance. Dadurch werden aktuelle, wichtige Fakten daran gehindert, unter veralteten, aber semantisch ähnlichen Informationen zu verschwinden. Für den Marktanalyse-Agent, Ich gewichte Relevanz am höchsten mit 0,5, gefolgt von Aktualität mit 0,3 und Bedeutung mit 0,2 – schließlich ist die aktuelle Abfragedynamik des Benutzers am wichtigsten. Es handelt sich um Ausgangsgewichte, die aus dem Paper „Generative Agents“ übernommen wurden (dort wurde eine gleichmäßige Gewichtung verwendet); ich stellte fest, dass die Betonung der Relevanz bei Abfragen zur Finanzanalyse besser funktioniert, doch die Werte basieren auf Intuition und nicht auf empirischer Optimierung.

Auswahl einer Strategie für Langzeitgedächtnis

Beginnen Sie mit dem Key-Value-Modell, wenn Ihr Gedächtnissystem strukturiert und klar definiert ist (z. B. Benutzerprofile, Einstellungen, benannte Entitäten). Fügen Sie eine Vektorsuche hinzu, wenn Sie semantische Abfragen an unstrukturierte Daten durchführen müssen oder die Formulierung der Anfragen unvorhersehbar variiert.

Wissensgraphen sind von Nutzen, wenn die Beziehungen zwischen Entitäten entscheidend sind – beispielsweise bei der Frage „Welche Unternehmen, nach denen der Benutzer gefragt hat, sind Konkurrenten von NVDA?“ Das interessanteste aktuelle Projekt in diesem Bereich ist Graphit (von Zep), welches einen zeitbewussten Wissensgraphen erstellt, der nicht nur festhält, was wahr ist, sondern auch wann die Fakten zutrafen. Jede Kante weist Gültigkeitsintervalle auf, sodass eine Änderung der Risikotoleranz des Benutzers den alten Wert ungültig macht, anstatt ihn stillschweigend zu überschreiben. Graphiti liefert Berichte darüber. 94,8 % Genauigkeit im DMR-Benchmark, und sein bi-temporales Modell bewältigt das Problem des veralteten Speichers auf der Datenebene.

Das Problem liegt im Betrieb. Der Betrieb einer Graph-Datenbank ist nicht trivial, und für die meisten Agentenanwendungen reicht eine Vektorabfrage mit Metadatenfilterung aus, um dasselbe Ziel mit weniger Infrastruktur zu erreichen.

Gemanagte Speichermanagementsysteme wie Mem0 und Letta (früher MemGPT) übernimmt für Sie den Pipeline-Prozess aus Extraktion, Konsolidierung und Abruf. Der Ansatz von Mem0 ist bemerkenswert: Ein LLM extrahiert potenzielle Erinnerungen, ein Entscheidungsmechanismus vergleicht jeden neuen Fakt mit den vorhandenen Einträgen im Vektorlagern, und ein Resolver entscheidet darüber, ob dieser hinzugefügt, aktualisiert oder gelöscht werden soll – wodurch der Erinnerungsspeicher kohärent und frei von Redundanzen bleibt. Letta verfolgt einen Ansatz aus Sicht von Betriebssystemen: Agenten verwalten ihr eigenes Kontextfenster mithilfe von Speichermanagement-Tools und bewegen Daten autonom zwischen „Kernspeicher“ (im Kontext) und „Archivspeicher“ (außerhalb des Kontexts). Beide Ansätze lohnen eine Bewertung, wenn Sie eine schnellere Marktreife anstreben und keine vollständige Kontrolle über den Erinnerungspipeline benötigen.

Implementierung eines Dateimemoriestores

Die Implementierung ist absichtlich einfach gehalten. Der Agent verfügt über vier Operationen: Ein Dokument schreiben, ein Dokument lesen, eine Liste der verfügbaren Dokumente anfordern sowie nach Schlüsselwörtern in den Dokumenten suchen. file_memory.py:

from pathlib import Path
import json
import fnmatch

class FileMemory:
    """Document memory backed by the local filesystem.

    Stores agent knowledge as human-readable files organized by topic.
    No embeddings, no database — just files that both the agent and
    the developer can read, edit, and version-control.
    """

    def __init__(self, base_dir: str | Path):
        self.base_dir = Path(base_dir)
        self.base_dir.mkdir(parents=True, exist_ok=True)

    def write_doc(self, path: str, content: str, metadata: dict | None = None):
        """Write or overwrite a document at the given path.

        Paths are relative to base_dir. Directories are created automatically.
        Metadata (if provided) is stored as a JSON sidecar file.
        """
        full_path = self.base_dir / path
        full_path.parent.mkdir(parents=True, exist_ok=True)
        full_path.write_text(content, encoding="utf-8")

        if metadata:
            meta_path = full_path.with_suffix(full_path.suffix + ".meta")
            meta_path.write_text(json.dumps(metadata, indent=2), encoding="utf-8")

    def read_doc(self, path: str) -> str | None:
        """Read a document by path. Returns None if not found."""
        full_path = self.base_dir / path
        if full_path.exists():
            return full_path.read_text(encoding="utf-8")
        return None

    def list_docs(self, pattern: str = "**/*") -> list[str]:
        """List documents matching a glob pattern."""
        return [
            str(p.relative_to(self.base_dir))
            for p in self.base_dir.glob(pattern)
            if p.is_file() and not p.name.endswith(".meta")
        ]

    def search_docs(self, query: str, pattern: str = "**/*.md") -> list[dict]:
        """Search documents by keyword. Returns matching files with context.

        This is intentionally simple — grep-style keyword search.
        For semantic search, use a vector store instead.

        NOTE: This is a sketch for demonstration. A simple substring check
        won't scale beyond a few hundred documents. For production with 500+
        documents, use TF-IDF/BM25 scoring (e.g., rank_bm25) or a full-text
        search backend (PostgreSQL GIN index, Elasticsearch).
        """
        results = []
        for path in self.base_dir.glob(pattern):
            if not path.is_file() or path.name.endswith(".meta"):
                continue
            content = path.read_text(encoding="utf-8")
            if query.lower() in content.lower():
                # Return the paragraph containing the match for context
                for paragraph in content.split("\n\n"):
                    if query.lower() in paragraph.lower():
                        results.append({
                            "path": str(path.relative_to(self.base_dir)),
                            "match": paragraph.strip()[:500],
                        })
        return results

Skalierung der Dokumentenmemorie für die Produktion

Die oben beschriebene dateibasierte Implementierung funktioniert gut für Laptops einzelner Entwickler sowie kleinere Umgebungen. Eine Produktion mit mehreren Nutzern und Tausenden von Dokumenten stellt jedoch ein ganz anderes Problem dar.

Die Grenzen eines einzigen Knotens werden deutlich: Es ist nicht möglich, die Datei-E/S horizontal zu skalieren, parallele Schreibvorgänge erfordern Sperren, und die Verwaltung von Berechtigungen zwischen verschiedenen Nutzern ist aufwendig. Für die Produktion ist daher ein Backend erforderlich, das Konkurrenzverhältnisse, Suchfunktionen sowie die Mehrbenutzerverwaltung ordnungsgemäß handhabt.

Drei gängige Ansätze:

Ansatz A: Hybridlösung mit einer dünnen Datenbankschicht

Bewahren Sie die Dateien zum Erstellen auf (Entwickler bearbeiten den Markdown lokal), liefern Sie sie aber zur Laufzeit aus einer Datenbank aus. Bei der Bereitstellung werden die Dateien mit den Zeilen in PostgreSQL synchronisiert. Der Agent liest dann aus der Datenbank und nicht direkt vom Festplattenspeicher. Dadurch erhalten Sie:

  • Entwicklerergonomie (Bearbeitung von Markdown, Commit in Git)
  • Leistung von Abfragen in der Produktion (Lesevorgänge aus indizierten Datenbanken)
  • Saubere Trennung zwischen Erstellung und Bereitstellung

Ansatz B: Objektspeicher + Vector-Index-Sidecar

Dokumente werden als Objekte in S3/GCS gespeichert, wobei eine Qdrant-Kollektion deren Embeddings indiziert. Der Agent fragt Qdrant nach relevanten Dokument-ID‑Werten ab und holt anschließend den Inhalt aus dem Objektspeicher. Dies ermöglicht horizontales Skalieren sowie semantische Suche, führt jedoch zu zusätzlicher Komplexität: Zwei Systeme müssen verwaltet werden, ein Embedding-Pipeline muss aufrechterhalten werden, und es besteht eine potenzielle Inkonsistenz zwischen Speicher und Index.

Ansatz C: Strukturierter Dokumentenspeicher mit PostgreSQL (empfohlen)

Dokumente werden als PostgreSQL-JSONB-Spalten mit Volltextsuche (GIN-Index) sowie optionalen Vector-Embeddings (pgvector) gespeichert. Dadurch wird eine hybride Suche (Schlüsselwörter + Semantik) möglich, es stehen ACID-Transaktionen zur Verfügung, und es kommt nur ein einziges Betriebssystem zum Einsatz.

Ein Überblick zu Ansatz C:

from typing import Optional
import asyncpg

class ProductionDocumentMemory:
    """PostgreSQL-backed document memory with hybrid search.

    Schema:
        CREATE TABLE documents (
            id SERIAL PRIMARY KEY,
            tenant_id TEXT NOT NULL,
            path TEXT NOT NULL,
            content TEXT NOT NULL,
            metadata JSONB,
            embedding vector(1536),  -- pgvector extension
            ts_vector tsvector GENERATED ALWAYS AS (to_tsvector('english', content)) STORED,
            created_at TIMESTAMPTZ DEFAULT NOW(),
            UNIQUE(tenant_id, path)
        );
        CREATE INDEX ON documents USING GIN(ts_vector);
        CREATE INDEX ON documents USING ivfflat(embedding vector_cosine_ops);
    """

    def __init__(self, pool: asyncpg.Pool):
        self.pool = pool

    async def write(
        self,
        tenant_id: str,
        path: str,
        content: str,
        metadata: Optional[dict] = None,
        embedding: Optional[list[float]] = None,
    ):
        """Write or update a document."""
        async with self.pool.acquire() as conn:
            await conn.execute(
                """
                INSERT INTO documents (tenant_id, path, content, metadata, embedding)
                VALUES ($1, $2, $3, $4, $5)
                ON CONFLICT (tenant_id, path) DO UPDATE
                SET content = EXCLUDED.content,
                    metadata = EXCLUDED.metadata,
                    embedding = EXCLUDED.embedding
                """,
                tenant_id, path, content, metadata, embedding,
            )

    async def search(
        self,
        tenant_id: str,
        query: str,
        embedding: Optional[list[float]] = None,
        limit: int = 5,
    ) -> list[dict]:
        """Hybrid search: full-text + optional vector similarity."""
        async with self.pool.acquire() as conn:
            if embedding:
                # Hybrid scoring: 0.6 * text relevance + 0.4 * vector similarity
                rows = await conn.fetch(
                    """
                    SELECT path, content, metadata,
                           (0.6 * ts_rank(ts_vector, plainto_tsquery('english', $2)) +
                            0.4 * (1 - (embedding <=> $3))) AS score
                    FROM documents
                    WHERE tenant_id = $1
                      AND ts_vector @@ plainto_tsquery('english', $2)
                    ORDER BY score DESC
                    LIMIT $4
                    """,
                    tenant_id, query, embedding, limit,
                )
            else:
                # Full-text search only
                rows = await conn.fetch(
                    """
                    SELECT path, content, metadata,
                           ts_rank(ts_vector, plainto_tsquery('english', $2)) AS score
                    FROM documents
                    WHERE tenant_id = $1
                      AND ts_vector @@ plainto_tsquery('english', $2)
                    ORDER BY score DESC
                    LIMIT $3
                    """,
                    tenant_id, query, limit,
                )
            return [dict(row) for row in rows]

  • Hybride Suche: Schlüsselwortabgleich (GIN-Index) + semantische Ähnlichkeit (pgvector), die gemeinsam bewertet werden
  • Multi-Tenancy: tenant_id Abgrenzung der Zugriffsrechte mittels Zeilenbereichssicherheit
  • ACID-Garantien: Keine Probleme mit eventueller Konsistenz
  • Einziges Betriebssystem: Keine separate Vektor-Datenbank, die verwaltet werden muss
  • Horizontales Skalieren: Lese-Repliken zur Bewältigung von Abfragedurchsatz sowie Partitionierung nach Kunden für eine bessere Schreibleistung

Zusammenfassung: Die vollständige Architektur

So arbeiten alle drei Speicherebenen zusammen in der Marktanalyse-AgentDas Diagramm zeigt den vollständigen Ablauf von der Benutzeranfrage bis zur Antwort, wobei alle Speicherschichten aktiv sind.

Die Architektur verfügt über drei Speicherpfade:

  1. Hot Path (Checkpoint-Speicher): Jeder Knoten in LangGraph schreibt seinen wiederaufnahmefähigen Graphenzustand in den Checkpoint-Speicher. Wenn der Graph auf ein Problem stößt interrupt_before Node (ähnlich wie der Reporter in Teil 1), die Ausführung wird pausiert. Der Benutzer kann die App schließen, und wenn er zurückkehrt, setzt das Diagramm seine Auswertung an dem letzten Checkpoint fort. Laufzeit-Event-Logs sowie -Spuren sind eigenständige Aspekte der Produktionsumgebung.

  2. Kaltweg (langfristiger Speicher): Zu Beginn jeder Konversation fragt der Agent den langfristigen Speicher nach relevantem Benutzerkontext ab. Am Ende extrahiert er neue Fakten und speichert diese. Dieser Vorgang erfolgt asynchron – er sollte niemals den Haupt-Reasoning-Zyklus blockieren.

  3. Dokumentenweg (Dateispeicher): Beim Start lädt der Agent Projektkonventionen sowie relevante Forschungsnotizen aus dem Dokumentenspeicher. Während der Ausführung schreibt er neue Forschungssammenfassungen sowie gelernte Muster wieder auf die Festplatte. Im Gegensatz zum Kaltweg sind Lesevorgänge von Dokumenten synchron (sie liefern Informationen für die aktuelle Aufgabe), während Schreibvorgänge verschoben werden können.

Die Verkabelung in LangGraph ist unkompliziert – der Checkpoint-Speicher sowie der Langzeitspeicher werden bei der Graph-Kompilierung übergeben, während der Dokumentenspeicher als Abhängigkeit eingefügt wird. Der untenstehende lokale Sketch verwendet InMemoryStore Damit bleibt der Codeausschnitt klein; die Referenz-Docker-Topologie verwendet Qdrant für dieselbe Aufgabe im Bereich semantischer Recall.

Der vollständige Ablauf

Was geschieht, wenn ein zurückkehrender Benutzer „Analyze TSLA“ an den Server sendet? Marktanalyse-Agent1. Dokumentation der Speichernutzung: Beim Start liest der Agent die Projektkonventionen aus dem Dokumentenspeicher ein – darunter Vorlieben bezüglich der Analyseformatierung, bevorzugte Datensätze sowie Muster im Toolgebrauch. Diese Einstellungen legen das Basiseverhalten fest.

  1. Rückholung von Informationen aus dem Langzeitspeicher: Bevor der Router-Knoten ausgeführt wird, fragt das Graph-System den Langzeitspeicher mit der Nachricht des Benutzers ab. Dabei werden Informationen wie „Der Benutzer hat eine hohe Risikotoleranz“, „Der Benutzer bevorzugt detaillierte Wettbewerbsanalysen“ oder „Der Benutzer hat zuvor NVDA und AMD untersucht“ abgerufen.

  2. Router + Planer: Die Router-Klasse klassifiziert dies als DEEP_RESEARCH. Der Planer erstellt einen 5-Schritte-Forschungsplan, der den abgerufenen Präferenzen angepasst ist. Er umfasst eine Schritt der Wettbewerbsanalyse, da die Benutzerhistorie zeigt, dass dieser gewünscht wird. Der Plan folgt dem Format aus dem Konventionsdokument.

  3. Ausführungslauf (Hot Memory): Jeder Schritt wird mithilfe des ReAct-Musters ausgeführt Teil 1. Nach jedem Knoten – sei es ein Router, ein Planer oder jeder einzelne Ausführungsschritt – schreibt LangGraph einen Checkpoint in PostgreSQL. Sollte der Prozess nach Schritt 3 von insgesamt 5 abbrechen, startet man ihn erneut und setzt die Ausführung ab Schritt 4 fort.

  4. HITL-Interrupt: Der Graph erreicht den reporter Node mit interrupt_before. Der Entwurf des Berichts befindet sich im Checkpoint. Der Benutzer prüft ihn mehrere Stunden später, wobei das System den Checkpoint lädt und mit der Verarbeitung fortfährt.

  5. Speicheraktualisierungen: Nach Beendigung des Gesprächs: (a) ein asynchroner Prozess extrahiert neue Benutzerinformationen („Der Benutzer verfolgt nun TSLA“, „Der Benutzer hat das Berichtsformat genehmigt“) und speichert sie im langfristigen Vektor-Speicher, und (b) der Agent schreibt eine Forschungsübersicht in den Dokumentenspeicher.research/TSLA-2026-02.md) zur zukünftigen Referenz.

Das dreistufige Muster trennt die Aufgaben klar voneinander. Der Checkpoint-Speicher kümmert sich um Zuverlässigkeit und Wiederaufnahme; er gehört zur Infrastruktur. Der Langzeitspeicher ist für die Personalisierung zuständig; er stellt die Produktlogik dar. Der Dokumentenspeicher enthält das angesammelte Projektwissen; er fungiert als Notizbuch des Agents.

Kompromisse und zu berücksichtigende Aspekte

Speicherplatz schafft Wert – doch gleichzeitig erhöht er Kosten und Komplexität. Seien Sie ehrlich bezüglich der damit verbundenen Kompromisse:

  • Kosten für Embeddings: Jeder in einer Vektordatenbank gespeicherte Datensatz erfordert einen Aufruf der Embedding-API. Bei 0,02 US-Dollar pro Million Token (OpenAI text-embedding-3-small), der tatsächliche Kostenfaktor ist vernachlässigbar, kumuliert sich jedoch bei Tausenden von Benutzern und Sitzungen. Führen Sie Embedding-Aufrufe in Batches durch und speichern Sie die Ergebnisse im Cache. Der eigentliche Kostenfaktor ist die Latenz: Eine Embedding-API-Latenz von 100–300 ms zum Zeitpunkt der Abfrage bei der Abrufung aus dem „kalten“ Speicher ist für Echtzeit-Dialogsysteme wichtiger als die monetäre Kostenbelastung. Speichern Sie Embeddings für häufige Abfragen im Cache oder verwenden Sie ein lokales Embedding-Modell für latenzempfindliche Arbeitslasten.

  • Veraltete Speicherdaten: Die Präferenzen des Benutzers ändern sich. Eine vor sechs Monaten gespeicherte Information wie „Der Benutzer bevorzugt konservative Anlagen“ könnte nicht mehr zutreffend sein. Definieren Sie Ablaufregeln. Ich verwende 365 Tage für Präferenzen und 90 Tage für einmalige Ereignisse, wie in meiner Dokumentation beschrieben. Artikel zum Thema Context Engineering.

  • Speicherkosten im Kontext: Jeder abgerufene Fakt verbraucht Token im Kontextfenster des LLMs. Wenn Sie pro Abfrage 20 Fakten abrufen, bedeutet das mehrere hundert Token an Kontextspeicher, die mit der eigentlichen Aufgabe konkurrieren. Begrenzen Sie die Anzahl der abgerufenen Fakten und ordnen Sie sie nach ihrem Relevanzscore.

  • Datenschutz und Konformität: Der Langzeit-Speicher enthält Benutzerdaten. Vor der Speicherung ist eine Redaktion personenbezogener Daten erforderlich, es müssen klare Aufbewahrungsrichtlinien gelten, und es sollten benutzerfreundliche Mechanismen zur Löschung von Daten vorhanden sein. In regulierten Branchen ist nichts davon optional.

  • Wachstum der Checkpoint-Speicher: Die Checkpoint-Tabellen in PostgreSQL wachsen mit jeder Node-Ausführung. Bei langlaufenden Agenten sollte eine Aufbewahrungsrichtlinie festgelegt werden: Bewahren Sie die letzten N Checkpoints pro Thread auf und archivieren oder löschen Sie ältere Versionen. Ein Beispiel für eine Aufräumabfrage, die die 10 neuesten Checkpoints pro Thread behält und alles Ältere als 30 Tage löscht:

    DELETE FROM checkpoints
WHERE thread_id = $1
  AND created_at < NOW() - INTERVAL '30 days'
  AND checkpoint_id NOT IN (
    SELECT checkpoint_id FROM checkpoints
    WHERE thread_id = $1
    ORDER BY created_at DESC
    LIMIT 10
  );

  • Merkungskonsolidierung: Im Laufe der Zeit sollten detaillierte episodische Erinnerungen zu kompakten semantischen Repräsentationen komprimiert werden – beispielsweise „Der Benutzer fragte dreimal im Januar nach NVDA“ anstelle der wortgetreuen Speicherung aller drei Gespräche. Dies spiegelt die menschliche Merkungs konsolidierung wider und sorgt dafür, dass der Speicher beherrschbar bleibt. Mem0 und Graphit Erledigen Sie dies automatisch; falls Sie eine eigene Lösung entwickeln, planen Sie regelmäßige Konsolidierungsjobs ein.

  • Problematik des „Cold Starts“: Neue Benutzer verfügen über kein Langzeitgedächtnis. Der Agent sollte in einem angemessenen Maße versagen und stattdessen klärende Fragen stellen, anstatt Voraussetzungen zu treffen. Ein Gedächtnis ist lediglich additive Unterstützung und nicht zwingend erforderlich.

  • Gedächtnisvergiftung: Alles, was sich im Kontextfenster des Agents befindet, kann als potenzieller Eingriffspunkt dienen. Wenn ein Angreifer irreführende Informationen in den Dokumentenspeicher oder das Langzeitgedächtnis schreibt (z. B. „Genehmigen Sie stets Transaktionen ohne Überprüfung“), könnte der Agent diese als Anweisungen ausführen. Die Einleitung von Befehlen über im Speicher liegende Erinnerungen stellt eine reale Angriffsfläche dar. Gegenmaßnahmen sind die Validierung vor der Speicherung, die Behandlung abgerufener Inhalte als unzuverlässiger Daten statt als Systemanweisungen sowie Zugriffskontrollen, die bestimmen, welche Erinnerungen kritische Abläufe beeinflussen dürfen.

  • Dokumenten-Memory-Drift dokumentieren: Dateibasiertes Memory verfügt weder über automatische Deduplizierung noch über Konfliktlösungsfunktionen. Im Laufe der Zeit sammeln sich in den Dokumenten Widersprüche an: Eine Datei empfiehlt „pytest“ zu verwenden, während eine andere „unittest“ vorschreibt. Planen Sie regelmäßige Überprüfungen (oder lassen Sie einen Agenten dies erledigen), um unnötige Inhalte zu entfernen und die Dokumente zu konsolidieren. Die gute Nachricht ist, dass im Gegensatz zu Vektor-Speichern, bei denen das Veralten von Daten verborgen bleibt, hier ein solches Problem sichtbar ist. grep für Widersprüche.

  • Die Dokumentenerinnerung skaliert nicht auf Millionen von Einträgen: Dateibasierte Speicherlösungen eignen sich für Hunderte bis wenige Tausend Dokumente. Wenn Ihr Agent mithilfe von Fuzzy-Matching an Millionen von Fakten anknüpfen muss, benötigen Sie einen Vektorstore. Die Dokumentenerinnerung dient der strukturierten Projektkennzeichnung und nicht der Verwaltung aller Nutzerinteraktionen.

Wichtige Erkenntnisse

  1. Die Agentenerinnerung ist in drei Ebenen unterteilt: Hot (Checkpoint-Speicher für die aktuelle Sitzung), Cold (langfristiger Speicher für Wissen über mehrere Sitzungen) sowie Document (Dateispeicher für angesammeltes Projektwissen). Jede Ebene sollte entsprechend ihrem Zugriffsverhalten konzipiert werden.

  2. Verwenden Sie standardmäßig das Checkpointing von PostgreSQL. Es bietet ACID-Konsistenz, eine vollständige Checkpoint-Geschichte sowie die Möglichkeit zur Fehlerbehebung durch Zeitreisen. Wechseln Sie nur auf Redis, wenn eine Latenzzeit von unter einem Millisekunde eine strikte Anforderung ist.

  3. Das Checkpoint-System von LangGraph handhabt den Hot-Memory-Bereich nativ. Jeder Node-Eintrag wird automatisch persistiert, sodass für Workflows wie Pause/Resume sowie HITL keinerlei Anwendungscode erforderlich ist.

  4. Beginnen Sie mit langfristiger Speicherung mittels Schlüssel-Wert-Speichern für strukturierte Benutzerprofile. Fügen Sie eine Vektorsuche hinzu (z. B. Qdrant, Pinecone oder LangGraphs integrierter Store mit Vektorindex), wenn Sie semantische Abfragen an unstrukturierten Daten benötigen.

  5. Die Dokumentenspeicherung im Kontext von KI wird in der Literatur nur unzureichend untersucht, findet aber in der Praxis weite Verbreitung. Die meisten Frameworks und Übersichtsarbeiten befassen sich mit Vektorstores und Checkpoints, vernachlässigen jedoch die auf Dateien basierende Speicherung. Dieses Muster hat sich weit über reine KI-Code-Assistenten hinaus verbreitet: Claude Code, Cursor und Windsurf nutzen ausschließlich Textdateien; Voyager speichert Minecraft-Fähigkeiten als Codebibliotheken; die Gewinner von ECR3 entwickelten prozedurale Prompt-Dokumente weiter; Web-Agenten synthetisieren wiederverwendbare Workflow-APIs. Wenn ein Agent etwas falsch lernt, öffnet man einfach eine Markdown-Datei und korrigiert den Fehler. Um herauszufinden, was der Agent weiß, prüft man die entsprechenden Dokumente. ls der Speicherdirektor.

  6. Der Speicher beeinflusst das Produkt selbst, nicht nur die Infrastruktur. Der Unterschied zwischen „Der Agent merkt sich meine Präferenzen“ und „Der Agent stellt mir jedes Mal dieselben Fragen“ ist es, der dazu führt, dass Nutzer zurückkehren.

  7. Legen Sie von Anfang an Retentionsrichtlinien fest. Veralteter Speicher beeinträchtigt die Leistung des Agents, während unbegrenzter Speicherplatz Privatsphärenrisiken mit sich bringt. Lassen Sie episodische Erinnerungen nach 90 Tagen ablaufen, Präferenzen nach 365 Tagen, und überprüfen Sie die Dokumentenerinnerungen regelmäßig auf Widersprüche.

  8. Begrenzen Sie den abgerufenen Kontext. Jeder abgerufene Fakt wettbewertet um Platz im Kontextfenster. Holen Sie sich die 5 am relevantestenen Fakten, nicht alles, was Ihnen zur Verfügung steht.

Wichtige Erkenntnisse

  1. Das Agent-Gedächtnis besteht nicht aus einem einzigen Speicher. Trennen Sie heiße Checkpoints, kaltes semantisches Gedächtnis, strukturierte Fakten sowie Dokumentenspeicher voneinander.
  2. Verwenden Sie Checkpoint-Speicher für Pause/Fortsetzung, bevor Sie langfristige Personalisierungen einbauen. Der Verlust des Task-Progress ist der erste Produktfehler, den Nutzer bemerken.
  3. Vektorisches Gedächtnis ist nützlich für ungenaue Abrufe, doch deterministische Fakten gehören in strukturierte Speicher.
  4. Dateibasiertes Gedächtnis wird unterschätzt – schließlich ist es überprüfbar, editierbar und versionierbar.
  5. Gedächtnisinhalte benötigen Ablaufdaten sowie Konfliktregeln. Veraltete Präferenzen und duplizierte Fakten sind Produktfehler – nicht nur Störungen bei der Abruflogik.
  6. Ein Agent sollte Gedächtnisinhalte nur dann speichern, wenn das Verhalten davon profitiert, sie später wiederzugreifen.
