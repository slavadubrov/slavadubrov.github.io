Nutzung von AI-Agenten-Tools im Jahr 2026: MCP, CLI, Skills und Codeausführung

Teil 3 der Serie „Engineering the Agentic Stack“

Teil 1 abgedeckte Schlussfolgerungszyklen Teil 2 belegte Speichermenge. Dieser Artikel befasst sich mit der Nutzung von AI-Agenten-Tools: Wie Agenten mit der Welt interagieren und warum die Tool-Design-Strategie wichtiger ist, als die meisten Teams erkennen.

Die Entwicklung bei den Tools änderte sich in den Jahren 2025–2026. MCP wurde zum de facto Standard für externe Dienste, doch Produktionsteams stoßen weiterhin auf seine Sicherheitslücken sowie den hohen Token-Verbrauch. Eine weitere zunehmend verbreitete Alternative sind Agenten, die Code schreiben und ausführen anstelle dessen, JSON-definierte Tools aufzurufen. Anthropic erzielte bei einem repräsentativen Workflow eine 98,7-prozentige Reduzierung der benötigten Tokens, und die im Paper „CodeAct“ dargestellten Ergebnisse zeigen 20 Prozent höhere Erfolgsraten bei Aufgaben.

In diesem Artikel werden JSON-basierte Toolaufrufe, MCP, Skills, CLI-Tools sowie Codeausführung miteinander verglichen und anschließend den Designprinzipien der Agent-Computer-Interface (ACI), die ich anwende, zugeordnet. Marktanalyse-Agent.

Zusammenfassung: Es gibt fünf Modalitäten für Tools von KI-Agenten, wobei jede ihre eigenen Vorteile aufweist. Der praktische Stack für das Jahr 2026 besteht aus Fähigkeiten zur Erzielung von Expertise, der CLI als Standard, MCP für externe Dienste sowie Codeausführung zur Orchestrierung mehrerer Schritte. Die Entwurfsprinzipien von ACI aus SWE-agent gelten für alle diese Ansätze.

Fünf Arten und Weisen, wie KI-Agenten Tools nutzen

In Teil 1, Ich habe gezeigt, wie die Reasoning-Läufe von KI-Agenten entscheiden, wie sie denken sollen. Die Tool-Modalitäten bestimmen hingegen wie sie handeln sollen. Derzeit gibt es fünf unterschiedliche Ansätze, von denen jeder andere Abwägungen hinsichtlich Tokenkosten, Flexibilität und Sicherheit aufweist.

1. Aufruf von JSON-Werkzeugen – Die Grundlage

Das ursprüngliche Muster: Sie definieren die Schemata der Werkzeuge als JSON, das LLM generiert strukturierte Funktionsaufrufe, und Ihr Code führt diese aus. Dieses Verfahren ist allgemein bekannt und funktioniert gut für kleine Werkzeugsets.

# Traditional tool definition — each tool consumes ~550-1,400 tokens (Apideck benchmark)
tools = [
    {
        "name": "get_stock_price",
        "description": "Get the current stock price for a ticker symbol",
        "input_schema": {
            "type": "object",
            "properties": {
                "ticker": {"type": "string", "description": "Stock ticker (e.g., NVDA)"}
            },
            "required": ["ticker"]
        }
    }
]

Bei 5–10 Tools ist das in Ordnung. Das Problem liegt bei der Skalierbarkeit: Jede Tool-Definition verursacht zusätzliche Kosten. 550–1.400 Tokens. Bei 20 Tools verbraucht man bereits 15–25 Tausend Tokens, noch bevor der Agent mit dem Reasoning beginnt.

2. MCP – Der universelle Standard (mit Einschränkungen)

Das Model Context Protocol ist der Standard, auf den sich die meisten Anbieter geeinigt haben. Im Dezember 2025 spendete Anthropic es an die Linux Foundation im Rahmen der Agentic AI Foundation (AAIF), die gemeinsam mit OpenAI und Block gegründet wurde und von Google, Microsoft sowie AWS als Platinmitglieder unterstützt wird. OpenAI fügte MCP-Unterstützung in seine Responses API hinzu. Derzeit gibt es mehr als 10.000 aktive MCP-Server sowie mehr als 97 Millionen monatliche Downloads der SDKs.

MCP hat Vorteile bei der Integration verschiedener SaaS-Lösungen unterschiedlicher Anbieter (z. B. Figma, Notion, Salesforce), bei der Unternehmenssteuerung mit Audit-Trail-Funktionen, bei Diensten ohne entsprechende CLI-Optionen sowie in Umgebungen, die eine OAuth-Orchestrierung erfordern. Für diese Anwendungsfälle ist MCP die klare Wahl.

In der Praxis ist die Situation jedoch komplizierter, als die offiziellen Zahlen vermuten lassen.

Die Sicherheitsoberfläche stellt das erste Problem dar. Der Anfälliges MCP-Projekt Erfasst 50 Schwachstellen in MCP-Servern, davon 13 als kritisch eingestuft, gemeldet von 32 Sicherheitsforschern. Zu den Angriffsklassen gehören Prompt-Injection, Fehler bei der Eingabenvalidierung, Authentifizierungsprobleme sowie Netzwerksicherheitslücken. Der erste echte bösartige MCP-Server in der Praxis erschien im September 2025: ein Paket mit dem Namen postmark-mcp Dass jede versendete E-Mail an eine Adresse des Angreifers per BCC weitergeleitet wurde, was vor der Entdeckung schätzungsweise 300–500 Organisationen betraf.

Tool Poisoning ist die Angriffsklasse, die mir am meisten Sorgen bereitet. Invariant Labs führte eine Demonstration durch. Dass solche vergifteten MCP-Tools Daten auslesen können, selbst wenn sie niemals aufgerufen werden. Schon das Lesen der Metadaten des Tools reicht aus, um den Angriff auszulösen. MCPTox-Benchmarks Durch Tests von 20 LLM-Agenten gegen 45 echte MCP-Server wurde eine Erfolgsrate bei Angriffen von bis zu 72,8 % festgestellt.

Der Token-Overhead stellt das betriebliche Problem dar. Ein Team, das MCP-Server für GitHub, Slack und Sentry betreibt (~40 Tools insgesamt), stellte fest 55.000 Tokens von Schema-Definitionen, die vor dem Aufruf durch den Benutzer injiziert werden. Ein weiterer gemeldeter Fall 143.000 von 200.000 verfügbaren Tokens (72 %) ausschließlich durch die Definitionen der Tools verbraucht.

Die Abschwächungsmaßnahmen funktionieren zwar, bestätigen aber das zugrunde liegende Problem. Bei Anthropic’s Das Tool Search Tool verringert die Schema-Overhead um 85 %., Die dynamische Tool-Ladung (3–5 relevante Tools pro Aufgabe) führt zu einer Reduktion von 96 %. Dabei handelt es sich um Workarounds für ein Protokoll, das ursprünglich ohne Berücksichtigung von Token-Budgets konzipiert wurde.

3. Fähigkeiten (SKILL.md) – Expertise statt Ausführung

Ende 2025 kam es zur Standardisierung von Agentenfähigkeiten als offenes Format (Einführung im Oktober 2025, Veröffentlichung als offener Standard im Dezember 2025). Dieser Unterschied ist wichtig: Tools bieten Funktionalitäten an (was Agenten tun können), während Fähigkeiten die Expertise darstellen (was Agenten über die Durchführung komplexer Aufgaben wissen).

Der SKILL.md-Standard definiert eine Fähigkeit als eine Markdown-Datei mit YAML-Frontmatter:

---
name: deploy
description: Deploy the application to production
argument-hint: "[environment]"
user-invocable: true
---
Deploy the application to the $0 environment (default: staging).
Steps:
1. Run the test suite
2. Build the production bundle
3. Deploy using the deploy script
4. Verify the deployment health check

Fähigkeiten nutzen progressive Disclosure: Beim Start werden lediglich die Metadaten (etwa 100 Tokens für das YAML-Frontmatter) geladen, während die vollständigen Anweisungen erst beim Aktivieren der Fähigkeit bereitgestellt werden. Im Vergleich dazu verbrauchen bei MCP etwa 40 Tools aus typischen Services ~55.000 Tokens Bevor mit der Analyse begonnen wird: Die plattformübergreifende Verbreitung Stand März 2026 umfasst Claude Code, OpenAI Codex CLI, Cursor, GitHub Copilot, Gemini CLI, Goose, Windsurf sowie Roo Code – ein klarer Trend. Victor Dibia hat diese Themen erforscht in seiner Analyse des Übergangs von Tools zu kodengesteuerten Agentenaktionen.

Wenn Fähigkeiten entscheidend sind: Übertragung von Fachwissen, komplexe mehrschrittige Verfahren, wiederkehrende Workflows wie Datenbankmigrationen oder Zahlungsanbindungen – sowie alle Aufgaben, bei denen der Agent wissen muss wie, und nicht nur was.

4. CLI/Bash – Die Renaissance von Unix

Die Unix-Terminal erwies sich als die effizienteste Token-basierte Agenten-Schnittstelle. Die Rechnung ist eindeutig: Ein Manifest für ein CLI-Tool kostet etwa 100 Tokens, während die entsprechenden MCP-Server-Schemata 50.000+ Tokens verbrauchen. Das bedeutet eine 4–32×-Unterschied gemessen mit Scalekit in 75 Benchmark-Ausführungen (32 Mal für Aufgaben wie die Ermittlung der Sprache eines Repositories sowie die Überprüfung von Lizenzen).

Modelle sind nativ in der Verwendung von CLI-Befehlen bewandert. Ihre Trainingsdaten umfassen Milliarden an Terminalinteraktionen aus Stack Overflow, GitHub-Repos und Dokumentationen. Sie verstehen diese Befehle auf nativer Ebene. git, docker, kubectl, gh, curl, und jq ohne jegliche Schema-Definitionen.

Ugo Enyiohas Leitfaden “Die Entwicklung von CLI-Tools, die KI-Agenten tatsächlich nutzen möchten” acht kodifizierte Designregeln:

  1. Eine strukturierte Ausgabe ist zwingend erforderlich — Unterstützung --json
  2. Ausgabekodes sind Teil des Kontrollflusses – verwenden Sie unterschiedliche Codes für verschiedene Fehlerarten.
  3. Befehle sollten idempotent sein.
  4. Selbst dokumentierend. --help mit realistischen Beispielen
  5. Entwurf für Komposierbarkeit — --quiet für reine Werte: Unterstützung für Standardeingabe
  6. Liefern --dry-run und --yes flags
  7. Unterstützung der Versionsinspektion
  8. Verarbeitung der Authentifizierung über Umgebungsvariablen

Die Kompromisse sind real. Die CLI verfügt nicht über die Typsicherheit von MCP, keine eingebauten Mechanismen für OAuth-Orchestrierung, keine Tool-Entdeckung sowie keinen Audit-Trail. Das Muster, auf das sich die meisten Teams einigen: Die CLI wird standardmäßig für Entwicklung und lokale Operationen verwendet, während MCP zur Integration mit externen Diensten sowie zur Unternehmenssteuerung eingesetzt wird.

5. Ausführung von Code – Der große Wandel

Dies ist der Veränderung im Agent-Tooling, die meiner Ansicht nach am bedeutendsten ist. Anstelle dass das LLM strukturierte JSON-Daten ausgibt, um vorgefertigte Funktionen nacheinander aufzurufen, erstellt der Agent ein vollständiges Python- oder Bash-Skript, das mehrere Tools aufruft, Ergebnisse mithilfe von Schleifen und Bedingungen verarbeitet und lediglich Zusammenfassungen in den Kontext des Modells zurückgibt.

Anthropic hat dies formalisiert. Programmgesteuerte Toolaufrufe (PTC) jetzt verfügbar in der Claude API. Die wissenschaftliche Grundlage ist die CodeAct-Paper (Wang et al., ICML 2024), die an 17 verschiedenen LLMs getestet haben und feststellten, dass Code-Aktionen im Vergleich zu JSON-Alternativen zu 20 % höhere Erfolgsraten bei der Aufgabenerledigung sowie 30 % weniger Schritte ermöglichten.

Die Produktionsbelege: Vercel hat ihren d0-Text-zu-SQL-Agent neu entwickelt durch das Entfernen von 80 % seiner Tools (von 15 auf 2): ExecuteCommand und ExecuteSQL). Der Erfolgsgrad der Aufgaben stieg von 80 % auf 100 %, die Ausführungszeit verringerte sich um das 3,5-fache und der Tokenverbrauch sank um 40 %. Ihre Formulierung: „Die besten Agenten sind vermutlich jene mit den wenigsten Tools.“

  • Cloudflare hat „Code Mode“ entwickelt, Es ermöglicht Agenten, stattdessen TypeScript zu verwenden, um ihre API aufzurufen, anstatt Tool-Schemata zu definieren – was den Kontextaufwand verringert. Ihre Begründung lautet: „LLMs verfügen über eine enorme Menge an echtem TypeScript in ihren Trainingsdatensätzen, doch es gibt nur eine begrenzte Anzahl künstlich erzeugter Beispiele für Tool-Aufrufe.“ Die PTC-Dokumentation von Anthropic. Herkömmliche Tools zur Kostenanalyse erfordern mehr als 20 separate Inferenzschritte – einen pro Teammitglied – wobei alle Zwischendaten über den Kontext weitergeleitet werden. Ein Workflow, der bei direkter Toolaufruf etwa 150.000 Tokens verbrauchte, wurde neu implementiert und benötigt nun nur noch rund 2.000 Tokens, was einer 98,7-%-Reduktion entspricht. Bei Ausführung von Code erstellt der Agent lediglich ein einziges Skript:
# Agent generates this code, executes in sandbox
import json
members = get_team_members("engineering")
over_budget = []
for m in members:
    expenses = get_expenses(m["id"], "Q3")
    total = sum(e["amount"] for e in expenses)
    if total > 5000:
        custom = get_custom_budget(m["id"])
        limit = custom["limit"] if custom else 5000
        if total > limit:
            over_budget.append({"name": m["name"], "spent": total, "limit": limit})
# Only this final summary returns to the LLM context
print(json.dumps(over_budget))

Das LLM sieht nur die finale JSON-Zusammenfassung und nicht die Tausende von Ausgabenpositionen, die im Sandbox-Modus verarbeitet werden. Token-Effizienz, natürliche Komposierbarkeit (Schleifen und Bedingungen sind inklusive) sowie echte Fehlerbehandlung.try/except Anstelle von natursprachlichem Reasoning bezüglich Fehler sowie durch bessere Privatsphärebedingungen – bei denen sensible Daten im Sandbox-Umfeld verbleiben – verbessern sich alle Aspekte gleichzeitig.

Szenarien, in denen der Aufruf von JSON-Tools weiterhin sinnvoll ist: einzelne, atomare Operationen, Umgebungen ohne Sandbox-Infrastruktur, kleinere Modelle mit eingeschränkter Codegenerierungsfähigkeit oder Auditanforderungen, die die Protokollierung jedes einzelnen Tool-Aufrufs erfordern.

Die Agent-Computer-Schnittstelle (ACI) für KI-Agenten-Tools

Der Begriff „Agent-Computer-Schnittstelle“ (ACI) wurde von John Yang, Carlos E. Jimenez und ihren Kollegen an der Princeton University geprägt. Paper zum SWE-Agent (NeurIPS 2024). Die Idee ist einfach: Genau wie Menschen von gut gestalteten Benutzeroberflächen (HCI) profitieren, sind LM-Agenten „eine neue Kategorie von Endnutzern mit eigenen Anforderungen und Fähigkeiten, die von speziell entwickelten Benutzeroberflächen Vorteile ziehen würden.“ Ablationsergebnisse Diese Ergebnisse wurden gespeichert. Der SWE-agent mit seiner vollständigen ACI-Implementierung erzielte auf dem SWE-bench Lite-Werkzeug 18,0 %, während eine herkömmliche Linux-Shell nur 7,3 % erreichte – das entspricht einer Verbesserung von 10,7 Prozentpunkten allein durch das Interface-Design. Allein die Linting-Funktion trug 3 Prozentpunkte dazu bei: Bei 51,7 % der von dem Agenten vorgenommenen Änderungen wurde zumindest ein Fehler bereits vor der Weiterverarbeitung vom Linter erkannt.

Anthropic hat ACI als grundlegendes Konzept in ihrer “Effektive Agenten entwickeln” Leitfaden, der dies als eines von drei Kernprinzipien auflistet: „Entwerfen Sie Ihre Agent-Computer-Schnittstelle sorgfältig durch umfassende Dokumentation und Tests der Tools.“ Ihre praktische Anleitung lautet: „Eine Faustregel ist, sich darüber Gedanken zu machen, wie viel Aufwand in Mensch-Computer-Schnittstellen gesteckt wird, und genauso viel Aufwand für die Erstellung guter Agent-Computer-Schnittstellen einzuplanen.“

Die vier Prinzipien in der Praxis

1. Aktionen sollten einfach und leicht verständlich sein. Der häufigste Fehler besteht darin, API-Endpunkte eins-zu-eins zu verpacken. Anstatt dessen list_users, list_events, create_event, implementieren schedule_event das die Verfügbarkeit prüft und den Zeitplan in einem einzigen Aufruf festlegt. Anstelle davon read_logs, implementieren search_logs das ausschließlich die relevanten Zeilen zusammen mit dem Kontext zurückgibt.

2. Aktionen sollten kompakt und effizient sein. Konsolidieren Sie wichtige Operationen in so wenigen Aktionen wie möglich. In der Marktanalyse-Agent, Ich kombiniere den Preisabruf mit grundlegenden Metriken zu einem einzigen get_stock_snapshot ein Tool anstelle von mehreren separaten Aufrufen für Preis, Volumen, Marktkapitalisierung und Kurs-Gewinn-Verhältnis.

3. Die Umgebungsfeedbacks sollten informativ, aber prägnant sein. Vermeiden Sie die Rückgabe von rohem HTML oder vollständigen API-Datenpaketen. Ersetzen Sie kryptische IDs durch semantisch aussagekräftige Namen. Anthropics Tests zeigte, dass das Hinzufügen eines response_format Eine Enum, die es den Agenten ermöglicht, entweder konservative Antworten mit etwa 72 Tokens oder ausführlichere Antworten mit rund 206 Tokens anzufragen – wobei sich die Tokenkosten um das Dreifache unterscheiden – hat die Leistung in der Praxis deutlich verbessert.

4. Schutzmechanismen sollten die Ausbreitung von Fehlern begrenzen. Die automatische Fehlererkennung hilft den Agenten, Fehler schnell zu erkennen und zu korrigieren. In SWE-Agent, ein benutzerdefinierter Dateieditor mit integrierter Linting-Funktion lehnt Syntaxfehler automatisch ab und erkennt somit 51,7 % der Bearbeitungsfehler des Agents, bevor diese sich weiter verschlimmern können. Ich wende dasselbe Prinzip im Market Analyst Agent an, indem ich die Argumente der Tools vor der Ausführung mithilfe von Pydantic-Schemata validiere:

from pydantic import BaseModel, Field, field_validator

class StockQuery(BaseModel):
    """Validated input for stock queries.

    Pydantic catches malformed tickers before the API call,
    preventing error propagation through the reasoning loop.
    """
    ticker: str = Field(description="Stock ticker symbol (e.g., NVDA)")
    period: str = Field(default="1mo", description="Time period: 1d, 5d, 1mo, 3mo, 1y")

    @field_validator("ticker")
    @classmethod
    def validate_ticker(cls, v: str) -> str:
        v = v.upper().strip()
        if not v.isalpha() or len(v) > 5:
            raise ValueError(f"Invalid ticker format: {v}")
        return v

    @field_validator("period")
    @classmethod
    def validate_period(cls, v: str) -> str:
        valid = {"1d", "5d", "1mo", "3mo", "6mo", "1y", "5y"}
        if v not in valid:
            raise ValueError(f"Invalid period: {v}. Must be one of {valid}")
        return v

Funktionsfähige Designmuster für Tools von KI-Agenten

Anthropics “Effektive Tools für Agenten entwickeln” Guide-Frames-Tools als „eine neue Art von Software, die einen Vertrag zwischen deterministischen Systemen und nicht-deterministischen Agenten widerspiegelt“. Hier sind die Muster, die sich aus der Produktionspraxis herauskristallisiert haben.

Toolbeschreibungen als Prompt-Engineering betrachten

Beschreibungen sollten folgendermaßen formuliert werden Mindestens 3 bis 4 Sätze sind erforderlich, um die Anforderungen an die Länge zu erfüllen. In technischen Dokumentationen für deutsche AI-Ingenieure wird häufig auf präzise Formulierungen geachtet, um Missverständnisse bei der Implementierung zu vermeiden. Die Übersetzung muss dabei stets den ursprünglichen Kontext und die fachliche Terminologie beibehalten. Zudem sollte die Struktur des Markdown-Textes unverändert bleiben, da sie für die korrekte Darstellung im Zielsystem entscheidend, einschließlich der Anwendungsfälle für das Tool, erforderlicher gegenüber optionellen Parametern, Ausgabeformat sowie Randfälle. Namensräume mit Präfixenasana_search, jira_search) hat „nicht unerhebliche Auswirkungen“ auf die Genauigkeit der Werkzeugauswahl. In den Experimenten von Anthropic übertrafen die von Claude optimierten Werkzeudescriptions die von Experten verfassten Versionen in den Bewertungsbenchmarks.

# Bad: vague, no context for when to use

    "name": "search",

Vermeiden Sie Identifikatoren niedriger Ebene.uuid, mime_type). Ersetzen Sie verschlüsselte IDs durch semantische Namen. Strukturieren Sie die Antwort so, dass der Agent ohne das Verarbeiten von Overhead-Daten logisch darauf schließen kann:

    response = api.get(f"/v1/quotes/{ticker}")

# Good: high-signal summary the agent can immediately reason about
def get_stock_price(ticker: str) -> dict:
    data = api.get(f"/v1/quotes/{ticker}").json()
    return {
        "ticker": ticker,
        "price": data["regularMarketPrice"],
        "change_pct": round(data["regularMarketChangePercent"], 2),
        "volume": data["regularMarketVolume"],
        "market_cap_b": round(data["marketCap"] / 1e9, 1),
        "pe_ratio": data.get("trailingPE"),

                   f"({'up' if data['regularMarketChangePercent'] > 0 else 'down'} "
                   f"{abs(data['regularMarketChangePercent']):.1f}%)"

Fehlerbehandlungsdesign für Widerstandsfähigkeit

Das in der Produktion bewährte Muster basiert auf vier Ebenen der Fehlertoleranz:

  1. Wiederholter Versuch mit exponentiellem Backoff bei vorübergehenden Fehlern
  2. Fallback-Strukturen für den Fall von Ausfällen der Anbieter
  3. Routenbasierte Klassifizierung von Fehlern – bei vorübergehenden Fehlern wird versucht, sie erneut zu verarbeiten; bei durch LLMs behobbaren Fehlern wird der Agent mit Kontext informiert; bei fehlern, die menschliches Eingreifen erfordern, erfolgt eine Eskalation
  4. Wiederherstellung aus Checkpoints zur Überlebensfähigkeit bei Abstürzen

## Anwendung dieser Muster: Der Marktanalyse-Agent

Erlauben Sie mir, zu zeigen, wie diese Prinzipien zusammenwirken. [Marktanalyse-Agent](https://github.com/slavadubrov/market-analyst-agent) von [Teil 1](/de/blog/2026/01/31/ai-agent-reasoning-loops/) ### Konsolidierung von Tools

Der Artikel Teil 1 zeigt die vereinfachte 5-Werkzeug-Oberfläche nach dieser Refaktorierung. Vor dieser Aufbereitung verfügte das ursprüngliche Design über 10+ Werkzeuge: `get_stock_price`, `get_company_metrics`, `get_market_cap`, `get_pe_ratio`, `get_volume`, und so weiter. Jeder davon war eine schmale Abdeckung für einen einzelnen API-Endpunkt. Der Agent musste abwägen, welche Kombination für jede Anfrage aufgerufen werden sollte.

Ich habe diese in 5 hochlevelige Tools zusammengefasst, wobei ich dem ACI-Prinzip kompakter, effizienter Aktionen folgte:

|                                                            | Nach (5 Tools)       | Warum                                                                               |
| ---------------------------------------------------------- | -------------------- | ----------------------------------------------------------------------------------- |
| `get_stock_price` + `get_company_metrics` + `get_pe_ratio` | `get_stock_snapshot` | Ein einziger Aufruf liefert alles, was für eine grundlegende Analyse benötigt wird. |
| `get_price_history` + `get_volume_history`                 | `get_price_history`  | In Kombination mit einer konfigurierbaren Zeitspanne sowie Indikatoren              |
| `search_news` + `search_press_releases`                    | `search_news`        | Einheitliche Suche mit Quellcode-Filterung                                          |
| `search_competitors` + `get_sector_data`                   | `search_competitors` | Gibt Wettbewerber mit relativen Metriken zurück                                     |
| `get_financials` + `get_balance_sheet` + `get_cash_flow`   | `get_financials`     | In Kombination mit dem Parameter statement_type vereint                             |

Dies verringert den Overhead des Tool-Schemas erheblich und macht die Werkzeugauswahl des Agents zuverlässiger, da weniger zweideutige Entscheidungen getroffen werden müssen.

### Strukturierte Ausgaben für Tool-Ergebnisse

Jedes Tool im Market Analyst Agent gibt eine durch Pydantic validierte Antwort zurück. Dadurch wird das Prinzip der ACI-Schutzmaßnahmen bereits an der Grenze zum Tool angewendet:

```python
class StockSnapshot(BaseModel):
    """Structured tool response — the agent never sees raw API noise."""
    ticker: str
    price: float
    change_pct: float
    volume: int
    market_cap_b: float
    pe_ratio: float | None
    summary: str  # Human-readable one-liner for direct use in reports

class NewsResult(BaseModel):
    """Each news item is pre-processed for agent consumption."""
    headline: str
    source: str
    date: str
    relevance_score: float  # Pre-ranked so the agent doesn't waste tokens sorting
    key_points: list[str]  # Extracted by the tool, not the agent

The summary Das Feld ist das Wichtigste. Es stellt dem Agenten einen sofort einsetzbaren String zur Verfügung, der ohne weitere Verarbeitung direkt in einen Bericht übernommen werden kann. key_points in NewsResult Sie werden serverseitig extrahiert, wodurch der Agent nicht gezwungen ist, Inferenz-Token für die Analyse des Artikelinhalts aufzuwenden.

Abwägungen und zu berücksichtigende Faktoren

Abgesehen von den oben genannten modalitätsbezogenen Einschränkungen gibt es noch einige allgemeine Aspekte, die die Auswahl beeinflussen:

  • Die Betriebskosten variieren je nach Dimension. Die Ausführung von Code spart Tokens, führt aber zu einer höheren Latenz beim Starten der Sandbox. MCP verkürzt die Entwicklungszeit für SaaS-Integrationen, verursacht jedoch zusätzliche Kosten durch die Serverbereitstellung. Die CLI lässt sich kostenlos starten, ist aber in großem Maßstab schwieriger zu verwalten. Optimieren Sie entsprechend Ihrem tatsächlichen Engpass – sei es die Kosten für Tokens, die Latenz oder die operative Komplexität.

  • Die Fähigkeiten des Teams sind entscheidend. Bei der Codeausführung wird vorausgesetzt, dass Ihre Agenten (sowie die dahinterliegenden Modelle) zuverlässigen Python- oder TypeScript-Code erzeugen können. Bei der CLI wird eine Vertrautheit mit Unix-Konventionen vorausgesetzt. MCP erfordert ein Verständnis von Übertragungsprotokollen und OAuth-Flüssen. Passen Sie die gewählte Modalität den Stärken Ihres Teams an.

  • Eine zu starke Konsolidierung von Tools kann problematisch sein. Wenn alles in ein einziges Mega-Tool zusammengefasst wird, wird der Parameterraum so komplex, dass der Agent ihn nicht mehr effektiv bewältigen kann. Für die meisten Agentensysteme ist eine Anzahl von 5 bis 10 gut abgegrenzten Tools ideal.

  • Fähigkeiten basieren auf Prompts und sind nicht verbindlich. Eine Fähigkeit stellen lediglich Anweisungen dar, denen der Agent folgen sollte, nicht jedoch Vorgaben, denen er unbedingt folgen muss. Für kritische Arbeitsabläufe sollten Fähigkeiten mit deterministischer Validierung kombiniert werden.

  • Die Anforderungen an Audits bestimmen die Wahl. Wenn Sie ein vollständiges Protokoll über jede Aufrufung eines Tools sowie deren Ergebnis benötigen, bieten MCP und die JSON-basierte Toolaufrufmethode standardmäßig strukturierte Auditrückverfolgungen. Die Ausführung von Code erzeugt zwar einen Skripttext samt Ausgabe, was nützlich ist, doch diese Ergebnisse sind schwieriger in einzelne Aktionen aufzuspalten, um sie für Compliance-Berichte zu nutzen.

Wohin sich die Tool-Ergonomie entwickelt

Drei Entwicklungsrichtungen sind besonders erwähnenswert.

Die erste betrifft Tool-RAG zur Skalierung. Wenn Toolsets auf Hunderte oder Tausende anwachsen, sinkt die Genauigkeit der naiven Toolauswahl auf 13,62 %. Die Das RAG-MCP-Paper Es zeigte sich, dass die Anwendung von Retrieval-augmented Generation bei der Werkzeugauswahl – durch das Indizieren von Werkzeudescriptions in einer Vektordatenbank sowie das Abrufen nur relevanter Werkzeuge pro Abfrage – eine Genauigkeit von 43,13 % erzielt, was einer Verbesserung um das 3,2-fache entspricht, während gleichzeitig die Anzahl der Prompt-Tokens um etwa 50 % reduziert wird.

Der zweite Ansatz beinhaltet Agenten, die ihre eigenen Werkzeuge erstellen. Der LATM-Framework („LLMs As Tool Makers“) hat ein zweiphasiges Paradigma etabliert, bei dem ein leistungsstarker LLM wiederverwendbare Python-Funktionen erstellt und ein leichterer LLM diese nutzt. ToolMaker (ACL 2025) wandelt GitHub-Repositorien autonom in LLM-kompatible Tools um – mit einer Erfolgsrate von 80 %. Der Fokus verschiebt sich dabei von der Nutzung von Tools hin zur Erstellung solcher Tools und schließlich zur Verwaltung von Tool-Bibliotheken.

Der dritte Ansatz ist der Dual-Protocol-Stack aus A2A und MCP. Googles Agent2Agent Protocol (A2A) löst Probleme, die MCP nicht bewältigen kann: die Kommunikation zwischen Agenten. MCP kümmert sich um die Integration von Agenten in Tools; A2A übernimmt die Entdeckung, Verhandlung sowie Delegation von Aufgaben zwischen Agenten. Die Kombination lautet daher: Mit Ihrem Framework entwickeln, mit MCP ausstatten und mit A2A kommunizieren.

  1. Behandeln Sie Toolbeschreibungen als Teil des Prompt-Engineerings. Eingabemuster verbessern die Genauigkeit erheblich. Von Claude optimierte Beschreibungen übertriffen solche, die von Experten manuell verfasst wurden. Investieren Sie genauso viel Aufwand in die Benutzerführung der Tools wie in die Benutzerführung bei menschlichen Interaktionen.

  2. Der praktische Stack für 2026 besteht aus Skills + CLI + MCP + Code Execution und ist je nach Anwendungsfall schichtweise aufgebaut. Skills dienen der Fachkompetenz, CLI für lokale Operationen, MCP für externe Dienste sowie Code Execution für mehrstufige Orchestrierung.
