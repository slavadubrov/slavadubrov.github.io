    Engenharia de Contexto para Agentes de IA: Janelas de Contexto, Memória, Ferramentas e Restrições

    TL;DR

    A engenharia de contexto é o processo que determina o que o modelo vê no momento de tomar uma decisão: instruções, exemplos, conhecimento, memória, competências, ferramentas e restrições. A maioria dos agentes em produção funciona ou falha nesta camada. Abaixo encontra-se o esquema ao qual sempre retorno, com os padrões que realmente utilizo.

    O mesmo agente que consegue executar uma demonstração de 30 minutos pode colapsar já no terceiro dia em ambiente de produção. Quase sempre, a falha não tem nada a ver com o modelo, mas sim com o conteúdo do contexto no momento em que foi tomada a decisão errada. Uma memória desatualizada. Um documento que já não é relevante. Uma descrição de ferramenta que se alterou. Uma instrução vaga.

    A engenharia de contexto é o que se faz para evitar que isso aconteça: projetar intencionalmente o que o modelo irá ver antes de cada decisão, dentro de determinados limites. Este artigo apresenta os padrões que utilizo.

    Por que a engenharia de contexto é importante para agentes de IA

    Um agente de suporte ao cliente funciona durante três semanas e lida com 200 tickets. Depois disso, começa a gerar detalhes de produtos falsos, a confundir clientes e a chamar as APIs erradas. O modelo não piorou; o contexto sim.

    Quatro mudanças em 2025 que tornam esta tarefa mais difícil do que antes. Os agentes deixaram de ser simples chatbots e passaram a executar ações, o que significa que uma única decisão errada no contexto pode afetar todo um plano composto por dez passos, em vez de resultar apenas numa resposta inadequada. A memória centralizada e padrões como o MCP permitem carregar o contexto pessoal de forma segura, mas apenas se a camada de processamento for projetada corretamente — sem governação adequada, corre-se o risco de vazar informações pessoais sensíveis ou comprometer a integridade do sistema. A recuperação híbrida de dados, o reclassificação de resultados e a recuperação baseada em grafos amadureceram, o que reduz as alucinações e o consumo de tokens, desde que as consultas sejam direcionadas para a estratégia adequada. E a maioria dos projetos piloto com agentes que enfrentam dificuldades o faz devido à falta de design de contexto e governação, e não à qualidade do modelo. Uma camada de contexto bem estruturada é o que resolve esses problemas.

    Conceitos fundamentais para iniciantes

    Alguns termos que aparecem ao longo do artigo:

    • Janela de contexto: a memória de trabalho do modelo. É o limite máximo de texto (em tokens) que ele consegue processar numa única decisão. Se esse limite for excedido, o modelo trava ou esquece as informações iniciais.
    • Tokens: unidade em que o texto é dividido. Geralmente, corresponde a entre 1.000 tokens e 750 palavras.
    • Orçamento de atenção: os modelos de linguagem prestam atenção a cada par de tokens; portanto, para n tokens existem n² relações. À medida que o contexto aumenta, esse orçamento fica mais limitado, fazendo com que os tokens intermediários sejam menos considerados em comparação com os iniciais e finais.
    • Embeddings: representações numéricas de texto. Eles permitem realizar buscas por significado, de modo que uma consulta para “cão” pode encontrar “filhote de cão”.
    • JSON Schema: um método padrão para descrever a estrutura dos dados JSON. É utilizado para forçar o modelo a gerar campos específicos, como {"answer": "...", "citations": [...]}.
    • MCP (Model Context Protocol): um padrão aberto que permite que modelos de IA comuniquem com dados e ferramentas externas através de uma interface comum. Pense nele como uma porta USB para conectar um agente aos seus ficheiros locais, bases de dados ou ao Slack.

    O que é a camada de contexto?

    Um pipeline associado a uma política que seleciona e estrutura as entradas em cada etapa, aplica controlos (formato, segurança, políticas) e fornece ao modelo o contexto necessário, na medida certa e no momento exato.

    Pense nela como a linha de montagem que prepara exatamente o que o modelo precisa para tomar uma boa decisão.

    O contexto é um recurso finito. Os modelos, tal como os seres humanos com memória de trabalho limitada, possuem um orçamento de atenção que se esgota à medida que o contexto aumenta. Cada novo token consome parte desse orçamento. O desafio de engenharia consiste em encontrar o conjunto mais pequeno de tokens de alto valor que permitam obter a resposta desejada.

    Não existe uma decomposição canónica única — diferentes equipas utilizam soluções distintas. A que eu uso tem a seguinte estrutura:

    1) Instruções

    Um contrato robusto para definir o comportamento: papel, tom, restrições, esquema de saída e objetivos de avaliação. Os modelos modernos respeitam hierarquias de instruções (sistema > desenvolvedor > utilizador), pelo que deve utilizar essa hierarquia de forma explícita em vez de colocar tudo num único bloco.

    Deve recorrer a instruções quando precisar de uma saída consistente (relatórios, SQL, chamadas API, JSON) ou quando tiver de aplicar políticas — remova dados PII, recuse pedidos para funcionalidades que não suporta e nunca inclua URLs internas. Os padrões eficazes na prática são os mais óbvios: mantenha os blocos de política separados das tarefas do utilizador, utilize esquemas JSON para gerir código determinístico e divida as mensagens de sistema, desenvolvedor e utilizador para que o modelo saiba de qual fonte provém cada instrução.

    Um exemplo simples de um bloco de política:

    SYSTEM RULES
- Role: support assistant for ACME.
- Always output valid JSON per AnswerSchema.
- If a request needs account data, ask for the account ID.
- Never include secrets or internal URLs.

    Raciocínio guiado por esquema (SGR)

    A atualização mais útil para “dar instruções ao modelo” consiste em tornar essas instruções estruturadas. Direcione o agente através de JSON Schemas que definem o plano, os argumentos das ferramentas, os resultados intermediários e a resposta final. O modelo emite e consome JSON em cada passo, sendo que o seu código o valida antes que qualquer outra ação seja realizada. Isto reduz a ambiguidade, torna as tentativas de repetição determinísticas e melhora a segurança, uma vez que os tipos e os campos obrigatórios são aplicados ao longo de todo o ciclo.

    O fluxo é simples. Defina os schemas para Plan, ToolArgs, StepResult, e FinalAnswer. Em cada passo, o modelo gera um JSON que corresponde a um deles. O seu código realiza a validação. Se a validação falhar, tente uma correção automática (por exemplo, preencha os campos obrigatórios em falta com valores padrão adequados). Se a correção também falhar, recuse a entrada e registre o evento.

    {
    "action": "call_tool",
    "tool": "search_tickets",
    "args": { "customer_id": "A-123", "limit": 10 },
    "expected_schema": "TicketList"
}

    e o seu código valida args de acordo com o esquema da ferramenta antes de chamar a API.

    2) Exemplos

    Alguns pares curtos de entrada/saída que demonstram o formato exato, o tom e os passos que o modelo deve seguir. Recorra a exemplos quando precisar de um modelo específico (tabelas, JSON, SQL, chamadas à API) ou quando desejar expressões específicas para um determinado domínio e um tom consistente.

    Algumas regras que continuo a reaprender. Mostre a estrutura exata de destino na sua demonstração canónica, e não uma paráfrase dela. Combine exemplos bons com exemplos ruins — contrastar um erro típico com o resultado desejado ensina mais do que dois outputs corretos. Associe o seu JSON Schema a duas ou três demonstrações curtas em vez de uma única longa. E mantenha-as curtas: muitas demonstrações pequenas e focadas superam quase sempre um único exemplo extenso.

    3) Conhecimento

    Fundamentar o modelo em factos externos: recuperação de vetores e palavras-chave, reclassificação, consultas a grafos, obtenção de dados da web, fontes empresariais. Isso é necessário quando o modelo não consegue obter a resposta a partir dos seus pesos — factos recentes, factos privados ou qualquer coisa que precise de citação.

    A pilha de recuperação a utilizar por padrão deve ser híbrida (BM25 combinado com método denso), com um reclassificador acima para reduzir o custo em tokens. Recorra à recuperação consciente de grafos (GraphRAG) quando a resposta exigir a intersecção de vários documentos, e ao RAG adaptativo quando um tipo de consulta não se aplicar a todos os casos — por vezes não se quer recuperação alguma, outras vezes apenas uma tentativa única, e outras ainda um ciclo iterativo.

    Os parâmetros que realmente influenciam a qualidade são menores do que o indicado na literatura especializada. Divida o texto por limites semânticos (parágrafos, secções) em vez de usar um tamanho fixo — a forma como se faz essa divisão é a decisão determinante para a qualidade da recuperação em documentos reais. Comece com um valor top-k entre 10 e 20 para abordagens híbridas e, em seguida, reduza esse valor para 3 a 5. Utilize o algoritmo MMR com um valor de λ em torno de 0,7 como padrão para garantir diversidade. E inclua sempre referências à fonte e citações diretas nos seus resultados, não porque o modelo possa gerar informações falsas sem elas, mas porque são esses elementos que permitem auditar as respostas.

    4) Memória

    Contexto duradouro ao longo de turnos e sessões. A memória de curto prazo armazena o estado da conversa. A memória de longo prazo guarda informações sobre o utilizador e a aplicação. A memória episódica armazena eventos. A memória semântica armazena entidades. Utilize a memória quando desejar personalização e continuidade, ou quando vários agentes precisam de coordenar-se ao longo de dias ou semanas.

    Os padrões que sobrevivem ao uso em ambiente de produção incluem: memórias de entidades (nomes, IDs, preferências) com políticas de expiração explícitas; recuperação direcionada a partir de um armazenamento de longo prazo suportado por vetores, pares chave-valor ou grafos; e associação da memória à compressão, de modo que, quando a memória de curto prazo se torna demasiado grande, ela seja resumida em vez de truncada. O processo de resumo é abordado em Estratégias de Compressão de Contexto [7].

    A política de expiração que defino por predefinição:

    • Preferências: 365 dias.
    • Eventos episódicos: 90 dias.
    • Estado de curto prazo: é limpo no final da sessão.
    • Entidades: não têm data de expiração, mas exigem validação periódica.

    5) Habilidades

    Especialização em domínios componíveis que os agentes podem descobrir e carregar conforme necessário. A da Anthropic Competências do Agente Um framework é o design de referência para capturar conhecimento procedural e partilhá-lo entre agentes.

    Criar uma “skill” para um agente é semelhante a elaborar um guia de integração para um novo colaborador. — Anthropic Engineering Blog

    Precisa de “skills” quando necessita de competências específicas de um determinado domínio (manipulação de PDF, git, análise de dados), quando deseja procedimentos reutilizáveis entre agentes ou organizações, ou quando quer especializar um agente sem ter de codificar comportamentos diretamente no prompt do sistema.

    O que é uma “skill”?

    Uma “skill” é uma pasta organizada que contém instruções, scripts e recursos que o agente pode descobrir e carregar sempre que necessário:

    • Um SKILL.md ficheiro com um nome, uma descrição (em frontmatter YAML) e as instruções da competência.
    • Ficheiros adicionais — referências, modelos — que a competência pode carregar.
    • Código: Python ou outros executáveis que o agente pode rodar como ferramentas.

    Padrão de competências: divulgação progressiva

    O princípio que permite que as competências se adaptem é a divulgação progressiva: carregar informações apenas quando necessárias. (Ver O princípio de divulgação progressiva Abaixo, para a versão geral.) Na inicialização, apenas os nomes e descrições das competências estão presentes no contexto — o que é suficiente para que o agente saiba o que está disponível. O ficheiro SKILL.md completo é carregado apenas quando a competência é ativada. Os ficheiros referenciados de forma mais profunda são carregados somente quando a competência ativada os necessita.

    Isto significa que o conteúdo relacionado com as competências é, na prática, ilimitado. O agente não paga custos de contexto pelo que não utiliza.

    ┌─────────────────────────────────────────────────────────┐
│ Context Window                                          │
├─────────────────────────────────────────────────────────┤
│ System Prompt                                           │
│ ├── Core instructions                                   │
│ └── Skill metadata (name + description only)           │
│     • pdf: "Manipulate PDF documents"                   │
│     • git: "Advanced git operations"                    │
│     • context-compression: "Manage long sessions"       │
├─────────────────────────────────────────────────────────┤
│ [User triggers task requiring PDF skill]                │
│                                                         │
│ → Agent reads pdf/SKILL.md into context                 │
│ → Agent reads pdf/forms.md (only if filling forms)      │
│ → Agent executes pdf/extract_fields.py (without loading)│
└─────────────────────────────────────────────────────────┘

    Boas práticas de competências

    As diretrizes da Anthropic resumem‑se a quatro pontos. Comece pela avaliação: execute o agente em tarefas representativas, identifique as lacunas e crie competências para as preencher. Estruture para escala: divida uma tarefa grande SKILL.md Em ficheiros separados, mantendo os caminhos distintos quando os contextos são mutuamente exclusivos. Observe como o agente utiliza as suas competências e itere nos nomes e descrições até que a precisão ao acioná-las seja elevada. Deixe também o agente ajudar: peça ao Claude para capturar abordagens bem-sucedidas e transformá-las em conteúdo de competência reutilizável.

    Considerações de segurança

    As competências introduzem vulnerabilidades por definição — elas conferem ao agente novas capacidades através de instruções e código. Instale apenas a partir de fontes em que confia. Realize uma auditoria antes da utilização, incluindo ficheiros anexos, dependências de código e quaisquer ligações de rede. Preste especial atenção às instruções que ligam a competência a serviços externos, pois é aí que a extração de dados geralmente começa.

    6) Ferramentas

    Chamadas de função que obtêm dados ou executam ações: APIs, bases de dados, pesquisas, operações com ficheiros, uso do computador. Precisa de ferramentas quando necessita de efeitos colaterais determinísticos e fidelidade dos dados, ou quando organiza ciclos de plano-chamada-verificação-contínua.

    Os padrões que utilizo por defeito são um planeamento baseado primeiro nas ferramentas, com validadores após cada chamada, saídas estruturadas entre etapas e soluções alternativas explícitas quando as ferramentas falham (retentativa, posterior redução de funcionalidades ou intervenção humana).

    Três conceitos sobre os quais vale a pena ser preciso. “Idempotente” significa que é seguro repetir a operação sem efeitos colaterais (GET sim, POST ou DELETE não). As “condições pós-execução” referem-se às verificações realizadas após cada chamada: resultado não vazio, estado igual a “ok” e JSON válido. A “cadeia de fallback” indica a sequência a seguir quando algo dá errado: tentar novamente, depois lidar com a situação de forma adequada e, por fim, escalonar para um humano.

    Uma observação sobre o MCP

    O MCP está a tornar-se o padrão para a forma como os agentes se conectam a ferramentas e a dados [6]. Em vez de escrever um wrapper de API personalizado para cada serviço, executa-se um servidor MCP para cada um, permitindo que o agente descubra automaticamente as ferramentas e recursos disponíveis.

    7) Restrições de Segurança

    Validação de entrada e saída, filtros de segurança, defesa contra tentativas de contorno de restrições, aplicação estrita de esquemas e políticas de conteúdo. As restrições de segurança são essenciais quando é necessário garantir conformidade e integridade da marca, bem como quando se deseja obter resultados corretos e um comportamento seguro.

    A arquitetura eficaz em produção inclui trilhos programáveis (regras de política associadas a ações), validadores de esquema e semântica (tipos, expressões regulares, avaliações), além de uma política central integrada com capacidades de monitorização (painéis de controle, testes de intrusão).

    A decisão entre reparar ou recusar é aquela que as equipas costumam errar. As violações de esquema dão direito a uma tentativa automática de reparação; se esta falhar, deve-se recusar o pedido indicando um erro claro. Já as violações de política dispensam completamente a tentativa de reparação — deve-se recusar imediatamente e sugerir uma alternativa segura.

    Quatro tipos comuns de proteções cobrem a maioria das necessidades em produção. As proteções de entrada capturam Dados Pessoais Identificáveis, injeções de prompts e conteúdos tóxicos antes que o modelo os processe. As proteções de saída garantem a conformidade com esquemas, políticas de conteúdo e consistência factual. As proteções de ferramentas lidam com limitação de taxa, verificações de permissões e limites de custo. As proteções de memória ocultam Dados Pessoais Identificáveis antes do armazenamento e garantem a expiração dos dados.

    Exemplo concreto: um bot de suporte a responder a um ticket

    Eis o que a camada de contexto monta quando um cliente pergunta “Porque é que a minha chave API não está a funcionar?”:

    • Instruções: atuar como um assistente de suporte útil da ACME, citar fontes e retornar JSON {answer, sources, next_steps}- Exemplos: dois pares curtos de P→R que ilustram o tom e a estrutura JSON, um sobre chaves API e outro sobre faturação.
    • Conhecimento: procure no centro de ajuda e nos manuais do produto por “API key troubleshooting”; inclua citações relevantes.
    • Memória: nome do cliente “Sam”, ID da conta “A-123”, plano “Pro”, última interação “Chave API criada há 3 dias”.
    • Habilidades: carregar o ticket-handling habilidade em procedimentos de resolução de problemas, políticas de escalonamento e modelos de resolução.
    • Ferramentas: search_tickets(customer_id), check_api_key_status(key), create_issue(description).
    • Regras de segurança: oculte os valores das chaves API nos resultados; se o esquema falhar, corrija-o uma única vez; se houver violação de política (por exemplo, um pedido para excluir dados de produção), recuse-o de forma educada.

    O modelo recebe todo este contexto estruturado, gera uma resposta, e as regras de segurança validam-na antes de ela ser enviada ao cliente.

    Análise aprofundada dos fundamentos do contexto

    Antes que os padrões façam sentido, é necessário compreender a sua estrutura.

    A anatomia do contexto

    O contexto possui vários componentes distintos, cada um com características e restrições diferentes.

    Promptes de sistema definem a identidade do agente, as suas restrições e as diretrizes de comportamento. Eles são carregados uma única vez no início da sessão e permanecem ativos durante toda a conversa. O desafio reside em encontrar o equilíbrio certo: suficientemente específicos para orientar efetivamente o comportamento, mas flexíveis o suficiente para permitir julgamento. Se forem demasiado genéricos, resultam em regras rígidas e pouco adaptáveis. Se forem demasiado detalhados, o modelo não dispõe de um sinal concreto ao qual se possa basear nas suas ações.

    Organize os promptes em secções distintas utilizando etiquetas XML ou títulos em Markdown:

    <BACKGROUND_INFORMATION>
You are a Python expert helping a development team.
Current project: Data processing pipeline in Python 3.9+
</BACKGROUND_INFORMATION>

<INSTRUCTIONS>
- Write clean, idiomatic Python code
- Include type hints for function signatures
- Add docstrings for public functions
</INSTRUCTIONS>

<TOOL_GUIDANCE>
Use bash for shell operations, python for code tasks.
File operations should use pathlib for cross-platform compatibility.
</TOOL_GUIDANCE>

    Definições de ferramentas especificam as ações que o agente pode realizar. Cada ferramenta possui um nome, descrição, parâmetros e formato de retorno. São as descrições que realmente orientam o comportamento do agente. Descrições inadequadas forçam o agente a adivinhar; as boas incluem contexto de utilização, exemplos e valores padrão sensatos. O princípio da consolidação: se um engenheiro humano não conseguir determinar com certeza qual ferramenta utilizar, o agente não irá obter melhores resultados. Esse é o sinal para fundi-las ou renomeá-las.

    Documentos recuperados trazem conteúdos relevantes para o contexto em tempo de execução, em vez de carregar tudo previamente. Esta abordagem just-in-time permite manter identificadores leves — como caminhos de ficheiros, consultas armazenadas e links da web — e carregar os dados reais apenas quando necessário.

    Histórico de mensagens representa a conversa entre o utilizador e o agente, incluindo consultas anteriores, respostas e raciocínios. Em tarefas de longa duração, este histórico pode tornar-se tão extenso que ocupe quase toda a janela de visualização. Funciona também como memória temporária, onde os agentes registam o progresso e o estado da tarefa.

    Saídas da ferramenta são os resultados das ações do agente: conteúdos de ficheiros, resultados de pesquisa, saídas de comandos, respostas de API. Nas trajetórias típicas de um agente, as observações acabam por representar mais de 80% do contexto total. [4]. Esse número representa a pressão exercida por técnicas como o mascaramento de observações e a compactação.

    Janelas de contexto e mecânicas de atenção

    Os modelos de linguagem calculam a atenção com base em relações entre pares de tokens. Para n tokens, isso resulta em n² relações. À medida que o contexto aumenta, a capacidade do modelo para capturar essas relações fica cada vez mais limitada.

    Os modelos também adquirem os seus padrões de atenção a partir dos dados de treino, que são predominantemente sequências mais curtas. Por isso, o modelo possui relativamente pouca experiência com dependências em todo o contexto. O resultado final é um orçamento de atenção que se esgota à medida que o contexto cresce.

    O princípio de divulgação progressiva

    A divulgação progressiva permite gerir o contexto de forma eficiente, carregando informações apenas quando necessário. Na inicialização, os agentes carregam apenas os nomes e descrições das competências — o suficiente para determinar quando uma competência pode ser relevante. O conteúdo completo é carregado apenas quando ativado para tarefas específicas.

    # Instead of loading all documentation at once:

# Step 1: Load summary

docs/api_summary.md # Lightweight overview

# Step 2: Load specific section as needed

docs/api/endpoints.md # Only when API calls needed
docs/api/authentication.md # Only when auth context needed

    Qualidade versus quantidade de contexto

    A suposição de que janelas de contexto maiores resolvem os problemas de memória foi refutada empiricamente. A engenharia de contexto consiste em identificar o conjunto mais pequeno possível de tokens de alto valor informativo.

    Várias forças incentivam a busca pela eficiência. O custo de processamento aumenta de forma desproporcional à extensão do contexto — não dobra simplesmente com o dobro de tokens, mas cresce exponencialmente em termos de tempo e recursos computacionais. O desempenho do modelo deteriora para além de certos limites de contexto, mesmo quando a janela técnica permitiria mais dados. Além disso, as entradas longas continuam a ser caras, mesmo com o uso de cache de prefixos.

    O princípio orientador é a informatividade em detrimento da exaustividade. Devem ser incluídos apenas os elementos relevantes para a decisão em questão, excluindo tudo o que não é essencial.

    Padrões de degradação do contexto

    Os modelos de linguagem apresentam padrões de degradação previsíveis à medida que o contexto aumenta. Conhecer esses padrões permite diagnosticar falhas e projetar sistemas resilientes.

    O fenómeno de perda no meio

    O padrão de degradação mais bem documentado é o “lost-in-middle”, no qual os modelos apresentam curvas de atenção em forma de U [1]. As informações no início e no final do contexto recebem uma atenção confiável; as informações no meio apresentam uma precisão de recuperação 10-40% mais baixa.

    O mecanismo é concreto. Os modelos alocam uma atenção massiva ao primeiro token (geralmente o token BOS) para estabilizar os estados internos, criando um “poço de atenção” que consome o orçamento de atenção disponível. [2]. À medida que o contexto aumenta, os tokens intermédios não conseguem obter um peso de atenção suficiente.

    A solução prática é colocar as informações críticas no início ou no final do contexto. Utilize estruturas de resumo que exponham as informações-chave em posições favorecidas pela atenção.

    # Organize context with critical info at edges

[CURRENT TASK] # At start

- Goal: Generate quarterly report
- Deadline: End of week

[DETAILED CONTEXT] # Middle (less attention)

- 50 pages of data
- Multiple analysis sections
- Supporting evidence

[KEY FINDINGS] # At end

- Revenue up 15%
- Costs down 8%
- Growth in Region A

    Para instruções que não podem, de forma alguma, ser perdidas, faça uma cópia delas tanto no início como no final. Como os modelos dão grande importância a ambos os extremos, a mesma instrução nas duas posições recebe atenção independentemente do comprimento do contexto. Esta é a abordagem adequada para restrições de sistema que devem ser sempre respeitadas, requisitos de formato de saída e políticas de segurança.

    # Example: Duplicating critical instructions

[SYSTEM - START]
CRITICAL: Always respond in JSON format. Never include PII.

[... long context with documents, history, tools ...]

[SYSTEM - REMINDER]
CRITICAL: Always respond in JSON format. Never include PII.

    Envenenamento de contexto

    O envenenamento de contexto ocorre quando ilusões, erros ou informações incorretas entram no contexto e se acumulam devido a referências repetidas. Uma vez envenenado, o contexto gera ciclos de retroalimentação que reforçam a crença errada.

    Geralmente, isso acontece por um de três canais: saídas de ferramentas com erros ou formatos inesperados, documentos recuperados com informações incorretas ou desatualizadas, e resumos gerados pelo modelo que introduzem ilusões e permanecem posteriormente.

    Percebemo-lo através de sintomas: qualidade reduzida das saídas em tarefas que anteriormente eram bem-sucedidas, agentes que utilizam ferramentas ou parâmetros incorretos, e ilusões persistentes que sobrevivem a tentativas de correção.

    A recuperação é mecânica: recorte o contexto até ao ponto anterior ao envenenamento, indique explicitamente o problema e solicite uma nova avaliação, ou reinicie com um contexto limpo, mantendo apenas as informações verificadas.

    A distração por contexto surge quando este se torna excessivamente longo, fazendo com que o modelo se concentre demais no que foi fornecido, em detrimento do que aprendeu durante o treino. Estudos demonstram que mesmo um único documento irrelevante pode reduzir o desempenho em tarefas que envolvem documentos relevantes. [8]Este efeito segue uma função de passo — a presença de qualquer fator de distração desencadeia uma degradação na qualidade das respostas.

    A ideia fundamental é que os modelos não conseguem ignorar contexto irrelevante. Eles são obrigados a prestar atenção a tudo o que lhes é fornecido, e essa própria atenção acaba por ser um fator de distração, mesmo quando as informações irrelevantes são claramente inúteis.

    As soluções incluem a filtragem de relevância antes do carregamento dos documentos, a utilização de nomes espaciais para facilitar a ignoração de seções irrelevantes, e a avaliação se as informações realmente precisam estar no contexto ou se podem ser obtidas através de chamadas a ferramentas.

    Confusão de contexto

    A confusão de contexto ocorre quando informações irrelevantes influenciam as respostas de forma a degradar a sua qualidade. Ao incluir algo no contexto, o modelo é forçado a dar-lhe atenção, o que pode levar à incorporação de dados desnecessários, ao uso de definições de ferramentas destinadas a tarefas diferentes, ou à aplicação de restrições provenientes de outro contexto.

    Sinais a observar: respostas que abordam o aspeto errado da consulta, chamadas a ferramentas adequadas para outra tarefa, e resultados que misturam requisitos de várias fontes.

    A solução passa pela segmentação explícita das tarefas (cada tarefa utiliza uma janela de contexto distinta), por transições claras entre contextos, e por uma gestão de estado que isola cada contexto.

    O conflito de contexto surge quando informações acumuladas entram em contradição direta, gerando orientações contraditórias. Isto difere do “envenenamento” de dados, em que apenas um elemento está incorreto — no caso de conflito, vários elementos corretos se contradizem entre si.

    Fontes comuns incluem recuperação de informação a partir de múltiplas fontes com dados contraditórios, conflitos de versão (informações desatualizadas e atuais presentes no contexto) e conflitos de perspetiva (pontos de vista válidos, mas incompatíveis).

    Resolva-o através da marcação explícita de conflitos, de regras de prioridade que determinem qual fonte deve ter precedência, e de filtragem de versões que exclua informações desatualizadas.

    Limiares de degradação específicos por modelo

    A investigação fornece dados concretos sobre o momento em que a degradação começa. O comprimento efetivo de contexto — ou seja, o limite em que os modelos mantêm um desempenho ótimo — é frequentemente significativamente menor do que o máximo anunciado. [3]:

    GPT-4o 128K tokens ~8K tokens A precisão do NIAH complexo diminui de 99% para 70% a 32K
    Claude 3.5 Sonnet 200 mil tokens ~4K tokens A precisão do NIAH complexo diminui de 88% para 30% a 32K.

Pontuações quase perfeitas em testes simples não refletem uma verdadeira compreensão de contextos extensos — tarefas de raciocínio complexo apresentam uma degradação muito mais acentuada.

    Estratégias de compressão de contexto

    Nota de terminologia: a compressão de contexto é um conceito abrangente que inclui a resumidão (para histórico de conversas e memória), o mascaramento de observações (para saídas de ferramentas) e o corte seletivo [5]. A resumo da memória mencionado em Memória A secção é uma aplicação destas estratégias de compressão mais abrangentes.

    Quando as sessões dos agentes geram milhões de tokens de histórico de conversação, a compressão deixa de ser opcional. A abordagem ingénua consiste numa compressão agressiva para minimizar os tokens por pedido. O objetivo correto de otimização são os tokens por tarefa. [4]: número total de tokens consumidos para concluir a tarefa, incluindo os custos associados à recuperação de dados quando a compressão resulta na perda de elementos críticos.

    Quando é necessária a compressão

    Ative a compressão quando as sessões dos agentes excederem os limites da janela de contexto, quando eles começarem a “esquecer” quais ficheiros modificaram, ao depurar uma longa sessão de programação que se estendeu por várias horas, ou quando o desempenho piorar significativamente em conversas prolongadas.

    Três abordagens prontas para produção

    A resumo iterativo ancorado mantém um resumo estruturado e persistente, com secções explícitas para a intenção da sessão, modificações de ficheiros, decisões e próximos passos. Quando a compressão é acionada, são resumidos apenas os trechos recém-truncados, que são posteriormente integrados no resumo existente. A eficácia desta abordagem deve-se à estruturação imposta: as secções dedicadas funcionam como uma lista de verificação que o resumidor deve preencher, evitando assim alterações silenciosas no conteúdo.

    A ler as métricas:

    • Rácio de compressão (98,6%): percentagem de tokens removidos. Um rácio de 98,6% significa que, de 100.000 tokens da história, restam apenas 1.400.
    • Pontuação de qualidade (3,70/5): medida através de avaliação baseada em testes — após a compressão, o agente é questionado sobre detalhes específicos da história truncada (que ficheiros modificámos, qual foi a mensagem de erro). 3,70/5 indica que o agente respondeu corretamente a aproximadamente 74% dos testes.
    • 0,7% de tokens adicionais: ao comparar o método iterativo ancorado (98,6%) com o método opaco (99,3%), a diferença é de 0,7%. Numa sessão com 100.000 tokens, isso corresponde a 1.400 tokens contra 700. Os 700 tokens adicionais (0,7% do total original) permitem ganhar 0,35 pontos de qualidade (3,70 vs 3,35) — o que resulta numa recuperação significativamente melhor de detalhes críticos para a tarefa.

    O problema da trilha de artefatos

    A integridade da trilha de artefatos é a dimensão mais fraca entre todos os métodos de compressão, obtendo uma pontuação entre 2,2 e 2,5 em 5,0. Os agentes de programação precisam de saber quais ficheiros criaram, modificaram ou leram, e a compressão costuma perder essa informação.

    Recomendação: implementar um índice separado de artefatos ou um registo explícito do estado dos ficheiros no esqueleto do agente, além da resumidificação geral.

    Avaliação baseada em sondas

    Métricas tradicionais como o ROUGE não conseguem avaliar a qualidade da compressão funcional. Um resumo pode obter uma pontuação elevada em termos de sobreposição léxica, mas ignorar o caminho do ficheiro que o agente realmente necessita.

    Avaliações baseadas em sondagem medem a qualidade diretamente, fazendo perguntas após a compressão:

    Tipo de Sonda O que é testado Exemplo de Pergunta
    Recuperação Retenção factual "Qual foi a mensagem de erro original?"
    Artefacto Acompanhamento de ficheiros "Quais ficheiros modificámos?"
    Continuação Planeamento de tarefas "O que devemos fazer a seguir?"
    Decisão Cadeia de raciocínio "Qual foi a decisão que tomámos em relação ao problema com o Redis?"

    Se a compressão preservou as informações corretas, o agente responde corretamente. Caso contrário, ele faz suposições ou gera ilusões.

    Técnicas de otimização de contexto

    A otimização de contexto aumenta a capacidade efetiva através da compressão, mascaramento, armazenamento em cache e particionamento. Estas técnicas baseiam-se no Estratégias de compressão de contexto Acima, aplicando-os de forma sistemática em ambiente de produção. Se feito corretamente, a otimização pode duplicar ou triplicar a capacidade efetiva de contexto sem a necessidade de um modelo maior.

    Estratégias de compactação

    A compactação resume o conteúdo do contexto quando se aproximam os limites, reiniciando posteriormente com esse resumo. O objetivo é condensar o conteúdo de forma de alta fidelidade, para que o agente possa continuar a operar com uma degradação mínima.

    A ordem de prioridade que sigo é: primeiro comprimir as saídas das ferramentas (substituindo-as por resumos), depois as conversas anteriores (resumindo diálogos antigos) e, em seguida, os documentos recuperados (resumindo-os caso existam versões mais recentes). Nunca se deve comprimir o prompt do sistema.

    O que deve ser preservado depende do tipo de conteúdo. Saídas de ferramentas: manter resultados, métricas e conclusões; descartar saídas brutas e verbosas. Turnos de conversação: manter decisões, compromissos e mudanças de contexto; descartar informações desnecessárias. Documentos recuperados: manter factos e afirmações-chave; descartar explicações complementares.

    Masking de observações

    As saídas das ferramentas podem representar mais de 80% do uso de tokens. [4]. Uma vez que um agente utilizou a saída de uma ferramenta para tomar uma decisão, manter toda a saída associada confere um valor cada vez menor.

    Categoria Ação Raciocínio
    Observações da tarefa atual Nunca mascarear Crítico para o trabalho atual
    Último turno Nunca mascarear Imediatamente relevante
    Raciocínio ativo Nunca mascarear Pensamento em andamento
    há 3+ turnos atrás Considere a aplicação de mascaramento Propósito provavelmente desempenhado
    Saídas repetidas Sempre mascareie Redundante
    Código de exemplo Sempre mascareie Sinal fraco

    Otimização da cache KV

    O cache KV armazena os tensores de Chave e Valor calculados durante a inferência. O armazenamento em cache de solicitações com prefixos idênticos evita a recálculo, o que reduz significativamente os custos e o tempo de resposta.

    # Stable content first (cacheable)
context = [system_prompt, tool_definitions]
# Frequently reused elements
context += [reused_templates]
# Unique elements last
context += [unique_content]

    Design para estabilidade de cache: evite conteúdos dinâmicos como datas e horas em prompts, utilize formatação consistente e mantenha a estrutura estável ao longo das sessões.

    Particionamento de contexto

    A otimização mais agressiva consiste em dividir o trabalho entre sub-agentes com contextos isolados. Cada sub-agente opera num contexto limpo, focado na sua subtarefa, sem carregar o contexto acumulado de outras subtarefas.

    Para agregação: valide se todas as partições foram concluídas, funda resultados compatíveis e faça um resumo caso a saída combinada continue a ser demasiado grande.

    Estrutura de decisão de otimização

    Quando otimizar: quando a utilização do contexto excede 70%, a qualidade da resposta se degrada em conversas prolongadas, os custos aumentam devido a contextos extensos ou a latência sobe com o comprimento da conversa.

    O que aplicar depende do fator que predomina. Se as saídas das ferramentas forem predominantes, deve-se utilizar mascaramento de observações. Se os documentos recuperados forem os mais relevantes, opte por resumo ou particionamento. Se o histórico de mensagens for dominante, aplique compactação combinada com resumo. Quando vários componentes tiverem peso igual, combine diferentes estratégias.

    Metas métricas razoáveis: compactação que resulte em uma redução de 50-70% nos tokens, mantendo uma degradação na qualidade inferior a 5%; mascaramento que leve a uma redução de 60-80% nas observações mascaradas; otimização do cache com uma taxa de acerto de 70% ou superior para cargas de trabalho estáveis.     Passo 7: iterar

    Adicione padrões avançados apenas quando se deparar com limites evidentes. Adicione mecanismos de reflexão quando a taxa de erros ultrapassar 5%. Adicione planificadores quando as tarefas exigirem regularmente mais do que três passos sequenciais. Adicione subagentes quando houver domínios distintos que necessitem de contextos isolados.

    Antipadrões

    Erros comuns que comprometem os sistemas agent-based. Evite-os.

    1. Overload do contexto

    Carregue todos os documentos, memórias e exemplos possíveis no contexto em cada consulta. O resultado é a deterioração do contexto — a relação sinal/ruído colapsa, e todos os padrões de degradação surgem simultaneamente. Veja Padrões de degradação de contexto Para obter detalhes sobre distrações e confusões, a solução consiste em rotear de forma adaptativa e aplicar compressão e mascaramento. Veja Técnicas de otimização de contexto### 2. Resultados de ferramentas não validados

    O agente chama uma ferramenta, recebe os dados e os transmite imediatamente ao modelo sem efetuar nenhuma verificação. Dados mal formatados causam falhas na lógica subsequente. Resultados nulos dão origem a alucinações. Este é um dos principais vetores para Envenenamento de contexto. Valide sempre os resultados das ferramentas em relação ao esquema e às condições pós-condição.

    3. Tudo de uma só vez

    A política do sistema, as diretrizes para desenvolvedores, os exemplos, as consultas dos usuários, a memória e o conhecimento são todos inseridos num único prompt monolítico. Não há separação de responsabilidades. A janela de contexto fica repleta de texto genérico duplicado. Separe as instruções duradouras do contexto específico de cada etapa, e utilize o Optimização da cache KV padrões acima.

    4. Memória ilimitada

    Armazene cada interação do utilizador para sempre e carregue tudo em cada consulta. O contexto enche-se de memórias obsoletas e irrelevantes, e você assume riscos de privacidade sem qualquer compensação. Veja o fenómeno de perda no meio Para entender por que o conteúdo intermédio acaba sendo ignorado, defina políticas de retenção, implemente recuperação por escopo e edite os dados PII.

    5. RAG em todo o lado

    Recupere documentos para cada consulta, mesmo para perguntas como “O que é 2+2?” ou “Olá.” Isso gera desperdício de latência e custos, além de introduzir ruído na recuperação que desencadeia Distração de contexto. Implemente um encaminhamento adaptativo RAG utilizando um classificador ou um pequeno conjunto de heurísticas. avaliação baseada em sondagem para detetar problemas de qualidade de compressão.

    Soluções rápidas: implemente estas hoje mesmo

    Se já tem um agente em produção e deseja melhorias imediatas, comece por aqui. Cada uma delas leva menos de um dia.

    1. Adicionar validação do esquema de saída

    Deteta a maioria dos erros antes que estes cheguem aos utilizadores.

    from jsonschema import validate, ValidationError

def validate_output(output: dict) -> dict:
    try:
        validate(instance=output, schema=ANSWER_SCHEMA)
        return output
    except ValidationError as e:
        repaired = auto_repair(output, e)
        validate(instance=repaired, schema=ANSWER_SCHEMA)
        return repaired

    2. Instrumentação de rastreio básico

    Depure de forma significativamente mais rápida quando ocorrem falhas.

    logger.info(json.dumps({
    "request_id": request_id,
    "query": query,
    "context_loaded": {"instructions": True, "memory": True, "knowledge": True},
    "tokens": {"input": 1200, "output": 150},
    "latency_ms": 1120,
    "result": "success"
}))

    3. Sistema de divisão vs. mensagens de utilizador

    Reduz significativamente o desperdício de tokens ao permitir Optimização da cache KV.

    messages = [
    {"role": "system", "content": SYSTEM_POLICY + DEVELOPER_GUIDELINES},
    {"role": "user", "content": f"Query: {query}\nMemory: {memory}\nKnowledge: {knowledge}"}
]

    4. Adicionar requisitos de citação

    Isso gera confiança, permite a auditoria e reduz as alucinações.

    5. Definir o prazo de expiração da memória

    Isso evita a poluição do contexto e os riscos à privacidade.

    def load_memory(customer_id: str) -> dict:
    entries = db.get_memory(customer_id)
    now = datetime.now()
    return {
        k: v for k, v in entries.items()
        if v.get("expires_at", now) > now
    }

    Conclusão

    A engenharia de contexto é a disciplina que distingue os agentes de demonstração dos agentes em produção. O modelo não piorou — foi o contexto que o fez. Os quatro pilares essenciais a conhecer são: fundamentos (o contexto é finito, a atenção é limitada e a divulgação progressiva é a chave para resolver o problema), padrões de degradação (perda de informação no meio do processo, envenenamento de dados, distrações, confusão e conflitos), compressão (uso de tokens por tarefa em vez de tokens por solicitação, resumos estruturados) e otimização (compactação, mascaramento, armazenamento em cache e particionamento).

    Comece com soluções rápidas. Adicione complexidade apenas quando enfrentar um limite real. E meça tudo — não é possível melhorar o que não se monitoriza.

    Principais Pontos

    1. A engenharia de contexto é a disciplina em tempo de execução responsável por reunir as informações corretas, na ordem adequada, para a próxima chamada do modelo.
    2. A janela de contexto não é memória. Trata-se do conjunto de dados com os quais o modelo pode trabalhar em cada passo.
    3. A recuperação de informações, a compressão, o escopo da memória, a formatação das saídas das ferramentas e a hierarquia de instruções influenciam todos a qualidade das respostas.
    4. O objetivo não é um contexto máximo, mas sim o menor contexto possível que preserve as informações relevantes para a tomada de decisões.
    5. A qualidade do contexto deve ser avaliada com base nos resultados das tarefas, e não apenas no número de tokens.
