    Design Orientado a Domínio para Agentes de IA: Contextos Delimitados, Ferramentas e Regras de Negócio

    TL;DR

    O design orientado a domínio (DDD) fornece às equipas de agentes de IA um vocabulário comum e interfaces claras entre os subsistemas. Utilize-o quando os seus prompts se desviaram dos objetivos de negócio e as regras estão dispersas por vários modelos.

    Por que o design orientado a domínios é importante para agentes de IA

    A maioria dos projetos de agentes não falha por causa de código de má qualidade. Eles falham porque as pessoas que criam os prompts e aquelas que realmente são responsáveis pelos processos de negócio não conseguem chegar a um acordo sobre o significado das coisas. A conformidade exige uma “verificação de política”, mas o resultado obtido é… process_data() método. Ninguém sabe o que ele faz, por isso os requisitos sofrem alterações e o sistema torna-se cada vez mais rígido. O DDD resolve isto ao colocar o domínio de negócio no centro. Não o esquema da base de dados, nem os modelos de prompt — mas sim o processo real do mundo exterior que se pretende modelar. Os benefícios práticos são:

    • Linguagem comum. Produto, operações e engenharia utilizam as mesmas palavras. Quando a área de conformidade menciona “solicitação de reembolso”, é exatamente isso que aparece no código, nos prompts e na documentação.
    • Escopo focado. Constrói-se apenas o que é realmente importante: os fluxos de trabalho essenciais e as regras de responsabilidade definida. Menos código genérico que se quebra com mudanças nos requisitos.
    • Adaptabilidade. Quando as políticas mudam, basta atualizar uma parte bem definida em vez de ter de analisar todo o sistema monolítico.

    Isto é particularmente relevante em áreas onde as regras mudam frequentemente: finanças, saúde e operações regulamentadas. O DDD oferece uma oportunidade real de manter o ritmo dessas mudanças.

    Blocos de construção estratégicos

    O DDD é um conjunto de padrões e ferramentas, e não uma única ideia. Geralmente, divide-se em duas partes:

    1. Design Estratégico: os aspetos relacionados com o “quadro geral”. Define-se aqui as fronteiras, as equipas e a forma como os sistemas comunicam entre si. Isto é essencial para sistemas multi-agente.
    2. Design Tático: os padrões a nível de código (Entidades, Agregados) que mantêm a lógica interna dos agentes organizada e limpa.

    Os conceitos abaixo são os que irá utilizar no dia a dia.

    Linguagem ubíqua

    O vocabulário partilhado que aparece em todo o lado: reuniões, documentação, comandos, nomes de métodos. Não existe uma camada de tradução entre a “linguagem empresarial” e a “linguagem de código”.

    Se a conformidade exigir “verificação de política”, o seu método será run_policy_check(), não process_data(). Se os médicos disserem “admitir paciente”, escreva admit_patient(), não add_user().

    class PatientRegistry:
    def admit_patient(self, patient_id: str) -> None:
        """Admit a patient to the registry - term used by medical staff."""
        ...

    Quando a linguagem utilizada no código corresponde à linguagem falada na equipa, as alterações nos requisitos manifestam-se como renomeações óbvias num único local. Deixa de ser necessário debater o assunto. process_data o que deveria ter sido feito.

    Contextos delimitados

    Os sistemas de grande escala necessitam de limites explícitos. Porquê? Porque a mesma palavra pode significar coisas diferentes em diferentes áreas da empresa.

    Tomemos o termo “produto” no comércio eletrónico. No contexto de Estoque, um produto é um item de catálogo com códigos SKU e quantidades disponíveis. No contexto de Faturação, trata-se de um item de fatura com regras de preços e cálculos de impostos. Em Gestão de Pedidos, é representado por uma quantidade e pela promessa de entrega.

    Contextos delimitados permitem que cada subdomínio tenha a sua própria definição sem conflitos. Camadas de tradução ou interfaces ligam-nos quando é necessário comunicar entre si.

    Isso mantém cada modelo compacto e evita a criação de um único objeto “produto” gigantesco que teria de satisfazer três equipas em simultâneo.

    Entidades e objetos de valor

    Estes são os blocos de construção básicos do seu modelo de domínio.

    Entidades possuem uma identidade que persiste ao longo do tempo. Uma Task com ID 123 A tarefa permanece a mesma mesmo que se altere a sua descrição, estado ou data de entrega. Duas entidades são consideradas iguais se possuírem o mesmo ID, independentemente dos seus atributos.

    from pydantic import BaseModel

class SupportTicket(BaseModel):
    ticket_id: str  # This is the identity
    customer: str
    issue: str
    status: str = "OPEN"

    def close(self) -> None:
        if self.status != "OPEN":
            raise ValueError("Ticket already closed")
        self.status = "CLOSED"

    Objetos de valor não possuem identidade. Eles são definidos inteiramente pelos seus atributos. Dois TimeSlot Objetos com os mesmos tempos de início e fim são intercambiáveis. Os objetos de valor são imutáveis; em vez de mutar um deles, cria-se um novo.

    from pydantic import BaseModel

class TimeSlot(BaseModel):
    start: str  # e.g., "2025-10-18 09:00"
    end: str    # e.g., "2025-10-18 10:00"

    @property
    def duration(self) -> int:
        # Compute duration from start to end
        ...

    Utilize entidades para representar elementos que possuem ciclos de vida.Order, User, AgentSession). Utilize objetos de valor para descrições e medições (EmailAddress, Priority, Location).

    from datetime import date
from pydantic import BaseModel, Field

class Task(BaseModel):
    id: str
    description: str
    completed: bool = False

class Plan(BaseModel):  # This is the aggregate root
    id: str
    tasks: list[Task] = Field(default_factory=list)

    def add_task(self, task: Task) -> None:
        # Business rule enforced here: no duplicate task IDs
        if any(t.id == task.id for t in self.tasks):
            raise ValueError("Task ID already exists")
        self.tasks.append(task)

    o código externo nunca entra em contato com tasks lista diretamente. Ele chama sempre add_task(). É isso que garante que a regra “sem IDs duplicados” não possa ser violada. Ao guardar dados numa base de dados, geralmente guarda-se todo o conjunto de informações de uma só vez.

    Repositórios

    Os repositórios ocultam a camada de persistência. Do ponto de vista do domínio, basta fazer a chamada necessária. save(plan) e get(plan_id). O facto de essas chamadas acabarem por ser processadas pelo Postgres ou pelo Redis é problema de outra pessoa.

    from abc import ABC, abstractmethod
from pydantic import BaseModel, Field

class Task(BaseModel):
    id: str
    description: str
    completed: bool = False

class Plan(BaseModel):
    id: str
    tasks: list[Task] = Field(default_factory=list)

class PlanRepository(ABC):
    """Domain layer defines the interface."""
    @abstractmethod
    def save(self, plan: Plan) -> None:
        ...

    @abstractmethod
    def get(self, plan_id: str) -> Plan | None:
        ...

class InMemoryPlanRepository(PlanRepository):
    """Infrastructure layer provides the implementation."""
    def __init__(self) -> None:
        self.storage: dict[str, Plan] = {}

    def save(self, plan: Plan) -> None:
        self.storage[plan.id] = plan

    def get(self, plan_id: str) -> Plan | None:
        return self.storage.get(plan_id)

    o código do seu domínio só conhece PlanRepository (a interface). A camada de infraestrutura integra a implementação real.

    Eventos de domínio

    Os eventos de domínio registam elementos importantes que ocorreram no seu sistema. A nomenclatura é no tempo passado (OrderPlaced, TaskCompleted, PaymentFailed) porque descrevem factos, e não comandos.

    Os eventos tornam os efeitos colaterais implícitos explícitos. Em vez de um módulo chamar diretamente outro quando algo acontece, o domínio gera um evento. Outras partes do sistema subscrevem-se e reagem de forma independente.

    from datetime import datetime
from pydantic import BaseModel

class TaskCompleted(BaseModel):
    task_id: str
    completed_at: datetime

    Quando uma tarefa é concluída, emite-se TaskCompleted. Um serviço de notificações pode estar atento a este evento e enviar um e-mail. Um serviço de relatórios pode registá-lo para fins analíticos. O ponto importante é que o agregador de tarefas não precisa de saber nada sobre e-mails ou análises; ele apenas anuncia o que aconteceu.

    É assim que a comunicação entre contextos permanece desacoplada. Isso também se adapta naturalmente a sistemas multi-agente, já que os agentes já reagem uns aos outros através de eventos.

    Traduzir DDD para arquiteturas de agente

    Os sistemas de agentes reais possuem fluxos de trabalho em várias etapas, saídas de LLM que estão incorretas numa certa percentagem de vezes e requisitos que mudam a cada trimestre. Os padrões do DDD coincidem perfeitamente com esses problemas.

    Contextos delimitados tornam-se agentes ou competências

    Cada agente (ou capacidade principal) corresponde a um contexto delimitado. Um orquestrador de investigação pode coordenar três agentes especializados:

    • Agente de Tendências — recolhe dados de mercado utilizando o seu próprio vocabulário e ferramentas
    • Agente de Conformidade — executa verificações de políticas com terminologia regulatória
    • Agente de Custos — estima despesas com regras específicas para finanças

    Cada um possui o seu próprio modelo, terminologia e invariantes. Eles comunicam-se através de interfaces ou eventos bem definidos.

    Até mesmo num sistema de agente único, pode ser necessário definir contextos internos. Um módulo de Planeamento e um módulo de Execução, cada um com o seu próprio modelo de domínio.

    Os prompts respeitam a linguagem universal

    Utilize termos específicos do domínio nos prompts do sistema, nas descrições de ferramentas e nas assinaturas de função. Se os especialistas em conformidade utilizarem a expressão “verificação de política”, essa frase exata deve constar nos seus prompts e no seu código. A vantagem é bastante prática: quando o rasto de atividades de um agente mostra run_policy_check, a equipa de conformidade consegue ler o conteúdo sem necessidade de tradutor.

    O estado torna-se entidades explícitas

    Os LLMs são frequentemente sem estado, mas os agentes reais registam uma grande quantidade de estado: sessões de conversação, objetivos, resultados intermediários, saídas de ferramentas. Modelem estes elementos como entidades ou objetos de valor:

    • ConversationSession entidade com ID e histórico de mensagens Task entidade que representa unidades de trabalho ToolOutput Objeto de valor para resultados imutáveis

    Uma vez que se tratem de objetos explícitos, pode-se associar a eles validações e regras de negócio. A Task Uma entidade pode recusar-se a ser concluída até que as suas dependências estejam finalizadas, sem que essa regra tenha de existir em três modelos de prompt diferentes.

    Os agregados expressam os planos do agente

    A Plan A raiz agregada gere a lista de tarefas e aplica todos os limites que são importantes para o negócio. Quando um LLM propõe adicionar 50 tarefas e a política definida for 10, a raiz agregada recusa as tarefas adicionais. Da mesma forma, se o modelo sugerir trabalhos duplicados, a raiz agregada rejeita essas propostas também. O modelo pode ser entusiasta; o domínio, no entanto, mantém-se sensato.

    Os eventos do domínio impulsionam a orquestração

    Os agentes geram eventos como ResearchCompleted, ThresholdExceeded, ou PolicyViolationDetected. Outros agentes ou serviços subscrevem-se e reagem. Nada está ligado de forma fixa, o que torna a adição de um novo ouvinte (ou de um novo agente) económica.

    As regras de negócio envolvem as ações da IA

    Os resultados gerados pelos LLM fluem através de serviços de domínio ou métodos de entidade em vez de irem diretamente para a base de dados. Se um modelo sugerir um reembolso que excede os limites da política, o seu RefundRequest valida e rejeita-o. O LLM pode improvisar; as regras de negócio têm a palavra final.

    A Camada Anticorrupção (ACL)

    Um LLM é probabilístico e, por vezes, comete erros de maneiras surpreendentes. O seu modelo de domínio tem de permanecer determinístico. Os dois não podem interagir diretamente.

    Essa é a função da Camada Anticorrupção (ACL).

    A ACL situa-se entre o modelo e o domínio. Ela traduz a saída bruta do LLM para os tipos estritos esperados pelo seu domínio.

    1. Ingerir texto bruto ou JSON proveniente do LLM.
    2. Validar a estrutura e os tipos com modelos Pydantic.
    3. Sanitizar os valores (sem preços negativos, sem transações com data futura, etc.).
    4. Traduzir DTOs (Data Transfer Objects) para entidades de domínio.

    Se a validação falhar, o ACL rejeita os dados e, frequentemente, devolve o erro para o LLM para que este possa tentar novamente. A ideia é simples: apenas dados válidos chegam alguma vez à lógica de negócio principal do sistema.

    Exemplo: um assistente de tarefas modelado com DDD

    Vamos construir um assistente pessoal de tarefas que processe pedidos como “Lembra-me de comprar leite amanhã” ou “O que está na minha lista de tarefas?”. Esta explicação aplica, uma por uma, as componentes do DDD mencionadas anteriormente.

    1. Mapear os contextos

    Comece por dividir o problema em subdomínios:

    • Gestão de Tarefas — tratamento de itens da lista de tarefas e lembretes (domínio principal)
    • Agendamento — eventos de calendário e reuniões
    • Notificações — envio de alertas e e-mails

    Vamos focar primeiro na Gestão de Tarefas. Os outros subdomínios podem evoluir como contextos delimitados separados ou agentes complementares.

    2. Falar a mesma linguagem

    Escolha o vocabulário em conjunto com as pessoas que realmente gerem o processo (ou utilize o senso comum para uma aplicação pessoal): “tarefa”, “prazo final”, “lembrete”, “prioridade”. Em seguida, utilize esses termos exatos em modelos de prompt, nomes de métodos e rótulos de interface. Não existe uma tradução separada para “negócios”.

    3. Capturar entidades, objetos de valor e eventos

    Agora, modele os conceitos fundamentais:

    • Entidade: Task com identidade (id) e estado mutávelcompleted)
    • Objeto de valor: Priority enum (imutável, definido pelo seu valor)
    • Evento de domínio: TaskCompletedEvent para sinalizar quando o trabalho estiver concluído
    from datetime import datetime, date, timezone
from enum import Enum
from pydantic import BaseModel, Field

class Priority(Enum):
    """Value object: priority is defined by its value alone."""
    LOW = 1
    NORMAL = 2
    HIGH = 3

class TaskCompletedEvent(BaseModel):
    """Domain event: announces a task was completed."""
    task_id: str
    time: datetime

class Task(BaseModel):
    """Entity: identity persists even as attributes change."""
    id: str
    description: str
    created_at: datetime = Field(default_factory=lambda: datetime.now(timezone.utc))
    due_date: date | None = None
    priority: Priority = Priority.NORMAL
    completed: bool = False

    def mark_completed(self) -> TaskCompletedEvent:
        """Business rule: can't complete an already-completed task."""
        if self.completed:
            raise ValueError("Task is already completed.")
        self.completed = True
        return TaskCompletedEvent(task_id=self.id, time=datetime.now(timezone.utc))

    A regra de negócio (não é possível concluir uma tarefa que já está concluída) encontra-se no método da entidade, e não num modelo de prompt. TaskList é a nossa raiz agregada. Ela contém vários Task entidades e aplica regras de consistência entre elas. Todas as modificações são processadas através dos métodos da raiz.

    from datetime import date
from pydantic import BaseModel, Field

class Task(BaseModel):
    id: str
    description: str
    due_date: date | None = None
    completed: bool = False

class TaskList(BaseModel):
    """Aggregate root: enforces invariants across all tasks."""
    owner: str
    tasks: list[Task] = Field(default_factory=list)

    def add_task(self, task: Task) -> None:
        """Business rule: no duplicate tasks on the same day."""
        if any(
            existing.description == task.description
            and existing.due_date == task.due_date
            for existing in self.tasks
        ):
            raise ValueError("A similar task on that date already exists.")
        self.tasks.append(task)

    def get_pending(self) -> list[Task]:
        """Query helper: find tasks that aren't done yet."""
        return [task for task in self.tasks if not task.completed]


TaskList.model_rebuild()  # Resolve forward references for Pydantic.

    O código externo nunca entra em contato com tasks directamente. Passa sempre por este caminho. add_task() ou outro método de raiz, que é o que garante que a regra “sem duplicados” seja cumprida de forma fiel.

    5. Envolver a persistência num repositório

    O repositório abstrai o armazenamento. A camada de domínio não sabe se as tarefas estão em memória ou no Postgres.

    from pydantic import BaseModel, Field

class Task(BaseModel):
    id: str
    description: str
    completed: bool = False

class TaskList(BaseModel):
    owner: str
    tasks: list[Task] = Field(default_factory=list)

TaskList.model_rebuild()  # Resolve forward references for Pydantic.

class TaskRepository:
    """Abstracts task storage - in-memory implementation for simplicity."""

    def __init__(self) -> None:
        self._data: dict[str, TaskList] = {}

    def get_task_list(self, owner: str) -> TaskList:
        """Retrieve a user's task list, or create a new empty one."""
        return self._data.get(owner, TaskList(owner=owner))

    def save_task_list(self, task_list: TaskList) -> None:
        """Persist changes to the task list."""
        self._data[task_list.owner] = task_list

    Em produção, substituir-se-ia isto por uma implementação suportada por base de dados (utilizando SQLAlchemy ou diretamente no Postgres), sem tocar no código do domínio.

    6. Executar o fluxo

    Quando um utilizador faz uma solicitação, o fluxo é o seguinte:

# User says: "Remind me to buy milk tomorrow"
# (In reality, an LLM would parse this into structured data)
user_input = "Remind me to buy milk tomorrow"
intent = "add_task"

# Initialize repository
repo = TaskRepository()

if intent == "add_task":
    # 1. Load the user's task list
    task_list = repo.get_task_list(owner="User123")

    # 2. Create a new task entity
    task = Task(
        id=str(uuid4()),
        description="buy milk",
        due_date=date.today() + timedelta(days=1),
    )

    # 3. Domain layer enforces business rules
    try:
        task_list.add_task(task)
        repo.save_task_list(task_list)
        print(f"Task '{task.description}' added for {task.due_date}.")
    except Exception as exc:
        print(f"Sorry, I couldn't add that task: {exc}")

    As camadas permanecem separadas:

    • Camada LLM analisa a linguagem natural para converter em dados estruturados (intenção + parâmetros)
    • Camada de domínio aplica as regras de negócio através de métodos de entidade
    • Camada de repositório gere a persistência sem interferir na lógica de domínio

    O LLM pode ser criativo na análise, mas é o domínio que define o que é consistente. Se tentar adicionar uma tarefa duplicada, a raiz agregada rejeitá-la-á. Não é necessário incluir uma cláusula especial no seu prompt para esse cenário.

    Ferramentas para dar vida ao modelo

    O DDD não exige frameworks especiais. No entanto, algumas ferramentas facilitam a implementação, especialmente para agentes de IA.

    FastAPI

    FastAPI corresponde perfeitamente às camadas DDD. Utilize roteadores para separar contextos delimitados./tasks, /schedule), modelos Pydantic para validação de requisições e respostas, bem como injeção de dependências para conectar os repositórios.

    Estruture o seu projeto em camadas:

    project/
├── domain/          # Pure business logic (entities, aggregates, value objects)
├── application/     # Use cases and command handlers
├── infrastructure/  # Repositories, databases, external APIs
└── interface/       # FastAPI routers and HTTP contracts

    Esta estrutura em camadas (às vezes designada por “arquitetura de cebola”) impede que as alterações se propaguem por toda a base de código. Substituir a base de dados implica modificar infrastructure/ e mais nada. Alterar a interface significa interferir diretamente nela. interface/ e mais nada.

    Pydantic e Pydantic AI

    Pydantic Aplica invariantes e valida dados em tempo de execução. Utilize-o para entidades, objetos de valor e, especialmente, para validar as saídas de modelos de linguagem grande.

    Pydantic AI leva isso ainda mais longe: impõe que as respostas dos LLM correspondam aos seus esquemas de domínio. Defina um AddTaskCommand Com os campos obrigatórios, o Pydantic AI valida a saída em JSON do modelo antes mesmo de o seu código a processar.

    Instrutor É outra opção disponível aqui. Ele corrige os clientes da OpenAI (e de outras plataformas) para que retornem modelos Pydantic diretamente, o que representa uma forma leve de implementar uma Camada Anticorrupção.DDDesign — fornece classes base para entidades, repositórios e objetos de valor construídos com Pydantic

    • Versátil — uma estrutura completa para DDD, CQRS e event sourcing, se pretender algo que já venha com muitas funcionalidades prontas para uso imediato

    A maioria dos desenvolvedores em Python ignora estas opções e utiliza classes simples com Pydantic, mas elas valem a pena ser exploradas em projetos de grande escala.

    Ferramentas orientadas a eventos

    Para eventos de domínio, considere:

    • intermitente — distribuidor de eventos leve em processoRedis-Py Pub/Sub ou RabbitMQ** — para eventos distribuídos entre serviços ou agentes
    • padrões de evento asyncio — se já estiver a utilizar async

    Os eventos são essenciais para a orquestração de múltiplos agentes. Um agente emite ResearchCompleted; outros subscrevem-se e reagem. Nenhum agente precisa de saber quem está a ouvir.

    Frameworks de agente

    LangChain, LangGraph, Haystack, Kernel Semântico, LlamaIndex, AutoGen, Google ADK, smolagents, e CrewAI Todos fornecem uma estrutura para os fluxos de trabalho dos agentes. Utilize-os na sua aplicação ou camada de infraestrutura e envolva-os em interfaces que pertencem à sua camada de domínio. Assim, a substituição de frameworks torna-se uma alteração controlada.

    Testes

    Uma vantagem prática do DDD: a camada de domínio pode ser testada sem que toda a pilha esteja em execução.

    • PyTest para testes unitários em entidades e agregados
    • Repositórios falsos (em memória) para testes de integração
    • Stubs de LLM que retornam resultados predeterminados

    O código do seu domínio nunca deve exigir um LLM ativo para executar os seus testes. O LLM é um detalhe de implementação. Os testes validam as regras de negócio.

    Lista de verificação para começar

    Uma ordem prática de operações ao iniciar um novo projeto de agente:

    1. Entrevistar especialistas no domínio. Elabore a linguagem comum utilizada. Anote-a por escrito.
    2. Mapear contextos delimitados. Desenhe os subdomínios e indique onde eles precisam comunicar-se entre si. Comece com um contexto central.
    3. Modelar entidades e objetos de valor. Quais elementos possuem identidade? Quais são apenas valores? Incorpore invariantes nos seus métodos.
    4. Definir raízes de agregado. Agrupe entidades relacionadas sob uma única raiz que aplique as regras de consistência.
    5. Criar interfaces de repositório. Ainda não implemente o armazenamento. Basta defini-las. save() e get(). O domínio permanece alheio à localização dos dados.
    6. Emitir eventos de domínio. Para alterações significativas (pedido feito, tarefa concluída), gere eventos. Conecte os ouvintes posteriormente, conforme necessário.
    7. Envolver as saídas do LLM em esquemas. Utilize modelos Pydantic para impor contratos. Textos em formato livre não devem chegar ao domínio.
    8. Adicionar orquestração. Crie serviços de aplicação que coordenem agentes por meio de comandos ou eventos estruturados.

    A regra realmente importante: comece pelo domínio, não pela pilha tecnológica. Entenda primeiro o problema de negócio. Modelá-lo de forma explícita. Depois, introduza as ferramentas de IA para servir esse modelo.

    Principais Pontos a Retentar

    1. O DDD ajuda os sistemas de agentes porque estes necessitam de limites de negócio claros, e não apenas de listas de ferramentas.
    2. Contextos delimitados impedem que um agente de uso geral acumule responsabilidades não relacionadas.
    3. A linguagem universal é crucial, pois prompts, ferramentas, esquemas e avaliações devem utilizar os mesmos termos do domínio.
    4. A orquestração de agentes torna-se mais simples quando cada agente corresponde a uma capacidade de domínio clara.
