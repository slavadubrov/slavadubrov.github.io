Guia de Serviço LoRAX: Milhares de Adaptadores LoRA em Kubernetes

Servir dezenas de modelos de linguagem grandes otimizados costumava exigir uma GPU por modelo. O LoRAX (LoRA eXchange) mantém um único modelo base na memória e substitui dinamicamente os adaptadores LoRA leves a cada pedido. O custo por token permanece praticamente constante à medida que se adicionam mais otimizações.

Este guia aborda o que é o LoRA, quando escolher o LoRAX em vez do vLLM, como implementá-lo em Kubernetes com o gráfico Helm oficial, e como utilizar as APIs compatíveis com REST, Python e OpenAI.

TL;DR: O LoRAX é útil quando vários adaptadores LoRA partilham um modelo base e exigem um serviço multi-tenant eficiente. As partes mais difíceis são o encaminhamento dos adaptadores, a gestão de memória, o agrupamento de operações, o autoescalonamento e a capacidade de observabilidade, e não a própria matemática por trás do LoRA.

Contexto: o que é o LoRA?

A Adaptação de Baixo Rank (LoRA) congela os pesos do modelo pré-treinado e insere matrizes pequenas de decomposição de rank em cada camada Transformer. Em vez de retreinar todo o modelo, treina-se um conjunto reduzido de “diferenças” que capturam o novo comportamento desejado.

Um ajuste fino completo num modelo de 7B resulta num ficheiro de 20GB ou mais. Um adaptador LoRA para o mesmo modelo tem cerca de 100MB. É essa diferença que torna possível o serviço dinâmico: é possível armazenar milhares de adaptadores no disco e carregar um deles para a memória da GPU em meros milissegundos.

O problema resolvido pelo LoRAX

O serviço de múltiplos modelos da forma tradicional é dispendioso. Cada modelo ajustado finamente necessita da sua própria memória GPU; portanto, para fornecer serviços a 50 modelos específicos de clientes, são necessárias 50 instalações, ou pelo menos 50 vezes mais memória. O custo aumenta linearmente com cada nova variante.

LoRAX é um projeto Apache 2.0 proveniente de Predibase Estende o Servidor de inferência para geração de texto no Hugging Face Com carregamento dinâmico de adaptadores, cache de pesos em camadas e agrupamento contínuo de múltiplos adaptadores. Juntos, estes mecanismos permitem servir centenas de adaptadores LoRA específicos de cada cliente numa única GPU da classe Ampere, sem perda de taxa de transferência ou latência.

O truque está no facto de o ajuste fino LoRA gerar apenas pesos delta pequenos, em vez de cópias completas do modelo. O LoRAX mantém apenas o modelo base na GPU e injeta os pesos dos adaptadores conforme necessário. Os adaptadores que não estão a ser utilizados não consomem qualquer espaço em VRAM.

Como funciona

Carregamento dinâmico de adaptadores

Os pesos dos adaptadores são injetados exatamente no momento certo para cada pedido. O modelo base permanece na memória da GPU, enquanto os adaptadores são carregados dinamicamente, sem bloquear outros pedidos. É possível catalogar milhares de adaptadores, mas os custos de memória aplicam-se apenas aos que estão ativamente a processar tráfego.

Cache de pesos em camadas

O LoRAX organiza os adaptadores em três níveis: VRAM da GPU para os mais utilizados, RAM do CPU para os menos frequentes, e disco para armazenamento a longo prazo. Esta hierarquia evita falhas devido à falta de memória e mantém os tempos de swap suficientemente rápidos para que os utilizadores não percebam diferença.

Agrupamento contínuo de múltiplos adaptadores

É aqui que o LoRAX altera o comportamento de agrupamento. Ele estende o agrupamento contínuo para funcionar em paralelo com diferentes adaptadores, permitindo que pedidos relacionados com ajustes finos diferentes compartilhem a mesma passagem de processamento. Os testes da Predibase mostram que processar 1M de tokens distribuídos por 32 adaptadores diferentes leva aproximadamente o mesmo tempo que processar 1M de tokens num único modelo.

TGI como base

LoRAX baseia-se na Text Generation Inference (TGI) da Hugging Face, pelo que herda as otimizações desta: FlashAttention 2, atenção paginada, kernels SGMV para inferência com múltiplos adaptadores e respostas em fluxo contínuo. Trata-se, portanto, da TGI acrescida de uma capacidade dinâmica de alternância de adaptadores.

O custo por token permanece praticamente constante

O gráfico ilustra bem este ponto. As implementações dedicadas (cinzento escuro) escalam de forma linear: duplicar os modelos significa duplicar o custo. Já o LoRAX (laranja) mantém o custo por token quase estável à medida que se adicionam mais adaptadores. Até mesmo os ajustes finos de APIs hospedadas por fornecedores como a OpenAI (cinzento claro) não conseguem competir com ele em cenários de workload com vários modelos.

Fluxo de solicitação

Quando utilizar o LoRAX

O LoRAX faz sentido em algumas situações específicas.

  • SaaS multi-tenant. Está a desenvolver uma plataforma na qual cada um dos 500 clientes recebe um chatbot otimizado com os seus dados. A abordagem tradicional exigiria 500 implementações de modelos. O LoRAX serve todos os 500 clientes a partir de uma única GPU, carregando o adaptador relevante sempre que chega um pedido de um cliente.
  • Encaminhamento por especialistas em domínios específicos. Mantém LLMs especializados para direito, medicina, finanças e engenharia. Em vez de quatro implementações separadas de 13B, o LoRAX executa um único modelo base LLaMA 2 de 13B e encaminha a solicitação para o adaptador adequado em função do domínio solicitado.
  • Experimentação rápida. Está a testar 10 abordagens de otimização em produção? Implemente o LoRAX uma única vez e alterne entre as variantes modificando o adapter_id parâmetro. Sem alterações na infraestrutura nem reinícios de serviço.
  • Implementações em ambientes com recursos limitados ou em edge. Um único NVIDIA A10G consegue alojar um modelo base quantizado de 7B, além de dezenas de adaptadores específicos para tarefas, em vez de uma GPU por modelo.

Arquitetura: hierarquia de memória e agendamento de solicitações

O LoRAX é construído com base numa hierarquia de memória de três níveis. Compreendê-la ajuda a prever o desempenho e a planear a capacidade necessária.

LoRAX trata cada adaptador como uma “visão” leve sobre o modelo base partilhado. O agendador agrupa os pedidos de forma a que a execução de 32 adaptadores diferentes seja tão rápida quanto a execução de um só, mesmo com um volume de tráfego de um milhão de tokens. Geralmente, cada adaptador ocupa entre 10 e 200 MB, em comparação com modelos completos que chegam a vários gigabytes.

Implementar o LoRAX em Kubernetes

O LoRAX é fornecido com diagramas Helm e imagens Docker, pelo que a implementação em Kubernetes é simples.

Pré-requisitos

Você precisará de:

  • Um cluster Kubernetes com GPUs NVIDIA (geração Ampere ou superior: A10, A100, H100) NVIDIA Container Runtime configurado em nós GPU kubectl e helm instalado localmente
  • Armazenamento persistente para caches de adaptadores; monte um PersistentVolume para /data dentro do pod

Início rápido com o chart Helm oficial

Helm É o gestor de pacotes para Kubernetes. Ele agrupa todos os recursos necessários a uma aplicação no Kubernetes (Deployments, Services, ConfigMaps, etc.) num único “chart”, permitindo assim implementar toda a estrutura com um único comando, em vez de gerir manualmente dezenas de ficheiros YAML.

# Clone the LoRAX repository and switch into it
git clone https://github.com/predibase/lorax.git
cd lorax

# Make sure kubectl can talk to your cluster
kubectl config current-context
kubectl get nodes

# Build chart dependencies (generates charts/lorax/charts/*.tgz)
helm dependency update charts/lorax

# Optional: render manifests locally to verify everything is templating
helm template mistral-7b-release charts/lorax > /tmp/lorax-rendered.yaml

# Deploy with default settings (Mistral-7B-Instruct)
helm upgrade --install mistral-7b-release charts/lorax

# Watch the pod come up
kubectl get pods -w

# Check logs to see model loading progress
kubectl logs -f deploy/mistral-7b-release-lorax

O gráfico cria uma instância de Deployment (uma réplica por predefinição) e um serviço ClusterIP que escuta na porta 80. Na primeira inicialização, o modelo base é descarregado da Hugging Face e carregado na memória da GPU, o que pode demorar alguns minutos dependendo da sua rede e da GPU utilizada. Nas reinicializações subsequentes, são reutilizados os pesos armazenados em cache a partir do volume persistente.

Dica: Se helm upgrade --install devolve Kubernetes cluster unreachableO seu contexto kubeconfig aponta para um cluster que está offline. Inicie o seu cluster local (Docker Desktop, kind, minikube) ou altere para um contexto acessível. kubectl config use-context. Em execução kubectl get nodes antes de implementar, confirma-se se o servidor API está ativo.

Personalizar o modelo base e o escalonamento

É possível substituir um modelo base por outro ou ajustar os recursos criando um ficheiro de valores personalizado. Eis um exemplo llama2-values.yaml:

# Use LLaMA 2 7B Chat instead of Mistral
modelId: meta-llama/Llama-2-7b-chat-hf

# Enable 4-bit quantization to save VRAM
modelArgs:
    quantization: "bitsandbytes"

# Scale to 2 replicas for high availability
replicaCount: 2

# Request exactly 1 GPU per pod
resources:
    limits:
        nvidia.com/gpu: 1

Implante com a sua configuração personalizada:

helm upgrade --install -f llama2-values.yaml llama2-chat-release charts/lorax

Execute esses comandos a partir do clonado lorax/ repositório para que o Helm possa localizar o diretório do chart.

O LoRAX suporta, de forma pronta para uso, os modelos open-source mais populares: LLaMA 2, CodeLlama, Mistral, Mixtral, Qwen e outros. Verifique o lista de compatibilidade de modelos para as adições mais recentes.

Exposição do serviço

O tipo de serviço predefinido é ClusterIP, que permite acesso apenas a partir do interior do cluster. Para tráfego externo, pode-se:

  • Criar um serviço LoadBalancer (em fornecedores cloud)
  • Configurar um Ingress com terminação TLS
  • Colocar uma gateway API à frente para autenticação e limitação de taxa

Limpeza

Quando terminar os testes, liberte os recursos GPU:

helm uninstall mistral-7b-release

Isso remove a Implantação, o Serviço e todos os pods. Os pesos do modelo em cache permanecem no PersistentVolume, a menos que sejam eliminados separadamente.

Trabalhar com as APIs LoRAX

Uma vez implantado, o LoRAX oferece três formas de interagir com ele: uma API REST compatível com o Hugging Face TGI, uma biblioteca de cliente em Python e um endpoint compatível com a OpenAI. Todas estas opções suportam a troca dinâmica de adaptadores.

API REST /generate o endpoint aceita cargas JSON com o seu prompt e parâmetros opcionais. Ao utilizar o modelo base sem qualquer adaptador:

# Basic request to the base model (no adapter)
curl -X POST http://localhost:8080/generate \
  -H "Content-Type: application/json" \
  -d '{
    "inputs": "Write a short poem about the sea.",
    "parameters": {
      "max_new_tokens": 64,
      "temperature": 0.7
    }
  }'

A resposta inclui o texto gerado e metadados como contagens de tokens e tempos de execução.

Adicionar um adapter_id parâmetro para apontar para um modelo afinado. Eis um exemplo que utiliza um adaptador especializado em matemática:

curl -X POST http://localhost:8080/generate \
  -H "Content-Type: application/json" \
  -d '{
    "inputs": "Natalia sold 48 clips in April, and then half as many in May. How many clips did she sell in total?",
    "parameters": {
      "max_new_tokens": 64,
      "adapter_id": "vineetsharma/qlora-adapter-Mistral-7B-Instruct-v0.1-gsm8k"
    }
  }'

Na primeira chamada com um novo adapter_id, o LoRAX descarrega o adaptador a partir do Hugging Face Hub e armazena-o em cache em /data. Os pedidos subsequentes utilizam a versão em cache. Pode também carregar adaptadores a partir de caminhos locais, definindo "adapter_source": "local" juntamente com um caminho de ficheiro.

Cliente Python

Para acesso programático, instale o lorax-client package:

pip install lorax-client

O cliente envolve a API REST numa interface limpa:

from lorax import Client

# Connect to your LoRAX instance (default port 8080)
client = Client("http://localhost:8080")

prompt = "Explain the significance of the moon landing in 1969."

# 1. Generate using the base model (no adapter loaded)
base_response = client.generate(prompt, max_new_tokens=80)
print("Base model:", base_response.generated_text)

# 2. Generate using a fine-tuned adapter
# The adapter_id can be a Hugging Face repo ID or a local path
adapter_response = client.generate(
    prompt,
    max_new_tokens=80,
    adapter_id="alignment-handbook/zephyr-7b-dpo-lora",
)
print("With adapter:", adapter_response.generated_text)

O cliente suporta transmissão em fluxo, parâmetros de decodificação (temperatura, top-p, penalização por repetição) e detalhes a nível de token. Consulte o referência de cliente para utilização avançada.

Ponta de extremidade compatível com OpenAI

O LoRAX implementa a API OpenAI Chat Completions sob o /v1 caminho. Isso permite integrar o LoRAX em ferramentas que exigem o formato de API da OpenAI: LangChain, Semantic Kernel ou aplicações personalizadas. model campo para especificar qual adaptador deve ser carregado:

import openai

# Point the OpenAI client at LoRAX
openai.api_key = "EMPTY"  # LoRAX doesn't require an API key by default
openai.api_base = "http://localhost:8080/v1"

# The model parameter becomes the adapter_id
# This allows seamless integration with tools like LangChain
response = openai.ChatCompletion.create(
    model="alignment-handbook/zephyr-7b-dpo-lora",
    messages=[
        {"role": "system", "content": "You are a friendly chatbot who speaks like a pirate."},
        {"role": "user", "content": "How many parrots can a person own?"},
    ],
    max_tokens=100,
)

print(response["choices"][0]["message"]["content"])

Dois casos de uso práticos resultam disso:

  1. Substituição imediata. Migre aplicações existentes dos modelos hospedados pela OpenAI para a sua própria infraestrutura, alterando apenas uma linha de configuração.
  2. Integração com ferramentas. Utilize o LoRAX com qualquer framework que já suporte a API da OpenAI, sem necessidade de código de adaptador personalizado.

A primeira solicitação a um novo adaptador apresenta maior latência, enquanto o LoRAX o descarrega e carrega. Em aplicações voltadas para o utilizador, planeie isso pré-carregando adaptadores populares ou exibindo um estado de carregamento.

Compromissos

O que o LoRAX faz bem

  • Muitos modelos numa única GPU. Centenas ou milhares de modelos otimizados numa única GPU, em vez de uma implementação por modelo. O custo permanece quase constante à medida que se adicionam adaptadores.
  • Sem memória ociosa. Os adaptadores são carregados conforme necessário. Os modelos não utilizados não consomem VRAM alguma. É possível manter um catálogo com 1.000+ modelos especializados e pagar apenas pelos poucos que estão ativamente a processar tráfego.
  • Desempenho estável. O agrupamento contínuo de vários adaptadores mantém a latência e o throughput próximos dos valores obtidos com um único modelo. Os testes da Predibase demonstram que a execução de 32 adaptadores em paralelo gera pouco overhead em relação à execução de um só.
  • Baseado no TGI da Hugging Face. Desenvolvido com base no Hugging Face TGI, pelo que beneficia do FlashAttention 2, atenção paginada, streaming e kernels SGMV para inferência com vários adaptadores.
  • Completo do ponto de vista operacional. Imagens Docker, charts Helm, métricas Prometheus, rastreio OpenTelemetry. Licenciado sob Apache 2.0, sem restrições comerciais.
  • Ampla compatibilidade de modelos. Funciona com LLaMA 2, CodeLlama, Mistral, Mixtral, Qwen e outros. Suporta quantização (4 bits através de bitsandbytes, GPTQ, AWQ) para reduzir o consumo de memória.

Limitações

  • Apenas LoRA. Todos os adaptadores devem resultar de um ajuste fino no estilo LoRA do mesmo modelo base. Os ajustes finos completos que geram modelos independentes não funcionarão sem conversão. Arquiteturas de base diferentes exigem implementações separadas de LoRAX.
  • Início em frio. O primeiro pedido após a inicialização carrega o modelo base na memória da GPU (de 30 a 90 segundos para modelos maiores). O primeiro pedido a um novo adaptador apresenta latência de download a partir da Hugging Face. Planeie a operação tendo isso em conta, através de verificações de saúde e pré-carregamento.
  • Problemas de cache sob carga intermitente. Se o tráfego atingir subitamente dezenas de adaptadores diferentes, o LoRAX tem de mover pesos entre a GPU, a RAM da CPU e o disco. A troca de adaptadores a partir da RAM demora cerca de 10 ms, mas um conjunto de trabalho muito grande pode causar abrandamentos temporários. Monitore a memória da GPU e as taxas de acerto do cache dos adaptadores.
  • Projeto em constante evolução. O LoRAX foi criado a partir do TGI no final de 2023 e sofre alterações rápidas. Espere atualizações frequentes e mudanças que possam quebrar funcionalidades, à medida que acompanha as evoluções do TGI original. Fixe versões em produção.

LoRAX vs. vLLM

vLLM É outro motor de servir de alto desempenho, que adicionou suporte a multi-LoRA mais recentemente. Os dois resolvem problemas diferentes.

FuncionalidadeLoRAXvLLM
Foco principalEscala massiva: centenas ou milhares de adaptadoresElevado débito: número máximo de tokens/s para um menor número de adaptadores ativos
ArquiteturaTroca dinâmica; transferência intensiva para CPU/discoLoteamento otimizado para execução concorrente de adaptadores ativos
Ideal paraSaaS de cauda longa: milhares de inquilinos, utilização esporádicaNíveis de alto tráfego: 5–10 adaptadores com elevado volume de utilização
BaseHugging Face TGIMecanismo Custom PagedAttention

Escolha o LoRAX se tiver uma grande quantidade de adaptadores (um por utilizador, na sua maioria inativos na maior parte do tempo), nos quais o cache em camadas se revela eficaz. Escolha o vLLM se tiver um conjunto pequeno de adaptadores muito ativos e a largura de banda bruta for o fator mais importante.

Introdução

Um roteiro prático do protótipo até à produção:

1. Comece por algo pequeno

Implemente o LoRAX utilizando o modelo de base que já está a utilizar, juntamente com 3 a 5 adaptadores representativos. Verifique se o carregamento dos adaptadores funciona corretamente e meça a latência de referência para o seu workload.

2. Medir e criar perfis

  • Acompanhe as taxas de sucesso do cache do adaptador e a memória da GPU sob tráfego realista.
  • Identifique os adaptadores mais utilizados (os 20% com maior volume de solicitações) e considere carregá-los antecipadamente na inicialização.
  • Medir a latência P50, P95 e P99 tanto para cargas no adaptador em cache como para cargas no adaptador frio.

3. Otimizar para a sua carga de trabalho

  • Se alguns adaptadores forem muito utilizados, aumente a alocação de memória GPU para manter mais deles ativos.
  • Se a utilização for distribuída de forma desigual entre centenas de adaptadores, ajuste o cache em camadas para equilibrar RAM e disco.
  • Utilize quantização (4-bit bitsandbytes ou GPTQ) se a VRAM estiver limitada.

4. Escalar horizontalmente

Uma vez compreendido o comportamento de instância única, adicione réplicas para garantir alta disponibilidade. Coloque um balanceador de carga à frente que direcione o tráfego de acordo com adapter_id Assim, os pedidos para o mesmo adaptador são direcionados à mesma réplica. Isso melhora a localidade no cache.

5. Monitorização

Configure painéis de controlo para a utilização da GPU, métricas do cache do adaptador e latência dos pedidos, discriminadas por adaptador. Fique atento a situações de “cache thrashing” durante picos de tráfego e ajuste a escala em conformidade.

Com o LoRAX, a execução de N ajustes finos torna-se um problema de roteamento num único GPU, em vez de um problema de provisionamento em N GPUs.

Principais Conclusões

  1. O fornecimento de adaptadores LoRA é um problema de roteamento e gestão de memória.
  2. Um modelo base acompanhado por vários adaptadores pode ser mais económico do que um modelo totalmente ajustado para cada utilizador.
  3. O agrupamento de pedidos só funciona se a camada de fornecimento compreender a localização dos adaptadores e o custo de substituição.
  4. O Kubernetes oferece vantagens operacionais, mas apenas se as métricas revelarem a latência a nível de adaptador, os acertos no cache e as falhas.
