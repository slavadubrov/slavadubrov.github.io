Guia de Ajuste Fino de LLMs: LoRA, QLoRA, DoRA, Unsloth, Axolotl e Implementação

A maioria dos projetos de ajuste fino que vi falhou não durante o treinamento, mas nas etapas anteriores e posteriores a ele: dados de má qualidade, modelo base inadequado e ausência de avaliação real. O próprio processo de treinamento é a parte mais simples. Este guia seria algo que eu gostaria de ter tido antes do meu primeiro ajuste fino sério — indicando quando realizá-lo, quando evitá-lo, os métodos eficazes atualmente (LoRA, QLoRA, DoRA, ORPO) e como levar um modelo de um notebook até um ponto de serviço.

TL;DR: Faça ajustes finos apenas quando o prompt e o RAG não conseguem alterar de forma fiável o comportamento, o formato, a latência ou o custo. Comece com LoRA ou QLoRA, crie um conjunto de dados limpo antes de modificar os hiperparâmetros, avalie-o com um conjunto de tarefas reservado e só então decida se um ajuste fino completo vale o custo de infraestrutura.

Deve mesmo fazer ajustes finos?

Antes de gastar horas em GPUs, decida se os ajustes finos são a ferramenta adequada para o problema que tem à sua frente.

Ajuste fino versus Raciocínio Guiado por Esquema (SGR)

Bibliotecas como xgrammar e outlines Restringir as saídas do modelo em tempo de inferência através de máquinas de estado finito. Estas funcionam com modelos base diretamente, sem necessidade de treino adicional.

O valor do SGR não reside apenas em saídas estruturadas. Trata-se da consistência: a mesma forma de entrada produz sempre a mesma forma de saída, sem ser necessário retrainar nada. Quando o simples uso de prompts gera resultados inconsistentes, o SGR fixa o padrão de saída no momento da decodificação.

Uma ordem prática:

  1. Comece com prompts e exemplos de few-shot para a formatação básica.
  2. Adicionar SGR (xgrammar ou outlines) quando o formato é inconsistente.
  3. Faça o ajuste fino apenas quando precisar de alterações de comportamento que um esquema não consegue impor.

Quando a matemática realmente funciona

O ajuste fino compensa-se quando o tráfego é elevado e os requisitos permanecem estáveis. Uma configuração RAG robusta costuma carregar 2.000 tokens de contexto em cada chamada: prompt do sistema, documentos recuperados e exemplos de few-shot. Trata-se de um custo adicional de contexto que se paga em cada solicitação.

Um modelo afinado absorve a maior parte dessas instruções nos seus pesos, fazendo com que o número de tokens necessários para um prompt passe de ~2.000 para ~50. Com volume suficiente, isso permite recuperar o custo computacional de treino em poucas semanas.

[!DICA] A configuração híbrida O padrão que observo frequentemente em produção: um modelo afinado de 8B combinado com um pequeno recuperador RAG para obter factos. Geralmente, supera um modelo de 70B+ com prompt em termos tanto de precisão como de custo.

Tipos de afinação

Existem três formas principais pelas quais a afinação pode ser realizada. Elas diferem em relação aos tipos de dados necessários e ao que são ensinados ao modelo.

1. Treino prévio contínuo (não supervisionado)

Treina-se o modelo base com mais texto bruto, sem etiquetas. Ele continua a realizar a previsão do token seguinte, tal como durante a fase de treino prévio inicial.

Quando utilizá-lo:

  • O domínio possui um vocabulário que o modelo base nunca viu (médico, jurídico, bases de código internas).
  • Possui grandes quantidades de texto relacionado com o domínio, mas sem pares de entrada/saída rotulados.
  • O modelo base tem dificuldades em lidar com terminologia específica do domínio.

Exemplo: treinar com milhões de notas clínicas para que o modelo aprenda abreviaturas médicas, nomes de medicamentos e fluxos de trabalho clínicos.

2. Ajuste fino supervisionado (SFT)

O SFT é realizado com pares de entrada/saída rotulados. É apresentado ao modelo a saída exata desejada para cada entrada.

Quando utilizá-lo:

  • Tem uma tarefa específica com um formato de entrada/saída bem definido.
  • Disponibiliza dados rotulados de qualidade, mesmo que em pequenas quantidades.
  • Precisa de um comportamento previsível para um formato de entrada conhecido.

Exemplo: treinar com pares de (descrição da consulta SQL, código SQL) para a conversão de texto em SQL.

{
    "input": "Get all users who signed up last month",
    "output": "SELECT * FROM users WHERE signup_date >= DATE_SUB(NOW(), INTERVAL 1 MONTH)"
}

3. Ajuste de instruções

O ajuste de instruções é um caso especial do SFT, concebido para fazer com que os modelos sigam uma grande variedade de instruções em linguagem natural. Os dados de treino consistem em pares (instrução, resposta) provenientes de diversas tarefas diferentes.

Quando utilizá-lo:

  • Pretende um assistente multiúso (como o ChatGPT ou o Claude).
  • O modelo precisa de ser capaz de lidar com solicitações variadas e abertas.
  • Está a desenvolver uma interface de chat.

Exemplo: treino com milhares de instruções diversas, como “Resumir este artigo”, “Escrever um poema sobre X” e “Explicar Y de forma simples”.

O pipeline de ajuste fino em 7 fases

O ajuste fino é um processo em várias etapas, e não uma única instrução. Cada fase apresenta os seus próprios modos de falha, e ignorar alguma delas geralmente resulta num modelo de qualidade inferior mais tarde.

Cada fase baseia-se na anterior:

  1. Preparação de dados — Limpe, elimine duplicatas e formate os seus dados (passo com maior impacto)
  2. Seleção do modelo — Escolha o modelo base adequado e carregue os pesos
  3. Configuração do treino — Defina o hardware, os hiperparâmetros e a estratégia de otimização
  4. Ajuste fino — Execute treinos do tipo SFT, DPO ou ORPO
  5. Avaliação — Faça testes de desempenho e valide a qualidade
  6. Implantação — Exporte e disponibilize o seu modelo
  7. Monitorização — Acompanhe o desempenho, faça manutenção e realize iterações

[!ADVERTÊNCIA] Os dados são a base A Fase 1 (preparação de dados) é o passo com maior potencial de retorno. Os algoritmos não conseguem corrigir falhas presentes nos dados utilizados para o treino. Geralmente, 500-1.000 exemplos cuidadosamente selecionados superam 50.000 exemplos ruidosos.

Fase 1: Preparação de dados

A maioria dos projetos de ajuste fino falha aqui, e não durante o treino. A preparação de dados moderna vai além da simples aplicação de expressões regulares em ficheiros CSV.

O pipeline de dados em 5 fases

As equipas que implementam este processo de forma séria dependem de ferramentas como o DataTrove (Hugging Face) e o Distilabel (Argilla), em vez de scripts personalizados.

1. Ingestão e filtragem

  • Ação: descartar respostas negativas (“Não consigo responder a isso”), dados com UTF-8 corrompido e idiomas não alvo.
  • Ferramentas: Trafilatura para extração, FastText para identificação de idioma.

2. Remoção de PII (obrigatória para empresas)

  • Ação: detetar e redigir e-mails, endereços IP e números de telefone antes do treino.
  • Ferramentas: Microsoft Presidio ou scrubadub.
  • Porquê: o treino com PII de clientes é um incidente de segurança iminente.

3. Deduplicação (MinHash LSH)

  • Ação: remover duplicatas quase idênticas para que o modelo não as memorize.
  • Ferramentas: O DataTrove lida bem com processamento em escala de terabytes.

4. Aumento sintético

  • Ação: utilizar um modelo “professor” mais potente (GPT-4o, DeepSeek-V3) para reescrever dados brutos em pares instrução-resposta limpos.
  • Ferramentas: Distilabel.
  • Por que é importante: é geralmente aqui que se obtém o maior ganho em qualidade.

5. Formatação

  • Ação: converter para um formato padrão (Alpaca ou ShareGPT).

Exemplos de formato de dados

Formato Alpaca (seguimento de instruções):

{
    "instruction": "Summarize the following text.",
    "input": "The text to be summarized...",
    "output": "This is the summary."
}

Formato ShareGPT/ChatML (conversacional):

{
    "conversations": [
        { "from": "user", "value": "Hello, who are you?" },
        { "from": "assistant", "value": "I am a helpful AI assistant." }
    ]
}

O que realmente importa

  • Qualidade em vez de quantidade. 500–1.000 exemplos cuidadosamente selecionados superam 50.000 exemplos de baixa qualidade.
  • Limpeza. Remova texto irrelevante, normalize os espaços em branco e mantenha a formatação consistente.
  • Equilíbrio. Distribua a cobertura por tópicos para que o modelo não se sobreajuste ao segmento que predomina.
  • A remoção de dados PII é indispensável em qualquer sistema de produção.

Matemática de memória

Para fazer o ajuste fino de um modelo de 72B, a GPU tem de suportar:

  • Peso do modelo com precisão de 16 bits: ~144 GB.
  • Gradientes e estado do otimizador: aproximadamente 2-3 vezes o tamanho do modelo, dependendo do otimizador (o AdamW está no valor mais elevado).
  • Ativações: aumentam com o comprimento do contexto, por exemplo, 32k tokens.

É por isso que o QLoRA (base de 4 bits + adaptadores LoRA) é a opção prática padrão para a maioria das equipas.

Fase 3: Métodos de treino (PEFT e LoRA)

Ajuste fino total vs PEFT

O ajuste fino total (FFT) atualiza todos os pesos. Um modelo de 7B com precisão de 16 bits necessita de cerca de 112 GB de VRAM apenas para ser treinado. Isso está fora do alcance da maioria das equipas.

O ajuste fino eficiente em termos de parâmetros (PEFT) treina apenas um pequeno subconjunto de parâmetros, congelando o resto. A matemática torna-se muito mais acessível.

LoRA: a opção padrão

A LoRA (Adaptação de Baixo Rank) é a técnica PEFT a ser aprendida primeiro. A ideia por trás dela é que as alterações nos pesos realmente necessárias durante o ajuste fino têm um “rank intrínseco” baixo, portanto não são necessárias atualizações de rank completo.

Em vez de atualizar uma matriz de pesos completa W (d × d), o LoRA aprende duas matrizes menores:

  • A (d × r) — projeção descendente
  • B (r × d) — projeção ascendente

A atualização é ΔW = B × A.

Com um rank de r=16, isso reduz os parâmetros treináveis em aproximadamente 10.000 vezes, diminuindo a VRAM de 120GB para 16GB.

Quando escolher qual

  • LoRA: comece por aqui. Rápido, eficiente em termos de memória e com excelente suporte.
  • QLoRA: quando se pretende fazer ajustes finos num modelo de 70B com GPUs de consumo.
  • DoRA: quando o teto de qualidade das LoRA se torna evidente, especialmente em tarefas de raciocínio mais complexas.
  • HFT: quando o LoRA não é suficiente, mas o ajuste fino completo é demasiado dispendioso.

DoRA: LoRA decomposta por pesos

DoRA (Adaptação de Baixo Rank Decomposta por Peso) divide cada peso pré-treinado em magnitude e direção, treinando-os separadamente. Geralmente, consegue reduzir significativamente a diferença entre o método LoRA e o ajuste fino completo.

Como funciona:

Em vez de tratar os pesos como uma entidade única, o DoRA divide os pesos pré-treinados em duas componentes:

  1. Magnitude — um escalar treinável por coluna que controla a “força”.
  2. Direção — atualizada com matrizes LoRA, que controlam “o quê”.

A atualização é W’ = m × (V + B × A), onde:

  • m = magnitude (treinável) V = direção (O / ||W||)
  • B × A = Atualização LoRA na direção desejada

O que você obtém com essa estrutura adicional:

  • Atualizações de parâmetros mais ricas sem abrir mão da eficiência.
  • Qualidade próxima à do ajuste fino completo.
  • Mesma economia de memória de aproximadamente 10× em relação ao LoRA.
  • Efeito particularmente notável em tarefas de raciocínio mais complexas.

Ajuste fino parcial (HFT)

O HFT situa-se entre o ajuste fino completo e os métodos PEFT:

  • Método: em cada rodada, metade dos parâmetros do modelo é congelada e a outra metade é atualizada.
  • Estratégia: a metade congelada muda a cada rodada.
  • Por que funciona: a metade congelada preserva o conhecimento prévio, enquanto a metade ativa aprende novos comportamentos.
  • Quando utilizá-lo: o LoRA não está a ser suficiente, mas você não pode permitir-se um ajuste fino completo.

Fusão de adaptadores para aprendizagem multi-tarefa

Em vez de ajustar um modelo monolítico para várias tarefas, treine um adaptador pequeno separado para cada tarefa, mantendo o modelo base congelado. Em seguida, pode fundir os adaptadores no momento da execução.

Métodos comuns de fusão:

  1. Concatenação — combina os parâmetros dos adaptadores e aumenta o rank efetivo. Rápido e simples.
  2. Combinação linear — soma ponderada dos adaptadores. Permite ajustes personalizados.
  3. SVD — decomposição matricial para fusão. Mais flexível, mas mais lenta.

Exemplo: um adaptador para resumo, outro para tradução, fundidos num único modelo multi-tarefa.

Fase 4: Ajuste fino e alinhamento de preferências

Às vezes, o SFT dá a resposta correta, mas com estilo inadequado: excessivamente verboso, fora de tom ou, ocasionalmente, inseguro. O alinhamento de preferências é o que resolve isso.

RLHF com PPO (a abordagem mais antiga)

A solução original consistia num pipeline em três fases:

  1. SFT — aprendizagem da tarefa.
  2. Modelo de recompensa — treino com preferências humanas (escolhidas vs. rejeitadas).
  3. PPO (Proximal Policy Optimization) — aprendizagem por reforço para otimizar a política.

Os principais problemas:

  • Complexidade na implementação e manutenção.
  • Custo elevado — é necessário treinar vários modelos.
  • Sensibilidade aos hiperparâmetros e tendência a treinos instáveis.

DPO (o sucessor mais simples)

O DPO (Direct Preference Optimization) elimina o modelo de recompensa explícito e o ciclo de RL. Continua a otimizar o objetivo do RLHF (maximização de recompensas com restrição de divergência KL), mas faz isso como um problema de aprendizagem supervisionada reparametrizado, em vez de através de aprendizagem por reforço:

{
    "prompt": "Explain quantum computing",
    "chosen": "Quantum computing uses qubits...",   # Preferred response
    "rejected": "Well, it's complicated..."        # Non-preferred response
}

O que obtém:

  • Caminho de código mais simples (sem modelo de recompensa separado, sem ciclo de RL).
  • Treino mais estável, uma vez que se trata de aprendizagem supervisionada tradicional.
  • Custo computacional reduzido.

O DPO venceu a maioria dos cenários práticos, mas o PPO ainda não foi completamente substituído:

  • O DPO pode gerar soluções tendenciosas em algumas configurações.
  • Um PPO bem ajustado continua a produzir resultados de ponta em tarefas mais difíceis, como a geração de código.
  • O sinal de recompensa explícito do PPO oferece orientações mais detalhadas para tarefas especializadas.

Se estiver a começar agora, deve optar primeiro pelo DPO (ou por ORPO abaixo) e recorrer ao PPO apenas se houver um motivo real para isso.

ORPO (único estágio)

O ORPO (Odds-Ratio Preference Optimization) combina o SFT e o alinhamento de preferências num único processo de treino. Para a maioria das equipas, isso representa uma simplificação significativa.

Como funciona: O ORPO utiliza uma perda combinada que realiza duas funções simultaneamente:

  1. Maximiza a probabilidade da resposta selecionada (aprendizagem da tarefa).
  2. Penaliza a resposta rejeitada através de um termo de razão de probabilidades (aprendizagem de preferências).

Hipermétricas importantes a conhecer:

from trl import ORPOConfig

config = ORPOConfig(
    learning_rate=8e-6,  # Very low, as recommended by the ORPO paper
    beta=0.1,            # Controls strength of preference penalty
    # ... other params
)
  • Taxa de aprendizagem: deve permanecer muito baixa (8e-6 no artigo do ORPO).
  • Beta: controla a intensidade com que a penalização por preferência atua (0,1 é um valor típico).

O que se obtém com o ORPO:

  • Uma fase de treino em vez de duas.
  • Sem modelo de recompensa.
  • Menos hiperparâmetros em comparação com o DPO.
  • O caminho mais curto do modelo bruto para o modelo alinhado.

Quando escolher qual método:

  • ORPO: a opção padrão para a maioria dos casos de uso.
  • DPO: quando se deseja maior controle sobre o alinhamento.
  • PPO: quando é necessário um sinal de recompensa explícito, como na geração de código.

Frameworks de ajuste fino

O ecossistema consolidou‑se em torno de quatro ferramentas principais.

Unsloth — velocidade e eficiência de memória

O Unsloth encapsula as funcionalidades do HuggingFace’s trl e transformers e adiciona uma camada de otimizações em cima disso:

  • Núcleos GPU Triton personalizados para atenção, RoPE e entropia cruzada, que eliminam o overhead do PyTorch.
  • Backpropagation eficiente em termos de memória, que recalcula as ativações durante a passagem reversa em vez de as armazenar na memória.
  • Operações fundidas que combinam vários passos (normalização por camada + linear e similares) em uma única chamada ao GPU.
  • Quantização de 4 bits integrada diretamente no fluxo QLoRA, com desquantização otimizada.

[!IMPORTANTE] A ordem de importação é importante O Unsloth aplica patches aos pacotes do HuggingFace no momento da importação. Importe e inicialize FastLanguageModel de unsloth antes de importar qualquer coisa de trl ou transformers, caso contrário, as correções não terão efeito.

# ✅ Correct order
from unsloth import FastLanguageModel  # Must be first!
from trl import SFTTrainer
from transformers import TrainingArguments

# ❌ Wrong order - will fail
from trl import SFTTrainer
from unsloth import FastLanguageModel  # Too late!

Optimizado para: treino com uma única GPU, prototipagem, notebooks do Colab e para quem quer controlar os custos relacionados com a GPU.

Vantagem principal: os kernels personalizados do Triton permitem um desempenho 2 a 5 vezes superior em comparação com a abordagem padrão da HuggingFace.

Configuração

As partes interessantes encontram-se em config.py:

# Model & Dataset
MODEL_NAME = "nvidia/Llama-3.1-Nemotron-Nano-4B-v1.1"  # 4B params, 128K context

# LoRA Configuration

LORA_ALPHA = 32    # Scaling factor - usually 2x LORA_R

    "q_proj", "k_proj", "v_proj", "o_proj",
    "gate_proj", "up_proj", "down_proj",
]

[!NOTA] Rácio alfa/rank Uma regra prática comum: defina alpha = 2 × rank (por exemplo, rank=16, alpha=32). Isto permite atualizações de peso mais intensas sem desestabilizar o treino.

Código principal de treino

from unsloth import FastLanguageModel
from trl import SFTTrainer
from transformers import TrainingArguments

# Load model with 4-bit quantization
model, tokenizer = FastLanguageModel.from_pretrained(
    model_name="nvidia/Llama-3.1-Nemotron-Nano-4B-v1.1",
    max_seq_length=4096,
    load_in_4bit=True,
)

# Add LoRA adapters
model = FastLanguageModel.get_peft_model(
    model,
    r=16,
    lora_alpha=32,
    target_modules=["q_proj", "k_proj", "v_proj", "o_proj",
                    "gate_proj", "up_proj", "down_proj"],
    use_gradient_checkpointing="unsloth",  # Big memory savings
)

# Train with SFTTrainer
trainer = SFTTrainer(
    model=model,
    tokenizer=tokenizer,
    train_dataset=dataset,
    max_seq_length=4096,
    packing=True,  # Key for efficiency!
    args=TrainingArguments(
        per_device_train_batch_size=2,
        gradient_accumulation_steps=4,
        learning_rate=2e-4,
        num_train_epochs=3,
        bf16=True,
    ),
)
trainer.train()

Fase 6: Implementação e formatos de saída

Após o treino, existem três formas de exportar o modelo:

1. Adaptador LoRA (padrão)

uv run finetune  # Saves ~100-500MB adapter
  • Tamanho: ~100-500 MB.
  • Ideal para: desenvolvimento, testes e utilização de vários adaptadores num único modelo base.
  • Vantagem adicional: é possível substituir os adaptadores sem ter de transferir novamente o modelo base.

2. Modelo fundido

uv run finetune --merge  # Creates standalone ~8-16GB model
  • Tamanho: ~8-16 GB (pesos de 16 bits completos).
  • Ideal para: partilha no HuggingFace, fornecimento via vLLM e implementações simples.
  • Compromisso: ficheiro maior, mas sem necessidade de uma dependência de modelo base separada.

3. Formato GGUF

uv run finetune --gguf q4_k_m  # Creates ~2-4GB quantized model
  • Tamanho: ~2-4 GB (quantização Q4_K_M).
  • Ideal para: inferência em CPU, Ollama, llama.cpp, implementação em dispositivos periféricos.
  • Opções: q4_k_m (balanceado), q5_k_m (qualidade superior) q8_0 (praticamente sem perdas).

Principais conclusões

  1. Percorra todo o pipeline. A preparação de dados é onde reside o maior impacto, e não o treinamento.
  2. Não recorra ao ajuste fino por padrão. Tente primeiro o prompting, seguido de SGR e depois RAG. Apenas recorra ao ajuste fino quando esses métodos não forem suficientes.
  3. Leve a preparação de dados a sério. Utilize pipelines modernos (como DataTrove e Distilabel) e remova todos os dados PII antes de qualquer implementação em ambiente empresarial.
  4. Escolha QLoRA como opção padrão, a menos que você disponha de 8 unidades H100. É a forma mais eficaz de ajustar modelos de 70B em hardware de consumo ou A100.
  5. Utilize ORPO para alinhamento. O treinamento em única fase é mais simples e, geralmente, rápido o suficiente.
  6. Recorra a DoRA quando o LoRA atingir seu limite de qualidade em tarefas de raciocínio mais complexas.
  7. Ative a funcionalidade de compactação de amostras. É o fator que gera a maior economia de tempo no processo de treinamento.
  8. Use Unsloth para prototipagem e Axolotl para produção.
  9. Exporte para formato GGUF sempre que o destino da implantação for local ou em edge devices.

Principais conclusões

  1. A maioria das equipas deve testar primeiro técnicas de prompting, recuperação de informação, roteamento e avaliação, antes de recorrer ao ajuste fino.
  2. A qualidade do conjunto de dados tem um papel determinante nos resultados do ajuste fino. Exemplos inadequados ensinam comportamentos incorretos mais rapidamente do que mais épocas de treinamento conseguem corrigi-los.
  3. O LoRA e o QLoRA são as opções práticas padrão para adaptação de modelos; o ajuste fino completo exige um esforço operacional muito maior.
  4. Avalie os modelos com exemplos específicos para cada tarefa, e não apenas com pontuações genéricas de benchmarks.
  5. A implantação é crucial: um modelo ajustado finamente que seja difícil de utilizar, monitorizar ou desfazer não está pronto para produção.
