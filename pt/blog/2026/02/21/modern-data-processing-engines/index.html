Motores Modernos de Processamento de Dados Comparados: Polars, DataFusion, Daft, Ray Data, Pandas e Spark

Durante anos, o Pandas foi responsável pelo processamento de tabelas em memória, enquanto o Apache Spark cuidava do processamento distribuído. Essa divisão funcionava bem quando os dados estavam estruturados.

Os trabalhos atuais já não são necessariamente estruturados. Os pipelines de imagens, áudio e vídeo coexistem agora com os pipelines tabulares, e, com a introdução da inferência em GPU, a coleta de lixo da JVM e o GIL do Python deixam de ser apenas incômodos para se tornarem verdadeiros gargalos. Esse é um dos principais motivos pelos quais uma nova geração de motores desenvolvidos com Rust e Apache Arrow ganhou força.

Ao longo do último ano, mudei a maior parte dos meus pipelines de um único nó para o Polars, e quis avaliar como ele se compara aos demais componentes dessa nova arquitetura. Por isso, realizei testes de desempenho com o Polars, DataFusion, Daft, Ray Data e Spark em conjuntos de dados reais — viagens de táxi em Nova Iorque para o lado tabular, e imagens do conjunto Food-101 para o lado multimodal.

Todo o código encontra-se no repositório engine-comparison-demo Clone-o e execute os testes de desempenho no seu próprio hardware — os seus resultados serão diferentes dos meus, e é exatamente nisso que consiste o objetivo.

TL;DR: Nos benchmarks otimizados para marketing, os motores nativos em Rust alcançam um ganho de velocidade de 94 vezes em relação ao Pandas num único nó. Em cargas de trabalho reais, os resultados são um pouco mais modestos, mas consistentes — tanto o Polars como o DataFusion constituem opções sólidas por padrão. Para pipelines multimodais, motores baseados em pipeline como o Ray Data e o Daft funcionam até 18 vezes mais rápido que o Spark, pois mantêm efetivamente as GPUs ocupadas. Não existe um vencedor absoluto; a escolha correta depende dos seus dados, da sua escala e do facto de as GPUs estarem envolvidas no processo.

Tipos de cargas de trabalho de processamento de dados

Os motores são especializados, pelo que a primeira pergunta a responder é qual é o tipo de carga de trabalho que realmente possui. A maior divisão ocorre entre dados estruturados e multimodais.

Dados Estruturados/Tabulares. Filtragem, agregação, junção. Dependente da CPU, geralmente em memória. Grande parte deste trabalho é executada em clusters distribuídos que, na verdade, não precisavam ser distribuídos desde o início — aproximadamente 90% das consultas processam menos de 1 TB de dados, o que hoje em dia se encaixa confortavelmente numa única máquina.

IA multimodal. Imagens, áudio, vídeo. O lado de inferência depende fortemente da GPU, mas o pré-processamento feito pela CPU é o verdadeiro fator limitante. Os modelos de aprendizagem profunda consomem dados mais rapidamente do que as CPUs conseguem decodificá-los, portanto, numa máquina padrão g6.xlarge (4 CPUs por GPU), a utilização da GPU fica abaixo de 20%. Todo o pipeline acaba a ser limitado pela velocidade com que consegue alimentar a GPU.

Os novos motores visam ambos os objetivos. O Rust confere-lhes velocidade, o Arrow permite a partilha de dados sem cópia entre processos, e a execução em fluxo permite-lhes lidar com conjuntos de dados maiores que a RAM.

Parte 1: Processamento em Nó Único

O gargalo do Pandas não é um erro. É uma questão de design. O Pandas carrega arquivos para a RAM de forma impulsiva, copia valores intermediários em quase todas as operações e permanece num único thread devido ao GIL. No Testes de desempenho PDS-H com um fator de escala de 10, o Pandas leva aproximadamente 365 segundos para concluir um conjunto padrão de consultas. O motor de streaming do Polars executa o mesmo trabalho em 3,89 segundos — cerca de 94 vezes mais rápido.

Polars: Processamento de Dados Tabulares

Polars substituiu o Pandas como padrão para ETL local de alto nível em muitas equipas com as quais conversei. Está escrito em Rust e utiliza execução lazy: em vez de executar cada linha à medida que é processada, ele cria um plano de consulta, otimiza-o (aplicação de predicados, eliminação de projeções e colunas desnecessárias) e, em seguida, executa-o em todos os núcleos de CPU através de lotes em fluxo contínuo.

A diferença aumenta à medida que os dados crescem. Em fator de escala 100 (~100 GB), o motor de streaming do Polars concluiu a tarefa em 23,94 segundos, contra 152,27 segundos para o seu próprio motor em memória — cerca de 6 vezes mais rápido, mesmo com dados que excedem a capacidade da RAM. engine_comparison_examples.ipynb:

import polars as pl

# scan_parquet reads only the schema — no data loaded yet
q = (
    pl.scan_parquet("yellow_tripdata_2024-01.parquet")
    .filter(
        (pl.col("trip_distance") > 5.0)
        & (pl.col("total_amount") > 30.0)
    )
    .group_by("payment_type")
    .agg(
        pl.col("total_amount").mean().alias("avg_fare"),
        pl.col("trip_distance").mean().alias("avg_distance"),
        pl.len().alias("trip_count"),
    )
)

# The entire plan is optimized and executed here, in parallel
result = q.collect()

A diferença em relação ao Pandas não está na API. Está na arquitetura. O Polars cria o plano primeiro e otimiza todo o pipeline antes de processar os dados. filter é encaminhado para o leitor Parquet, fazendo com que grupos inteiros de linhas sejam ignorados. Apenas as colunas referenciadas na consulta são baixadas do disco. O Pandas não consegue fazer nada disso — cada linha é processada assim que é analisada, e cópias intermediárias acabam espalhadas por todo o lado.

Apache DataFusion: Motor de Consultas Extensível

Enquanto o Polars é uma biblioteca que se utiliza diretamente, DataFusion é o motor sobre o qual se constroem outros motores. Ele fornece energia InfluxDB 3.0, GreptimeDB, e da Apple Acelerador Comet Spark.

Em novembro de 2024, a DataFusion tornou-se a o motor de único nó mais rápido para consultar ficheiros Parquet no ClickBench, à frente do DuckDB, chDB e ClickHouse no mesmo hardware. A equipa da Embucket executou posteriormente TPC-H com fator de escala 1000 (~1 TB) num único nó Além disso, esse é um valor útil para se ter em mente sempre que alguém diz precisar de um cluster. engine_comparison_examples.ipynb:

from datafusion import SessionContext

ctx = SessionContext()
ctx.register_parquet("taxi", "yellow_tripdata_2024-01.parquet")

# SQL executed directly against Parquet — no intermediate copies
df = ctx.sql("""
    SELECT payment_type,
           COUNT(*)           AS trip_count,
           AVG(trip_distance) AS avg_distance,
           AVG(total_amount)  AS avg_fare
    FROM taxi
    WHERE trip_distance > 5.0 AND total_amount > 30.0
    GROUP BY payment_type
    ORDER BY trip_count DESC
""")
result = df.to_pandas()

DataFusion tem como ponto forte o seu modularidade. As APIs de extensão abrangem catálogos personalizados, fornecedores de tabelas, regras de otimização e planos de execução, e é por isso que são utilizadas sempre que alguém está a criar uma plataforma de dados personalizada ou a integrar um motor de consultas no seu próprio produto.

Daft: Processamento de Dados Multimodal

Estúpido é aquele que leva a sério os dados multimodais. Imagens, áudio, vídeo, embeddings — são tipos reais no DataFrame, e não bytes opacos. O Pandas trata uma imagem como um array de bytes, o Spark serializa-a através da JVM, mas o Daft dispõe de expressões nativas em Rust para decodificar imagens, buscar URLs e gerar tensores, sem a necessidade de laços em Python no meio do processo. engine_comparison_examples.ipynb:

import daft

df = daft.read_parquet("yellow_tripdata_2024-01.parquet")

result = (
    df.where(
        (daft.col("trip_distance") > 5.0)
        & (daft.col("total_amount") > 30.0)
    )
    .groupby("payment_type")
    .agg(
        daft.col("total_amount").mean().alias("avg_fare"),
        daft.col("trip_distance").mean().alias("avg_distance"),
        daft.col("trip_distance").count().alias("trip_count"),
    )
    .collect()
)

A API parecerá familiar se você conhecer o Pandas, mas o motor é a mesma pilha Rust + Arrow encontrada no Polars e no DataFusion. O Daft revela seu verdadeiro valor quando as “linhas” do seu DataFrame são imagens, PDFs ou tensores.

Com este tamanho (~660 MB em formato Parquet), qualquer motor moderno consegue processar rapidamente os dados. O valor de 94x citado para Polars em comparação com Pandas provém do PDS-H, um benchmark analítico sintético — no meu conjunto de dados com 41 milhões de linhas, observei uma melhoria real e consistente em relação ao Pandas, mas longe dos 94x. O resultado mais interessante é que Polars, DataFusion, Daft e a versão em Rust puro via Polars-rs apresentam desempenhos praticamente semelhantes nesse tipo de carga de trabalho. DataFusion supera os demais em termos de tempo total de processamento. Polars é o mais fácil de utilizar e representa uma opção versátil e confiável.

O Spark paraleliza as partições entre os núcleos, mas a sua execução baseada em fases (síncrona em lote) é o ponto negativo. Dentro de uma única tarefa, cada passo (transferência, decodificação, classificação) é executado sequencialmente num único núcleo, sem qualquer sobreposição assíncrona. Além disso, todas as tarefas de uma fase têm de ser concluídas antes que a fase seguinte comece. As GPUs ficam ociosas enquanto os CPUs realizam o pré-processamento; os CPUs ficam ociosos enquanto as GPUs executam a inferência. Cada fase materializa a sua saída antes que a próxima comece, o que exerce pressão adicional sobre a memória.

Execução em pipeline é a alternativa. Em vez de executar as fases uma após a outra, o motor sobreposições de operações de E/S, trabalho da CPU e inferência da GPU, de modo que o componente mais lento é o único a aguardar em determinado momento.

Parte 3: Processamento Distribuído

Quando uma única máquina já não é suficiente, é preciso decidir como distribuir o trabalho. É aqui que as diferenças arquitetónicas entre os motores começam realmente a fazer diferença.

Apache Spark

Spark continua a ser a minha primeira escolha para ETL de tabelas em escala de petabytes, especialmente quando há operações de ordenação complexas. A tolerância a falhas graças à trilha de execução dos RDDs funciona eficazmente, o ecossistema é vasto (Databricks, EMR) e as junções de vários TB são tarefas bem consolidadas. No entanto, ele deixa a desejar em cargas de trabalho de IA. As tarefas são agendadas para executores JVM que não têm suporte a GPUs, fazendo com que estas fiquem ociosas enquanto o pré-processamento é realizado pela CPU. distributed_spark.ipynb:

from pyspark.sql import SparkSession
from pyspark.sql import functions as F

spark = SparkSession.builder \
    .appName("TaxiETL") \
    .config("spark.sql.adaptive.enabled", "true") \
    .getOrCreate()

orders = spark.read.parquet("s3a://lake/taxi/*.parquet")
zones = spark.read.csv("s3a://lake/taxi_zones.csv", header=True)

# Spark excels at this: joining massive tables with a shuffle
result = (
    orders
    .filter(F.col("trip_distance") > 5.0)
    .join(zones, orders.PULocationID == zones.LocationID, "inner")
    .groupBy("Borough")
    .agg(
        F.sum("total_amount").alias("total_revenue"),
        F.avg("total_amount").alias("avg_fare"),
    )
    .orderBy(F.desc("total_revenue"))
)

Dados do Ray: Utilização da GPU

Ray Data foi projetado para cargas de trabalho de IA. Em vez das barreiras de fase do Spark, utiliza um modelo de streaming que mantém as GPUs alimentadas. A funcionalidade que justifica a sua existência é o agendamento misto de recursos: pode declarar que um agente deseja “1 GPU, 4 CPUs” e outro apenas CPUs, sendo que o Ray resolve o resto.

A Amazon relatou ter conseguido poupanças mais de 120 milhões de dólares por ano ao migrar cargas de trabalho específicas do Spark para o Ray. Os resultados do seu PoC mostraram uma eficiência de custos 91% superior e 13 vezes mais dados por hora; nos ambientes de produção, a eficiência de custos por GiB de dados de entrada no S3 ficou 82% melhor.

De distributed_ray.ipynb:

import ray

ray.init()

ds = ray.data.read_images("s3://my-bucket/food101/")

class ImageClassifier:
    def __init__(self):
        import torch
        from torchvision.models import resnet18, ResNet18_Weights
        self.model = resnet18(weights=ResNet18_Weights.DEFAULT).cuda()
        self.model.eval()
        self.preprocess = ResNet18_Weights.DEFAULT.transforms()

    def __call__(self, batch):
        import torch
        tensors = torch.stack([
            self.preprocess(img) for img in batch["image"]
        ]).cuda()
        with torch.no_grad():
            preds = self.model(tensors)
        return {
            "prediction": preds.argmax(dim=1).cpu().numpy(),
            "confidence": preds.max(dim=1).values.cpu().numpy(),
        }

# ActorPoolStrategy creates persistent GPU workers
predictions = ds.map_batches(
    ImageClassifier,
    compute=ray.data.ActorPoolStrategy(size=4),
    num_gpus=1,
    batch_size=64,
)
predictions.write_parquet("s3://output/predictions/")

Um detalhe importante a saber: à medida que se adicionam mais CPUs por GPU, o Ray Data apresenta crescimento, enquanto outros motores atingem um platô. Em Os benchmarks da Anyscale, passar de uma proporção CPU-GPU de 4:1 para 32:1 resultou num aumento de velocidade de 3 vezes na inferência de imagens, pois os processadores CPU finalmente conseguiram acompanhar o desempenho da GPU.

Daft (Distribuído)

O Daft escala horizontalmente através dos seus Motor da frota: um trabalhador Swordfish por nó, com o Flotilla a gerir o agendamento em todo o cluster. O Swordfish trata da execução local em Rust e das operações de E/S dos pipelines através de fluxo em pequenos lotes, garantindo que cada nó permaneça ocupado sem esperar pela fase seguinte.

Em quatro cargas de trabalho multimodais, o Daft com Flotilla funcionou 2–18 vezes mais rápido que o Spark, No caso mais exigente (deteção de objetos em vídeo com YOLO11n), o Spark demorou mais de 3,5 horas, enquanto o Daft concluiu a tarefa em menos de 12 minutos — uma diferença de 18 vezes, principalmente porque o Spark gasta esse tempo com serialização e barreiras entre fases.

Um ponto a ter em conta: a Anyscale, empresa responsável pelo Ray, publicou os seus próprios benchmarks Mostra como o Ray Data reduz ou elimina a diferença em instâncias com alto consumo de CPU através de ajustes manuais. Atualmente, os dois métodos alternam-se a cada trimestre. distributed_daft.ipynb:

import daft
from daft import col

# Daft's Flotilla engine distributes work across the cluster
df = daft.read_parquet("s3://data-lake/pdf_metadata/*.parquet")

# Download PDFs — Daft parallelizes downloads in Rust
df = df.with_column("pdf_bytes", col("pdf_url").url.download())

# Define a GPU UDF for embedding generation
@daft.udf(return_dtype=daft.DataType.list(daft.DataType.float32()))
class TextEmbedder:
    def __init__(self):
        from sentence_transformers import SentenceTransformer
        self.model = SentenceTransformer("all-MiniLM-L6-v2", device="cuda")

    def __call__(self, text_col):
        texts = text_col.to_pylist()
        embeddings = self.model.encode(texts, batch_size=32)
        return [emb.tolist() for emb in embeddings]

# Daft schedules CPU downloads and GPU embeddings simultaneously
df = df.with_column("embedding", TextEmbedder(col("text")))
df.write_parquet("s3://output/embeddings/")

Parte 4: O Papel do Rust e do Arrow

Abaixo da competição, a convergência é a história mais interessante. Polars, Daft, DataFusion e o núcleo do Ray partilham todos a mesma base: Apache Arrow para o formato em memória e Rust para a concorrência.

Isso não é apenas uma questão de elegância arquitetónica. O Interface Arrow PyCapsule (__arrow_c_stream__ o protocolo permite transferir dados entre motores sem necessidade de serialização: realize o processamento em DataFusion e envie o resultado para Polars ou PyArrow sem qualquer custo adicional; faça o pré-processamento multimodal em Daft e introduza os lotes em formato Arrow diretamente num PyTorch DataLoader.

from datafusion import SessionContext
import polars as pl

# Compute in DataFusion (Rust-native execution)
ctx = SessionContext()
ctx.register_parquet("events", "events.parquet")
df = ctx.sql("""
    SELECT user_id, COUNT(*) as event_count
    FROM events WHERE event_type = 'purchase'
    GROUP BY user_id HAVING COUNT(*) > 5
""")

# Zero-copy conversion to Arrow, then to Polars
arrow_table = df.to_arrow_table()
df_polars = pl.from_arrow(arrow_table)

# Continue analysis in Polars
result = df_polars.with_columns(
    pl.col("event_count").rank().alias("rank")
).sort("rank")

Então, por que Rust em vez da JVM?

  1. Sem pausas de coleta de lixo. A propriedade e o empréstimo substituem a GC, o que significa ausência de pausas “stop-the-world” e sem picos imprevisíveis de latência. Os erros de memória insuficiente que surgem em larga escala nos sistemas JVM desaparecem na maior parte dos casos.

  2. Sem custo adicional de cabeçalhos de objeto. A JVM adiciona de 12 a 16 bytes de cabeçalho de objeto a cada objeto. Em bilhões de objetos pequenos, isso já representa um grande consumo de memória por si só. O Rust não incorre nesse custo.

  3. Segurança em threads em tempo de compilação. O sistema de tipos do Rust rejeita corridas de dados antes mesmo da construção do binário, portanto a execução multithreadada não apresenta os habituais bugs de concorrência sutis.

Juntamente com o layout de memória colunar do Arrow, o gargalo de serialização que consome até 30% dos ciclos do CPU nas pilhas JVM legadas desaparece.

Parte 5: Adoção de Múltiplos Motores

A resposta sincera é que não se escolhe apenas um motor. Eles são combinados entre si.

Em produção, isto geralmente acontece através de um mecanismo de retransmissão de dados. Os dados brutos armazenados num “lake” são limpos e combinados por meio do Spark (ou Dask, se a sua equipa utiliza exclusivamente Python) para formar tabelas no formato Parquet ou Delta. Essas tabelas são posteriormente processadas pelo Ray Data ou Daft para as tarefas finais — inferência em GPU, geração de embeddings, transformações multimodais — para as quais o Spark não foi concebido. Para análises ad hoc e desenvolvimento local, o Polars e o DuckDB fornecem feedback imediato sem a necessidade de configurar um cluster.

O que torna possível esta arquitetura são os formatos abertos: Parquet, Delta, Iceberg, Arrow. Cada motor da stack lê e escreve esses formatos de forma nativa, pelo que a transferência de dados entre etapas consiste simplesmente na troca de um caminho de ficheiro.

Escala Diagonal

Durante muito tempo, a opção era escalar verticalmente (um equipamento maior) ou escalar horizontalmente (mais equipamentos). Polars Cloud está a fazer algo intermédio, que é designado por escala diagonal: escala horizontalmente enquanto lê do armazenamento em nuvem para maximizar o I/O, e depois converge para um único nó grande assim que os filtros e agregações reduzem os dados, evitando completamente o embaralhamento distribuído.

A parte contraintuitiva: uma instância maior e mais cara pode acabar por ser mais barata por tarefa, porque eleva a utilização da GPU a um nível suficientemente alto, fazendo com que o tempo total de execução diminua mais rapidamente do que a taxa horária aumenta.

Principais Conclusões

  1. Não distribua se não for necessário. O Polars e o DataFusion conseguem processar até ~1 TB numa única máquina, com acelerações consistentes em relação ao Pandas (e 94x em benchmarks analíticos intensivos como o PDS-H). Recorra a um cluster apenas quando realmente atingir os limites de um único nó, e não antes.

  2. Os limites do Pandas são de natureza arquitetónica. Execução imediata, single-threading, cópias intermediárias — estas são escolhas de design intencionais que se tornam gargalos quando os dados aumentam de tamanho. A execução diferida, com pushdown de predicados e poda de colunas, é o que muda isso.

  3. A execução em pipeline supera as barreiras entre fases para trabalhos com GPU. As fases do Spark deixam os GPUs ociosos enquanto os CPUs fazem o pré-processamento. O Ray Data e o Daft sobreporão as tarefas de E/S, CPU e GPU, de modo a que nenhum recurso fique à espera da próxima fase.

  4. Utilize cada motor para aquilo para o qual é mais adequado. Spark para ETL em petabytes, Polars e DuckDB para análises ad hoc, Ray Data para pipelines de treino com GPU, Daft para carregamento de dados multimodais, DataFusion para motores de consulta personalizados.

  5. Rust + Arrow é a base comum. Não é por acaso que Polars, DataFusion, Daft e Ray convergem aqui — elimina pausas de GC, sobrecarga de serialização entre processos e todo um conjunto de erros de concorrência.

Se não tiver a certeza por onde começar: comece com o Polars numa única máquina e, depois, adicione o Daft ou o Ray Data quando os GPUs entrarem em cena. Use o Spark apenas quando realmente tiver dados em petabytes.
