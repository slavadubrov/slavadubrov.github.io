Melhores ferramentas locais de LLM para macOS

As ferramentas locais de LLM no macOS são divididas por função: fornecimento simples, exploração via GUI, controlo avançado da inferência e experimentação nativa para Apple Silicon. Uma única ferramenta não precisa de cumprir todas as quatro funções.

Configuração padrão: Ollama para uma API local. LM Studio para a exploração de modelos. llama.cpp quando é necessário controlo do ambiente de execução GGUF. MLX quando se pretende trabalhar com Python de forma nativa no Apple Silicon, perto do modelo.

Qual deve ser instalado primeiro?

Instale primeiro o Ollama se estiver a desenvolver software. Muitas aplicações sabem como comunicar com ele, e a API local é suficiente para protótipos, testes e pequenas ferramentas internas. É o caminho mais curto de “preciso de um modelo local” a “a minha aplicação consegue chamar um modelo local.”

Instale primeiro o LM Studio se estiver a escolher um modelo. É útil para navegar por modelos, alterar definições, comparar resultados e executar um servidor local compatível com a OpenAI, sem precisar de projetar o fluxo de trabalho manualmente.

Instale primeiro o llama.cpp se se interessar pela mecânica da inferência. O comprimento do contexto, a quantização, os parâmetros do Metal, o processamento de prompts, os tamanhos de lote e o comportamento do servidor são mais fáceis de analisar quando se está mais próximo do momento de execução.

Utilize o MLX quando fizer algo além de fornecer um modelo de chat. Ele é adequado para experimentos com modelos nativos do Apple Silicon, conversão, ajuste fino e fluxos de trabalho em Python, onde a memória unificada faz parte do design.

Notas de hardware

A memória unificada é, de facto, o verdadeiro limite no Apple Silicon. Um modelo que cabe num MacBook Pro de 64 GB pode tornar-se inutilizável num MacBook Air de 8 GB. A quantização ajuda, mas o comprimento do contexto pode, silenciosamente, dominar o consumo de memória. Faça testes de desempenho com a estrutura real do prompt, em vez de se focar apenas no nome do modelo.

Para ferramentas locais de pequeno porte, um modelo da classe 7B ou 8B costuma ser mais útil do que um modelo maior e sobrecarregado. No âmbito da programação, um contexto extenso e a integração com ferramentas podem ser mais importantes do que a classificação bruta em benchmarks. Para tarefas de QA em documentos, a qualidade da recuperação de informações costuma determinar a escolha do modelo local.

O que evitar

Não transforme a configuração de LLMs locais num projeto de benchmark permanente, a menos que o desempenho seja o objetivo final. Comece utilizando o Ollama ou o LM Studio. Demonstre primeiro que a inferência local traz benefícios. Só então passe para soluções como o llama.cpp ou o MLX quando houver motivos concretos para isso.

Não compare modelos apenas através de uma interface de chat, se o trabalho real envolver extração estruturada, edição de código ou síntese de respostas com base em RAG. Escreva um pequeno script de avaliação com prompts representativos.

Leitura adicional LLMs locais no macOS abrange a configuração prática. Variantes de LLM de Código Aberto e Formatos de Ficheiro explica o GGUF, GPTQ, AWQ, modelos base e modelos de instrução. Configuração do MacBook para Engenharia de IA abrange a configuração mais ampla da estação de trabalho.
