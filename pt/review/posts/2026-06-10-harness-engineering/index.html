    Harness engineering for AI agents: designing the loop around the model

    Part 6 of the Engineering the Agentic Stack series

    In a synthetic checkout run, a coding agent edits src/checkout.py, runs the nearest unit test, receives exit code 0, and reports success. The browser checkout still fails.

    The run stopped because its acceptance rule treated the unit test as enough evidence. That rule lives in the harness, the program around the LLM that builds each model request, executes or rejects tool calls, and decides when the evidence is sufficient. For this task, the fix is specific: add the browser test to the acceptance gate and keep the loop open when it fails.

    TL;DR: The first version needs one model, a few narrow tools, and an end-to-end acceptance check. Read failed traces before adding machinery. When you compare a new component, keep the model version, task set, grader, and total budget unchanged across both variants. Remove it when the result stays the same.

    One proposed checkout edit moves from context to authorization, execution, verification, and a stopping decision. The harness can record, reject, or verify each transition without relying on the final message from the model.

    What the harness owns

    OpenAI's Codex loop walkthrough describes the basic cycle. The harness assembles a prompt, asks the model for the next action, executes a tool call, appends the result, and asks again until the system accepts the result or hands control back to the user.

    That cycle spans four parts that framework documentation often blurs:

    Term Job Coding-agent example
    Model Proposes text, a tool call, or a final answer Suggests an edit to checkout.py
    Reasoning loop Chooses the next move from the available context Inspect, edit, test, inspect again
    Harness Builds context, exposes tools, applies policy, records state, and checks completion Allows edits under src/ and requires browser checkout
    Runtime Keeps the process alive and isolated Worker, sandbox, session store, queue

    The boundary between harness and runtime varies by implementation. Look at the failure itself: a poor plan points toward model instructions or reasoning, while an edit outside the allowed directory points toward tool policy. A dead worker belongs to the runtime.

    OpenAI's own harness-engineering case study describes a team that made one application instance bootable per worktree, wired browser automation into the agent environment, and exposed logs, metrics, and traces. A task such as "keep these four user journeys under two seconds" became testable because the agent could run the application and query the same signals an engineer would inspect. The case study is product-specific. The task became checkable only after the application and its performance signals were available inside the agent environment.

    Follow one action through the loop

    In the checkout task, suppose the model asks to replace calculate_discount in src/checkout.py. Several things happen before that edit counts as progress:

    1. The context builder supplies the task, repository instructions, relevant files, prior tool results, and the current plan.
    2. The model proposes an edit_file call with a path and replacement text.
    3. The tool boundary validates the arguments, checks the path against the allowed scope, and asks for approval if the operation needs it.
    4. The runtime applies the edit in the sandbox and returns a structured result.
    5. The harness runs the checks attached to the task. Here that means checkout unit tests followed by an end-to-end browser test.
    6. A failed check becomes new context for the next model turn. A passing result is recorded as evidence for completion.

    The model's tool call is a proposal, which lets ordinary code reject ../../secrets.env, block a destructive command, or stop a run that has exhausted its budget before the call reaches the runtime. At step 5, ordinary code can inspect an exit status directly without asking the same model to interpret its own work.

    An agent's final message carries little weight by itself, so I want the trace to answer a concrete question: which check passed, against which artifact, after which side effects?

    Decide where each rule is enforced

    The browser test from the opening gives us one concrete rule: the agent may report completion only after that test passes. A prompt can state the rule and ask the agent to run the test, yet the model can still forget the instruction or decide that a unit test is enough. The acceptance gate reads the browser test's exit code and rejects completion when the test fails. No model judgment is needed.

    The table shows where each rule goes:

    Layer Rules it can handle Checkout example
    Prompt or skill Search order, coding conventions, preferred plan shape Read AGENTS.md before editing checkout code
    Tool boundary Argument validation, file scope, approvals, capability limits Permit writes only under src/
    Deterministic code Budgets, timeouts, retries, tests, release gates Keep the run open while browser checkout fails
    Separate evaluator Visual quality, rubric-based review, adversarial inspection Compare the checkout page with a design brief

    In the checkout run, the prompt tells the model which instructions to read, while the tool boundary limits what it may change and deterministic code decides whether the required tests passed. A separate evaluator handles artifact judgments that those checks cannot make.

    The tool boundary needs more than a natural-language tool description. For a call such as create_test_order, the harness needs a contract with:

    • validated arguments, so malformed input is rejected before execution
    • a structured result, so later checks do not have to parse free-form text
    • a side-effect class, so the harness knows whether the call only reads state or may change it
    • a timeout and retry policy, so a lost response does not trigger an unlimited sequence of calls
    • a permission category, so the harness can require the right approval before execution

    The description helps the model choose the tool. The harness checks the contract before execution. That distinction matters when one server supplies the tool metadata and another client decides whether the call is allowed. The MCP specification tells clients to treat tool annotations as untrusted unless they came from a trusted server.

    Timeouts make the side-effect class matter. In a second synthetic checkout run, create_test_order creates the order, but the HTTP response is lost. Repeating the request without checking the service would create a second order.

    Reads are simpler to retry. The harness may also retry a mutation when an idempotency key or a deduplication record protects it. The service can then recognize the repeated request, skip the duplicate mutation, and return the first call's result to the harness. AWS's idempotent API guidance describes this pattern with a client request identifier.

    Without that protection, the harness needs to check whether the order already exists before it retries. If the result cannot be recovered safely, the next step may require compensation or a human decision.

    The tool may now run. The harness must still decide whether the resulting checkout change is acceptable. Start with checks that have an objective result, such as a schema validator and the test suite. Add a separate evaluator when the requirement depends on judgment, such as visual hierarchy or the quality of an explanation, and verify its rubric against human decisions before using it as an acceptance gate.

    Long runs need a handoff

    Now let the checkout change grow into a two-hour migration. The first session changes six files, starts a local payment sandbox, finds one failing browser test, and reaches its context limit. A conversation summary helps the next model turn, but it cannot restart the sandbox or prove which files are currently modified.

    The next session has to reconstruct three different kinds of state:

    State What it contains How it can fail
    Conversation Messages, tool calls, observations Old details crowd out the current task
    Workspace Files, processes, browser state, external objects The transcript says a service is running after it died
    Task Plan, completed checks, pending approval, next action The next session repeats finished work

    Compaction and a handoff solve different problems. Compaction replaces older conversation detail with a shorter summary so the same session can continue. It does not recreate the workspace for another worker.

    A progress handoff is written for that next worker. It records the current branch, changed files, last test command, its output, and the next unresolved step. If the old conversation has accumulated bad assumptions or too much stale detail, a context reset starts a new worker from this handoff.

    A 10-minute documentation edit may need none of these mechanisms. A migration that crosses sessions may need a handoff even when the conversation still fits in the context window, because the next worker must reconstruct the workspace and task status.

    Anthropic's experiments with long-running coding agents used git history and a progress file between sessions. Its later harness-design report separates compaction from a fresh-context handoff and records the extra orchestration, tokens, and wall time required by that design.

    Use the trace to separate different failures

    Across ten synthetic runs of the checkout task, six call the order API and receive a 503 before an order is created. In two runs, the order exists but the response disappears. The remaining two update the code successfully and skip the required browser test.

    One retry policy cannot repair all ten runs. The first six need bounded retries. The next two need idempotency or a status lookup before another request, while the last two need an acceptance gate tied to the missing browser test.

    A chat transcript records what the agent saw, but it cannot prove whether the service created an order before the response disappeared. To reconstruct a side effect, the trace needs to preserve five events in order:

    1. the tool call proposed by the model
    2. the policy check or human authorization that allowed it
    3. the result of runtime execution
    4. the postcondition check against the workspace or external service
    5. the final decision to continue or accept the run
    Repeated symptom Small change to try What to measure
    A transient read error ends the run Bounded retry with backoff Recovery rate, extra calls, wall time
    A resumed session repeats work Structured progress handoff Resume time, repeated actions, stale decisions
    The agent finishes with a broken artifact Deterministic acceptance check Escaped defects, false passes
    Visual defects survive passing tests Fresh evaluator with a written rubric Defects found, false rejects, review minutes
    The agent edits outside its scope Narrower tool permission Blocked calls, overrides, policy violations

    Before adding a component, write down the failure it should prevent and the metric that should move. A progress file should reduce resume time enough to repay the cost of writing and reading it. A fresh reviewer should catch defects that tests and self-review miss without creating too many false rejects. If the expected metric does not improve, extra planners, critics, and retries only add work around the original defect.

    Measure one change at a time

    Each proposed component now has a failure it should prevent and a metric it should change. An ablation tests that expectation by changing or removing one harness component while the rest of the experiment stays fixed. For example, does editor linting help this model on this task suite?

    Use the following protocol:

    1. Freeze the model version, task instances, environment, grader, and prompts outside the component under test.
    2. Give both variants the same total token, time, and dollar budget.
    3. Choose the number of trials or stopping rule before running the comparison.
    4. Run the same task instances in both variants. Repeat stochastic trials.
    5. Report the mean together with the spread or confidence interval.
    6. Count every started trial, including timeouts, policy stops, harness crashes, and evaluator failures.

    Success rate alone can hide an expensive component. At minimum, track escaped defects, cost and wall time per completed task, tool errors, duplicate effects, review minutes, and operator overrides. Choose the main metric based on where the product pays for failure. A two-point success gain can still be a bad trade if it doubles the review queue.

    A progress handoff gives the running example a concrete A/B test. Both variants use the same model, tasks, grader, and total budget. The handoff is the only switch, and the measured outcome is repeated work after the checkout migration resumes.

    The SWE-agent paper fixes GPT-4 Turbo on the 300-task SWE-bench Lite split and reports 18.0% resolved with its full interface, compared with 11.0% for the shell-only agent. The paper also changed individual interface features:

    Interface change Resolved
    Full SWE-agent interface 18.0%
    Editor without linting 15.0%
    Full file instead of a 100-line viewer 12.7%
    Full observation history instead of the last five 15.0%

    These numbers belong to that model, benchmark, and $4 per-task budget. Because the model stayed fixed while individual interface features changed, the rows show the effect of those interface choices under the paper's conditions.

    LangChain published a broader fixed-model comparison for deepagents-cli. It reports a Terminal-Bench 2.0 increase from 52.8% to 66.5% with gpt-5.2-codex fixed while its team changed the system prompt, tools, and middleware.

    Several parts of the harness changed together. The post also omits a confidence interval, a fixed-total-budget comparison, and a per-change ablation table. I read it as a vendor case study of trace-driven tuning. The bundled change set cannot isolate any one component.

    Anthropic describes one-at-a-time removal after a model upgrade. In its long-running application harness, an evaluator checked 27 criteria for one sprint. With a later model, some tasks no longer benefited enough from that evaluator to justify its cost. The team removed harness pieces one at a time and checked the result, including components that had helped the previous model.

    Try the method in the companion lab

    The harness-demo project is a small, deterministic exercise with four configurations and 12 synthetic tasks covering flaky tools, lost progress, optimistic self-review, and ambiguous completion.

    The lab uses a deterministic simulation instead of an LLM call. Its token counts and outcomes are synthetic, so each failure remains easy to connect to the component that handles it. Production-agent performance requires sampled tasks, real model calls, and a product grader.

    Run it from the project directory:

    make check
make run
make failures

    make run prints the comparison matrix. make failures shows why each configuration missed a task. The full configuration passes the synthetic suite and spends extra simulated tokens, so the output puts cost beside pass rate.

    The failure checks are plain Python:

    if task.flaky_tool and not config.retry_policy:
    failures.append("flaky tool call was not retried")

if task.needs_progress_file and not config.progress_handoff:
    failures.append("next session lost task state")

if task.needs_fresh_evaluator and not config.evaluator:
    failures.append("self-review missed an implementation gap")

    Start with one loop and one acceptance check

    The first coding-agent harness I would build is small. One capable model gets the repository instructions and a few narrow tools inside a sandbox. The acceptance check exercises the finished artifact.

    The harness records each tool call, its result, the cost, and the final check in one trace. When the first useful failure appears, that trace shows what happened without forcing the engineer to reconstruct the run from terminal logs and chat transcripts.

    Retries enter the loop after traces show transient tool errors. When sessions repeat work or lose the next step, use a progress handoff. A separate evaluator belongs there when tests pass while the same class of subjective defect continues to escape.

    For each addition, record who owns it, what it costs, and which regression test justifies keeping it. Run the removal test again after a model upgrade because a component that helped the previous model may only add cost to the new one.

    Six months later, someone who sees progress_handoff=True should be able to find the failed traces that justified it and the regression cases that still keep it there.
