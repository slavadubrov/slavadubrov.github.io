uv en macOS: Gestión de versiones de Python, proyectos y herramientas

Inicio rápido

# Install uv
brew install uv

# For new projects (modern workflow)
uv init                # create project structure
uv add pandas numpy    # add dependencies
uv run train.py        # run your script

# For existing projects (legacy workflow)
uv venv                             # create virtual environment
uv pip install -r requirements.txt  # install dependencies
uv run train.py                     # run your script

# Run tools without installing them
uvx ruff check .       # run linter
uvx black .            # run formatter

# Run a single-file script with its own dependencies (no project needed)
uv run fetch.py        # uv reads the inline deps and runs it

Resumen: Utilice uv como proyecto y gestor de herramientas predeterminado de Python en macOS. Mantén Homebrew para los paquetes del sistema e instala las versiones de Python a través de uv python, fijar dependencias en pyproject.toml, y utilice uvx Para herramientas CLI puntuales.

¿Por qué usar uv para Python en macOS?

Si ya utilizas Python desde hace tiempo, probablemente manejes al mismo tiempo pip, virtualenv, pip-tools, pyenv, y poetry Dependiendo del proyecto. uv Hace lo mismo que todos esos, pero con un único archivo binario. Está escrito en Rust y, en mi máquina, instala paquetes aproximadamente de 10 a 100 veces más rápido que el antiguo conjunto de herramientas. La ventaja más importante para mí es que ya no tengo que pasar de una herramienta a otra en medio de una tarea.

Reemplaza pip y pip-tools para la gestión de paquetes, pyenv para instalar versiones de Python, virtualenv/venv para los entornos, pipx para ejecutar herramientas, y poetry o pdm para los flujos de trabajo de proyectos.

Instalación de uv

La forma más sencilla de instalarlo uv En macOS se hace a través de Homebrew:

brew install uv

uv Detecta automáticamente la arquitectura de tu Mac (Apple Silicon o Intel), por lo que no es necesario realizar ninguna configuración adicional.

Para mantenerla actualizada:

brew upgrade uv
# OR
uv self update

Conceptos fundamentales

uv cubre tres casos de uso:

  1. Proyectos — crear una aplicación o biblioteca con dependencias.
  2. Scripts — ejecutar un script Python de un solo archivo con dependencias integradas.
  3. Herramientas — ejecutar utilidades de línea de comandos (como ruff o httpie) a nivel global.

1. Gestión de proyectos

Para los nuevos proyectos, uv utiliza el estándar pyproject.toml para la configuración y multiplataforma uv.lock para obtener compilaciones reproducibles.

Crear un nuevo proyecto:

uv init my-project
cd my-project

Esto crea un pyproject.toml, una .gitignore, y un hello.pyAdicionar dependencias:

# Runtime dependencies
uv add pandas requests

# Development dependencies
uv add pytest ruff --dev

Ejecuta tu código:

uv run hello.py

uv gestiona el entorno virtual en .venv Para usted. No es necesario activarlo manualmente.

2. Gestión de versiones de Python

uv instala y gestiona las versiones de Python por ti, guardándolas en ~/.cache/uv. Si has estado utilizando pyenv Para esto, puede omitirlo.

Instalar una versión específica:

uv python install 3.12

Fija una versión para tu proyecto:

uv python pin 3.11

Esto escribe un .python-version archivo. En el siguiente uv runUtiliza la versión fijada y la descarga si es necesario. Así, tu equipo y los procesos CI acaban utilizando la misma versión de Python sin necesidad de coordinación adicional.

3. Ejecutar herramientas con uvx y uv tool install

uvx (un alias para uv tool run) ejecuta las herramientas de línea de comandos de Python sin añadirlas al entorno global ni a las dependencias del proyecto.

# Run a linter
uvx ruff check .

# Run a formatter
uvx black .

# Start a temporary Jupyter server
uvx --from jupyterlab jupyter lab

Cada herramienta se ejecuta en su propio entorno temporal, por lo que no existe conflicto de versiones con lo que ya esté instalado en tu proyecto.

Dos detalles a los que recurre constantemente:

  • Fijar la versión directamente en el código con @: uvx ruff@0.6.0 check ., o forzar la versión más reciente con uvx ruff@latest.
  • ¿El nombre del comando no coincide con el paquete? Utilice --from. El paquete jupyterlab envía los jupyter comando, por lo que está uvx --from jupyterlab jupyter lab. Lo mismo para --from httpie http.
  • Necesita una dependencia adicional para un plugin? Ádíguela con --with: uvx --with mkdocs-material mkdocs serveSi utilizas una herramienta a diario, instálala una sola vez en lugar de crear un entorno nuevo cada vez:
uv tool install ruff      # now `ruff` is on your PATH
uv tool list              # see what's installed
uv tool upgrade --all     # update everything
uv tool uninstall ruff

La regla práctica: uvx para ejecuciones puntuales o en entornos de CI. uv tool install Para los scripts que utilizas a diario.

Proyectos heredados (requirements.txt)

requirements.txt Ha funcionado bien hasta ahora. Se trata de una lista simple de paquetes sin archivo de bloqueo, sin separación entre dependencias de aplicación y de desarrollo, y sin registro alguno que explique por qué se ha fijado cada paquete. Eso está bien mientras no intentes reproducir una compilación de hace ocho meses y descubras que lo que se entendía por “fijado” era simplemente lo que PyPI resolvía en ese momento.

Si eres el propietario del proyecto, migralo. uv lee tu lista existente directamente en un pyproject.toml:

uv init --bare                        # minimal pyproject.toml, no scaffolding
uv add -r requirements.txt            # import runtime dependencies
uv add --dev -r requirements-dev.txt  # and dev ones, if you split them

Ahora dispone de uno. pyproject.toml y uno real uv.lock que fija las versiones resueltas exactas. Confirma que todo haya llegado sin problemas. uv pip freeze, eliminar el antiguo requirements.txt, y no mires atrás.

Pero quizás has estado ejecutando el mismo requirements.txt Dado que es desde Python 3.6, y como no te interesa mi opinión al respecto… Está bien. uv sigue funcionando como sustituto directo para pip y venv, no se requiere migración:

# Create a virtual environment
uv venv

# Install dependencies
uv pip install -r requirements.txt

# Run it
uv run python app.py

Los mismos comandos que ya conoces, pero más rápidos. El antiguo flujo de trabajo no empeora en absoluto; simplemente ya no es necesario utilizarlo.

Scripts de un solo archivo con dependencias incrustadas

Esta es la funcionalidad que cambió por completo la forma en que escribo código temporal y de unión. Un script en Python puede declarar sus propias dependencias dentro del propio archivo, en un bloque de comentarios definido por PEP 723. No pyproject.toml, no requirements.txt, no hay entorno virtual que crear — uv run Lee el bloque, crea un entorno en caché y ejecuta el archivo.

Es importante aclarar qué es lo que realmente realiza la función aquí: no se trata de una característica del lenguaje Python, ni tampoco es… uv Invención. El bloque es simplemente un comentario, por lo que queda sin formato. python fetch.py Lo ignora y falla debido a la falta de importaciones. PEP 723 es un estándar común, por lo que cualquier ejecutor compatible lee el mismo bloque. pipx runHatch y PDM también lo soportan. uv resulta ser la forma más rápida de utilizarlo, y por eso el resto de esta sección utiliza uv run.

He aquí todo el contenido en un único archivo. fetch.py:

# /// script
# requires-python = ">=3.12"
# dependencies = [
#     "httpx",
#     "rich",
# ]
# ///
import httpx
from rich import print

print(httpx.get("https://api.github.com/repos/astral-sh/uv").json())

Ejecútalo:

uv run fetch.py

La primera ejecución resuelve e instala httpx y rich Se carga en un entorno en caché; las ejecuciones posteriores lo reutilizan y se inician de forma instantánea.

No es necesario escribir el bloque de metadatos a mano. uv Crearé la estructura básica y la editaré por usted:

# Create a new script with the block pre-filled
uv init --script fetch.py --python 3.12

# Add or remove dependencies
uv add --script fetch.py httpx rich
uv remove --script fetch.py rich

Para un experimento rápido, incluso puedes omitir por completo el bloque y pasar las dependencias por la línea de comandos:

uv run --with httpx --with rich fetch.py

Por qué es ideal para scripts de automatización y mejora de habilidades

Me apoyo en esto para los scripts pequeños que no merecen ser considerados proyectos: una corrección puntual de datos, un auxiliar para CI, un Habilidad Claude Code script, un trabajo cron. El script es la unidad. Puedes subirlo a un gist, comprometerlo en cualquier repositorio o entregárselo a un compañero de equipo. uv run Es lo único que necesitan para ejecutarlo correctamente.

Háganlo directamente ejecutable con un shebang:

#!/usr/bin/env -S uv run --script
# /// script
# dependencies = ["httpx"]
# ///
import httpx
...
chmod +x fetch.py
./fetch.py        # uv handles the environment behind the scenes

El -S divide el shebang en argumentos separados para env supera las pruebas --script hasta uv### Bloqueo y fijación de scripts

Para los scripts que deben seguir funcionando meses después, existen dos opciones.

Bloquee la resolución exacta junto al script:

uv lock --script fetch.py   # writes fetch.py.lock

o fija un punto en el tiempo para uv Solo tiene en cuenta los paquetes publicados antes de una fecha determinada; es útil para garantizar la reproductibilidad sin necesidad de un archivo de bloqueo:

# /// script
# dependencies = ["httpx"]
# [tool.uv]
# exclude-newer = "2025-04-17T00:00:00Z"
# ///

Consejos y trucos útiles

Algunas herramientas más que uso con frecuencia y que no son evidentes en la guía de inicio rápido.

Sincronización desde el archivo de bloqueo en CI. uv sync instala exactamente lo que se encuentra en uv.lock. Añadir --frozen para que se produzca un error evidente si el archivo de bloqueo está desactualizado, en lugar de volver a resolverlo de forma silenciosa; exactamente lo que se desea en los entornos de CI y en las construcciones con Docker.

uv sync --frozen

Agrupe sus dependencias de desarrollo. Más allá de --dev, se pueden definir grupos de dependencias nombrados en pyproject.toml (docs, lint, test) y sincroniza únicamente lo que necesites:

uv add --group docs mkdocs-material
uv sync --only-group docs

Inspecciona el árbol de dependencias. uv tree Muestra qué se ha incorporado y qué es — y --outdated Actualizaciones de flags:

uv tree
uv tree --outdated

Exportar a requirements.txt cuando una herramienta posterior todavía lo requiera:

uv export --format requirements-txt > requirements.txt

Ejecutar un script de Python único con paquetes adicionales, sin necesidad de proyecto:

uv run --with pandas --with matplotlib python

Gestionar la caché cuando el espacio en disco se vuelve limitado o una compilación falla:

uv cache clean      # wipe the whole cache
uv cache prune       # remove only unused entries

Por qué es rápido

Hay tres aspectos clave aquí. Está escrito en Rust, por lo que no existe la sobrecarga de inicio de Python en cada llamada. Además, almacena globalmente las versiones precompiladas — una vez numpy Si está instalado en un proyecto, instalarlo en otro es, básicamente, instantáneo (en macOS utiliza enlaces de tipo copy-on-write). Además, descarga e instala los paquetes de forma paralela en lugar de uno por uno.

Conclusiones clave

  1. Utilizar uv Para las versiones de Python, los entornos virtuales, las dependencias del proyecto y las herramientas puntuales, en lugar de mezclar varios gestores de paquetes.
  2. Deje que Homebrew se encargue de los paquetes del sistema, y no de la aislación de proyectos Python.
  3. Haga un commit. pyproject.toml y uv.lock De este modo, la configuración del proyecto es reproducible en una máquina nueva.
  4. Preferir uvx para CLI ocasionales cuando no se desea contaminar un entorno global.
