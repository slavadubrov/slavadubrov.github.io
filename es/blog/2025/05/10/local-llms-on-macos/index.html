LLMs locales en macOS: Ollama, LM Studio, llama.cpp, MLX y Apple Silicon

Enviar cada petición a una API de terceros se vuelve algo tedioso, sobre todo cuando la mitad de las solicitudes son cosas como “reescribe este párrafo” o “¿cuál es el esquema JSON para esto?”. Los LLMs locales resolvieron ese problema en mis Macs con Apple Silicon más rápido de lo que esperaba. Un modelo de 7 mil millones de parámetros, cuantizado en 4 bits, funciona sin problemas en un MacBook de 16 GB, y todo el proceso se realiza directamente desde el teclado.

Por lo tanto, la pregunta pendiente es qué aplicación utilizar para ejecutarlos. Ollama, LM Studio, llama.cpp, MLX y algunas otras opciones utilizan motores de inferencia similares y los mismos archivos GGUF. La diferencia radica en el nivel de complejidad que hay entre el usuario y el modelo: por un lado, basta con hacer doble clic y escribir; por otro, es necesario compilar desde el código fuente y luego consultar la documentación correspondiente.

Resumen: Utiliza Ollama para una API local sencilla, LM Studio para una interfaz gráfica y un endpoint compatible con OpenAI, llama.cpp cuando necesites un control directo sobre la inferencia de GGUF, y MLX si deseas flujos de trabajo en Python nativos para Apple Silicon.

Conceptos clave para modelos LLM locales en macOS

  1. Inferencia: ejecución del modelo para generar texto.
  2. Cuantización (GGUF): técnica para reducir el tamaño del modelo con una pérdida mínima de calidad. Verás nombres de archivo como llama-3-8b-Q4_K_M.gguf. El Q4 part se refiere a la cuantización de 4 bits, la cual utiliza una cantidad de RAM mucho menor que los pesos completos de 16 bits.
  3. Apple Silicon (Metal): Los chips de la serie M de Apple comparten la RAM entre la CPU y la GPU (Apple denomina esto “Memoria Unificada”). Por eso, un MacBook con 32 GB o 64 GB puede cargar modelos que en un PC requerirían una GPU dedicada costosa.

Requisitos previos

  • Hardware: lo que necesitas es una Mac con Apple Silicon (de la serie M1 a M4). Las Macs Intel también funcionan, pero son notablemente más lentas.
  • RAM:
    • 8GB: suficiente para modelos pequeños (Mistral 7B, Llama 3 8B).
    • 16GB+: ideal para modelos más grandes y para trabajar con múltiples tareas al mismo tiempo.
  • Espacio en disco: los modelos ocupan mucho espacio. Planifica al menos 10-20GB para tener una biblioteca básica útil.

1. Ollama: La opción “simplemente funciona”

Descargar: ollama.com

Considere a Ollama como el “Docker para LLMs”. Él encapsula el llama.cpp El motor se incluye en un paquete nativo para macOS, carga los modelos bajo demanda y dirige el procesamiento automáticamente a Metal. Solo hay que instalarlo, ejecutar una orden y ya se puede comenzar a chatear. Este es el camino más sencillo si lo que se busca es contar con un LLM funcional y un endpoint HTTP al que dirigir el código.

Flujo de trabajo de ejemplo

# 1. Download and run Llama 3 (it auto-downloads if needed)
ollama run llama3

# 2. Use it in your code via the local API
curl http://localhost:11434/api/generate -d '{
  "model": "llama3",
  "prompt": "Explain quantum computing to a 5-year-old",
  "stream": false
}'

Ventajas y desventajas

Configuración más sencilla (arrastrar y soltar) .dmg)La aplicación principal es de código cerrado.
CLI limpioollama list, ollama pull)Menor control detallado sobre los parámetros de generación
Amplia biblioteca de modelos preconfigurados

2. LM Studio: el explorador visual

Descargar: lmstudio.ai

LM Studio es la opción orientada principalmente a interfaces gráficas de usuario. Cuenta con un navegador al estilo App Store para buscar directamente en HuggingFace, admite el formato MLX de Apple junto con GGUF (que puede ser más rápido en algunos Macs), y ofrece un servidor local compatible con OpenAI. Por lo tanto, el código de cliente existente que ya se comunica con OpenAI funciona, en su mayoría, sin cambios.

Flujo de trabajo de ejemplo

Puede utilizar el SDK oficial de Python, o cualquier cliente compatible con OpenAI dirigido al servidor local:

# Using the official LM Studio SDK
from lmstudio import LMStudio

client = LMStudio()
response = client.complete(
    model="llama-3-8b",
    prompt="Write a haiku about debugging."
)
print(response.content)

Ventajas y desventajas

✅ Ventajas❌ Desventajas
Interfaz pulida y fácil de usarLa interfaz gráfica es de código cerrado.
Soporte nativo tanto para modelos GGUF como para MLXDescarga más grande (~750 MB)
RAG integrado (chatea con tus PDFs)

3. llama.cpp: La herramienta para usuarios avanzados

Repositorio: github.com/ggml-org/llama.cpp

Este es el motor que casi todas las demás herramientas utilizan como base. Si desea obtener el máximo rendimiento, acceder a las últimas funcionalidades en cuanto se lanzan, o integrar un LLM en su propia aplicación en C++, este es el punto de partida. Es de nivel bare-metal y ligero, pero la contrapartida es que debe gestionar todo usted mismo: descargas, formatos y decenas de parámetros de la línea de comandos.

Flujo de trabajo de ejemplo

# 1. Install via Homebrew
brew install llama.cpp

# 2. Download a model manually (e.g., from HuggingFace)
huggingface-cli download TheBloke/Llama-3-8B-Instruct-GGUF --local-dir .

# 3. Run inference with full control
llama-cli -m llama-3-8b-instruct.Q4_K_M.gguf \
  -p "Write a python script to sort a list" \
  -n 512 \
  --temp 0.7 \
  --ctx-size 4096

Ventajas y desventajas

✅ Ventajas❌ Desventajas
Control total sobre cada parámetroCurva de aprendizaje pronunciada (solo CLI).
Licencia MIT (código abierto)Gestión manual de modelos
Muy ligero (<30 MB)

4. GPT4All: Privacidad en primer lugar y RAG

Descargar: gpt4all.io

GPT4All se basa en dos principios fundamentales: la privacidad y los documentos. Su característica más destacada, LocalDocs, permite indicarle a la aplicación una carpeta con archivos PDF, notas o código para poder interactuar directamente con su contenido mediante chat. Todo funciona sin conexión y sin recopilar datos de telemetría. Es la forma más sencilla de tener una configuración RAG operativa en tu propio equipo, sin necesidad de escribir ningún código.

Ventajas y desventajas

✅ Ventajas❌ Desventajas
LocalDocs RAG funciona correctamente sin necesidad de configuraciones adicionales.Solo GUI (sin modo sin interfaz)
Completamente sin conexión e íntimoUso de recursos mayor que el de Ollama
Multiplataforma (Mac, Windows, Linux)

5. KoboldCPP - Para narradores

Repo: github.com/LostRuins/koboldcpp

Una bifurcación de llama.cpp Dirigido a la escritura creativa y a los RPG de estilo de mesa. Funciona como una aplicación web local que cuenta con herramientas para la generación de texto extenso: “Información del mundo”, memoria de personajes y técnicas para mantener la coherencia narrativa, con el objetivo de que el modelo siga funcionando adecuadamente incluso al manejar miles de tokens. Su público objetivo son escritores y quienes gestionan RPG basados en texto; si lo que buscas principalmente es chatear o programar, la interfaz habitual resultará demasiado limitada.

Flujo de trabajo de ejemplo

# 1. Download the single binary
wget https://github.com/LostRuins/koboldcpp/releases/latest/download/koboldcpp-mac.zip

# 2. Run it (launches a web server)
./koboldcpp --model llama-3-8b.gguf --port 5001 --smartcontext

Ventajas y desventajas

✅ Ventajas❌ Desventajas
Herramientas potentes para la escritura creativaUI de nicho (no es ideal para programación/chats)
Ejecutable de un solo archivo (sin instalación)Licencia AGPL (restricciones para uso comercial)

Mención honorífica: MLX-LM

Si eres un desarrollador de Python en Apple Silicon, echa un vistazo a MLX-LM de Apple. Se trata de un framework optimizado para los chips de la serie M, y en el hardware adecuado suele ser la forma más rápida de ejecutar un modelo localmente. La desventaja es que es menos flexible que Ollama: más código en Python y menos restricciones.

Qué elegir realmente

  • Comienza con Ollama si lo que deseas es tener algo funcionando ya mismo.
  • Opta por LM Studio si prefieres explorar los modelos de forma visual primero.
  • Elige llama.cpp cuando necesites un control total sobre el proceso de inferencia.

Conclusiones clave

  1. Ollama es la vía más rápida para disponer de un servidor de modelos local destinado al desarrollo.
  2. LM Studio es la opción más adecuada cuando se necesita una interfaz gráfica de escritorio y una exploración sencilla de los modelos.
  3. llama.cpp ofrece el control más directo sobre los modelos GGUF y las opciones de ejecución de bajo nivel.
  4. MLX es la opción nativa de Apple Silicon para quienes se sienten cómodos escribiendo en Python y utilizando los formatos de modelos MLX.
