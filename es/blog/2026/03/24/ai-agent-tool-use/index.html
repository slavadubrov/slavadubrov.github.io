Uso de herramientas de agentes de IA en 2026: MCP, CLI, Skills y ejecución de código

Parte 3 de la serie “Ingeniería de la pila agente”

Parte 1 bucles de razonamiento cubiertos. Parte 2 Memoria ocupada. Este artículo trata sobre el uso de herramientas por parte de agentes de IA: cómo interactúan estos agentes con el mundo y por qué el diseño de las herramientas es más importante de lo que la mayoría de los equipos reconocen.

La situación relativa a las herramientas cambió entre 2025 y 2026. MCP se convirtió en el estándar de facto para los servicios externos, pero los equipos de producción siguen topándose con sus deficiencias de seguridad y el alto consumo de tokens. El otro enfoque que está ganando adopción son los agentes que escriben y ejecutan código en lugar de llamar a herramientas definidas mediante JSON. Anthropic midió una reducción del 98,7 % en los tokens en un flujo de trabajo representativo, y el artículo de CodeAct indica tasas de éxito en las tareas un 20 % más altas.

Este artículo compara la llamada a herramientas mediante JSON, MCP, Skills, herramientas CLI y la ejecución de código, y luego las relaciona con los principios de diseño de la Interfaz Agente-Computadora (ACI) que yo aplico en Agente de Análisis de Mercado.

TL;DR: Existen cinco modalidades de herramientas para agentes de IA, cada una con sus ventajas específicas. La combinación práctica para 2026 es la siguiente: habilidades especializadas para el conocimiento profundo, CLI como opción por defecto, MCP para servicios externos, y ejecución de código para la orquestación de múltiples pasos. Todos ellos se rigen por los principios de diseño ACI provenientes de SWE-agent.

In Parte 1Mostré cómo los bucles de razonamiento de los agentes de IA determinan cómo pensar. Las modalidades de herramienta, por su parte, deciden cómo actuar. Actualmente existen cinco enfoques distintos, cada uno con diferentes compromisos en cuanto al costo de tokens, la flexibilidad y la seguridad.

1. Llamadas a herramientas JSON: el punto de partida

El patrón original consiste en definir los esquemas de las herramientas en formato JSON; el LLM genera llamadas a funciones estructuradas, y su código las ejecuta. Este enfoque está ampliamente reconocido y funciona correctamente para conjuntos de herramientas pequeños.

# Traditional tool definition — each tool consumes ~550-1,400 tokens (Apideck benchmark)
tools = [
    {
        "name": "get_stock_price",
        "description": "Get the current stock price for a ticker symbol",
        "input_schema": {
            "type": "object",
            "properties": {
                "ticker": {"type": "string", "description": "Stock ticker (e.g., NVDA)"}
            },
            "required": ["ticker"]
        }
    }
]

Con 5-10 herramientas, está bien. El problema es la escala: cada definición de herramienta tiene un costo. 550-1,400 tokens. Con 20 herramientas, ya se gastan entre 15 y 25 mil tokens antes de que el agente siquiera comience a razonar.

2. MCP: El estándar universal (con matices)

El Model Context Protocol es el estándar al que se han alineado la mayoría de los proveedores. Anthropic lo donó a la Linux Foundation bajo la Agentic AI Foundation (AAIF) en diciembre de 2025, en colaboración con OpenAI y Block, contando con el apoyo de Google, Microsoft y AWS como miembros de nivel platino. OpenAI incorporó soporte para MCP en su API Responses. Actualmente existen más de 10,000 servidores MCP activos y más de 97 millones de descargas mensuales de SDK.

MCP destaca en escenarios como la integración entre diferentes plataformas SaaS (Figma, Notion, Salesforce), la gobernanza empresarial con trazabilidad de auditorías, servicios que no disponen de equivalentes mediante línea de comandos, y entornos que requieren orquestación mediante OAuth. Para estos casos de uso, MCP es sin duda la opción más adecuada.

Sin embargo, la realidad en entornos de producción es algo más compleja de lo que sugieren estas cifras superficiales.

La superficie de seguridad es el primer problema. El Proyecto MCP vulnerable Detecta 50 vulnerabilidades en servidores MCP, de las cuales 13 están clasificadas como Críticas, reportadas por 32 investigadores de seguridad. Las categorías de ataque incluyen inyección de prompts, fallos en la validación de entradas, brechas de autenticación y vulnerabilidades de seguridad de red. El primer servidor MCP malicioso en el mundo real Apareció en septiembre de 2025.: un paquete llamado postmark-mcp que enviaba por copia oculta cada correo electrónico enviado a la dirección del atacante, afectando a unas 300–500 organizaciones antes de que se descubriera el problema.

La envenenamiento de herramientas es el tipo de ataque que más me preocupa. Invariant Labs mostró Esos herramientas MCP contaminadas pueden extraer datos incluso cuando nunca se invocan. Basta con que el modelo lea los metadatos de la herramienta para desencadenar el ataque. Pruebas de rendimiento de MCPTox Al probar 20 agentes de LLM contra 45 servidores MCP del mundo real, se observó que las tasas de éxito en los ataques alcanzaron hasta el 72,8%.

El sobrecoste por tokens representa el problema operativo principal. Un equipo que gestiona servidores MCP para GitHub, Slack y Sentry (un total de aproximadamente 40 herramientas) descubrió 55,000 tokens de las definiciones de esquema inyectadas antes de que el usuario pregunte algo. Otra queja reportada 143.000 de 200.000 tokens disponibles (72 %) Consumido únicamente por las definiciones de herramientas.

Las medidas de mitigación funcionan, pero confirman el problema subyacente. De Anthropic’s La herramienta de búsqueda de herramientas reduce la sobrecarga del esquema en un 85%.. El cargamento dinámico de herramientas (3-5 herramientas relevantes por tarea) permite una reducción del 96 %. Se trata de soluciones temporales para un protocolo que no fue diseñado teniendo en cuenta los presupuestos de tokens.

3. Habilidades (SKILL.md) — Expertise, no ejecución

A finales de 2025 se produjo la estandarización de las habilidades de agente como formato abierto (lanzado en octubre de 2025 y publicado como estándar abierto en diciembre de 2025). Esta distinción es importante: las herramientas proporcionan capacidades (lo que pueden hacer los agentes), mientras que las habilidades aportan conocimiento especializado (lo que saben los agentes sobre cómo llevar a cabo tareas complejas).

El estándar SKILL.md define una habilidad como un archivo Markdown con frontmatter en YAML:

---
name: deploy
description: Deploy the application to production
argument-hint: "[environment]"
user-invocable: true
---
Deploy the application to the $0 environment (default: staging).
Steps:
1. Run the test suite
2. Build the production bundle
3. Deploy using the deploy script
4. Verify the deployment health check

Las habilidades emplean divulgación progresiva: únicamente los metadatos (~100 tokens en el frontmatter YAML) se cargan al iniciar, mientras que las instrucciones completas se cargan cuando se activa la habilidad. Compárese esto con MCP, donde unos ~40 herramientas de servicios típicos consumen ~55,000 tokens Antes de iniciar cualquier razonamiento: la adopción multiplataforma a fecha de marzo de 2026 incluye Claude Code, OpenAI Codex CLI, Cursor, GitHub Copilot, Gemini CLI, Goose, Windsurf y Roo Code, lo que representa una tendencia clara. Victor Dibia realizó una exploración en su análisis del cambio de las herramientas a acciones de agente impulsadas por código.

Cuando las habilidades son clave: la transferencia de conocimiento especializado en un dominio, procedimientos complejos de múltiples pasos, flujos de trabajo recurrentes como migraciones de bases de datos o integraciones de pagos, y cualquier tarea en la que el agente necesite saber cómo hacer algo en lugar de solo qué hacer.

4. CLI/Bash: El renacimiento de Unix

Resultó que la terminal de Unix es la interfaz de agente más eficiente en términos de tokens. Las cifras son abrumadoras: un manifiesto de herramienta CLI cuesta alrededor de 100 tokens, mientras que los esquemas equivalentes de servidores MCP consumen 50,000+ tokens. Eso representa una diferencia enorme. 4-32x de diferencia medida por Scalekit en 75 ejecuciones de pruebas de rendimiento (32 veces para tareas como la identificación del lenguaje del repositorio y la búsqueda de licencias).

Los modelos son hablantes nativos de la línea de comandos. Sus datos de entrenamiento incluyen miles de millones de interacciones en la terminal provenientes de Stack Overflow, repositorios de GitHub y documentación. Entienden esto de forma nativa. git, docker, kubectl, gh, curl, y jq sin ninguna definición de esquema.

La guía de Ugo Enyioha “Cómo crear herramientas de línea de comandos que los agentes de IA realmente quieran utilizar” Se han codificado ocho reglas de diseño:

  1. El resultado estructurado es obligatorio — se debe ofrecer soporte --json
  2. Los códigos de salida son parte del flujo de control: utilice códigos distintos para diferentes tipos de errores.
  3. Los comandos deben ser idempotentes.
  4. Autodocumentación. --help con ejemplos realistas
  5. Diseño para la componibilidad — --quiet Para valores sin formato, soporte para stdin
  6. Proporcionar --dry-run y --yes flags
  7. Soporte para la introspección de versiones
  8. Gestión de autenticación mediante variables de entorno

Los compromisos son reales. La CLI carece de la seguridad tipológica de MCP, de la orquestación integrada de OAuth, de la detección de herramientas y del registro de auditoría. El patrón al que parece llegar la mayoría de los equipos es: utilizar la CLI como opción predeterminada para el desarrollo y las operaciones locales, y MCP para la integración con servicios externos y la gobernanza empresarial.

5. Ejecución de código: el gran cambio

Este es el cambio en las herramientas de agente que considero más significativo. En lugar de que el LLM emita JSON estructurado para invocar funciones predefinidas una por una, el agente escribe un script completo en Python o Bash que llama a múltiples herramientas, procesa los resultados mediante bucles y condicionales, y solo devuelve resúmenes finales al contexto del modelo.

Anthropic formalizó esto mediante Llamadas a herramientas de forma programática (PTC), ahora disponible en la API de Claude. La base académica es la Artículo CodeAct (Wang et al., ICML 2024), que realizaron pruebas en 17 modelos de lenguaje grande y descubrieron que las acciones basadas en código lograron tasas de éxito en las tareas hasta un 20 % más altas, así como 30 % menos pasos en comparación con las alternativas basadas en JSON.

La evidencia de producción: Vercel ha reconstruido su agente d0 de texto a SQL al eliminar el 80 % de sus herramientas (de 15 a 2): ExecuteCommand y ExecuteSQL). El éxito de la tarea aumentó del 80 % al 100 %, el tiempo de ejecución disminuyó 3,5 veces y el consumo de tokens se redujo en un 40 %. Su forma de expresarlo es: “Los mejores agentes podrían ser aquellos que disponen de la menor cantidad de herramientas”.

  • Cloudflare desarrolló el “Modo Código”. permitiendo que los agentes escriban código en TypeScript para llamar a su API en lugar de definir esquemas de herramientas, lo cual reduce la sobrecarga de contexto. Su razonamiento es el siguiente: “Los LLMs cuentan con una enorme cantidad de código en TypeScript del mundo real en sus conjuntos de entrenamiento, pero solo un pequeño conjunto de ejemplos artificiales de llamadas a herramientas”. La documentación de PTC de Anthropic. Las herramientas tradicionales utilizadas para el análisis de gastos requieren más de 20 pasos de inferencia independientes, uno por cada miembro del equipo, y todos los datos intermedios deben pasar por el contexto. Un flujo de trabajo que consumía aproximadamente 150.000 tokens al utilizar llamadas directas a herramientas fue reimplementado para solo unos 2.000 tokens, lo que supone una reducción del 98,7%. Con la ejecución de código, el agente escribe un único script:
# Agent generates this code, executes in sandbox
import json
members = get_team_members("engineering")
over_budget = []
for m in members:
    expenses = get_expenses(m["id"], "Q3")
    total = sum(e["amount"] for e in expenses)
    if total > 5000:
        custom = get_custom_budget(m["id"])
        limit = custom["limit"] if custom else 5000
        if total > limit:
            over_budget.append({"name": m["name"], "spent": total, "limit": limit})
# Only this final summary returns to the LLM context
print(json.dumps(over_budget))

El LLM solo ve el resumen JSON final, y no los miles de elementos de gasto procesados en el entorno de pruebas. Eficiencia en el uso de tokens, composibilidad nativa (bucles y condicionales sin coste adicional), manejo real de errores.try/except En lugar de recurrir al razonamiento en lenguaje natural para analizar errores, la privacidad (ya que los datos sensibles permanecen dentro del entorno aislado) también mejora de forma simultánea.

Cuando seguir utilizando herramientas JSON tiene sentido: operaciones atómicas individuales, entornos sin infraestructura de aislamiento, modelos más pequeños con capacidad limitada de generación de código, o requisitos de auditoría que exigen que se registre cada llamada a una herramienta por separado.

La interfaz agente-computadora (ACI) para herramientas de agentes de IA

El término “Interface entre Agente y Ordenador” (ACI) fue acuñado por John Yang, Carlos E. Jiménez y sus colegas en Princeton en su Artículo sobre SWE-agent (NeurIPS 2024). La idea es sencilla: al igual que los seres humanos se benefician de interfaces bien diseñadas (HCI), los agentes de LM constituyen “una nueva categoría de usuarios finales con sus propias necesidades y capacidades, que también se beneficiarían de interfaces desarrolladas específicamente para ellos”. resultados de ablación Se realizó una copia de seguridad. El SWE-agent, con su ACI completo, logró un 18,0 % en SWE-bench Lite, frente al 7,3 % que se obtuvo únicamente con una shell estándar de Linux; esto representa una mejora de 10,7 puntos porcentuales solo gracias al diseño de la interfaz. La función de control de errores por linting, por sí sola, aportó 3 puntos porcentuales: el 51,7 % de las ediciones realizadas por el agente presentaban al menos un error detectado por el lint antes de que pudiera propagarse.

Anthropic adoptó el ACI como un concepto fundamental en su “Desarrollo de agentes eficaces” Guía, presentándola como uno de los tres principios fundamentales: “Diseña cuidadosamente tu interfaz agente-computadora mediante una documentación y pruebas exhaustivas de las herramientas”. Su consejo práctico: “Una regla práctica es considerar la cantidad de esfuerzo que se invierte en las interfaces humano-computadora, y planificar invertir la misma cantidad en crear buenas interfaces agente-computadora”.

Los cuatro principios en la práctica

1. Las acciones deben ser simples y fáciles de comprender. El error más frecuente es envolver los puntos finales de la API de forma uno a uno. En lugar de list_users, list_events, create_event, implementar schedule_event que determina la disponibilidad y programa las tareas en una sola llamada. En lugar de read_logs, implementar search_logs que devuelve únicamente las líneas relevantes junto con su contexto.

2. Las acciones deben ser compactas y eficientes. Consolide las operaciones importantes en el menor número posible de acciones. En la Agente de Análisis de Mercado, combino la obtención de precios con métricas básicas en una sola get_stock_snapshot herramienta que evita la necesidad de realizar llamadas separadas para obtener el precio, el volumen, el valor de mercado y la relación P/E.

3. La retroalimentación del entorno debe ser informativa pero concisa. Evite devolver HTML sin procesar o cargas completas de la API. Convierta los ID poco claros en nombres semánticos. Las pruebas de Anthropic demostró que al añadir un response_format El uso de un enum que permite a los agentes solicitar respuestas concisas (~72 tokens) o detalladas (~206 tokens), lo que supone una diferencia de costo en tokens de 3 veces, mejoró notablemente el rendimiento en entornos reales.

4. Las restricciones deben mitigar la propagación de errores. La detección automática de errores ayuda a los agentes a reconocer y corregir errores de forma rápida. En SWE-agentUn editor de archivos personalizado con linting integrado rechaza automáticamente los errores de sintaxis, detectando el 51,7 % de los errores en las ediciones realizadas por los agentes antes de que puedan agravarse. Aplico el mismo principio en el Market Analyst Agent al validar los argumentos de las herramientas mediante esquemas Pydantic antes de su ejecución:

from pydantic import BaseModel, Field, field_validator

class StockQuery(BaseModel):
    """Validated input for stock queries.

    Pydantic catches malformed tickers before the API call,
    preventing error propagation through the reasoning loop.
    """
    ticker: str = Field(description="Stock ticker symbol (e.g., NVDA)")
    period: str = Field(default="1mo", description="Time period: 1d, 5d, 1mo, 3mo, 1y")

    @field_validator("ticker")
    @classmethod
    def validate_ticker(cls, v: str) -> str:
        v = v.upper().strip()
        if not v.isalpha() or len(v) > 5:
            raise ValueError(f"Invalid ticker format: {v}")
        return v

    @field_validator("period")
    @classmethod
    def validate_period(cls, v: str) -> str:
        valid = {"1d", "5d", "1mo", "3mo", "6mo", "1y", "5y"}
        if v not in valid:
            raise ValueError(f"Invalid period: {v}. Must be one of {valid}")
        return v

Patrones de diseño para herramientas de agentes de IA que funcionan

Anthropic’s “Desarrollo de herramientas eficaces para agentes” define los marcos de guía como “un nuevo tipo de software que refleja un contrato entre sistemas deterministas y agentes no deterministas”. A continuación se presentan los patrones que han surgido de la experiencia en entornos de producción.

Tratar las descripciones de herramientas como ingeniería de prompts

Las descripciones deben funcionar como Se requiere un mínimo de 3 a 4 oraciones en la traducción., abarcando cuándo utilizar la herramienta, parámetros obligatorios frente a opcionales, formato de salida y casos límite. Nomenclatura con prefijosasana_search, jira_search) tiene «efectos no triviales» en la precisión de la selección de herramientas. En los experimentos de Anthropic, las descripciones de herramientas optimizadas para Claude superaron a las redactadas por expertos humanos en los benchmarks de evaluación.

# Bad: vague, no context for when to use
tools = [{
    "name": "search",
    "description": "Search for items",
}]

# Good: specific, with input examples and edge cases
tools = [{
    "name": "stock_search_news",
    "description": (
        "Search for recent news articles about a specific stock or company. "
        "Use this tool when the user asks about recent events, earnings, "
        "announcements, or market-moving news for a specific ticker. "
        "Returns up to 10 articles sorted by relevance. "
        "For broad market news (not ticker-specific), use market_overview instead."
    ),
    "input_schema": {
        "type": "object",
        "properties": {
            "query": {
                "type": "string",
                "description": "Search query. Examples: 'NVDA earnings Q3 2025', 'Tesla delivery numbers'"
            },
            "max_results": {
                "type": "integer",
                "description": "Max articles to return (1-10, default 5)",
                "default": 5
            }
        },
        "required": ["query"]
    }
}]

Pruebas internas mostró que los ejemplos de entrada (input_examples field) ha mejorado de forma sustancial la precisión en el manejo de parámetros complejos.

Devolver una salida de alto nivel y legible por máquinas

Evitar identificadores de bajo nivel (uuid, mime_type). Asigne nombres semánticos a los ID enigmáticos. Estructure la respuesta de modo que el agente pueda razonar sobre ella sin tener que analizar texto genérico.

# Bad: raw API response dumped to agent
def get_stock_price(ticker: str) -> dict:
    response = api.get(f"/v1/quotes/{ticker}")
    return response.json()  # 500+ tokens of nested JSON

# Good: high-signal summary the agent can immediately reason about
def get_stock_price(ticker: str) -> dict:
    data = api.get(f"/v1/quotes/{ticker}").json()
    return {
        "ticker": ticker,
        "price": data["regularMarketPrice"],
        "change_pct": round(data["regularMarketChangePercent"], 2),
        "volume": data["regularMarketVolume"],
        "market_cap_b": round(data["marketCap"] / 1e9, 1),
        "pe_ratio": data.get("trailingPE"),
        "summary": f"{ticker} at ${data['regularMarketPrice']:.2f} "
                   f"({'up' if data['regularMarketChangePercent'] > 0 else 'down'} "
                   f"{abs(data['regularMarketChangePercent']):.1f}%)"
    }

Diseño de manejo de errores para la resiliencia

El patrón que ha demostrado ser eficaz en entornos de producción consiste en cuatro capas de tolerancia a fallos:

  1. Reintentos con retroceso exponencial para errores transitorios
  2. Cadenas de fallback del modelo ante interrupciones del proveedor
  3. Enrutamiento por clasificación de errores: los errores transitorios se vuelven a intentar, los errores recuperables mediante LLM se devuelven al agente con contexto, y los errores que requieren intervención humana se escalan
  4. Recuperación mediante puntos de control para sobrevivir a caídas bruscas

Los equipos informan que las fallas irreparables han disminuido del 23 % al menos del 2 % con este enfoque de cuatro capas, tal como se documenta en Patrones de resiliencia de las herramientas de Anthropic.

from tenacity import retry, stop_after_attempt, wait_exponential

@retry(
    stop=stop_after_attempt(3),
    wait=wait_exponential(multiplier=1, min=2, max=10),
)
def call_stock_api(ticker: str) -> dict:
    """Fetch stock data with automatic retry on transient failures.

    Layer 1: Exponential backoff handles rate limits and network blips.
    If all retries fail, the error propagates to the agent with
    enough context to decide whether to try a different approach.
    """
    response = httpx.get(
        f"https://api.example.com/v1/quotes/{ticker}",
        timeout=10.0,
    )
    response.raise_for_status()
    return response.json()

Esto redujo de forma significativa la sobrecarga asociada al esquema de herramientas y aumentó la fiabilidad en la selección de herramientas por parte del agente, ya que hay menos opciones ambiguas entre las que decidir.

Salidas estructuradas para los resultados de la herramienta

Cada herramienta del Market Analyst Agent devuelve una respuesta validada con Pydantic. Esto aplica el principio de límites de seguridad de ACI en los bordes de las herramientas:

class StockSnapshot(BaseModel):
    """Structured tool response — the agent never sees raw API noise."""
    ticker: str
    price: float
    change_pct: float
    volume: int
    market_cap_b: float
    pe_ratio: float | None
    summary: str  # Human-readable one-liner for direct use in reports

class NewsResult(BaseModel):
    """Each news item is pre-processed for agent consumption."""
    headline: str
    source: str
    date: str
    relevance_score: float  # Pre-ranked so the agent doesn't waste tokens sorting
    key_points: list[str]  # Extracted by the tool, not the agent

El summary El campo es el que más importancia tiene. Proporciona al agente una cadena lista para usar que puede incorporarse directamente en un informe sin necesidad de procesamiento adicional. key_points en NewsResult Se extraen del lado del servidor, lo que evita que el agente gaste tokens de inferencia en el análisis del contenido de los artículos.

Compromisos y consideraciones

Más allá de las limitaciones específicas de cada modalidad mencionadas anteriormente, existen algunos aspectos transversales que influyen en la elección:

  • El costo operativo varía según la dimensión. La ejecución de código ahorra tokens, pero aumenta la latencia inicial en el entorno aislado. MCP acelera el desarrollo de integraciones SaaS, aunque incrementa la carga por la implementación del servidor. La interfaz CLI es gratuita para iniciar, pero resulta más difícil de gestionar a gran escala. Optimice según su cuello de botella real, ya sea el costo de tokens, la latencia o la complejidad operativa.

  • Las habilidades del equipo son clave. La ejecución de código presupone que sus agentes (y los modelos subyacentes) pueden generar código fiable en Python o TypeScript. La CLI requiere conocimiento de las convenciones de Unix. MCP exige comprensión de protocolos de transporte y flujos OAuth. Ajuste la modalidad a las fortalezas de su equipo.

  • La consolidación de herramientas puede llevarse demasiado lejos. Si combina todo en una sola herramienta masiva, el espacio de parámetros se vuelve demasiado complejo para que el agente lo maneje. Para la mayoría de los sistemas de agentes, el punto óptimo es contar con entre 5 y 10 herramientas bien delimitadas.

  • Las habilidades se basan en prompts, no son obligatorias. Una habilidad son instrucciones que el agente debería seguir, no reglas estrictas que debe cumplir. En flujos de trabajo críticos, combine las habilidades con validaciones deterministas.

  • Los requisitos de auditoría determinan la elección. Si necesita un registro completo de cada llamada a una herramienta y su resultado, MCP y las llamadas a herramientas en formato JSON ofrecen registros de auditoría estructurados de forma predeterminada. La ejecución de código genera un script y sus resultados, lo cual es útil, pero resulta más difícil descomponerlo en acciones individuales para la elaboración de informes de cumplimiento.

Hacia dónde se dirige la ergonomía de las herramientas

Hay tres tendencias que merecen ser seguidas.

La primera es el uso de herramientas RAG para escalar la capacidad. A medida que los conjuntos de herramientas pasan de cientos a miles, la precisión en la selección de herramientas de forma intuitiva disminuye hasta el 13,62%. El Artículo sobre RAG-MCP Se demostró que la aplicación de la generación reforzada por recuperación en la selección de herramientas —es decir, indexando las descripciones de las herramientas en una base de datos vectorial y recuperando únicamente las herramientas relevantes para cada consulta— permite alcanzar una precisión del 43,13 %, lo que representa una mejora de 3,2 veces, al tiempo que se reducen los tokens de los prompts en aproximadamente un 50 %.

El segundo enfoque consiste en que los agentes creen sus propias herramientas. marco LATM (“LLMs As Tool Makers”) definió un paradigma en dos fases en el que un LLM potente genera funciones de Python reutilizables, y un LLM de menor capacidad las utiliza. ToolMaker (ACL 2025) transforma de forma autónoma los repositorios de GitHub en herramientas compatibles con LLMs, logrando una tasa de éxito del 80 %. La frontera tecnológica está pasando del uso de herramientas a su creación, y posteriormente a la gestión de bibliotecas de herramientas.

El tercero es la pila de protocolos duales A2A + MCP. El Protocolo Agent2Agent de Google (A2A) aborda lo que MCP no puede hacer: la comunicación entre agentes. MCP se encarga de la integración agente-herramienta; A2A se ocupa del descubrimiento, la negociación y la delegación de tareas entre agentes. La fórmula combinada es: construye con tu framework, equipa con MCP y comunícate con A2A.

Conclusiones clave

  1. El uso de herramientas tiene cinco modalidades distintas, cada una con un punto óptimo claro. No uses MCP para todo; adapta la modalidad a la tarea.

  2. La ejecución de código en lugar de llamar a herramientas mediante JSON representa el cambio más importante en las herramientas para agentes. PTC de Anthropic, la reducción de herramientas en Vercel y Code Mode de Cloudflare convergen en la misma idea: permitir que los agentes escriban código en lugar de llamar a esquemas.

  3. La sobrecarga de tokens es el cuello de botella operativo. 55K+ tokens para unas 40 herramientas MCP frente a unos 100 tokens para los manifiestos CLI no es una diferencia menor. Es la diferencia entre que un agente tenga el 65 % frente al 95 % de su contexto para razonar.

  4. Los principios de diseño ACI son más importantes que el protocolo. SWE-agent demostró que solo el diseño de la interfaz explicaba 10,7 puntos porcentuales en las pruebas de rendimiento. Acciones simples, retroalimentación concisa y medidas de seguridad integradas son aplicables independientemente de si se utiliza MCP, CLI o ejecución de código.

  5. Consolida las herramientas de forma agresiva. Cinco herramientas bien diseñadas superan en rendimiento a veinte herramientas más detalladas. Cada herramienta es una decisión que obligas al modelo a tomar, así que reduce el espacio de decisión.

  6. Trate las descripciones de herramientas como parte del diseño de prompts. Los ejemplos de entrada mejoran significativamente la precisión. Las descripciones optimizadas para Claude superan en rendimiento a las redactadas por expertos humanos. Invierta el mismo esfuerzo en la experiencia de usuario de las herramientas que lo haría en la experiencia de usuario humana.

  7. La arquitectura práctica para 2026 consiste en Skills + CLI + MCP + Ejecución de código, organizada según el caso de uso. Skills para la especialización, CLI para operaciones locales, MCP para servicios externos, y ejecución de código para la orquestación de múltiples pasos.
