Mejores herramientas locales de LLM para macOS

Las herramientas locales de LLM en macOS se clasifican según su función: servicio sencillo, exploración mediante interfaz gráfica, control de inferencia a nivel bajo y experimentación nativa en Apple Silicon. Una sola herramienta no necesita realizar las cuatro tareas.

Configuración por defecto: Ollama para una API local. LM Studio para la exploración de modelos. llama.cpp cuando se necesita control del entorno de ejecución GGUF. MLX cuando se desea trabajar con Python de forma nativa en Apple Silicon, cerca del modelo.

¿Cuál debería instalar primero?

Instale primero Ollama si está desarrollando software. Muchas aplicaciones saben cómo comunicarse con él, y la API local es suficiente para prototipos, pruebas y herramientas internas de pequeño tamaño. Es el camino más corto desde “Necesito un modelo local” hasta “Mi aplicación puede llamar a un modelo local”.

Instale primero LM Studio si va a elegir un modelo. Es útil para explorar modelos, modificar configuraciones, comparar resultados y ejecutar un servidor local compatible con OpenAI sin tener que diseñar el flujo de trabajo por sí mismo.

Instale primero llama.cpp si le interesan los aspectos técnicos de la inferencia. La longitud del contexto, la cuantización, las opciones de Metal, el procesamiento de prompts, los tamaños de lote y el comportamiento del servidor son más fáciles de analizar cuando se está cerca del momento de ejecución.

Utilice MLX cuando vaya a hacer algo más que simplemente ofrecer un modelo de chat. Es adecuado para experimentos con modelos nativos de Apple Silicon, conversión, ajuste fino y flujos de trabajo en Python donde la memoria unificada forma parte del diseño.

Notas de hardware

La memoria unificada es la verdadera limitación en Apple Silicon. Un modelo que cabe en un MacBook Pro de 64 GB puede resultar inutilizable en un MacBook Air de 8 GB. La cuantización ayuda, pero la longitud del contexto puede influir de forma significativa en el consumo de memoria. Realice pruebas con la estructura real de los prompts en lugar de centrarse únicamente en el nombre del modelo.

Para herramientas locales de pequeño tamaño, un modelo de clase 7B o 8B suele ser más útil que uno más grande y sobrecargado. En el ámbito del desarrollo de software, un contexto extenso y la integración con herramientas pueden ser más importantes que una simple clasificación en pruebas de rendimiento. En cuanto a la verificación de calidad de documentos, la calidad de la recuperación de información suele ser el factor determinante a la hora de elegir un modelo local.

Qué no hacer

No convierta la configuración de un LLM local en un proyecto permanente basado en pruebas de rendimiento, a menos que el objetivo sea mejorar el rendimiento del producto. Comience utilizando Ollama o LM Studio. Demuestre primero que la inferencia local es beneficiosa. Solo entonces, cuando tenga una razón concreta, pase a utilizar llama.cpp o MLX.

No compare modelos únicamente a través de una interfaz de chat si la carga de trabajo real consiste en extracción estructurada, edición de código o síntesis de respuestas mediante RAG. Escriba un pequeño script de evaluación con prompts representativos.
