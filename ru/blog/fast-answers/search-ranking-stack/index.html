Лучшая стек-архитектура для ранжирования результатов поиска в продуктах ИИ

Хорошая стек-архитектура для ранжирования — это своего рода воронка. В первую очередь важна низкая стоимость восстановления всех возможных результатов. Высокая точность ранжирования имеет меньшее значение. Большинство команд сталкиваются с проблемами, когда заменяют лексикальный поиск на эмбеддинги вместо того, чтобы использовать их совместно.

Стандартная архитектура: начните с алгоритма BM25 и фильтров. Добавьте метод плотного поиска для улучшения семантического восстановления результатов. Объедините разреженные и плотные кандидаты с помощью метода RRF или аналогичного подхода. Переранжируйте полученный список с использованием кросс-энкодера. Используйте переранжирование с помощью больших языковых моделей только для очень небольшого набора кандидатов, где прирост качества оправдывает задержку и дополнительные затраты.

Почему BM25 по-прежнему имеет место в стеке технологий

Эмбеддинги хорошо справляются с задачей определения семантической схожести. Однако они не могут служить надежной заменой точным токенам. Коды ошибок, имена функций, SKU продуктов, юридические формулировки и имена людей зачастую передают необходимую информацию именно благодаря точному тексту. BM25 по-прежнему остается надежной базовой моделью, поскольку она стимулирует учет тех терминов, которые пользователь действительно ввел.

Метод плотного поиска повышает показатель воспроизводимости, когда пользователь не знает точного словаря. Подходящим решением не является ни BM25, ни эмбеддинги. Для лексической воспроизводимости используйте BM25, для семантической — эмбеддинги, а для объединения результатов применяйте метод слияния.

Когда добавлять ранжировщик

Добавьте кросс-энкодер, если подходящие документы находятся где-то в числе первых 50 результатов, но не в их верхней части. Это самый ясный признак того, что генерация кандидатов работает, но требуется дополнительная помощь в ранжировании.

Не добавляйте ранжировщик на основе LLM до использования кросс-энкодера, если набор кандидатов слишком мал. Кроме того, необходимо иметь достаточно тонкий критерий релевантности, чтобы оправдать связанные с этим затраты. Ранжирование с использованием LLM может помочь, но оно дорогое и медленное. Сравните его с более дешевыми вариантами ранжирования.

Последовательность оценки

  1. Отметьте небольшой набор реальных запросов.
  2. Оцените работу BM25 в одиночку.
  3. Добавьте метод плотного поиска и измерьте изменение показателя воспроизводимости.
  4. Добавьте метод слияния и измерьте показатели nDCG и Recall@k.
  5. Добавьте ранжирование с использованием кросс-энкодера и измерьте Precision@1 и nDCG.
  6. Добавьте ранжирование с использованием LLM только в том случае, если это улучшает качество при учете затрат и времени обработки.
  7. Отслеживайте показатели в реальном времени: доля запросов без результатов, частота переформулирования запросов, коэффициент кликов, коррекция ответов, задержка обработки на уровне p95 и затраты.
