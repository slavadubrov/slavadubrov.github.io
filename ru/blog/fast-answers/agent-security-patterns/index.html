    Лучшие паттерны безопасности AI-агентов в 2026 году

    Безопасность агентов — это безопасность действий. Чат-бот может сказать что-то неверное и поставить вас в неловкое положение. Агент может вызвать инструмент с реальными учетными данными и изменить состояние production.

    Мой базовый принцип: не давайте агенту возможности, которые он никогда не должен использовать. Начинайте с узких инструментов, проверок политик перед вызовом инструмента, изоляции в песочнице, credentials с ограниченной областью, шлюзов человеческого подтверждения, трасс и guardrails. Фильтрация вывода помогает, но не является границей безопасности.

    Рейтинг паттернов

    Паттерн Приоритет Защищает от Примечание по реализации
    Инструменты с наименьшими привилегиями P0 Избыточной агентности Не предоставляйте инструменты, которые агент никогда не должен использовать.
    Проверки политик перед вызовом инструмента P0 Опасных действий Проверяйте конкретное действие непосредственно перед выполнением.
    Песочницы P0 Ущерба через файлы, shell, браузер и сеть Изолируйте код и недоверенный контент.
    Человеческие подтверждения P0 Необратимых или регулируемых действий Ставьте шлюз на запись, deployment, платежи, внешние отправки и привилегированные изменения.
    Credentials с ограниченной областью P0 Избыточных полномочий credentials и сбоев confused deputy Используйте отдельные узкие scopes для каждого сервера и инструмента.
    Изоляция MCP-серверов P1 poisoning инструментов, shadowing инструментов, межсерверных атак Не смешивайте недоверенные серверы и мощные инструменты в одном контексте без ревью.
    Аудит-трассы P1 Неизвестной истории инцидента Сохраняйте запрос пользователя, вызов инструмента, аргументы, результат, решение политики и подтверждающего.
    Guardrails P1 Небезопасного входного и выходного текста Полезны, но недостаточны для полномочий инструмента.
    Red-team evals P1 Известных путей атаки Тестируйте prompt injection, poisoning инструментов, эксфильтрацию данных и обходы разрешений.

    Что внедрять в первую очередь

    Сначала удаляйте возможности. Если агенту не нужна запись в GitHub, не выдавайте ему write token. Если ему нужна только доступность календаря, не давайте полный доступ к mailbox. Наименьшие привилегии лучше, чем строгий prompt.

    Затем поставьте границу политики перед каждым вызовом инструмента. Проверяйте точное имя инструмента, аргументы, целевой ресурс, пользователя, окружение и побочный эффект. Запрос пользователя может выглядеть безобидно, тогда как сгенерированная shell-команда — нет.

    Добавьте песочницы для выполнения кода, автоматизации браузера, доступа к файлам и обработки недоверенных документов. Песочница не делает действие корректным, но снижает ущерб от скомпрометированного результата инструмента или ошибившейся модели.

    Используйте человеческое подтверждение для необратимых действий. Не подтверждайте каждый шаг. Подтверждайте границы: production deployment, удаление данных, отправку email, движение денег, изменения разрешений и регулируемые решения.

    Риски, специфичные для MCP

    MCP полезен, потому что стандартизирует доступ к инструментам. Он рискован, потому что описания инструментов, схемы, идентичности серверов, OAuth-scopes и выводы инструментов становятся частью среды принятия решений модели.

    Для MCP я бы закрепил в code review такие правила:

    • проверять описания инструментов и схемы перед одобрением
    • предпочитать узкие credentials для каждого сервера
    • изолировать недоверенные MCP-серверы от чувствительных инструментов
    • отслеживать изменения определений инструментов после установки
    • считать вывод инструмента недоверенным вводом
    • логировать каждый сервер, инструмент, аргумент и результат

    Guardrails недостаточно

    Guardrails могут валидировать ввод и вывод. Они не решают вопросы наименьших привилегий, области credentials, sandboxing, poisoning инструментов или политики подтверждения. Сохраняйте их, но размещайте после проектирования возможностей и перед пользовательским выводом.
