    Автоматический перевод

    Эта статья была автоматически переведена с оригинальной английской версии.

    uv на macOS: управление версиями Python, проектами и инструментами

    Быстрое начало

    # Install uv
brew install uv

# For new projects (modern workflow)
uv init                # create project structure
uv add pandas numpy    # add dependencies
uv run train.py        # run your script

# For existing projects (legacy workflow)
uv venv                             # create virtual environment
uv pip install -r requirements.txt  # install dependencies
uv run train.py                     # run your script

# Run tools without installing them
uvx ruff check .       # run linter
uvx black .            # run formatter

# Run a single-file script with its own dependencies (no project needed)
uv run fetch.py        # uv reads the inline deps and runs it

    Кратко: Используйте uv в качестве стандартного менеджера проектов и инструментов для Python на macOS. Оставьте Homebrew для системных пакетов, устанавливайте версии Python через uv python, фиксация зависимостей в pyproject.toml, и используйте uvx для одноразовых инструментов командной строки.

    Почему использовать uv для Python на macOS?

    Если вы уже некоторое время работаете с Python, скорее всего, вы регулярно используете pip, virtualenv, pip-tools, pyenv, и poetry в зависимости от проекта. uv Он выполняет ту же функциональность, что и все остальные инструменты, но в виде одного бинарника.

    Инструмент написан на Rust, и на моем компьютере он устанавливает пакеты примерно в 10–100 раз быстрее, чем старый набор инструментов. Самым большим преимуществом для меня стало то, что мне больше не приходится переключаться между разными инструментами в процессе работы.

    Он заменяет его. pip и pip-tools для управления пакетами, pyenv для установки версий Python, virtualenv/venv для сред выполнения. pipx для запуска инструментов, и poetry или pdm для рабочих процессов проекта.

    Установка uv

    Самый простой способ установки uv В macOS это делается с помощью Homebrew:

    brew install uv

    uv Автоматически определяет архитектуру вашего Mac (Apple Silicon или Intel), поэтому дополнительная настройка не требуется.

    Чтобы поддерживать его в актуальном состоянии:

    brew upgrade uv
# OR
uv self update

    Основные концепции

    uv покрывает три сценария использования:

    1. Проекты — создание приложения или библиотеки с зависимостями.
    2. Скрипты — запуск однофайлового Python-скрипта с встроенными зависимостями.
    3. Инструменты — запуск утилит командной строки (например ruff или httpie) в глобальном масштабе.

    1. Управление проектами

    Для новых проектов, uv использует стандартный формат pyproject.toml для конфигурации и работы в кроссплатформенной среде uv.lock для воспроизводимых сборок.

    Начать новый проект:

    uv init my-project
cd my-project

    Это создаёт pyproject.toml, а .gitignore, и ещё один hello.pyДобавить зависимости:

    # Runtime dependencies
uv add pandas requests

# Development dependencies
uv add pytest ruff --dev

    Запустите свой код:

    uv run hello.py

    uv управляет виртуальной средой в .venv для вас. Вам не нужно вручную его активировать.

    2. Управление версиями Python

    uv устанавливает и управляет версиями Python от вашего имени, храня их в ~/.cache/uv. Если вы уже используете pyenv Для этого его можно просто убрать.

    Установка конкретной версии:

    uv python install 3.12

    Закрепите версию для вашего проекта:

    uv python pin 3.11

    Этот код выполняет запись. .python-version файл. В следующем uv run, он использует закреплённую версию и загружает её при необходимости. Ваша команда и система CI автоматически получают одну и ту же версию Python без какой-либо дополнительной координации.

    3. Запуск инструментов с uvx и uv tool install

    uvx (псевдоним для uv tool run) запускает инструменты командной строки Python без их добавления в глобальную среду или зависимости проекта.

    # Run a linter
uvx ruff check .

# Run a formatter
uvx black .

# Start a temporary Jupyter server
uvx --from jupyterlab jupyter lab

    Каждый инструмент запускается в собственной временной среде, поэтому не возникает конфликтов версий с тем, что уже установлено в вашем проекте.

    Два аспекта, к которым я постоянно обращаюсь:

    • Фиксируйте версию внутри текста с @: uvx ruff@0.6.0 check ., или принудительно использовать самую новую версию с uvx ruff@latest.
    • Имя команды не совпадает с именем пакета? Используйте --from. Пакет jupyterlab отправляет jupyter команда, поэтому это uvx --from jupyterlab jupyter lab. То же самое и для --from httpie http.
    • Требуется дополнительная зависимость для плагина? Добавьте её с --with: uvx --with mkdocs-material mkdocs serveЕсли вы используете тот или иной инструмент ежедневно, установите его один раз вместо того, чтобы каждый раз создавать новую среду.
    uv tool install ruff      # now `ruff` is on your PATH
uv tool list              # see what's installed
uv tool upgrade --all     # update everything
uv tool uninstall ruff

    Эмпирическое правило: uvx для одноразовых запусков или запусков в рамках CI, uv tool install для ваших повседневных задач.

    Проекты устаревшей версии (requirements.txt)

    requirements.txt У этого подхода был неплохой период использования. Речь идет о простом списке пакетов без файла lockfile, без разделения зависимостей приложения и зависимостей разработки, а также без какой-либо информации о причинах фиксации версий пакетов. Всё это работает нормально до тех пор, пока вы не попытаетесь воспроизвести сборку, созданную восемь месяцев назад, и не обнаружите, что под «фиксированной версией» на самом деле подразумевалась та версия, которую в тот момент случайно выбрал PyPI.

    Если вы являетесь владельцем проекта, выполните его миграцию. uv читает ваш существующий список непосредственно в pyproject.toml:

    uv init --bare                        # minimal pyproject.toml, no scaffolding
uv add -r requirements.txt            # import runtime dependencies
uv add --dev -r requirements-dev.txt  # and dev ones, if you split them

    Теперь у вас есть pyproject.toml и настоящий uv.lock что фиксирует именно те версии, которые были успешно разрешены. Подтвердите, что всё было получено целостно. uv pip freeze, удалить старый requirements.txtи не оглядывайтесь назад.

    Но, возможно, вы продолжаете запускать тот же самый requirements.txt С версии Python 3.6, и вас не интересует мое мнение по этому поводу. Ладно. uv по-прежнему может использоваться в качестве прямой замены pip и venv, миграция не требуется:

    # Create a virtual environment
uv venv

# Install dependencies
uv pip install -r requirements.txt

# Run it
uv run python app.py

    Те же самые команды, которые вы уже знаете, только быстрее. Старый способ работы ни в коей мере не ухудшается — просто теперь вам больше не нужно им пользоваться.

    Скрипты в одном файле с встроенными зависимостями

    Это именно та функция, которая изменила мой подход к написанию временного и «клеевого» кода. Скрипт на Python может указать собственные зависимости прямо внутри файла, в блоке комментариев, определённом PEP 723. Нет. pyproject.toml, нет requirements.txt, виртуальную среду создавать не нужно — uv run считывает блок, создаёт кэшированную среду и запускает файл.

    Стоит уточнить, что именно здесь выполняет работу: это не особенность языка Python, и это не… uv Изобретение. Этот блок представляет собой лишь комментарий, поэтому его можно оставить без изменений. python fetch.py игнорирует это и выдает ошибку из-за отсутствующих импортов. PEP 723 представляет собой общий стандарт, поэтому любой совместимый запускающий инструмент будет читать тот же блок кода. pipx runHatch и PDM также поддерживают его. uv это как раз самый быстрый способ его использования, и именно поэтому в остальной части этого раздела применяется uv run.

    Вот вся структура в одном файле. fetch.py:

    # /// script
# requires-python = ">=3.12"
# dependencies = [
#     "httpx",
#     "rich",
# ]
# ///
import httpx
from rich import print

print(httpx.get("https://api.github.com/repos/astral-sh/uv").json())

    Запустите его:

    uv run fetch.py

    Первая запуском процедура обеспечивает решение проблем и установку. httpx и rich загружается в кэшированную среду; при последующих запусках она используется повторно, что обеспечивает мгновенный старт.

    Вам не нужно вручную вводить блок метаданных. uv Я создам структуру и внесу необходимые изменения для вас:

    # Create a new script with the block pre-filled
uv init --script fetch.py --python 3.12

# Add or remove dependencies
uv add --script fetch.py httpx rich
uv remove --script fetch.py rich

    Для быстрого эксперимента можно даже полностью пропустить этот блок и передать зависимости через командную строку:

    uv run --with httpx --with rich fetch.py

    Почему это отлично подходит для скриптов, направленных на развитие навыков и автоматизацию

    Я использую его для небольших скриптов, которые не заслуживают статуса отдельного проекта: одноразовой коррекции данных, вспомогательных инструментов в CI и т. д. Навык Claude Code скрипт, задача cron. Сам скрипт и является единицей выполнения. Его можно разместить в сервисе Gist, сохранить в любом репозитории или передать коллеге. uv run это единственное, что необходимо для его корректной работы.

    Сделайте файл напрямую исполняемым с помощью строки запуска:

    #!/usr/bin/env -S uv run --script
# /// script
# dependencies = ["httpx"]
# ///
import httpx
...

    chmod +x fetch.py
./fetch.py        # uv handles the environment behind the scenes

    Этот -S разделяет строку с шебангом на отдельные аргументы, так что env проходит проверку --script до конца uv### Заблокировка и фиксация скриптов

    Для скриптов, которые должны продолжать работать в течение многих месяцев, существует два варианта.

    Заблокируйте точное решение рядом со скриптом:

    uv lock --script fetch.py   # writes fetch.py.lock

    Или задайте конкретную точку во времени uv учитывает только пакеты, выпущенные до указанной даты — удобно для воспроизводимости без использования файла lockfile:

    # /// script
# dependencies = ["httpx"]
# [tool.uv]
# exclude-newer = "2025-04-17T00:00:00Z"
# ///

    Советы и хитрости, которые стоит знать

    Ещё несколько инструментов, которые я регулярно использую и которые не очевидны при быстром старте.

    Синхронизация из файла lockfile в среде CI. uv sync устанавливает ровно то, что указано uv.lock. Добавить --frozen чтобы произошла явная ошибка при использовании устаревшего файла блокировки вместо тихого повторного получения данных — именно это и нужно при сборке в CI и в Docker.

    uv sync --frozen

    Группируйте свои зависимости для разработки. Помимо этого --dev, вы можете определить группы зависимостей с именами в pyproject.toml (docs, lint, test) и синхронизируйте только то, что вам нужно:

    uv add --group docs mkdocs-material
uv sync --only-group docs

    Проверьте дерево зависимостей. uv tree показывает, что было загружено и что именно — и --outdated Обновления флагов:

    uv tree
uv tree --outdated

    Экспорт в requirements.txt при том, что следующий инструмент всё ещё ожидает его наличия:

    uv export --format requirements-txt > requirements.txt

    Запустить одноразовую сессию Python с дополнительными пакетами, проект не требуется:

    uv run --with pandas --with matplotlib python

    Управление кэшем при ограниченном пространстве на диске или сбоях в процессе сборки:

    uv cache clean      # wipe the whole cache
uv cache prune       # remove only unused entries

    Почему он быстрый

    Здесь важны три фактора. Код написан на Rust, поэтому при каждом вызове отсутствует нагрузка, связанная с запуском интерпретатора Python. Также осуществляется глобальное кэширование готовых библиотек — один раз numpy если он установлен в одном проекте, то его установка в другом занимает практически мгновение (в macOS используются ссылки типа copy-on-write). Кроме того, он загружает и устанавливает пакеты параллельно, а не по одному.

    Краткое резюме

    Задача Старый способ Способ UV
    Установка Python pyenv install 3.12 uv python install 3.12
    Новый проект mkdir proj && cd proj && python -m venv .venv uv init proj
    Установка пакета pip install pandas && pip freeze > requirements.txt uv add pandas
    Запустить скрипт source .venv/bin/activate && python script.py uv run script.py
    Скрипт + зависимости проект или вручную созданная среда venv только для одного файла блок встроенного формата PEP 723 + uv run
    Запустить инструмент один раз pipx run black uvx black
    Установка инструмента pipx install ruff uv tool install ruff
    Воспроизводимый CI pip install -r requirements.txt uv sync --frozen

    Я переключил свой рабочий процесс на Python uv Сразу после выхода продукта я перешёл на него и больше не возвращался к старой версии. Если вы всё ещё используете устаревшую технологическую базу, попробуйте его в своём следующем проекте — именно так я его тестировал перед полным переходом.

    Основные выводы

    1. Использовать uv для версий Python, виртуальных сред, зависимостей проекта и одноразовых инструментов вместо использования нескольких менеджеров пакетов.
    2. Позвольте Homebrew заниматься системными пакетами, а не изоляцией Python-проектов.
    3. Сохраните изменения pyproject.toml и uv.lock Таким образом, настройка проекта становится воспроизводимой на новом устройстве.
    4. Рекомендуется uvx для эпизодического использования CLI-инструментов в тех случаях, когда не требуется загрязнять глобальную среду.
