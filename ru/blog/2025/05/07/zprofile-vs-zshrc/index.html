Файлы для запуска Zsh: ~/.zprofile против ~/.zshrc на macOS и Linux

Если ваш терминал работает медленно или переменные окружения загружаются не там, где ожидается, скорее всего, проблема связана с порядком запуска Zsh.

Разделение между ~/.zprofile и ~/.zshrc это одна из наиболее распространённых причин путаницы при переходе на Zsh, особенно в macOS, где стандартные настройки отличаются от тех, что используются в Linux.

Кратко

~/.zprofile Это предназначено для настройки среды. Он запускается один раз при входе, что в macOS означает один раз на каждой вкладке терминала. Поместите ваш PATH, EDITOR, и системы управления версиями вроде fnm или pyenv Вот.

~/.zshrc Это раздел для интерактивной настройки. Он запускается каждый раз при старте новой оболочки. Здесь следует хранить псевдонимы команд, темы подсказок и привязки клавиш.

Порядок запуска оболочки

Чтобы понимать, куда помещать различные элементы, необходимо знать момент загрузки файлов. В Zsh существует определённая иерархия.

Оболочка при входе и интерактивная оболочка

Оболочка при входе — это первая оболочка, которая открывается после аутентификации. В macOS каждая новая вкладка или окно терминала по умолчанию является оболочкой при входе. В Linux же открытие терминала обычно запускает необолочку при входе, а интерактивную оболочку, из-за чего и возникает большинство проблем при работе в нескольких операционных системах.

Интерактивная оболочка — это любая оболочка, в которой можно вводить команды.

Вот что на самом деле происходит при открытии терминала в macOS:

Порядок загрузки

  1. ~/.zshenv (необязательно). Выполняется в любом шелле, включая неконсольные скрипты. Не размещайте здесь вывод или сложную логику — это может нарушить работу скриптов, которые импортируют вашу конфигурацию. Используйте его только для переменных окружения, которые должны существовать везде, что для большинства пользователей встречается редко.
  2. ~/.zprofile. Работает только во время запуска оболочки для входа. Это этап настройки. ~/.zshrc. Запускается в интерактивных оболочках. Этап настройки под ваши нужды.
  3. ~/.zlogin (необязательно). Выполняется в самом конце процесса инициализации оболочки при входе.

Куда что попадает? ~/.zprofile: слой среды

Именно здесь настраиваются элементы, которые наследуются всеми остальными процессами — пути, переменные, менеджеры версий языка.

Что должно находиться здесь:

  • PATH модификации.
  • Переменные среды: EDITOR, LANG, GOPATH, JAVA_HOME.
  • Инициализация инструмента, влияющая на окружение: pyenv, rbenv, fnm, cargo. Эти значения необходимо вычислять только один раз. Если вы внесёте их в .zshrc, они пересчитываются каждый раз при открытии под-шелла или запуске скрипта, что тратит время и может привести к появлению дублирующихся записей в вашем PATH.
# ~/.zprofile

# 1. Set up your PATH
# Local bin first so your tools override system ones
export PATH="$HOME/.local/bin:/opt/homebrew/bin:$PATH"

# 2. Global variables
export EDITOR="nvim"
export VISUAL="nvim"
export LANG="en_US.UTF-8"

# 3. Version managers (the heavy ones)
# Doing this here keeps shell startup fast
eval "$(fnm env --use-on-cd)"
eval "$(pyenv init -)"

~/.zshrc: интерактивный слой

Именно здесь вы настраиваете оболочку, в которую фактически вводите команды.

Что должно находиться здесь:

  • Псевдонимы: alias g='git'.
  • Ваш запрос: Starship, Powerlevel10k, Pure.
  • Результаты: compinit.
  • Связи клавиш: bindkey.
  • Опции оболочки: setopt autocd, setopt histignorealldupsЭти параметры имеют значение только тогда, когда за клавиатурой находится человек. Скрипт, работающий в фоновом режиме, не требует ввода от пользователя и не использует алиасы Git.
# ~/.zshrc

# 1. Prompt
autoload -Uz promptinit && promptinit
prompt pure

# 2. Aliases
alias ll='ls -lah'
alias g='git'
alias gs='git status'

# 3. Shell options
setopt autocd              # cd by typing the directory name
setopt histignorealldups   # skip duplicate history entries
setopt share_history       # share history across tabs

# 4. Completions
autoload -Uz compinit && compinit

Почему не поместить всё в один файл?

Возможно, у вас возникает вопрос: если .zprofile сначала выполняется, почему бы не задать свои псевдонимы там и не пропустить этот шаг? .zshrc полностью?

Всё сводится к вопросу наследования против переопределения.

Переменные среды наследуются

Когда вы export EDITOR="vim" В родительской оболочке каждый дочерний процесс наследует её: подоболочки, скрипты, программы. Вы устанавливаете её один раз, и она распространяется по всей иерархии, вот почему .zprofile это подходящее место для export### Псевдонимы и функции не

Псевдонимы вроде alias g='git' Функции оболочки являются локальными для текущей оболочки. Они не передаются в дочерние оболочки.

Если вы определите псевдоним в .zprofile, он находится в вашей оболочке входа верхнего уровня. Как только вы введёте zsh Чтобы запустить подшелл или скрипт, такой псевдоним больше не существует. Чтобы псевдонимы были доступны везде, их необходимо переопределять в каждом новом интерактивном шелле. Именно это и происходит. .zshrc предназначено для../deploy.sh), он запускает новую неконсольную оболочку. Ему не нужен ваш промпт, ему не нужны ваши псевдонимы Git, и он определённо не собирается ждать oh-my-zsh для завершения загрузки. Сохранение интерактивной конфигурации в .zshrc это означает, что скрипты выполняются быстро и без ошибок, при этом не происходит проникновения ваших личных настроек.

Распространённые ошибки

Применение nvm или pyenv в .zshrc

Вы открываете новую вкладку терминала, и проходит два–три секунды, прежде чем вы можете что-либо ввести. Менеджеры версий обычно требуют длительной инициализации. .zshrc запускается каждый раз. Переместите их в ~/.zprofile и задержка исчезает.

Рост PATH

Ваш $PATH в итоге одни и те же каталоги перечисляются пять раз. Причина почти всегда заключается в export PATH="$HOME/bin:$PATH" находится в состоянии покоя .zshrc: при каждой перезагрузкеsource ~/.zshrc) или подшелл добавляет путь снова. Переместить PATH определения ~/.zprofile### Перезагрузка после внесения изменений

Изменения ~/.zprofile не применяется к вашей текущей оболочке, поскольку .zprofile Читается только при входе. Вы можете либо закрыть вкладку и открыть новую (это самый простой вариант), либо запустить source ~/.zprofile вручную. .zshrc, просто:

source ~/.zshrc

Конфигурация, работающая на macOS и Linux

Если вы переключаетесь между разными устройствами (macOS на работе, Linux дома, или наоборот), возникают определённые сложности. Терминалы Linux часто запускаются в режиме неавторизованных оболочек, из-за чего они пропускают ~/.zprofile полностью.

Обычным решением является указание источника .zprofile из .zshrc когда он ещё не загружен:

# ~/.zshrc

# On Linux/non-login shells, make sure the environment is set
if [[ -o interactive && ! -o login ]]; then
    [[ -f ~/.zprofile ]] && source ~/.zprofile
fi

# ... rest of your interactive config

Два правила охватывают большую часть этого случая. Переменные наследуются вниз по дереву процессов, поэтому export принадлежит к .zprofile. Псевдонимы и функции не наследуются, поэтому они должны находиться в .zshrc и их необходимо переопределять для каждого нового интерактивного шелла. Если новые вкладки работают медленно, причиной почти всегда является высокая нагрузка. pyenv init или nvm.sh находится в состоянии покоя .zshrc вместо того чтобы .zprofile.

Основные выводы

  1. Используйте .zprofile для настройки оболочки входа, такой как инициализация переменной PATH в Terminal под macOS.
  2. Используйте .zshrc для реализации интерактивного режима оболочки, такого как алиасы, настройка промпта, автодополнение и опции оболочки.
  3. Не дублируйте изменения в PATH в обоих файлах; дублирование затрудняет отладку порядка запуска.
  4. В случае сомнений размещайте код инициализации среды .zprofile и интерактивные удобства в .zshrc.
