    Сравнение современных движков обработки данных: Polars, DataFusion, Daft, Ray Data, Pandas и Spark

    На протяжении многих лет Pandas обрабатывал табличные данные в оперативной памяти, в то время как Apache Spark занимался распределённой обработкой. Такое разделение справлялось с задачами при наличии структурированных данных.

    Однако современные нагрузки уже не всегда являются структурированными. Пайплайны для обработки изображений, аудио и видео теперь сосуществуют с табличными, а появление инференса на GPU превращает процессы сбора мусора в JVM и ограничения GIL в Python из просто незначительных неудобств в серьёзные узкие места. Именно по этой причине получила широкое распространение новое поколение движков, построенных на Rust и Apache Arrow.

    За последний год я перевёл большинство своих однодольных пайплайнов на Polars и захотел выяснить, как он справляется по сравнению с остальными компонентами нового стека. Поэтому я провёл тестирование Polars, DataFusion, Daft, Ray Data и Spark на реальных наборах данных — данных о поездках такси в Нью-Йорке для табличных задач и изображениях из проекта Food-101 для мультимодальных задач.

    Весь код хранится в репозиторий engine-comparison-demo. Скопируйте проект и запустите тесты на собственном оборудовании — ваши результаты будут отличаться от моих, и именно в этом суть.

    Кратко: На тестах, ориентированных на маркетинговые цели, движки, написанные нативно на Rust, обеспечивают ускорение в 94 раза по сравнению с Pandas на одном узле. В реальных рабочих нагрузках показатели будут не такими высокими, но стабильными — Polars и DataFusion являются хорошими стандартными выборами. Для мультимодальных пайплайнов движки вроде Ray Data и Daft работают в 18 раз быстрее Spark, поскольку они эффективно используют GPU. Универсального победителя нет; оптимальный выбор зависит от типа данных, масштаба задачи и наличия GPU в процессе обработки.

    Типы рабочих нагрузок по обработке данных

    Движки специализируются на определённых задачах, поэтому первый вопрос, на который нужно ответить, — это тип рабочей нагрузки, с которой вы сталкиваетесь. Основное разделение происходит между структурированными данными и мультимодальными.

    Структурированные/табличные данные. Фильтрация, агрегация, объединение. Зависит от производительности CPU, обычно выполняется в памяти. Значительная часть этих операций осуществляется на распределённых кластерах, которые изначально не требовались для такой работы — примерно 90% запросов обрабатывают менее 1 ТБ данныхчто в наши дни может комфортно работать на одной машине.

    Мультимодальный ИИ. Изображения, аудио, видео. При выполнении прогнозов критически важна работа с GPU, но именно предварительная обработка на CPU становится основным барьером. Модели глубокого обучения потребляют данные быстрее, чем CPU успевают их декодировать, поэтому на стандартном g6.xlarge (4 CPU на каждую GPU); уровень использования GPU падает ниже 20%. Вся цепочка обработки в итоге ограничивается скоростью подачи данных в GPU.

    Новые движки ориентированы на решение обоих этих вопросов. Rust обеспечивает высокую скорость выполнения, библиотека Arrow позволяет осуществлять обмен данными без копирования между процессами, а технология потоковой обработки даёт возможность работать с наборами данных, превышающими объём оперативной памяти.

    Часть 1: Обработка на одном узле

    Проблема с задержками в Pandas — это не баг, а особенности самой архитектуры. Pandas активно загружает файлы в оперативную память, копирует промежуточные данные практически при каждой операции и использует только один поток из-за ограничений GIL. В данном контексте... тесты производительности PDS-H При коэффициенте масштабирования 10 библиотека Pandas требует примерно 365 секунд, чтобы завершить выполнение стандартного набора запросов. Сервис потоковой обработки Polars выполняет ту же работу за 3,89 секунды — что на 94 раза быстрее.

    Polars: обработка табличных данных

    Polars Он заменил Pandas в качестве стандартного инструмента для серьезной локальной обработки данных типа ETL во многих командах, с которыми я общался. Он написан на Rust и использует отложенную выполнимость: вместо того чтобы обрабатывать каждую строку по мере её поступления, он формирует план запроса, оптимизирует его (применение условий на этапе обработки, упрощение проекции данных, упрощение выборки столбцов) и затем выполняет операции на всех ядрах CPU в виде потоковых пакетов.

    Разрыв между ними увеличивается по мере роста объемов данных. В коэффициент масштабирования 100 (~100 ГБ), движок потоковой обработки Polars завершил работу за 23,94 секунды по сравнению с 152,27 секундами у его собственного внутреннего движка в памяти — что на в 6 раз быстрее при обработке данных, превышающих объем оперативной памяти. engine_comparison_examples.ipynb:

    import polars as pl

# scan_parquet reads only the schema — no data loaded yet
q = (
    pl.scan_parquet("yellow_tripdata_2024-01.parquet")
    .filter(
        (pl.col("trip_distance") > 5.0)
        & (pl.col("total_amount") > 30.0)
    )
    .group_by("payment_type")
    .agg(
        pl.col("total_amount").mean().alias("avg_fare"),
        pl.col("trip_distance").mean().alias("avg_distance"),
        pl.len().alias("trip_count"),
    )
)

# The entire plan is optimized and executed here, in parallel
result = q.collect()

    Разница с Pandas заключается не в API, а в архитектуре. Polars сначала формирует план обработки и оптимизирует весь поток данных до того, как начать работу с ними. filter это данные передаются в читатель формата Parquet, в результате чего пропускаются целые группы строк. С диска считываются только те столбцы, которые упомянуты в запросе. Pandas не способен к подобному подходу — каждая строка обрабатывается сразу после парсинга, и промежуточные копии оказываются повсюду.

    Apache DataFusion: Расширяемый движок запросов

    В то время как Polars представляет собой библиотеку, которую используют напрямую, DataFusion это движок, на основе которого строятся другие движки. Он обеспечивает работу InfluxDB 3.0, GreptimeDB, и Apple’s Ускоритель Comet Spark.

    В ноябре 2024 года DataFusion стала самый быстрый одноузловой движок для запросов к файлам Parquet в ClickBench, опережая DuckDB, chDB и ClickHouse на том же оборудовании. Позже команда Embucket провела TPC-H с коэффициентом масштабирования 1000 (~1 ТБ) на одном узле Кроме того, это важное значение, которое стоит иметь в виду каждый раз, когда кто-то говорит о необходимости кластера. engine_comparison_examples.ipynb:

    from datafusion import SessionContext

ctx = SessionContext()
ctx.register_parquet("taxi", "yellow_tripdata_2024-01.parquet")

# SQL executed directly against Parquet — no intermediate copies
df = ctx.sql("""
    SELECT payment_type,
           COUNT(*)           AS trip_count,
           AVG(trip_distance) AS avg_distance,
           AVG(total_amount)  AS avg_fare
    FROM taxi
    WHERE trip_distance > 5.0 AND total_amount > 30.0
    GROUP BY payment_type
    ORDER BY trip_count DESC
""")
result = df.to_pandas()

    Сильная сторона DataFusion — это её модульность. API расширений охватывают пользовательские каталоги, поставщиков таблиц, правила оптимизатора и планы выполнения, и именно поэтому они используются каждый раз, когда кто-то создаёт собственную платформу обработки данных или встраивает движок запросов в свой продукт.

    Daft: Мультимодальная обработка данных

    Глупость это та библиотека, которая серьёзно относится к мультимодальным данным. Изображения, аудио, видео, эмбеддинги — всё это реальные типы в DataFrame, а не непрозрачные байты. Pandas рассматривает изображение как массив байтов, Spark сериализует его через JVM, но Daft предоставляет нативные выражения на Rust для декодирования изображений, получения URL и формирования тензоров без необходимости использования циклов на Python. engine_comparison_examples.ipynb:

    import daft

df = daft.read_parquet("yellow_tripdata_2024-01.parquet")

result = (
    df.where(
        (daft.col("trip_distance") > 5.0)
        & (daft.col("total_amount") > 30.0)
    )
    .groupby("payment_type")
    .agg(
        daft.col("total_amount").mean().alias("avg_fare"),
        daft.col("trip_distance").mean().alias("avg_distance"),
        daft.col("trip_distance").count().alias("trip_count"),
    )
    .collect()
)

    Этот API покажется знакомым тем, кто работал с Pandas, однако под капотом используется та же самая стековая архитектура Rust + Arrow, что и в Polars и DataFusion. Преимущества Daft проявляются тогда, когда «строки» в вашем DataFrame представляют собой изображения, PDF-файлы или тензоры.

    При таком размере данных (около 660 МБ в формате Parquet) любой современный движок обрабатывает их с высокой скоростью. Цифра в 94 раза быстрее у Polars по сравнению с Pandas взята из теста PDS-H — синтетического аналитического критерия оценки. На моем наборе данных из 41 миллиона строк я действительно зафиксировал стабильное ускорение по сравнению с Pandas, но расстояние до показателя в 94 раза так и не было преодолено. DataFusion незначительно опережает остальные по общему времени обработки. Polars остается самым удобным в использовании инструментом и надежным универсальным решением.

    Часть 2: Работа с мультимодальными данными

    Таблицевые процессы ETL обычно сокращают объем данных: фильтрация, агрегация, запись меньшего объема информации, чем было считано. Пайплайны мультимодальных ИИ делают противоположное — один путь к документу может привести к созданию десятков фрагментов текста и векторов представлений.

    Spark обрабатывает изображения и аудио как бинарные данные. Для работы с ними на Python необходимо осуществить переход от JVM к Python через Py4J, выполнить обработку с помощью библиотек Pillow или OpenCV, а затем снова сериализовать результат. Этот цикл «запрос-ответ» в значительной степени определяет общее время выполнения задачи в реальных условиях.

    Выполнение в виде пайплайнов и использование GPU

    Spark параллелизует обработку разделов данных между ядрами процессора, однако его способ выполнения, основанный на этапах (блочная синхронная обработка), представляет собой серьезный недостаток. В рамках одной задачи каждый шаг (загрузка, декодирование, классификация) выполняется последовательно на одном ядре без возможности асинхронного параллелизма. К тому же каждая задача внутри определенного этапа должна быть завершена до начала следующего этапа. GPU остаются неиспользуемыми во время предварительной обработки на CPU, а CPU — во время выполнения вычислений на GPU. Каждый этап генерирует свой результат до начала следующего, что создает дополнительное давление на объем памяти.

    Выполнение с использованием конвейера — это альтернативный подход. Вместо последовательной обработки этапов движок совмещает операции ввода-вывода, вычислительные задачи на CPU и инференс на GPU, благодаря чему в любой момент времени ожидает только самый медленный компонент.

    Polars и DataFusion отсутствуют здесь, поскольку у них нет встроенных операций с изображениями — обработка изображений приходится выполнять с использованием последовательного Python, что делает сравнение несправедливым. Вот ключевые цифры: операции с изображениями, реализованные нативно на Rust в Daft, работают в 3,6 раза быстрее, чем Pandas + Pillow, а чистый Rust обеспечивает скорость в 4,2 раза выше, чем Pandas, на всем протяжении пайплайна.

    Часть 3: Распределённая обработка

    Когда одной машины становится недостаточно, необходимо определить способ распределения задач. Именно здесь начинают проявляться архитектурные различия между различными движками.

    Apache Spark

    Spark по-прежнему остается моим выбором для обработки табличных данных в масштабе петабайт при выполнении операций ETL, сопровождающихся сложными процессами сортировки и перемешивания данных. Механизмы обеспечения отказоустойчивости через прослеживаемость объектов RDD действительно эффективны, экосистема инструментов крайне обширна (Databricks, EMR), а операции объединения данных объемом в несколько терабайт уже давно являются стандартной практикой. Однако Spark теряет преимущества при работе с задачами искусственного интеллекта. Задания запускаются на исполнителях JVM, которые не поддерживают GPU, в результате чего графические процессоры остаются без дела и вынуждены ждать предварительной обработки на центральном процессоре. distributed_spark.ipynb:

    from pyspark.sql import SparkSession
from pyspark.sql import functions as F

spark = SparkSession.builder \
    .appName("TaxiETL") \
    .config("spark.sql.adaptive.enabled", "true") \
    .getOrCreate()

orders = spark.read.parquet("s3a://lake/taxi/*.parquet")
zones = spark.read.csv("s3a://lake/taxi_zones.csv", header=True)

# Spark excels at this: joining massive tables with a shuffle
result = (
    orders
    .filter(F.col("trip_distance") > 5.0)
    .join(zones, orders.PULocationID == zones.LocationID, "inner")
    .groupBy("Borough")
    .agg(
        F.sum("total_amount").alias("total_revenue"),
        F.avg("total_amount").alias("avg_fare"),
    )
    .orderBy(F.desc("total_revenue"))
)

    Ray Data: Использование GPU

    Данные Ray был разработан с учётом нагрузок, связанных с ИИ. Вместо барьеров этапов в Spark он использует модель потоковой обработки это обеспечивает постоянную загрузку GPU. Основной механизм, делающий эту функцию целесообразной, — это смешанное планирование ресурсов: вы можете указать, что один процесс требует «1 GPU и 4 CPU», а другой — только CPU, после чего Ray сам определяет оставшееся распределение.

    Компания Amazon сообщила о сокращении затрат более 120 миллионов долларов в год путем переноса определённых рабочих нагрузок из Spark в Ray. Результаты их пилотного тестирования показали на 91% лучшую эффективность затрат и возможность обработки в 13 раз большего объёма данных в час; в режиме производства была достигнута эффективность затрат на уровне 82% выше на каждый ГиБ входных данных в S3.

    Из distributed_ray.ipynb:

    import ray

ray.init()

ds = ray.data.read_images("s3://my-bucket/food101/")

class ImageClassifier:
    def __init__(self):
        import torch
        from torchvision.models import resnet18, ResNet18_Weights
        self.model = resnet18(weights=ResNet18_Weights.DEFAULT).cuda()
        self.model.eval()
        self.preprocess = ResNet18_Weights.DEFAULT.transforms()

    def __call__(self, batch):
        import torch
        tensors = torch.stack([
            self.preprocess(img) for img in batch["image"]
        ]).cuda()
        with torch.no_grad():
            preds = self.model(tensors)
        return {
            "prediction": preds.argmax(dim=1).cpu().numpy(),
            "confidence": preds.max(dim=1).values.cpu().numpy(),
        }

# ActorPoolStrategy creates persistent GPU workers
predictions = ds.map_batches(
    ImageClassifier,
    compute=ray.data.ActorPoolStrategy(size=4),
    num_gpus=1,
    batch_size=64,
)
predictions.write_parquet("s3://output/predictions/")

    Один важный момент, который стоит знать: по мере увеличения количества ядер CPU на одну GPU, производительность Ray Data растет, тогда как у других движков она достигает плато. тесты производительности Anyscale, переход с соотношения CPU к GPU 4:1 на 32:1 привёл к ускорению обработки изображений в 3 раза, поскольку процессоры наконец смогли справляться с нагрузкой, задаваемой графическим процессором.

    Daft (Distributed)

    Daft обеспечивает масштабируемость за счёт своей архитектуры Двигатель флотилии: по одному рабочему процессу Swordfish на узел, при этом слой Flotilla отвечает за планирование в масштабе кластера. Swordfish обрабатывает локальную экспекуцию кода на Rust и управляет вводом-выводом в потоках с использованием метода небольших партий, благодаря чему каждый узел остается занятым без необходимости ожидания следующей стадии обработки.

    В ходе тестирования четырех мультимодальных нагрузок система Daft в сочетании с Flotilla показала В 2–18 раз быстрее, чем Spark. На самом сложном задании (обнаружение объектов в видео с использованием YOLO11n) Spark потребовал более 3,5 часов, тогда как Daft завершил работу менее чем за 12 минут — в 18 раз быстрее, причем это связано в основном с тем, что Spark тратит время на сериализацию данных и обработку промежуточных этапов выполнения.

    Одно ограничение: Anyscale, компания-разработчик Ray, опубликовала их собственные тесты производительности Показано, как Ray Data сокращает или устраняет разрыв в производительности на инстансах с высокой нагрузкой CPU при ручной настройке. В настоящее время эти два подхода поочередно применяются каждый квартал. distributed_daft.ipynb:

    import daft
from daft import col

# Daft's Flotilla engine distributes work across the cluster
df = daft.read_parquet("s3://data-lake/pdf_metadata/*.parquet")

# Download PDFs — Daft parallelizes downloads in Rust
df = df.with_column("pdf_bytes", col("pdf_url").url.download())

# Define a GPU UDF for embedding generation
@daft.udf(return_dtype=daft.DataType.list(daft.DataType.float32()))
class TextEmbedder:
    def __init__(self):
        from sentence_transformers import SentenceTransformer
        self.model = SentenceTransformer("all-MiniLM-L6-v2", device="cuda")

    def __call__(self, text_col):
        texts = text_col.to_pylist()
        embeddings = self.model.encode(texts, batch_size=32)
        return [emb.tolist() for emb in embeddings]

# Daft schedules CPU downloads and GPU embeddings simultaneously
df = df.with_column("embedding", TextEmbedder(col("text")))
df.write_parquet("s3://output/embeddings/")

    Часть 4: Роль Rust и Arrow

    На фоне соревнований именно процесс конвергенции представляет собой более интересную историю. Polars, Daft, DataFusion и ядро Ray имеют общую основу: Apache Arrow для формата в памяти — Rust, а для обеспечения многопоточности — также Rust.

    Это не просто архитектурная изящность. Интерфейс Arrow PyCapsule (__arrow_c_stream__ протокол позволяет передавать данные между движками без необходимости их сериализации: выполняйте вычисления в DataFusion, передавайте результаты в Polars или PyArrow без каких-либо затрат; осуществляйте мультимодальную предобработку в Daft и сразу подавайте полученные данные формата Arrow в DataLoader PyTorch.

    from datafusion import SessionContext
import polars as pl

# Compute in DataFusion (Rust-native execution)
ctx = SessionContext()
ctx.register_parquet("events", "events.parquet")
df = ctx.sql("""
    SELECT user_id, COUNT(*) as event_count
    FROM events WHERE event_type = 'purchase'
    GROUP BY user_id HAVING COUNT(*) > 5
""")

# Zero-copy conversion to Arrow, then to Polars
arrow_table = df.to_arrow_table()
df_polars = pl.from_arrow(arrow_table)

# Continue analysis in Polars
result = df_polars.with_columns(
    pl.col("event_count").rank().alias("rank")
).sort("rank")

    Почему же Rust вместо JVM?

    1. Отсутствие пауз из-за сборки мусора. Механизмы владения и заимствования заменяют собой сборщик мусора, что исключает паузы типа «stop-the-world» и непредсказуемые скачки задержек. Ошибки OOM, возникающие в масштабных системах на JVM, в основном исчезают.

    2. Отсутствие дополнительных заголовков объектов. JVM добавляет 12–16 байт заголовка к каждому объекту. При работе с миллиардами небольших объектов это уже составляет значительную часть оперативной памяти. В Rust таких затрат нет.

    3. Безопасность потоков на этапе компиляции. Система типов Rust блокирует возникновение конкурентных ситуаций ещё до сборки бинарного файла, поэтому при многопоточной работе не возникает тех типов скрытых ошибок конкурентности, характерных для других языков.

    В сочетании с столбчатой структурой памяти, предоставляемой библиотекой Arrow, исчезает проблема бутылочного горлышка при сериализации, которая ранее потребляла до 30% циклов процессора в традиционных стеках JVM.

    Часть 5: Использование нескольких движков

    Честный ответ заключается в том, что не выбирают один движок — их комбинируют.

    В производственной среде это обычно реализуется с помощью реле. Чистые данные из хранилища обрабатываются и объединяются с помощью Spark (или Dask, если в команде используется только Python) в таблицы формата Parquet или Delta. Эти таблицы затем передаются в Ray Data или Daft для выполнения финальных операций — инференции на GPU, генерации эмбеддингов, мультимодальных преобразований — для которых Spark изначально не предназначался. Для спонтанных анализов и локальной разработки Polars и DuckDB позволяют получать мгновенную обратную связь без необходимости запуска кластера.

    Такая архитектура становится возможной благодаря открытым форматам: Parquet, Delta, Iceberg, Arrow. Каждый компонент этой стек-технологии поддерживает чтение и запись в этих форматах нативно, поэтому передача данных между этапами осуществляется просто через путь к файлу.

    Краткое описание сценариев использования движков

    Задача Рекомендуемый инструмент Компромисс
    Ад-хок-анализ на ноутбуке Polars / DuckDB Скорость > Масштаб
    Масштабные процессы ETL в SQL (петабайты) Apache Spark Надёжность > Скорость
    Масштабирование, реализованное нативно в Python Dask Простота важнее оптимизации
    Обучение больших языковых моделей / компьютерное зрение Ray Data Использование GPU > Функции ETL
    Загрузка мультимодальных данных Daft Опыт разработчика > Экосистема
    Разработка собственных движков запросов DataFusion Расширяемость > Встроенные функции

    Диагональное масштабирование

    Долгое время существовал только два варианта решения: масштабирование вверх (использование более мощной машины) или масштабирование в стороны (добавление дополнительных машин). Polars Cloud в процессе выполняется какой-то промежуточный шаг, который называется диагональным масштабированием: сначала происходит горизонтальное масштабирование при чтении данных из облачного хранилища для максимизации объёма ввода-вывода, а затем, после того как фильтрация и агрегация сокращают объём данных, система сводится к одному крупному узлу, полностью исключая процедуру распределённой перестановки данных.

    Парадоксально, но более крупная и дорогая инстанс может оказаться дешевле за одну задачу, поскольку она позволяет довести загрузку GPU до высокого уровня, в результате чего общее время выполнения сокращается быстрее, чем растёт стоимость за час.

    Основные выводы

    1. Не распространяйте данные, если это не необходимо. Polars и DataFusion способны обрабатывать до ~1 ТБ на одной машине, обеспечивая стабильное ускорение по сравнению с Pandas (в 94 раза на нагруженных аналитических тестах вроде PDS-H). Обращайтесь к кластеру только тогда, когда действительно достигнуты пределы одного узла, а не раньше.

    2. Ограничения Pandas связаны с архитектурой. Режим быстрой обработки, однопоточность, создание промежуточных копий — это осознанные дизайнерские решения, которые превращаются в узкие места при больших объёмах данных. Режим отложенной обработки с применением операций к предикатам и оптимизацией столбцов меняет ситуацию.

    3. Пайплайнная обработка эффективнее, чем разделение на этапы, для работы с GPU. В Spark этапы заставляют GPU оставаться неактивными во время предварительной обработки на CPU. Ray Data и Daft совмещают операции ввода-вывода, работы CPU и GPU, так что ни один ресурс не тратится в ожидании следующего этапа.

    4. Используйте каждый инструмент для его основных задач. Spark — для ETL на уровне петабайт, Polars и DuckDB — для спонтанных анализов, Ray Data — для пайплайнов обучения на GPU, Daft — для загрузки мультимодальных данных, DataFusion — для кастомных движков запросов.

    5. Rust + Arrow — это общая основа. Не случайно Polars, DataFusion, Daft и Ray все сводятся к этой технологии — она устраняет паузы из-за сборки мусора, нагрузку от сериализации между процессами и целый класс ошибок конкурентности.

    Если вы не уверены, с чего начать: начните с Polars на одной машине, затем переходите к Daft или Ray Data при использовании GPU. Spark подходит только тогда, когда у вас действительно есть петабайты данных.
