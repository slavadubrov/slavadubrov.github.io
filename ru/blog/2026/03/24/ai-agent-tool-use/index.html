    Использование инструментов AI-агентов в 2026 году: MCP, CLI, навыки и выполнение кода

    Часть 3 серии «Инженерия стека агентов»

    Часть 1 обработанные циклы рассуждения, Часть 2 Занятая память. В этой статье рассматривается использование инструментов в AI-агентах: как агенты взаимодействуют с внешним миром и почему проектирование инструментов имеет большее значение, чем осознают большинство команд.

    Ситуация с инструментарием изменилась в 2025–2026 годах. MCP стал фактическим стандартом для внешних сервисов, однако команды, работающие в продакшене, по-прежнему сталкиваются с его проблемами в области безопасности и дополнительной нагрузкой на токены. Другим подходом, который набирает популярность, являются агенты, которые пишут и выполняют код вместо вызова инструментов, определённых в формате JSON. Компания Anthropic зафиксировала сокращение использования токенов на 98,7% в типичном рабочем процессе, а в статье CodeAct сообщается о повышении скорости успешного выполнения задач на 20%.

    В данной статье проводится сравнение методов вызова инструментов через JSON, MCP, функции Skills, инструменты CLI и выполнение кода; затем это сравнение связывается с принципами проектирования интерфейса агента и компьютера (ACI), которые я применяю в своей работе. Агент аналитика рынка.

    Кратко: существует пять модальностей использования инструментов в ИИ-агентах, у каждой из которых есть свои преимущества. Практический набор инструментов на 2026 год: специализированные функции для глубокой экспертизы, CLI в качестве стандартного интерфейса, MCP для взаимодействия с внешними сервисами, выполнение кода для оркестрации многократных операций. Принципы проектирования ACI из проекта SWE-agent лежат в основе всех этих подходов.

    Пять способов использования инструментов ИИ-агентами

    В Часть 1, Я показал, как циклы рассуждений ИИ-агента определяют как происходит мышление. Модальности инструментов определяют как следует действовать. В настоящее время существует пять различных подходов, каждый из которых имеет свои компромиссы с точки зрения стоимости токенов, гибкости и уровня безопасности.

    1. Вызов инструментов через JSON — базовый подход

    Исходная схема работы: вы определяете схемы инструментов в формате JSON, большая языковая модель генерирует структурированные вызовы функций, а ваш код выполняет их. Этот подход хорошо известен и эффективно работает для небольших наборов инструментов.

    # Traditional tool definition — each tool consumes ~550-1,400 tokens (Apideck benchmark)
tools = [
    {
        "name": "get_stock_price",
        "description": "Get the current stock price for a ticker symbol",
        "input_schema": {
            "type": "object",
            "properties": {
                "ticker": {"type": "string", "description": "Stock ticker (e.g., NVDA)"}
            },
            "required": ["ticker"]
        }
    }
]

    С 5–10 инструментами всё в порядке. Проблема заключается в масштабируемости: определение каждого инструмента требует дополнительных ресурсов. 550–1 400 токенов. При использовании 20 инструментов вы тратите от 15 до 25 тысяч токенов ещё до того, как агент начнёт выполнять логические операции.

    2. MCP — универсальный стандарт (с оговорками)

    Протокол контекста модели является стандартом, на котором согласовались большинство поставщиков. В декабре 2025 года компания Anthropic передала его в Linux Foundation в рамках фонда Agentic AI Foundation (AAIF) совместно с OpenAI и Block; проект поддерживают Google, Microsoft и AWS в качестве платиновых членов. OpenAI добавила поддержку MCP в свой API Responses. В настоящее время существует более 10 000 активных серверов MCP, а количество ежемесячных загрузок SDK превышает 97 миллионов.

    MCP демонстрирует преимущества в таких сценариях, как интеграция SaaS-сервисов разных производителей (Figma, Notion, Salesforce), обеспечение корпоративного управления с возможностью отслеживания действий, работы с сервисами, не имеющими аналогов в формате CLI, а также в средах, требующих оркестрации через OAuth. Для этих случаев MCP представляет собой очевидный выбор.

    Однако реальная ситуация в производственной среде сложнее, чем показывают эти цифры.

    Первой проблемой является площадь уязвимостей. The Уязвимый проект MCP Отслеживает 50 уязвимостей в серверах MCP, 13 из которых отнесены к критической категории; обнаружены 32 исследователями безопасности. К классам атак относятся инъекции запросов, сбои валидации входных данных, пробелы в механизмах аутентификации и уязвимости сетевой безопасности. Первый реальный злонамеренный сервер MCP появилось в сентябре 2025 года: пакет с названием postmark-mcp этот механизм пересылал каждое отправляемое письмо по адресу злоумышленника через поле BCC, что повлияло на примерно 300–500 организаций до того, как ситуация была обнаружена. Компания Invariant Labs продемонстрировала Эти заражённые инструменты MCP могут вывозить данные даже в тех случаях, когда они никогда не вызываются. Для запуска атаки достаточно лишь того, чтобы модель прочитала метаданные данного инструмента. тесты производительности MCPTox Проведённые тесты 20 агентов больших языковых моделей против 45 реальных серверов MCP показали, что уровень успешности атак достигал 72,8%.

    Проблемой с точки зрения эксплуатации является накладные расходы, связанные с токенами. Одна команда, управляющая серверами MCP для GitHub, Slack и Sentry (всего около 40 инструментов), обнаружила 55 000 токенов определений схемы, вставляемых до того, как пользователь задаст какой-либо вопрос. Ещё один зарегистрированный случай 143 000 из 200 000 доступных токенов (72%) используется исключительно для определений инструментов.

    Меры по снижению рисков действительно эффективны, однако они лишь подтверждают наличие основной проблемы. У компании Anthropic... Инструмент поиска компонентов снижает нагрузку на схему на 85%.. Динамическая загрузка инструментов (3–5 соответствующих инструментов на задачу) позволяет достичь сокращения нагрузки на 96%. Это временные решения для протокола, который изначально не предусматривал учёт лимитов токенов.

    3. Навыки (SKILL.md) — экспертиза, а не выполнение операций

    К концу 2025 года была стандартизирована концепция навыков агентов в виде открытого формата (опубликован в октябре 2025 года и официально зарегистрирован как открытый стандарт в декабре 2025 года). Различие между этими понятиями важно: инструменты обеспечивают функциональные возможности (то, что могут делать агенты), тогда как навыки представляют собой экспертизу (знания агентов о способах выполнения сложных задач).

    Стандарт SKILL.md определяет навык как файл в формате Markdown с YAML-метаданными:

    ---
name: deploy
description: Deploy the application to production
argument-hint: "[environment]"
user-invocable: true
---
Deploy the application to the $0 environment (default: staging).
Steps:
1. Run the test suite
2. Build the production bundle
3. Deploy using the deploy script
4. Verify the deployment health check

    Навыки используют принцип постепенного отображения: при запуске загружаются только метаданные (около 100 токенов в формате YAML frontmatter), а полные инструкции — при активации навыка. Для сравнения, в MCP около 40 инструментов из типичных сервисов потребляют... ~55 000 токенов До начала любых рассуждений. По состоянию на март 2026 года кроссплатформенная поддержка включает Claude Code, OpenAI Codex CLI, Cursor, GitHub Copilot, Gemini CLI, Goose, Windsurf и Roo Code — это явный тренд. Виктор Дибия исследовал в своем анализе сдвига от использования инструментов к действиям агентов, основанным на коде.

    Когда преимущество имеют навыки: передача экспертизы в конкретной области, сложные многоэтапные процедуры, повторяющиеся рабочие процессы, такие как миграция баз данных или интеграция платежных систем, а также любые задачи, при которых агенту необходимо знать как, а не просто что.

    4. CLI/Bash — Возрождение Unix

    Оказалось, что терминал Unix — это самый эффективный с точки зрения количества токенов интерфейс для агентов. Расчеты однозначны: манифест инструмента CLI занимает около 100 токенов, тогда как соответствующие схемы сервера MCP потребляют 50 000+ токенов. Это в Разница 4–32×, измеренная с помощью Scalekit за 75 запусков тестов (32 раза для задач вроде определения языка репозитория и поиска лицензии).

    Модели являются нативными пользователями командной строки. Их данные обучения включают миллиарды интеракций с терминалом из Stack Overflow, репозиториев GitHub и документации. Они нативно понимают git, docker, kubectl, gh, curl, и jq без каких-либо определений схемы.

    Руководство Уго Эньиохи «Разработка инструментов командной строки, которые действительно хотят использовать ИИ-агенты» Закодировано восемь правил проектирования:

    1. Структурированный вывод является обязательным — необходима поддержка --json
    2. Коды выхода представляют собой элемент управления потоком — используйте разные коды для различных типов ошибок.
    3. Команды должны быть идемпотентными.
    4. Самодокументирующиеся. --help с реалистичными примерами
    5. Проектирование с учётом композиционности — --quiet для простых значений — поддержка stdin
    6. Предоставить --dry-run и --yes flags
    7. Поддержка инспекции версий
    8. Обработка аутентификации с помощью переменных окружения

    Компромиссы здесь реальны. CLI не обеспечивает типобезопасности MCP, встроенной оркестрации OAuth, возможностей поиска инструментов и ведения журнала операций. Схема, к которой, как я вижу, склоняются большинство команд, заключается в следующем: CLI используется по умолчанию для разработки и локальных операций, а MCP — для интеграции с внешними сервисами и реализации корпоративного управления.

    5. Выполнение кода — ключевой сдвиг

    Именно это изменение в инструментах агентов кажется мне наиболее значимым. Вместо того чтобы LLM генерировал структурированный JSON для последовательного вызова заранее определённых функций, агент пишет полный скрипт на Python или bash, который вызывает несколько инструментов, обрабатывает результаты с использованием циклов и условий, а затем возвращает в контекст модели только итоговые сводки.

    Anthropic сформулировал это более формально Программное вызов инструментов (PTC), теперь GA в API Claude. Академической основой является Статья CodeAct (Wang et al., ICML 2024), которые провели тестирование на 17 больших языковых моделях и обнаружили, что использование кодовых операций позволяет достигать на 20% более высоких показателей успешного выполнения задач и на 30% меньшего количества шагов по сравнению с альтернативными JSON-решениями.

    Доказательства из продакшена: Версел переписал свой агент d0 для преобразования текста в SQL путем удаления 80 % своих инструментов (с 15 до 2): ExecuteCommand и ExecuteSQL). Уровень успешности задач вырос с 80% до 100%, время выполнения сократилось в 3,5 раза, а расход токенов — на 40%. Их формулировка: «Лучшими агентами могут оказаться те, у которых наименьше инструментов».

    • Cloudflare разработал функцию «Code Mode». позволяет агентам писать код на TypeScript для вызова своего API вместо определения схем инструментов, что снижает нагрузку, связанную с обработкой контекста. Их аргументация такова: «В наборах данных, используемых для обучения больших языковых моделей, имеется огромное количество реальных примеров кода на TypeScript, однако представлено лишь небольшое количество искусственно созданных примеров вызовов инструментов». Документация Anthropic по PTC. Традиционные инструменты для анализа расходов требуют более 20 отдельных проходов инференса — по одному на каждого сотрудника — причем вся промежуточная информация передается через контекст. members = get_team_members("engineering")
over_budget = []
for m in members:

    Интерфейс агента и компьютера (ACI) для инструментов ИИ-агентов

    Термин «Интерфейс агента и компьютера» (ACI) был введён Джоном Янгом, Карлосом Э. Хименесом и их коллегами из Принстона в их Статья о SWE-agent (NeurIPS 2024). Идея довольно проста: так же, как люди получают выгоду от хорошо спроектированных интерфейсов (HCI), агенты на основе языковых моделей представляют собой «новую категорию конечных пользователей с собственными потребностями и возможностями, которые также могут извлечь пользу из специально разработанных интерфейсов». результаты абляции Этот результат был получен в результате создания резервной копии. Агент SWE-agent с полноценной поддержкой ACI показал эффективность 18,0% на тестовой платформе SWE-bench Lite, тогда как при использовании только стандартного оболочки Linux этот показатель составил 7,3%. Таким образом, только за счёт оптимизации интерфейса было достигнуто улучшение на 10,7 процентных пунктов. Одни лишь механизмы проверки кода обеспечили дополнительное улучшение на 3 процентных пункта: 51,7% всех изменений, внесённых агентом, содержали хотя бы одну ошибку, выявленную инструментом проверки до того, как она смогла распространиться дальше.

    Anthropic взяла ACI в качестве основополагающего концепта в своей «Создание эффективных агентов» Руководство, выделяющее это как один из трех основных принципов: «Тщательно проектируйте интерфейс агента и компьютера с помощью подробной документации инструментов и их тестирования». Практические рекомендации: «Один из эмпирических правил заключается в том, чтобы учитывать объем усилий, затрачиваемых на интерфейсы человека и компьютера, и планировать вложить столько же усилий в создание качественных интерфейсов агента и компьютера».

    Четыре принципа на практике

    1. Действия должны быть простыми и понятными. Самая распространённая ошибка — однозначное соответствие API-эндпоинтов друг другу. Вместо того чтобы list_users, list_events, create_event, реализовать schedule_event который определяет наличие ресурсов и планирует задачи в одном вызове. Вместо того чтобы read_logs, реализовать search_logs который возвращает только соответствующие строки вместе с контекстом.

    2. Действия должны быть компактными и эффективными. Объединяйте важные операции в как можно меньшее количество действий. В данном случае Агент аналитика рынка, Я объединяю получение цен с базовыми метриками в единый get_stock_snapshot инструмент, позволяющий получить всю необходимую информацию за один вызов, вместо того чтобы выполнять отдельные запросы для получения данных о цене, объеме торгов, рыночной капитализации и коэффициенте P/E.

    3. Отчеты об состоянии среды должны быть информативными, но лаконичными. Избегайте возвращения необработанного HTML или полных данных из API. Заменяйте туманные идентификаторы на понятные семантические названия. тестирование в Anthropic показало, что добавление response_format enum, позволяющий агентам запрашивать краткие ответы (около 72 токенов) или подробные (около 206 токенов), при этом разница в стоимости обработки составляет в 3 раза, значительно улучшила производительность в реальных условиях.

    4. Механизмы контроля должны снижать распространение ошибок. Автоматическое обнаружение ошибок помогает агентам быстро выявлять и исправлять недочеты. В SWE-agentспециальный редактор файлов с встроенной проверкой синтаксиса автоматически отклоняет ошибки синтаксиса, выявляя 51,7% ошибок, возникающих при редактировании агентом, ещё до того, как они успевают усугубиться. Я применяю тот же принцип в агенте Market Analyst, проверяя аргументы инструментов с помощью схем Pydantic перед их выполнением:

    from pydantic import BaseModel, Field, field_validator

class StockQuery(BaseModel):
    """Validated input for stock queries.

    Pydantic catches malformed tickers before the API call,
    preventing error propagation through the reasoning loop.
    """
    ticker: str = Field(description="Stock ticker symbol (e.g., NVDA)")
    period: str = Field(default="1mo", description="Time period: 1d, 5d, 1mo, 3mo, 1y")

    @field_validator("ticker")
    @classmethod
    def validate_ticker(cls, v: str) -> str:
        v = v.upper().strip()
        if not v.isalpha() or len(v) > 5:
            raise ValueError(f"Invalid ticker format: {v}")
        return v

    @field_validator("period")
    @classmethod
    def validate_period(cls, v: str) -> str:
        valid = {"1d", "5d", "1mo", "3mo", "6mo", "1y", "5y"}
        if v not in valid:
            raise ValueError(f"Invalid period: {v}. Must be one of {valid}")
        return v

    Возвращать высококачественный, машинопонимаемый результат

    Избегайте использования низкоуровневых идентификаторовuuid, mime_type). Преобразуйте загадочные идентификаторы в семантические названия. Структурируйте ответ таким образом, чтобы агент мог логически работать с ним, не разбирая шаблонный текст:

    # Bad: raw API response dumped to agent
def get_stock_price(ticker: str) -> dict:
    response = api.get(f"/v1/quotes/{ticker}")
    return response.json()  # 500+ tokens of nested JSON

# Good: high-signal summary the agent can immediately reason about
def get_stock_price(ticker: str) -> dict:
    data = api.get(f"/v1/quotes/{ticker}").json()
    return {
        "ticker": ticker,
        "price": data["regularMarketPrice"],
        "change_pct": round(data["regularMarketChangePercent"], 2),
        "volume": data["regularMarketVolume"],
        "market_cap_b": round(data["marketCap"] / 1e9, 1),
        "pe_ratio": data.get("trailingPE"),
        "summary": f"{ticker} at ${data['regularMarketPrice']:.2f} "
                   f"({'up' if data['regularMarketChangePercent'] > 0 else 'down'} "
                   f"{abs(data['regularMarketChangePercent']):.1f}%)"
    }

    Проектирование обработки ошибок для повышения устойчивости

    В производственных средах наиболее эффективной оказалась модель четырехуровневой толерантности к сбоям:

    1. Повторная попытка с экспоненциальным откладыванием времени для временных ошибок
    2. Использование цепочек резервных решений в случае отказа поставщиков услуг
    3. Маршрутизация на основе классификации ошибок — временные ошибки обрабатываются путем повторной попытки, ошибки, которые можно исправить с помощью LLM, возвращаются агенту вместе с контекстом, а ошибки, требующие вмешательства человека, передаются на более высокий уровень обработки
    4. Восстановление с использованием точек контроля для выживания при сбоях системы

    Как отмечено в документации, благодаря этому четырехуровневому подходу команды сообщают о необратимых сбоях в количестве, снизившемся с 23% до менее 2% Шаблоны устойчивости инструментов Anthropic.

    from tenacity import retry, stop_after_attempt, wait_exponential

@retry(
    stop=stop_after_attempt(3),
    wait=wait_exponential(multiplier=1, min=2, max=10),
)
def call_stock_api(ticker: str) -> dict:
    """Fetch stock data with automatic retry on transient failures.

    Layer 1: Exponential backoff handles rate limits and network blips.
    If all retries fail, the error propagates to the agent with
    enough context to decide whether to try a different approach.
    """
    response = httpx.get(
        f"https://api.example.com/v1/quotes/{ticker}",

    response.raise_for_status()

    Компромиссы и аспекты, которые следует учитывать

    Помимо вышеупомянутых особенностей, связанных с конкретными форматами данных, выбор оптимального подхода определяется ещё рядом общих факторов:

    • Рабочие затраты зависят от различных параметров. Выполнение кода позволяет сэкономить токены, но увеличивает время запуска в среде изоляции. Механизм MCP сокращает время разработки интеграций с SaaS-сервисами, однако добавляет нагрузку на развертывание серверов. Интерфейс CLI позволяет начать работу бесплатно, но его сложнее контролировать в масштабах. Оптимизируйте решение с учётом вашего реального узкого места — будь то стоимость токенов, задержки или операционная сложность.

    • Навыки команды играют ключевую роль. Для выполнения кода необходимо, чтобы ваши агенты (и модели, стоящие за ними) могли генерировать надёжный код на Python или TypeScript. Интерфейс CLI требует знания конвенций Unix. Механизм MCP предполагает понимание протоколов передачи данных и процессов авторизации OAuth. Подбирайте формат данных в соответствии с сильными сторонами вашей команды.

    • Консолидация инструментов может привести к чрезмерной сложности. Если объединить всё в один гигантский инструмент, пространство параметров станет слишком сложным для работы агента. Оптимальным вариантом для большинства систем агентов является наличие 5–10 хорошо специализированных инструментов.

    • Навыки реализуются через промпты, а не являются обязательными правилами. Навык представляет собой инструкции, которым агент должен следовать, но не жёсткие ограничения, которые он обязан соблюдать. Для критически важных рабочих процессов сочетайте использование навыков с методами детерминистичной верификации.

    • Требования к аудиту определяют выбор инструментов. Если вам необходим полный журнал каждого вызова инструмента и его результата, MCP и механизмы вызова инструментов в формате JSON обеспечивают структурированные записи аудита из коробки. Выполнение кода генерирует скрипт и его вывод, что полезно, но затрудняет разбиение процесса на отдельные действия для целей составления отчетов о соответствии требованиям.

    Как развивается эргономика инструментов

    Стоит отслеживать три основных направления.

    Первое — это использование технологий RAG для масштабирования. По мере увеличения количества наборов инструментов до сотен или тысяч точность простого выбора подходящего инструмента снижается до 13,62%. The Статья RAG-MCP Было показано, что применение метода генерации с усилением на основе поиска к процессу выбора инструментов (индексация описаний инструментов в векторной базе данных и извлечение только релевантных инструментов для каждого запроса) обеспечивает точность 43,13%, что на 3,2 раза выше исходных показателей, при этом количество токенов в промпте сокращается примерно на 50%.

    Второй подход заключается в использовании агентов для создания собственных инструментов. фреймворк LATM («LLMs As Tool Makers») сформулировал парадигму в два этапа, при которой мощный LLM генерирует повторно используемые функции на Python, а более легкий LLM задействует их в своей работе. ToolMaker (ACL 2025) автономно преобразует репозитории GitHub в инструменты, совместимые с LLM, с коэффициентом успеха 80%. Границы развития смещаются от использования инструментов к их созданию, а затем — к управлению библиотеками инструментов.

    Третьим является стек с двойными протоколами A2A + MCP. Протокол Agent2Agent от Google (A2A) решает те задачи, которые невозможно решить с помощью MCP: коммуникацию между агентами. MCP отвечает за интеграцию агента с инструментами, тогда как A2A занимается обнаружением агентов, ведением переговоров и делегированием задач. Совокупный подход заключается в следующем: разрабатывайте приложение с использованием вашей фреймворковой среды, оснащайте его инструментами MCP и обменивайтесь данными через протокол A2A.

    1. Рассматривайте описания инструментов как элементы инжиниринга промптов. Примеры входных данных значительно повышают точность работы системы. Описания, оптимизированные для Claude, превосходят по эффективности те, что написаны экспертами вручную. Вкладывайте столько же усилий в пользовательский интерфейс инструментов, сколько и в интерфейсы, предназначенные для людей.

    2. Практическая стек-архитектура 2026 года включает компоненты Skills + CLI + MCP + выполнение кода, расположенные слоями в зависимости от сценария использования: Skills — для обеспечения экспертизы, CLI — для локальных операций, MCP — для взаимодействия с внешними сервисами, выполнение кода — для организации многоэтапных процессов.
