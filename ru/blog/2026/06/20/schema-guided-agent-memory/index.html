ИИ-агент Память: типизированное состояние с управлением по схеме для долгоживущих систем

Агенты с длительным временем выполнения часто получают устаревшие данные, поскольку обычные семантическая память не содержат правил для определения того, какое значение считается актуальным.

Когда пользователь изменяет крайний срок действия паспорта с 15 июля на 30 июня, vector search может получить оба значения. Слой памяти с поддержкой состояний обязан фиксировать, что значение от 30 июня имеет приоритет перед более ранним.

Кратко: рассматривайте память долговечного агента как типизированное состояние приложения. Извлекайте кандидаты в элементы памяти через границу структурированного вывода, храните записи с учётом контекста абонента, окон действительности, механизма замены, информации о происхождении и версионирования схемы, а затем на этапе чтения извлекайте самый актуальный текущий фрагмент данных. Зарезервируйте vector search для методов нечёткого поиска. Читайте изменяемые данные из ограниченных по контексту актуальных записей.

Ловушка контекстного окна

контекстное окно представляет собой входные данные, доступные при одном вызове модель. Приложение может передавать сообщения в последующие вызовы, однако правила работы приложения должны определять, какие старые факты остаются актуальными и кто имеет право на их просмотр.

Для агентов с длительным временем работы необходимо сохранять предпочтения пользователя, статус задач, информацию о клиентах, решения, принятые с использованием инструментов, записи, касающиеся соответствия требованиям, а также прошлые ошибки. Простой способ — добавлять краткие сводки или сохранять старые записи в векторном хранилище. Такой подход работает до тех пор, пока какой-либо из сохранённых фактов не изменится.

В настоящее время у агента могут существовать два срока истечения действия паспортов, два предпочтительных формата или два решения, касающихся проекта. Semantic search способен извлекать данные по обоим вариантам. Краткое резюме может перезаписать информацию о одном из них. В расширенном контексте устаревшая версия данных может сохраняться рядом с актуальной. Такая архитектура позволяет восстанавливать предыдущие значения, не налагая при этом обязательства соблюдать текущие.

Договор должен содержать ответы на конкретные вопросы:

  • Что считается истинным в настоящий момент?
  • Что считалось истинным 2 июня?
  • Кто это сказал?
  • Какому тенанту принадлежит эта информация?
  • Какой более старый факт был заменён этим?
  • Могу ли я удалить или истечь срок действия этой информации?

Память агента с управлением по схеме (Schema-Guided Agent Memory, SGAM) хранит такие ответы в виде полей и связей, вместо того чтобы оставлять их неявно в простом тексте.

Что означает SGAM

Три идеи с похожими названиями определяют объём применения SGAM.

Диалог с управлением схемой (SGD) — это Ориентированный на задачи диалог Google 2019 датасет. В его схеме описываются сервис APIs, намерения и слоты, что позволяет диалогу модель отслеживать состояние сервисов, ранее не встречавшихся у него. Это полезный пример использования схем для отслеживания, причём его диапазон применения ограничен диалоговыми сервисами.

Schema-Guided Memory (SGM) — это исследовательский термин, введённый Мэй и соавторами в По моему мнению: система качественной проверки на основе долгосрочной персонализированной референтной памяти.. В данной статье осуществляется сравнение текстовой формы Descriptive Memory (DM) с элементами памяти типа «ключ-значение» с фиксированной схемой. Обе формы представления содержат одинаковую исходную информацию, но в разных структурах.

В данной статье термин Schema-Guided Agent Memory (SGAM) означает инженерный паттерн, при котором схемы управляют процессами записи, обновления, retrieval и удаления данных. Именно схема определяет состояние приложения и его жизненный цикл.

ATM-Bench

Вопрос о хранении данных между форматами SGM и DM заключается в следующем: следует ли сохранять информацию в виде свободного текста или использовать структурированные поля с именами? Однако у агента в режиме производства существует ещё одна проблема до момента сохранения данных — ему необходимо на основе неструктурированного диалога сформулировать предложение по обновлению памяти. Руководство схемой Ризонинг (SGR) Этот алгоритм определяет последовательность действий: сначала анализируются имеющиеся доказательства, выявляются субъект и атрибут, проверяется, изменяет ли факт текущее состояние, после чего формируется кандидат на запись. После этого вызова модель SGAM применяет правила управления хранилищем и жизненного цикла данных.

Отдельный процесс извлечения модель и управление владением памятью

Запись в память проходит через три уровня. Механизм Structured Output (SO) обеспечивает соблюдение формы кандидатского объекта. Метод Schema-Guided Ризонинг (SGR) кодирует последовательность шагов, которые модель должен выполнить для достижения данного кандидата. Система Schema-Guided Agent Memory (SGAM) отвечает за сохранение кандидата в качестве стабильного состояния после вызова модель.

Решение, маршрут или план обычно устаревают вместе с текущим запросом. Повторная обработка может ознакомиться с кандидатом в памяти спустя несколько дней или использовать его для выбора tool call. Такой более длительный срок службы требует правил хранения, которые SGR не предоставляет.

SGR ограничивает один вызов модель путем задания его ризонинг топологии. При записи в память схема может требовать наличия исходных данных, нормализованных значений для субъекта и атрибутов, сравнения с текущим состоянием, прежде чем будет применено предложенное обновление. Инструменты вроде Pydantic или JSON Schema описывают этот алгоритм. Реализации, основанные непосредственно на провайдере, Structured Output, или механизмы управляемой декодировки рантайм, такие как XGrammar, не позволяют модель пропускать поля или возвращать данные в ином формате.

Схема не может гарантировать получение корректного результата. Однако она делает необходимый путь принятия решения явным и доступным для анализа, включая доказательства и сравнения, на основе которых был выбран кандидат.

SGAM определяет, что происходит после того, как объект существует. Необходимо ли его хранение? Заменяет ли он более старый факт? Какие арендаторы могут его видеть? Является ли он актуальным или историческим? Какая эпизод из исходных данных его подтверждает?

Пути записи и чтения выполняют разные задачи

SGAM имеет путь записи и путь чтения. Только путь записи изменяет хранящееся состояние. Путь чтения отбирает записи для текущего запроса.

Поток входных данных представляет собой путь записи:

  1. Записывать необработанный эпизод из сообщений, результатов работы инструментов или бизнес-событий.
  2. Извлекать варианты ввода с помощью structured output.
  3. Проверять соответствие шаблону и отклонять некорректные записи.
  4. Урегулировывать конфликты, закрывать устаревшие данные и сохранять информацию о происхождении.
  5. Загружать запись в хранилище SGAM.

Поток запросов представляет собой путь чтения:

  1. Начните с вопроса пользователя.
  2. Определите, требуется ли информация о текущем состоянии или о состоянии в определённый момент времени.
  3. Примените фильтрацию по арендатору, типу памяти, теме, атрибуту и окну действительности.
  4. Добавляйте векторное или графовое расширение только в том случае, если поиска по точному состоянию недостаточно.
  5. Соберите минимально необходимый набор контекстной информации для модель.

Читайте эту диаграмму слева направо, разделив её на два канала. В верхнем канале происходит запись в память, а в нижнем — её чтение. Для обоих процессов используется один и тот же механизм хранения.

Тот же самый таблицевой формат позволяет обрабатывать как текущие запросы, так и запросы к конкретному моменту времени, после чего трейс результат возвращается к соответствующей исходной записи. schema_version поддерживает миграции, при этом retention_policy указывает задачам на удаление, что ещё необходимо устранить.

Используйте RAG для загрузки документов, а SGAM — для поддержания изменяемого состояния. Vector search по‑прежнему присутствует в системе для реализации методов размытого поиска, кластеризации и расширения результатов. Текущее значение mira.passport_deadline Данные должны поступать из записи ограниченной памяти, а не из того элемента чанк, который случайно занял первое место в рейтинге.

Пример устаревшего факта

Рассмотрим синтетический двухэпизодный трейс, представленный в виде базовой модели DM и реестра SGAM:

e1: Mira prefers concise answers. Her passport deadline is 2026-07-15.
e2: Mira corrected the deadline. It is now 2026-06-30.

Базовая конфигурация DM сохраняет оба эпизода в виде обычного текста, что позволяет инструментам поиска по тексту возвращать соответствующие результаты. e1 поскольку в нём содержатся соответствующие термины. SGAM извлекает введённый факт из каждой эпизодической записи, привязанный к арендатору, субъекту и атрибуту. Он должен возвращать e2 в качестве текущего состояния и сохранить e1 для выполнения исторического поиска.

Приведённая ниже функция представляет собой код пути записи SGAM. Она будет находиться в модуле хранения, таком как sgam_store.py. У объекта DM нет аналога, поскольку он не сохраняет отдельную текущую строку для каждого атрибута. Перед тем как вызывающий код вставит замену, эта функция закрывает текущую строку:

def close_previous_fact(db: sqlite3.Connection, fact: MemoryFact) -> int | None:
    row = db.execute(
        """
        select fact_id
        from memory_facts
        where tenant_id = ?
          and subject = ?
          and attribute = ?
          and valid_to is null
        order by valid_from desc
        limit 1
        """,
        (fact.tenant_id, fact.subject, fact.attribute),
    ).fetchone()
    if not row:
        return None

    db.execute(
        "update memory_facts set valid_to = ? where fact_id = ?",
        (fact.valid_from, row["fact_id"]),
    )
    return int(row["fact_id"])

Когда e2 прибывает — функция устанавливает valid_to на основе извлечённого факта e1 для e2временная метка. Затем вызывающий код вставляет новый факт с открытым состоянием valid_to. У механизма DM отсутствует соответствующий шаг обновления, поэтому старый текст может по-прежнему иметь более высокий ранг, чем исправленная версия.

Выполнение запросов к текущим и историческим дедлайнам в отношении обеих репрезентаций приводит к разным результатам:

Naive text memory:
  returned episode: e1 -> passport deadline is 2026-07-15

SGAM current state:
  mira.passport_deadline = 2026-06-30
  valid_from=2026-06-03T10:00:00Z, source=e2

SGAM point-in-time state:
  on 2026-06-02, mira.passport_deadline = 2026-07-15

В режиме производства необходимо сопоставить эту транзакцию с процедурой структурированной экстракции на этапе записи данных. Транзакция базы данных обновляет временные параметры действительности. Механизм модель извлекает кандидата на факт, но не определяет, какая сохранённая строка остаётся актуальной.

Сформируйте путь записи до построения графа

Сначала необходимо определить, что именно продукт имеет право запоминать. Вопрос выбора между графами и векторами рассматривается позже.

Агент поддержки может запоминать уровень аккаунта, открытые заявки и настройки контактов. Он не должен автоматически переводить каждого недовольного пользователя в состояние профиля. Агент по кодированию может запоминать конвенции репозитория и нерешённые задачи. Он не должен сохранять личную запись вечно только потому, что она была извлечена один раз.

Начните с пути записи и рассматривайте память как небольшую мутацию состояния:

  1. Укажите тип памяти, субъект, диапазон тенантов и класс хранения данных.
  2. Извлеките кандидатские записи с помощью structured output.
  3. Проверьте содержимое записи с использованием Pydantic или слоя схем, уже применяемого в вашей архитектуре.
  4. Устраните конфликты перед вставкой записи, включая определение того, заменяет ли новая запись старую.
  5. Сохраняйте указатель на исходный источник — это может быть оригинальная эпизодическая запись, результат работы инструмента, файл, тикет или подтверждение от пользователя, которые легли в основу создания записи.
  6. Записывайте версию схемы вместе с каждой записью, а не ограничиваться указанием её только в коде приложения.

Первый хранилище типа SGAM может представлять собой реляционную таблицу с столбцом JSON и несколькими индексами. Графическая структура становится полезной в тех случаях, когда необходимо обрабатывать связи между объектами, такие как связь «клиент — счёт», «счёт — политика», «задача — результат» или «проект — решение».

Работа с «горячими» путями и фоновыми записями

Прямая экстракция данных оправдана в тех случаях, когда следующий этап обработки зависит от только что полученной информации. Если пользователь указывает: «Запомните, что я предпочитаю краткие ответы», системе не требуется выполнение еженощных задач, чтобы изменить своё поведение.

В большинстве случаев нет необходимости в немедленной записи данных. Достаточно сохранить необработанный эпизод вместе с информацией о арендаторе, сессия, и метаданными инструмента, после чего фоновый работник сможет извлечь подходящие кандидатуры позже. Благодаря методу консолидации на основе рекуррентности работник временно хранит слабые сигналы и утверждает тот или иной факт лишь после повторения аналогичных доказательств или подтверждения пользователя. Это приводит к задержке обновления информации. Такой подход приемлем в ситуациях, когда «пользователь часто запрашивает экспорт в формате CSV», но представляет риск в случаях, когда «клиент изменил адрес доставки».

Сохраняйте детерминированный путь чтения. В первую очередь соблюдайте ограничения по тенантам и срокам действия данных, применяя алгоритмы размытого поиска retrieval лишь в тех случаях, когда это позволяет добавить полезную контекстуальную информацию.

  1. Фильтрация по арендатору, типу памяти и окну действительности.
  2. Получение точного структурированного состояния в первую очередь, затем семантических аналогов.
  3. Использование векторного или графового расширения для получения подтверждающих данных, связанных сущностей и примеров; это не должно рассматриваться как источник достоверности текущих фактов.
  4. Сбор минимального возможного контекста, на основе которого можно ответить на вопрос.

Рассматривайте миграцию схемы как изменение продукта, поскольку она влияет на то, что агент может вспоминать, цитировать или удалять. Кроме того, она может изменить список исторических фактов, которые считаются актуальными в текущий момент. Планируйте скрипты миграции, процедуры восстановления данных, периоды одновременного чтения и логику удаления в рамках одного и того же релиза.

Когда SGAM оправдывает добавление сложности

Используйте SGAM в тех случаях, когда факты могут изменяться со временем:

  • предпочтения пользователя, которые можно обновлять или отзывать
  • сведения о клиенте или аккаунте с требованиями к аудиту
  • состояние задачи для долгосрочно работающих ассистентов
  • память проекта кодирующего агента
  • multi-agent общее состояние
  • примечания по соответствию стандартам, где важна происхожденность данных
  • временные вопросы вроде «Что мы считали до миграции?»

SGAM представляет собой чрезмерное решение в тех случаях, когда память имеет ограниченный срок жизни, используется в эксплоративных целях или её можно легко пересчитать. Если агенту требуется лишь небольшой объём информации для поддержания континуитета, достаточно чекпоинт вместе с сокращённой историей сообщений. Для проверки качества статических документов может потребоваться лишь RAG. Кроме того, в сферах с крайне нестабильными структурами, где схемы меняются ежедневно, использование типизированной памяти будет замедлять работу команды.

Чек-лист оценки

Оцените жизненный цикл памяти, а также итоговый результат. Система может сгенерировать правдоподобный ответ даже в том случае, если она записала неверный факт, извлекла устаревшую информацию или вышла за пределы определённой зоны доступа пользователя.

Я использую ту же поэтапную сегментацию, что и в моем Статья по оценке RAG. Оценивайте этап, на котором может возникнуть сбой, вместо того чтобы ограничивать анализ только генерируемым текстом. Методика трейс из статья об оценке агентов Это также имеет значение, поскольку ошибка памяти зачастую проявляется в истории выполнения задач ещё до того, как приводит к получению результата.

Я бы протестировал SGAM с использованием механизма повторного воспроизведения. Необходимо загрузить фиксированную последовательность эпизодов в модуль записи в память, проверить состояние журнала после каждого значимого хода, а затем задавать вопросы о текущем состоянии и состоянии в определённый момент времени на основе полученных данных.

Ограничения и замечания

SGAM — это мой собственный термин для описания определённого паттерна, а не стандарт. В существующих проектах проблема рассматривается иначе. память LangGraph и LangMem Опишите краткосрочные и долгосрочные хранилища, профили, коллекции, операции записи в «горячих» путях и фоновые менеджеры памяти. Зеп Графити используется термин «граф временного контекста». Летта поддерживает блоки памяти с возможностью редактирования, при этом Mem0 предоставляет слой управляемой памяти. Microsoft GraphRAG, Графы свойств LlamaIndex, а также подход Cognee предполагают моделирование соответствующих частей проблемы в виде графов знаний.

Профиль пользователя, журнал эпизодов, граф документов и блок памяти, доступный для редактирования агентом, позволяют решать различные проблемы, связанные с retrieval и обновлением данных. Я отношу SGAM к категории устойчивой памяти, которая отражает текущее состояние приложения, в связи с чем требует наличия схемы структуры, механизмов проверки корректности данных, отслеживания происхождения информации, урегулирования конфликтов, правил хранения и процедур миграции.

Записанная в памяти информация может по-прежнему содержать ошибки. Схема помогает легче отслеживать некорректные записи, но не делает их достоверными. Для обеспечения надёжности всё равно требуется доверие к исходному источнику, подтверждение пользователя для чувствительных данных, политика разрешения конфликтов, возможность удаления записей и постоянный мониторинг.

Миграция схем — это сложная задача. Как только данные становятся частью состояния системы, вы берете на себя ответственность за версионирование, восполнение недостающих данных, обработку старых записей и правила их удаления. Если пропустить этот процесс, старые записи будут существовать дольше, чем сама семантика или политика хранения, которые их создали.
