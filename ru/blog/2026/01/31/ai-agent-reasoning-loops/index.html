Циклы рассуждения ИИ-агентов в 2026 году: ReAct против ReWOO против «Планирование и выполнение»

Часть 1 серии «Инжиниринг стека агентов»

Создание полезных ИИ-агентов в настоящее время в основном сводится к проектированию системы, а не к инжинирингу промптов. Самое важное решение — это цикл рассуждений агента: способ, которым система планирует действия, вызывает инструменты, анализирует результаты и определяет момент прекращения работы.

В этой статье сравниваются три подхода к циклу работы ИИ-агентов, которые стоит знать в 2026 году: ReAct, ReWOO и Plan-and-Execute. В качестве примера используется агент-аналитик рынка, который я разработал в LangGraph; полный код доступен на GitHub.

Кратко: ReAct обладает гибкостью, но требует больших ресурсов; ReWOO работает быстро при предсказуемых рабочих процессах; модель типа «Планирование и выполнение» подходит для многократного анализа. Промышленный ИИ-агент может переключаться между циклами рассуждений в зависимости от запроса, используя общее состояние и сохраняемые узлы LangGraph, чтобы каждая задача выполнялась с использованием цикла, соответствующего её структуре.

Почему циклы рассуждений ИИ-агента имеют значение

Год назад ключом к эффективному использованию LLM была правильная формулировка промпта. Сейчас же речь идет о проектировании графов: о том, как связываются процессы рассуждения, использование инструментов и хранение информации.

Цикл рассуждений находится в сердце такого графа. Он определяет, когда модель должна анализировать информацию, когда вызывать инструмент и когда прекращать работу. Неправильный выбор приведет к трате лишних токенов на ненужные вызовы, увеличению времени задержки на каждый ответ или к сбою агента при получении от инструмента неожиданных данных.

Три основных паттерна рассуждений ИИ-агента

ReAct: мышление, действие, наблюдение, повторение

ReAct (Yao и др., 2022) представляет собой первоначальную модель интерактивных агентов. Она работает по следующей циклической структуре:

  1. Мышление: агент формирует «мысль» с целью разложения задачи на составные части и планирования следующего шага.
  2. Действие: на основе этой мысли агент вызывает соответствующий инструмент.
  3. Наблюдение: агент анализирует полученный результат, что позволяет обновить его представление для последующего этапа мышления.

Преимущества:

  • Основание в реальных наблюдениях снижает количество вымышленных фактов.
  • Агент может динамически менять стратегию в зависимости от только что увиденного.
  • «Бумажка» позволяет вести отчет о ходе логических рассуждений.

Недостатки:

  • История операций накапливается и перерабатывается на каждом шаге, поэтому задержки и затраты растут с увеличением длины цикла.
  • Это неэффективно, когда вызовы инструментов можно было запланировать заранее — именно эту нишу и заполняет ReWOO.
  • Без условия остановки или лимита шагов цикл будет выполняться бесконечно.

Идеально для: исследовательских задач, отладки, ситуаций, когда невозможно предсказать следующий шаг.

ReWOO: запланируйте всё заранее

ReWOO (Reasoning Without Observation) представляет собой более эффективную версию архитектуры ReAct. Секрет её эффективности заключается в разделении процессов рассуждения и выполнения инструментов: вместо того чтобы останавливаться для наблюдения за каждым действием, подход ReWOO позволяет спланировать всю последовательность вызовов инструментов за один проход.

  1. План: один вызов большой языковой модели формирует полный план вызовов инструментов с использованием местоимений-заменителей.#E1, #E2) — для результатов, которые ещё не существуют.
  2. Рабочий процесс: исполнитель, не являющийся LLM, запускает запланированные инструменты последовательно или параллельно, заполняя места замены.
  3. Решатель: окончательный вызов LLM обрабатывает собранные данные и формирует ответ.

Преимущества:

  • По эффективности использования токенов примерно в 5 раз выше, чем у ReAct, поскольку не требуется хранение повторяющейся истории «мысль — действие — наблюдение».
  • Более низкая задержка. Нет необходимости пересылать историю на каждом шаге.
  • Планировщик можно оттачивать отдельно, без использования реальной среды.

Недостатки:

  • Хрупкость при некорректном поведении инструментов. План предполагает, что всё работает как надо.
  • Требует предсказуемых рабочих процессов.
  • В отсутствие явной логики возврата к альтернативе ошибка в плане продолжает влиять на работу.

Идеально подходит для: быстрых анализов, проверки статуса, панелей управления. Для любых сценариев, где поведение инструментов предсказуемо.

Планирование и выполнение: гибридный подход

Планирование и выполнение он располагается между этими двумя этапами. В статье описывается простая схема разделения работы, которую приняли большинство современных агентов:

  1. Этап планирования: сначала агент генерирует план, разделяющий задачу на более мелкие подзадачи.
  2. Этап выполнения: затем агент выполняет эти подзадачи.

Исходная статья была посвящена методу запуска без предварительной настройки. Современные фреймворки, такие как LangGraph, развили эту концепцию до полноценного шаблона оркестрации с последовательным выполнением действий и выбором модели для каждого шага (мощная модель для планирования и более экономичная — для выполнения).

Преимущества:

  • Иерархическое рассуждение, соответствующее способу, которым человек-эксперт разбивает проект на этапы.
  • Возможность перепланирования. Можно приостановить работу и переоценить ситуацию, если результат определённого шага оказался неожиданным.
  • Специализация модели. Механизм планирования может быть ресурсоёмким, тогда как механизм выполнения — дешёвым.
  • Ограниченная сложность, при этом после каждого шага предусмотрена чёткая точка контроля.

Недостатки:

  • Более высокая задержка по сравнению с ReWOO из-за последовательной обработки шагов.
  • Необходимость управления большим количеством состояний. — Чрезмерная сложность для одноразовых запросов.

Идеально подходит для: сложного многоэтапного анализа, исследовательских задач, любых случаев, требующих синтеза в конце.

1. ReAct: паттерн «мышление по мере выполнения»

Каково это ощущение: словно человек отлаживает проблему. «Я попробую вот это… хорошо, это не сработало, давайте попробуем что-то другое».

Преимущество: способность справляться с абсолютно неизвестными ситуациями. Если результат поиска выявляет новую тему, агент может изменить стратегию в следующем шаге.

Недостаток: склонен к входу в цикл при неудачной операции. Самый дорогой вариант с точки зрения количества токенов.

2. Планирование и выполнение: паттерн, ориентированный на задачу

Каково это ощущение: словно вы руководитель проекта. «Вот план из 5 шагов. Давайте выполним шаги с 1 по 5, а затем проверим, закончили ли мы».

Преимущество: позволяет агенту сосредотачиваться на общей цели. Повышает вероятность успешного выполнения длительных и сложных задач.

Недостаток: если на первом шаге происходит сбой, приводящий к нарушению выполнения шагов с 2 по 5, агент может продолжать выполнять повреждённый план, не замечая этого до тех пор, пока не станет слишком поздно.

3. ReWOO: шаблон компилятора

Каково это ощущение: словно пишешь небольшую программу. «Мне нужны данные из инструмента A и инструмента B, после чего я объединю их в инструменте C».

Преимущества: значительно более высокая скорость и ниже стоимость. План компилируется один раз с использованием мест замены.#E1 (для вывода первого инструмента), после чего выполнение продолжалось без дополнительных вызовов больших языковых моделей до финальной синтезации.

Недостаток: слепота во время выполнения. Если первый инструмент сообщает «Я не могу найти этого человека», агент всё равно будет выполнять следующие шаги, которые зависели от наличия этого человека.

Что выбрать

  • ReAct — когда агент ведёт чат с пользователем и должен немедленно отреагировать.
  • Plan-and-Execute — при автоматизации длительных, многократных операций, таких как подготовка исследовательского отчёта.
  • ReWOO — когда у вас предсказуемый поток задач и вы хотите сократить расходы на API примерно на 80%.

Пример реализации: агент-аналитик рынка

Чтобы проиллюстрировать это на практике, я создал Агент аналитика рынка который использует все три шаблона в одном кодбейсе в рамках задачи по исследованию рынка. LangGraph для оркестрации. Все три шаблона используют один и тот же объект состояния, поэтому маршрутизатор может выбирать, какой из них запустить для каждого запроса:

Определение состояния

Общее состояние хранит всю информацию, необходимую агенту в разных режимах работы:

class PlanStep(BaseModel):
    """A single step in the research plan."""
    step_number: int
    description: str
    tool_hint: str | None = None
    completed: bool = False
    result: str | None = None

class UserProfile(BaseModel):
    """Structured user context loaded from long-term memory."""
    risk_tolerance: str | None = None
    investment_horizon: str | None = None

class AgentState(BaseModel):
    """Main state for the Market Analyst Agent graph."""

    # Identity and profile context for memory-backed personalization
    user_id: str
    user_profile: UserProfile = Field(default_factory=UserProfile)

    # Message history with LangGraph's add_messages reducer
    messages: Annotated[list, add_messages] = Field(default_factory=list)

    # Execution mode (set by router)
    execution_mode: ExecutionMode | None = None

    # Plan-and-Execute state
    plan: list[PlanStep] = Field(default_factory=list)
    current_step_index: int = 0

    # ReWOO state
    rewoo_plan: list[ReWOOPlanStep] = Field(default_factory=list)

    # Research results
    research_data: ResearchData | None = None

    # HITL output
    draft_report: DraftReport | None = None
    report_approved: bool = False

Шаблон 1: Реализация по принципу «планирование и выполнение»

Шаблон «планирование и выполнение» идеально подходит для задач, требующих многократной обработки данных. Ключевой момент заключается в разделении процессов планирования и выполнения: сначала используется сильная модель для формирования первоначального плана, а затем применяется цикл ReAct для выполнения каждого шага с возможностью реагирования на результаты работы инструментов.

Как это соответствует шаблону:

  1. Одна фаза планирования. Один вызов большой языковой модели позволяет сформировать полный план в виде списка описаний шагов.
  2. Структурированный вывод с использованием метода Schema-Guided Reasoning, что гарантирует корректность формата JSON.
  3. Пока не происходит выполнение инструментов. Планировщик определяет лишь то, что нужно сделать, но не как это сделать.
  4. Шаги в читаемом формате. Каждый шаг представлен в виде текста, который будет интерпретироваться исполнителем.
# System prompt guides the LLM to think like a research analyst
# creating a strategic plan, not immediate tool calls
PLANNER_SYSTEM_PROMPT = """You are a senior investment research analyst.
Break down stock analysis requests into 4-6 research steps covering:
1. Current price and basic metrics
2. Recent news and announcements
3. Competitor analysis (if relevant)
4. Financial health assessment
5. Risk factors
6. Investment thesis synthesis

Output as JSON with step_number, description, and tool_hint."""

# Schema-Guided Reasoning: Enforce structure with Pydantic
class PlanOutput(BaseModel):
    """Structured output for the planner."""

    steps: list[PlanStep] = Field(description="Research steps to execute")
    ticker: str = Field(description="The stock ticker being analyzed")

def planner_node(state: AgentState) -> dict:
    """Generate a research plan from the user's request.

    This is Phase 1 of Plan-and-Execute: creating the high-level strategy.
    """

    # Use a powerful model for strategic planning
    llm = ChatAnthropic(model="claude-sonnet-4-5-20250929", temperature=0)

    # Apply Schema-Guided Reasoning to guarantee valid plan structure
    # This prevents common formatting errors that would break execution
    structured_llm = llm.with_structured_output(PlanOutput)

    # Context from long-term memory personalizes the plan
    profile_context = f"""
User Profile:
- Risk Tolerance: {state.user_profile.risk_tolerance}
- Investment Horizon: {state.user_profile.investment_horizon}
"""

    # Single LLM call creates the complete plan
    result: PlanOutput = structured_llm.invoke([
        SystemMessage(content=PLANNER_SYSTEM_PROMPT + profile_context),
        HumanMessage(content=f"Create a research plan for: {last_user_message}"),
    ])

    # State update: Store the plan and initialize tracking
    return {
        "plan": result.steps,           # The sequential steps to execute
        "current_step_index": 0,        # Start at step 0
        "research_data": ResearchData(ticker=result.ticker),  # Initialize data container
    }

Это llm.with_structured_output(PlanOutput) Эта строка означает Schema-Guided Reasoning (SGR), который я уже рассматривал. предыдущая публикация. Принудительное изменение PlanOutput Под схемой понимается структура, обеспечивающая то, что планировщик всегда возвращает допустимый список шагов. LangGraph затем использует эти структурированные результаты для управления детерминированным потоком выполнения через условные рёбра.

Вариант 2: выполнение по принципу ReAct

После формирования плана исполнитель запускает каждый шаг как отдельный цикл ReAct. Это соответствует второму этапу: каждый шаг достаточно мал, чтобы цикл «мышление — действие — наблюдение» оставался сфокусированным, и агент мог реагировать на любые данные, возвращаемые инструментом.

Как соотносится часть, связанная с ReAct:

  1. Итеративное выполнение. Один шаг за раз с обратной связью от наблюдений.
  2. Цикл «мышление — действие — наблюдение» выполняется внутри create_react_agent3. Результаты предыдущего шага используются в качестве контекста для текущих вычислений.
  3. Агент выбирает инструменты на основе описания шага.
  4. Он может изменить подход в ходе шага в зависимости от результата, возвращаемого инструментом.
# Tools available for the ReAct agent to choose from
TOOLS = [
    get_stock_snapshot,
    get_price_history,
    search_news,
    search_competitors,
    get_financials,
]

def executor_node(state: AgentState) -> dict:
    """Execute the current step using a ReAct agent.

    This is Phase 2 of Plan-and-Execute: adaptive execution of each planned step.
    Each step runs as a mini ReAct loop until completion.
    """

    # Get the current step from the plan
    current_step = state.plan[state.current_step_index]

    # Build context from what we've learned so far
    # This matters: each step builds on previous observations
    previous_context = ""
    for step in state.plan[:state.current_step_index]:
        if step.result:
            previous_context += f"\nStep {step.step_number}: {step.result}\n"

    # Create a ReAct agent for this step
    # LangGraph's create_react_agent implements the full Thought-Action-Observation loop:
    # 1. Agent generates a "thought" about what tool to call
    # 2. Agent calls the tool ("action")
    # 3. Tool returns result ("observation")
    # 4. Agent decides: call another tool or finish
    react_agent = create_react_agent(
        model=ChatAnthropic(model="claude-sonnet-4-5-20250929"),
        tools=TOOLS,
    )

    # Invoke the ReAct loop for this single step
    # The agent will loop internally until it completes the step
    result = react_agent.invoke({
        "messages": [
            SystemMessage(content=EXECUTOR_SYSTEM_PROMPT),
            HumanMessage(content=f"""Execute Step {current_step.step_number}:
{current_step.description}

Ticker: {state.research_data.ticker}
Previous findings: {previous_context}"""),
        ]
    })

    # Extract the final answer from the ReAct agent's message history
    # The last message contains the synthesis after all tool calls
    updated_plan = list(state.plan)
    updated_plan[state.current_step_index] = PlanStep(
        step_number=current_step.step_number,
        description=current_step.description,
        completed=True,
        result=result["messages"][-1].content,  # Final observation
    )

    # State update: Mark step complete and advance to next
    return {
        "plan": updated_plan,
        "current_step_index": state.current_step_index + 1,
    }

Это и есть основной цикл «Планирование — Выполнение»: мощная модель формирует план, после чего механизм ReAct выполняет каждый шаг с полной адаптивностью.

Шаблон 3: ReWOO для быстрых снимков состояния

Для кратких брифингов ReWOO пропускает этап взаимосвязанного рассуждения и запускает инструменты параллельно. Планировщик заранее генерирует скомпилированный скрипт вызовов инструментов, а исполнитель запускает его без дальнейшего участия больших языковых моделей.

Структура этого процесса:

  1. Три фазы (Планировщик → Рабочий процесс → Решатель), без циклов.
  2. Вызовы инструментов являются ссылками. #E1, #E2 места замены для результатов, которые пока отсутствуют.
  3. Во время выполнения LLM не используется. Рабочий процесс просто запускает инструменты.
  4. Независимые инструменты работают параллельно.
  5. В конце производится один вызов синтеза, обрабатывающий всю информацию сразу.

Этап 1: планировщик ReWOO (создает полную структуру выполнения заранее)

class ReWOOPlanStep(BaseModel):
    """A step in the ReWOO plan with variable placeholders.

    Key difference from Plan-and-Execute's PlanStep:
    - Contains actual tool_name and tool_args (not just description)
    - Uses variable references (#E1) for dependencies
    """
    step_id: str  # e.g., "#E1" - becomes a variable
    description: str
    tool_name: str     # Exact tool to call
    tool_args: dict    # May contain variable refs like {"price": "#E1"}
    depends_on: list[str] = []  # For dependency ordering
    result: str | None = None

class ReWOOPlanOutput(BaseModel):
    """Structured output for ReWOO planner."""
    steps: list[ReWOOPlanStep] = Field(description="Planned tool calls with variables")

def rewoo_planner_node(state: AgentState) -> dict:
    """Generate a complete plan of tool calls upfront.

    This is the key difference from Plan-and-Execute: instead of creating
    human-readable step descriptions, we create EXACT tool calls that
    the worker will execute blindly.
    """

    llm = ChatAnthropic(model="claude-sonnet-4-5-20250929", temperature=0)

    # Schema-Guided Reasoning ensures valid tool call specifications
    structured_llm = llm.with_structured_output(ReWOOPlanOutput)

    ticker = state.research_data.ticker if state.research_data else "UNKNOWN"

    # Single LLM call to plan ALL tool executions
    result: ReWOOPlanOutput = structured_llm.invoke([
        SystemMessage(content=REWOO_PLANNER_PROMPT),
        HumanMessage(content=f"""Create a ReWOO plan for: {query}

Ticker: {ticker}

Output tool calls with:
- step_id: Variable name (#E1, #E2, etc.)
- description: What this accomplishes
- tool_name: Exact tool from the list
- tool_args: Dictionary of arguments
- depends_on: List of step_ids this depends on"""),
    ])

    # State update: Store the complete execution plan
    # Worker will execute this without any LLM involvement
    return {"rewoo_plan": result.steps}

Этап 2: рабочий процесс ReWOO (запускает инструменты без использования логики больших языковых моделей)

def rewoo_worker_node(state: AgentState) -> dict:
    """Execute all planned tools in parallel (no LLM calls).

    This is the key efficiency: Worker is "dumb" - it just runs tools
    according to the plan. No LLM calls = massive token savings.
    """

    results = {}  # Store results keyed by step_id (e.g., "#E1": "$150.23")

    # Execute ALL independent steps in parallel using ThreadPoolExecutor
    # This is where ReWOO gets its speed advantage
    with ThreadPoolExecutor(max_workers=5) as executor:
        futures = {
            executor.submit(execute_tool, step): step
            for step in state.rewoo_plan
            if not step.depends_on  # Only independent tools for parallel batch
        }

        # Collect results as they complete
        for future in as_completed(futures):
            step = futures[future]
            results[step.step_id] = future.result()
            # No LLM reasoning here - just store the raw tool output

    # State update: Store results for the Solver phase
    return {"rewoo_plan": updated_steps}

Этап 3: решатель ReWOO (синтезирует все результаты в одном вызове большой языковой модели)

def rewoo_solver_node(state: AgentState) -> dict:
    """Synthesize all tool results into a flash briefing.

    This is the second efficiency gain: Instead of interleaving
    LLM calls with tool execution (like ReAct), we make ONE
    final synthesis call with all gathered data.
    """

    # Build context from ALL tool results at once
    tool_results = []
    for step in state.rewoo_plan:
        if step.result:
            tool_results.append(f"### {step.description}\n{step.result}")

    context = "\n\n".join(tool_results)

    # Single LLM call to synthesize everything
    structured_llm = llm.with_structured_output(FlashBriefingOutput)
    result = structured_llm.invoke([
        SystemMessage(content=REWOO_SOLVER_PROMPT),
        HumanMessage(content=f"Create a flash briefing from this data:\n\n{context}"),
    ])

    return {"draft_report": result}

Ключевое отличие заключается в том, что ReWOO заранее планирует каждый вызов инструмента с местоимениями-заменителями (#E1, #E2), выполняет их параллельно без вызовов больших языковых моделей в процессе и синтезирует результаты в одном вызове в конце. Благодаря этому такой подход остается экономичным для предсказуемых рабочих процессов.

Понимание кода: в чём различия между каждым шаблоном

Эти три шаблона отличаются по времени и способу вызова большой языковой модели:

ШаблонВызовы LLM во время выполненияОбновления состоянияОсновной шаблон кода
Планирование и выполнение1 — для планирования + 1 за каждый шагПоследовательное выполнение шаговplanner_node() → цикл: executor_node() → reporter_node()
ReAct (внутри каждого шага)Множественные шаги в один цикл (циклы мышления и действия)Накопленная история сообщенийcreate_react_agent() внутренне выполняет циклы до завершения шага
ReWOO1 — на этапе планирования + 0 во время выполнения + 1 — при синтезеПараллельное дополнение инструментовrewoo_planner_node() → rewoo_worker_node() → rewoo_solver_node()

Различия между ними заключаются в том, что генерирует планировщик. Именно он определяет всё, что происходит далее в работе системы.

  1. Функция Plan-and-Execute генерирует описания шагов в формате, понятном человеку:

    # Planner output (list of PlanStep objects)
plan = [
    PlanStep(
        step_number=1,
        description="Get current price and key financial metrics",
        tool_hint="get_stock_price"
    ),
    PlanStep(
        step_number=2,
        description="Search for recent news and earnings",
        tool_hint="search_news"
    ),
    # ... more steps
]

    Экзекутор читает каждое описание и определяет, какие инструменты необходимо вызвать. Этот подход гибок, однако каждый шаг сопряжён с затратами в виде вызова большой языковой модели.

  2. В алгоритме ReAct отсутствует заранее определённый план. Он использует итеративное рассуждение:

    # No planning phase - ReAct works step-by-step with accumulated messages
messages = [
    HumanMessage(content="Execute Step 1: Get current price"),
    AIMessage(content="I'll call get_stock_price"),
    ToolMessage(tool_call_id="1", content="$132.45"),
    AIMessage(content="Now I need metrics..."),
    # ... agent continues until step complete
]

    Множественные вызовы больших языковых моделей на каждом шаге с адаптацией под полученные данные. Максимальная гибкость, но и максимальные затраты.

  3. ReWOO формирует чёткие, исполняемые спецификации вызовов инструментов:

    # Planner output (list of ReWOOPlanStep objects)
rewoo_plan = [
    ReWOOPlanStep(
        step_id="#E1",
        tool_name="get_stock_price",
        tool_args={"ticker": "NVDA"}
    ),
    ReWOOPlanStep(
        step_id="#E2",
        tool_name="search_news",
        tool_args={"query": "NVDA earnings", "limit": 5}
    ),
    # ... all tool calls planned upfront
]

    Рабочий процесс выполняется автономно, без участия LLM. Весь процесс рассуждений находится в планировщике и решателе. Это вариант с наименьшими затратами среди трех.

Поток памяти и состояний:

  • Планирование и выполнение: состояние проходит через plan → current_step_index → research_data- ReAct: состояние накапливается внутри messages массив (полная история переписки).
  • ReWOO: состояние прослеживается по шагам rewoo_plan, с result поля, заполняемые оператором.

Сборка всего воедино: подключение графа

Вот как три шаблона сосуществуют в одной системе LangGraph. Они используют общий AgentState и существуют в одной графе. Маршрутизатор выбирает путь для каждого запроса, поэтому речь идет об одном агенте с тремя режимами выполнения, а не о трех агентах, действующих независимо.

LangGraph использует декларативный подход к описанию структуры связей:

def create_graph(checkpointer=None):
    builder = StateGraph(AgentState)

    # Add nodes
    builder.add_node("router", router_node)
    builder.add_node("planner", planner_node)
    builder.add_node("executor", executor_node)
    builder.add_node("reporter", reporter_node)
    builder.add_node("rewoo_planner", rewoo_planner_node)
    builder.add_node("rewoo_worker", rewoo_worker_node)
    builder.add_node("rewoo_solver", rewoo_solver_node)

    # Define edges
    builder.add_edge(START, "router")
    builder.add_conditional_edges("router", route_after_router, {
        "planner": "planner",
        "rewoo_planner": "rewoo_planner",
    })

    # Deep Research path
    builder.add_edge("planner", "executor")
    builder.add_conditional_edges("executor", route_after_executor, {
        "executor": "executor",  # Loop back for more steps
        "reporter": "reporter",  # Done with plan
    })
    builder.add_edge("reporter", END)

    # Flash Briefing path (ReWOO)
    builder.add_edge("rewoo_planner", "rewoo_worker")
    builder.add_edge("rewoo_worker", "rewoo_solver")
    builder.add_edge("rewoo_solver", END)

    return builder.compile(
        checkpointer=checkpointer,
        interrupt_before=["reporter"],  # HITL pause for approval
    )

Автоматический выбор шаблона с помощью маршрутизатора

Для того чтобы выбирать подходящий цикл для каждого запроса, я добавил классификатор маршрутизатора. Он использует метод рассуждений на основе схемы для обеспечения надежности классификации:

class ExecutionMode(str, Enum):
    """Execution mode for the agent."""

    DEEP_RESEARCH = "deep_research"  # Plan-and-Execute + ReAct (thorough)
    FLASH_BRIEFING = "flash_briefing"  # ReWOO (fast, token-efficient)

class RouterOutput(BaseModel):
    """Structured output for the router."""

    mode: ExecutionMode  # DEEP_RESEARCH or FLASH_BRIEFING
    ticker: str
    reasoning: str

ROUTER_SYSTEM_PROMPT = """Classify the user's request:

1. **deep_research**: Complex analysis requiring synthesis
   - Examples: "Analyze strategic risks", "investment thesis"

2. **flash_briefing**: Quick snapshots, simple data retrieval
   - Examples: "quick snapshot", "current price"

Default to deep_research if unclear."""

structured_llm = llm.with_structured_output(RouterOutput)

При таком подходе пользователи не должны выбирать режим работы. Маршрутизатор самостоятельно направляет данные типа «текущая цена» в модуль ReWOO, а данные типа «инвестиционная тезис» — в модуль Plan-and-Execute.

Основные выводы

  1. Режим ReAct является стандартным вариантом благодаря своей гибкости, однако он сопряжён с увеличением расхода токенов и задержек.
  2. Модуль ReWOO превосходит по скорости и стоимости тогда, когда инструменты надёжны, а результаты предсказуемы.
  3. Модуль Plan-and-Execute оптимален для сложного анализа, требующего финальной синтезации данных.
  4. Маршрутизатор может выбирать подходящий режим для каждого запроса, что освобождает пользователя от необходимости делать этот выбор самостоятельно.
  5. Управление состоянием имеет решающее значение. Именно возможность сохранения состояния с помощью LangGraph позволяет осуществлять прерывания и восстановление работы системы.
