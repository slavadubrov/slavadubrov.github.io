    Model Quantization in 2026: From Foundations to Production Serving

    Quantization is not a single optimization. It is a set of trade-offs between weight memory, activation compute, KV cache size, runtime kernels, calibration data, and quality regression.

    The useful question is not "How many bits can I remove?" The useful question is "Which bottleneck am I solving?" A 4-bit checkpoint can fit a model into VRAM and still be slower than expected if the serving kernel is wrong. FP8 can improve high-throughput serving and still fail on unsupported hardware. KV cache quantization can matter more than weight quantization once long context and concurrency dominate memory.

    TL;DR: Do not pick a quantization method based on bit-width alone. Start with your system's bottleneck. If your model is too large for GPU memory, use 4-bit weight-only compression (such as AWQ or GPTQ). If you serve at high throughput on modern GPUs, use 8-bit floating-point (FP8) for faster hardware calculation. If long conversations or high concurrency consumes your VRAM, compress the Key-Value (KV) cache. If you run models locally on laptops or CPUs, use GGUF. Avoid low-precision formats such as NF4 for production serving; keep them for parameter-efficient fine-tuning.

    1. The Operating Decision

    Before choosing a quantization path, decide what is actually limiting your system.

    Decode the Acronyms

    To navigate the options, it helps to understand the terminology:

    • W{x}A{y} (Weight/Activation Precision): Tells you how many bits are used for weights (\(W\)) and activations (\(A\)). For example, W4A16 uses 4-bit weights to save memory, while activations remain in 16-bit precision for high-quality calculations.
    • Precision Formats (e.g., FP8, INT8, INT4, NF4): The numerical representations defining how numbers are represented and stored. For instance, FP8 uses 8 bits to represent floating-point numbers, while INT4 uses 4-bit integers.
    • Quantization Algorithms (e.g., GPTQ, AWQ, SmoothQuant): The mathematical methods used to scale, round, and fit model parameters into a target precision format with minimal accuracy loss.
    • GGUF: A file format and runtime optimized for running models on consumer CPUs, Apple Silicon (Macs), and edge devices using the llama.cpp ecosystem.
    • NF4 (NormalFloat 4-bit): A specialized 4-bit data format optimized for weights with a normal distribution, primarily used to reduce memory during parameter-efficient fine-tuning (QLoRA).
    • KV Cache: The Key-Value cache, which serves as the model's short-term memory during a chat session so the model doesn't have to recompute previous tokens.
    • GEMM (General Matrix Multiply): The standard mathematical operation for multiplying matrices (e.g., multiplying input activations by model weights), which makes up almost all LLM computation.

    Understand the Bottleneck

    LLM execution runs in two main phases:

    1. Prefill (Compute-Bound): The model processes the initial prompt. The bottleneck is raw computation (FLOPs) as the GPU performs massive matrix multiplication.
    2. Decode (Memory-Bandwidth-Bound): The model generates text one token at a time. The bottleneck is memory transfer speed, because the GPU must load every single weight from VRAM into its compute cores just to generate a single token.

    Quantization targets these phases differently:

    • 4-bit weight-only formats (like W4A16) speed up memory transfer by reducing model weight sizes. This speeds up the decode phase for single users (which is bottlenecked by VRAM bandwidth). However, it does not speed up the math: the GPU must dequantize the 4-bit weights back to 16-bit float inside its registers before performing calculations, so the actual multiplications still run on standard 16-bit Tensor Cores.
    • 8-bit weight-activation formats (like FP8 W8A8) speed up both memory transfer and arithmetic calculations. The GPU does not need to dequantize the weights; it performs matrix multiplications directly in 8-bit using native low-precision Tensor Cores. This speeds up both the compute-bound prefill phase and high-concurrency decoding.
    Target Bottleneck Target Format & Precision Quantization Algorithm / Tool Why It Works What to Validate
    Model too large for VRAM (Memory-bound decode) W4A16 (INT4) AWQ or GPTQ (via llm-compressor or GPTQModel) Cuts weight VRAM footprint by 75% to reduce weight loading time. Perplexity, reasoning tasks (MMLU/GPQA), coding (HumanEval).
    High throughput / serving (Compute-bound prefill/batch) W8A8 (FP8 or INT8) FP8 PTQ (via llm-compressor / ModelOpt) or SmoothQuant Runs matrix math natively on low-precision Tensor Cores. Throughput (tokens/s), Time to First Token (TTFT), accuracy.
    Long context / high concurrency (KV cache bound) FP8 KV cache Dynamic or calibrated scaling (via vLLM / TensorRT-LLM) Halves KV cache footprint to fit larger batches or longer history. Long-context retrieval (needle-in-a-haystack), degradation.
    Local CPU, Apple Silicon, Edge (Consumer RAM limits) GGUF (mixed low-bit) llama.cpp / Ollama K-quants or IQ-quants Uses custom runtime kernels optimized for CPU/Metal/unified memory. Prompt latency, memory footprint, general text quality.
    Low-memory fine-tuning (Adapter training memory) NF4 (4-bit NormalFloat) QLoRA (via bitsandbytes / peft) Fits base model in a single GPU; merge back for serving. Fine-tuning loss, merged model accuracy, task performance.
    Local image generation (Diffusion pipelines) W4A4 (INT4) or FP8 SVDQuant (Nunchaku) or torchao / ModelOpt Handles volatile activation outliers to prevent visual artifacts. Visual inspection (FID/LPIPS), prompt alignment, latency.

    This framing also explains why benchmark posts can disagree. One guide may measure an H200 vLLM path with Marlin kernels. Another may explain quantization math without serving kernels. A third may focus on static INT8 calibration. All can be correct inside their own deployment context.

    2. The Core Concept: From Float to Buckets

    Quantization maps high-precision values into a smaller set of representable values. You save memory and bandwidth, but you introduce rounding error.

    Think of rounding prices to the nearest dollar. You store fewer digits, but you lose cents. That is acceptable for a rough budget and unacceptable for payroll. Neural networks have the same shape of problem. A BF16 weight can represent subtle variation. INT4 gives you 16 buckets. If the rounded values preserve the model's useful behavior, you gain capacity. If a sensitive layer or outlier channel is rounded badly, reasoning, instruction following, or visual quality can regress.

    Symmetric and Asymmetric Mapping

    Quantization maps a continuous set of float values \(x \in [\beta, \alpha]\) to a discrete grid of integer or low-precision floating-point values.

    To understand the formulas below, it helps to identify the variables:

    • \(x\): The original, high-precision continuous input value (e.g., in BF16 or FP32).
    • \(x_q\): The resulting low-precision quantized value (an integer coordinate).
    • \(s\) (Scale): The step size between representable values, acting as a scaling multiplier.
    • \(z\) (Zero-Point): An integer offset representing where 0.0 in full precision maps in the target quantized grid.
    • \([\beta, \alpha]\): The minimum (\(\beta\)) and maximum (\(\alpha\)) values of the input tensor.
    • \([q_{\min}, q_{\max}]\): The boundary values of the target integer range (e.g., for signed 4-bit integers, this range is \([-7, 7]\)).

    Two primary mapping approaches are used:

    • Symmetric quantization centers the grid around zero and sets the zero point \(z\) to 0: $\(s = \frac{\max(|x|)}{q_{\max}}\)$ $\(x_q = \text{clip}\left(\text{round}\left(\frac{x}{s}\right), q_{\min}, q_{\max}\right)\)$ Trade-off: Because the zero point is always zero, the hardware doesn't need to subtract an offset during runtime calculation, making symmetric math much faster on the GPU.

    • Asymmetric quantization uses a shifted zero point to cover skewed ranges more efficiently: $\(s = \frac{\alpha - \beta}{q_{\max} - q_{\min}}\)$ $\(z = \text{round}\left(\frac{-\beta}{s}\right) + q_{\min}\)$ $\(x_q = \text{clip}\left(\text{round}\left(\frac{x}{s}\right) + z, q_{\min}, q_{\max}\right)\)$ Trade-off: By shifting the grid, asymmetric quantization covers asymmetric distributions (such as activations that are strictly positive after a GELU or ReLU layer) without wasting representable integer buckets on non-existent negative numbers.

    The scale (\(s\)) and zero point (\(z\)) are only the beginning. Quality also depends on scaling granularity, protected tensors, calibration data, and the execution kernel.

    Scaling Granularity

    • Per-tensor: one scale for the whole tensor. It is simple, but one outlier can waste most of the grid.
    • Per-channel: one scale per output channel or matrix row. This is a common default for LLM weights.
    • Per-group or block: one scale per small block, such as 128 weights. This is the practical sweet spot for many 4-bit weight-only formats.

    For weights, group-wise symmetric or asymmetric quantization is common. For activations, dynamic scales are often safer because activation distributions change with the prompt, token position, batch, and layer.

    3. When to compute ranges: PTQ vs. QAT

    The performance and quality of a quantized model depend heavily on when the scaling ranges are calculated. Quantization approaches fall into two main families:

    1. PTQ (Post-Training Quantization): Compressing a model after it has finished training. This is fast and requires no access to the training setup.
    2. QAT (Quantization-Aware Training): Simulating quantization during the training process, allowing the model to adapt its parameters to compensate for rounding errors.

    Within these families, the scale and zero-point ranges can be computed at different times:

    Quantization Method Range Calculation Timing Best Suited For System Trade-off
    Dynamic PTQ At runtime for each activation path on-the-fly. Quick experimentation and handling unpredictable activation outliers. Minor runtime math overhead; limited by hardware support.
    Static PTQ Offline before serving, using representative calibration data. Production serving where speed is the priority (e.g., INT8/W8A8). Requires a calibration dataset that matches real production traffic.
    Weight-Only PTQ Offline before serving, calculated only for static model weights. Memory-bound serving (e.g., W4A16) to fit large models in VRAM. Model weights are compressed, but activation compute stays in 16-bit.
    Quantization-Aware Training (QAT) During training, simulated using fake-quantization operators. Highly sensitive models where PTQ causes excessive quality loss. Requires full training infrastructure, data, and significant compute time.

    Understanding this distinction is critical because calibration is not ceremony. If your production prompts are long Retrieval-Augmented Generation (RAG) traces, calibrating a static model on short Wikipedia paragraphs can lead to bad production results despite clean offline benchmarks. Calibration datasets must reflect actual serving conditions: sequence lengths, formatting conventions, domain-specific vocabulary, system prompt instructions, and refusal patterns.

    4. The 2026 precision landscape

    Modern model deployment makes use of several precision families. The right choice depends entirely on your targeted serving runtime and hardware capabilities.

    Format Storage Size (per value) Recommended Use Case Main Watchpoint
    BF16 / FP16 2 bytes Baseline quality, training-friendly serving, broad compatibility. High VRAM consumption and high memory bandwidth traffic.
    FP8 (OCP) 1 byte High-throughput W8A8 serving on NVIDIA Ada, Hopper, or newer GPUs. Requires native hardware FP8 Tensor Cores and runtime library support.
    INT8 1 byte W8A8 serving on older or non-NVIDIA hardware using SmoothQuant. Vulnerable to activation outliers; requires static calibration.
    INT4 0.5 bytes W4A16 serving when weight memory is the primary system bottleneck. Higher quality regression, especially on smaller or reasoning-heavy models.
    FP4 / NVFP4 ~0.5 bytes Experimental low-precision serving on Blackwell-era architectures. Hardware-specific, version-sensitive compiler and runtime support.
    GGUF Variable (mixed bit) Local consumer CPU, Apple Silicon, desktop, and edge environments. Not optimized for concurrent, high-throughput cloud server deployments.
    NF4 0.5 bytes Low-memory parameter-efficient fine-tuning (QLoRA) setups. Optimized for adapter training, not a default serving export format.

    BF16 vs. FP16

    • BF16 (Bfloat16) allocates 8 bits for the exponent (matching the range of FP32) and 7 bits for the mantissa (fraction). This large exponent range makes it highly resistant to overflow and underflow, which is why it is the default format for modern model training and baseline inference.
    • FP16 allocates 5 bits for the exponent and 10 bits for the mantissa. While it has slightly higher precision, its narrow exponent range makes it prone to overflow spikes during multi-layer inference, requiring careful activation scaling.

    FP8 (E4M3 vs. E5M2)

    FP8 formats are standardized by the OCP (Open Compute Project). The two common variants distribute their 8 bits differently:

    • E4M3 (1 sign bit, 4 exponent bits, 3 mantissa bits) favors numerical precision over dynamic range. It is the default format for forward-pass weights and activations because it preserves subtle weight variations.
    • E5M2 (1 sign bit, 5 exponent bits, 2 mantissa bits) favors range over precision. It has the same exponent range as FP16, making it suitable for training gradients and highly volatile activation layers where values can spike.

    In a comprehensive empirical study, Kurtic et al. (ACL 2025, "Give Me BF16 or Give Me Death") analyzed over 500,000 evaluations on the Llama-3.1 family. They found that FP8 (W8A8) is effectively lossless across all model scales. Unlike integer-only methods, the floating-point distribution matches the natural weight spreads of LLMs, preserving complex reasoning and coding capabilities.

    Microscaling Formats (MXFP8 / NVFP4)

    Next-generation hardware (such as NVIDIA Blackwell) introduces microscaling formats. Instead of scaling a whole tensor (per-tensor) or a whole row (per-channel), microscaling splits tensors into tiny blocks that share a scale factor.

    • MXFP8: Splits tensors into blocks of 32 elements sharing an 8-bit scale factor.
    • NVFP4 (Blackwell-native): Blackwell's native 4-bit floating-point format splits elements into tiny blocks of 16 values. Each block shares an FP8 (E4M3) scale factor, alongside a second-level FP32 scaling factor per tensor. This microscaling overhead brings the actual storage to ~4.5 bits per value.

    According to NVIDIA, NVFP4 reduces the model VRAM footprint by approximately 3.5x relative to FP16 and 1.8x relative to FP8, while keeping accuracy degradation to 1% or less on key reasoning and language tasks (e.g. DeepSeek-R1, Llama-3.3).

    5. Weight-only vs. weight-activation quantization

    The notation WxAy specifies the bit precision used for model weights (\(W\)) and activations (\(A\)).

    Weight-Only Quantization (W4A16 / W8A16)

    Weights are stored in low-precision (e.g., 4-bit or 8-bit), while activations remain in standard 16-bit (BF16/FP16) precision. During model execution, the GPU loads the compressed weights from VRAM, dequantizes them back to 16-bit in its fast cache (SRAM), and performs standard 16-bit matrix calculations.

    • Connection to Bottleneck: This is highly effective during the decode phase (generating text one token at a time). Because the GPU is memory-bandwidth-bound during decode, cutting weight sizes by 75% dramatically reduces VRAM transfer traffic, speeding up generation.
    • System Limitation: Do not expect compute-bound speedups. Because weights must be dequantized back to 16-bit on-the-fly and the mathematical multiply-accumulate operations are still performed in 16-bit, W4A16 does not speed up the processing of large prompt batches (the prefill phase) or high-concurrency workloads.

    Weight-Activation Quantization (W8A8 / W4A4)

    Both model weights and layer activations are compressed into low precision. This allows the serving engine to skip dequantization and run mathematical calculations directly in low precision (using native integer or FP8 Tensor Cores).

    • Connection to Bottleneck: This is optimized for the prefill phase and high-concurrency serving where compute (FLOPs) is the system bottleneck. Keeping activations in low precision allows the GPU to use its high-throughput 8-bit Tensor Cores.
    • System Limitation: This requires strict hardware compatibility and static calibration. If the GPU or runtime does not support low-precision Tensor Cores, the system will fallback to dequantizing activations, introducing massive latency overhead.

    6. Kernels matter as much as algorithms

    The quantization algorithm (like GPTQ or AWQ) determines how the weight numbers are rounded and scaled. The runtime kernel (the low-level CUDA program running on the GPU) determines whether those quantized weights are actually executed fast.

    To evaluate these methods, we track several key metrics:

    • Perplexity: A language model accuracy metric measuring how confused the model is by test text (lower is better; increases show quality loss).
    • Pass@1: The percentage of coding tasks solved correctly on a single attempt (higher is better).
    • Throughput: Total generated tokens per second across all concurrent users (higher is better).
    • TTFT (Time to First Token): The latency before the user receives the first generated character (lower is better).

    JarvisLabs' vLLM benchmark on Qwen2.5-32B-Instruct with an NVIDIA H200 illustrates this connection:

    Quantization / Kernel Perplexity (Lower is Better) Pass@1 (Higher is Better) Throughput (tok/s) TTFT (ms)
    FP16 baseline 6.56 56.1% 461 tok/s 57.7 ms
    AWQ 6.84 51.8% 68 tok/s 277.8 ms
    GPTQ 6.90 46.3% 277 tok/s 107.1 ms
    Marlin-GPTQ 6.97 45.7% 712 tok/s 51.9 ms
    Marlin-AWQ 6.84 51.8% 741 tok/s 73.5 ms
    GGUF Q4_K_M 6.74 51.8% 93 tok/s 958.0 ms
    bitsandbytes 6.67 51.8% 168 tok/s 135.3 ms

    Do not copy those numbers into your capacity plan. They are specific to one model, one hardware class, and one benchmark environment. The lesson is more durable: AWQ and GPTQ are offline mathematical algorithms, not runtime engines. The execution kernel path decides whether a 4-bit model runs fast or just takes up less space.

    For instance, the benchmark shows a massive speedup when switching from basic AWQ (68 tok/s) to Marlin-AWQ (741 tok/s). While both use the exact same 4-bit weights, the standard AWQ kernel dequantizes weights slowly and runs standard math. The Marlin kernel is a highly optimized CUDA program that fuses weight dequantization and matrix multiplication into a single assembly-level pipeline, maximizing GPU register utilization and memory bandwidth.

    Use vLLM's own benchmark commands to measure your target stack:

    vllm bench serve \
  --model ./outputs/Qwen2.5-32B-Instruct-AWQ-W4A16 \
  --dataset-name sharegpt \
  --num-prompts 200 \
  --input-len 1024 \
  --output-len 256

    Measure the baseline BF16 model, the quantized checkpoint, and any kernel-specific variants under the same prompt mix. Track throughput, TTFT, inter-token latency, memory consumption, and downstream task quality.

    7. The quantization algorithm menu

    Quantization algorithms use different mathematical approaches to choose the scales, zero points, and integer values.

    Algorithm Common Format Primary Objective Key Trade-off Status
    GPTQ W4A16 Minimizes layer-wise reconstruction error using second-order (Hessian) math. High calibration time and cost. Production standard for W4A16 GPU serving.
    AWQ W4A16 / W4A8 Protects the 1% most important channels (outliers) based on activation scales. Needs calibration data; requires fused kernels. Production standard for W4A16 GPU serving.
    SmoothQuant W8A8 Scales down activation spikes and scales up weights to make both easier to fit in 8-bit. Needs scaling-strength tuning per model family. Practical standard for high-throughput INT8.
    QuaRot / SpinQuant W4A4 / W4A8 Mathematically rotates weights and activations to remove outliers entirely. Requires complex online rotation transforms at runtime. Advanced low-bit research and production frontier.
    HQQ W4A16 / W2A16 Fast, data-free optimization that quantizes weights without calibration datasets. Can trail AWQ/GPTQ quality at very low bit-widths (e.g., 2-bit). Good for fast local quantization.
    QLoRA (NF4) NF4 NormalFloat representation designed for normally distributed weights. Great for training adapters, but slow for production serving. QLoRA default for parameter-efficient fine-tuning.
    llama.cpp K-quants / IQ-quants Mixed low-bit Groups weights into blocks, using mixed bit-widths for sensitive layers. Not designed for high-concurrency CUDA cloud serving. Desktop, edge, and llama.cpp default.

    Modern Toolchain and Tool Deprecations (2026)

    The model compression software landscape has consolidated significantly. Legacy quantization packages such as AutoGPTQ (archived in April 2025) and AutoAWQ (deprecated in late 2025) are no longer actively maintained. In their place, two primary toolchains have become standard:

    • llm-compressor (developed by Neural Magic / Red Hat / vLLM): The primary engine used to produce compressed-tensors checkpoints (for W4A16, FP8 W8A8, and other configurations) that are natively consumed by serving backends like vLLM.
    • GPTQModel: The active successor to AutoGPTQ, providing updated Marlin/Machete kernels, MoE VRAM-saving strategies, and native disk-offload support.

    Adjacent Compression Techniques

    Sparsity and distillation are adjacent optimization techniques that can combine with quantization but require separate validation loops.

    • 2:4 Structured Sparsity: A hardware-accelerated compression format where exactly two out of every four weights in a matrix are forced to zero. NVIDIA Ampere and newer GPUs can skip computing these zero values, doubling math throughput. Algorithms like SparseGPT and Wanda enforce this pattern after training, while newer approaches like SlideSparse offer flexible sparse patterns.
    • Distillation: Replacing a large model with a smaller "student" model trained to mimic the large "teacher." While distillation can beat quantization for narrow, specialized tasks, it is expensive and requires custom training loops and large datasets.

    8. Serving memory: weights, KV cache, and runtimes

    A common capacity planning mistake is sizing only the quantized model weights. Total active serving memory includes several components:

    \[\text{VRAM}_{\text{serve}} \approx \text{Quantized Weights} + \text{KV Cache} + \text{Runtime Activations} + \text{Engine Overhead}\]

    Offline Quantization Memory

    Quantization tools (such as llm-compressor) process models sequentially, one transformer block at a time. The tool loads a single layer into GPU memory, runs calibration and Hessian-derived math, writes the compressed block back to disk, and moves on.

    Because of this block-by-block process, peak memory during quantization is small:

    \[\text{GPU Peak} \approx \text{Largest Layer (BF16)} + \text{Calibration Activations} + \text{Method Buffers}\]

    The host system still requires enough CPU RAM and disk storage to stream the unquantized weights.

    Online Serving Memory

    At serve time, the entire model must stay resident in GPU memory along with the Key-Value (KV) cache. For long-context generation or high batch sizes, the KV cache can exceed the size of the model weights:

    \[\text{KV Cache (Bytes)} = 2 \times L \times H_{\text{kv}} \times D \times S_{\text{ctx}} \times B_{\text{batch}} \times \text{BytesPerValue}\]

    To understand this calculation:

    • \(L\): The number of layers in the model.
    • \(H_{\text{kv}}\): The number of key-value attention heads. Modern models use Grouped-Query Attention (GQA), where multiple query heads share a single KV head, reducing the cache footprint by 8x.
    • \(D\): The dimension of each head (typically 128 or 256).
    • \(S_{\text{ctx}}\): The context length (input prompt length + generated output tokens).
    • \(B_{\text{batch}}\): The active concurrent serving batch size.
    • \(\text{BytesPerValue}\): The size of the cache precision (2 bytes for BF16/FP16, 1 byte for FP8/INT8).

    For a 70B model serving a 32k context, the KV cache can easily dominate VRAM. In these scenarios, quantizing the KV cache to FP8 using systems like PagedAttention is more effective for capacity than squeezing weight bits further.

    9. Sizing, selection, and hardware matrix

    Use the decision flowchart to map your hardware constraints, then run serving benchmarks to finalize your configuration.

    Weights-Only Footprint

    To estimate weight footprint sizes:

    \[\text{Weight Size (GB)} \approx \frac{\text{Parameter Count (B)} \times \text{Bits}}{8}\]
    Model Size BF16 Weights FP8/INT8 Weights INT4 Weights
    7B / 8B ~14 - 16 GB ~7 - 8 GB ~3.5 - 4 GB
    14B ~28 GB ~14 GB ~7 GB
    32B / 34B ~64 - 68 GB ~32 - 34 GB ~16 - 17 GB
    70B ~140 GB ~70 GB ~35 GB
    109B MoE ~218 GB (total) ~109 GB ~55 GB

    Note: For Mixture of Experts (MoE) models, the active parameters per token are smaller, but the full model weights must still reside in VRAM unless offloaded.

    Hardware Compatibility and Architectures

    Hardware architectures differ fundamentally in their low-precision execution capabilities:

    • CPU: Lacks massive parallel execution pipelines. GGUF formats rely on CPU vector math instruction sets (AVX-512, AMX) via llama.cpp to achieve local speedups.
    • Apple Silicon: Uses a unified memory architecture (UMA) where the CPU and GPU share the same high-bandwidth RAM. This enables running large models (30B+) locally without dedicated multi-GPU server setups.
    • NVIDIA Ampere (A100, RTX 3090): Supports hardware INT8 Tensor Cores. It does not support native FP8. Standard serving choices are INT4 weight-only (W4A16) or static INT8.
    • NVIDIA Ada (RTX 4090, L4) & NVIDIA Hopper (H100, H200): Support native FP8 Tensor Cores (both E4M3 and E5M2 formats), enabling high-throughput W8A8 serving.
    • NVIDIA Blackwell: Introduces dedicated hardware support for 4-bit floating point (NVFP4) and microscaling blocks, further reducing weight-activation footprint.

    10. Calibration and evaluation gate

    Quantization is not complete when the model loads. It is complete when the compressed model passes your quality promotion gate for the target workload.

    Calibration Checklist

    • Use representative datasets: Gather active prompts from the target workload, including RAG traces, SQL queries, agent chat histories, and code generation tasks.
    • Match sequence lengths: Calibrate with sequence lengths that match production serving, not just short single-turn examples.
    • Protect sensitive layers: Keep word embeddings (embed_tokens) and the output projection head (lm_head) in higher precision (16-bit) to prevent generation instability.
    • Use sufficient samples: Use between 128 and 512 representative prompts to compute stable activation scales.
    • Redact data: Filter out private user data or secrets before using internal production logs for calibration.

    Evaluation Checklist

    • Perplexity: Measure on standard datasets (like WikiText) to check for general language degeneration.
    • Downstream tasks & reasoning degradation: Run standard benchmarks targeting your domain (e.g., HumanEval for coding, MMLU for knowledge, AIME/MATH-500 for reasoning). Watch out for reasoning-heavy architectures (like DeepSeek-R1 or QwQ); evaluations show they are highly sensitive to sub-4-bit or unrotated W4A4 quantization, which can cause reasoning accuracy to collapse by 7% or more.
    • Format compliance: Verify that the model still follows JSON schemas, markdown formats, and system instructions correctly.
    • Long-context checks: Run needle-in-a-haystack tasks to ensure KV cache quantization has not broken context retrieval.
    • Performance benchmarks: Measure throughput (tokens/sec), TTFT, and maximum concurrent batch capacity under simulated load.

    11. Practical workflow with the companion repository

    The companion repository at slavadubrov/model-compression-demo provides a lightweight implementation for planning and estimating quantization requirements. It uses uv for dependency management.

    Clone the repository and list the supported algorithms:

    git clone https://github.com/slavadubrov/model-compression-demo.git
cd model-compression-demo
uv run python demo.py list-algorithms

    Plan VRAM requirements for a specific hardware target and batch size:

    uv run python demo.py plan \
  --model-preset llama3-8b \
  --goal fit-memory \
  --hardware ampere \
  --context 4096 \
  --concurrency 4

    Estimate model weight and KV cache footprint:

    uv run python demo.py estimate \
  --model-preset llama3-8b \
  --scheme w4a16 \
  --context 8192 \
  --concurrency 8

    Generate a reference quantization script recipe:

    uv run python demo.py recipe --algorithm gptq-w4a16

    Dry-run a calibration job to test dataset formats before running full GPU processing:

    uv run python demo.py quantize \

    12. Diffusion and image models are a separate path

    Diffusion and Diffusion Transformer (DiT) models process data differently from autoregressive LLMs. Instead of generating text token-by-token, they run multiple denoising steps. Activation distributions shift significantly at different steps of the denoising process, meaning naive quantization can cause severe visual artifacts.

    Apply these rules when quantizing image pipelines:

    1. Keep the VAE (Variational Autoencoder) in 16-bit: The VAE translates latent noise back into actual pixels. Even minor quantization errors here will cause color banding, noise, or broken image structures.
    2. Quantize the DiT or U-Net backbone first: This transformer backbone contains the majority of model parameters, offering the largest memory and compute savings when compressed.
    3. Isolate text encoders: Quantizing text encoders (like T5-XXL) separately prevents prompt alignment and text rendering from degrading.
    4. Use specialized algorithms: Apply specialized diffusion methods like SVDQuant (which absorbs outlier activations into low-rank components) to avoid visual degradation.
    5. Evaluate visually: Measure visual quality (using prompts checking text rendering, color balance, and prompt adherence) alongside latency and VRAM. Text perplexity metrics do not apply here.

    13. Default serving stack

    For Enterprise Server Deployments (LLMs)

    1. Establish a solid BF16 baseline in your target serving engine (e.g., vLLM or TensorRT-LLM).
    2. If high throughput is your goal and you run Ada, Hopper, or Blackwell GPUs, implement FP8 (W8A8).
    3. If memory limits prevent the model from fitting in VRAM, implement AWQ or GPTQ (W4A16) using Marlin kernels.
    4. If serving long contexts or high concurrency, enable FP8 KV cache and verify retrieval quality.
    5. Only promote models to production when both quality evaluation gates and throughput benchmarks pass.

    For Local and Edge Inference

    1. Start with GGUF Q4_K_M or Q5_K_M mixed-precision weights as your default choice.
    2. Upgrade to GGUF Q8_0 if memory allows and maximum reasoning quality is required.
    3. Avoid going below 4-bit quantization unless extreme RAM constraints make it mandatory.

    For Model Fine-Tuning

    1. Use NF4 (QLoRA) to train parameter-efficient adapter layers on single GPUs.
    2. Evaluate the adapter inside your targeted application framework before merging.
    3. Merge the adapter back into the base model weights, then quantize the merged model into GGUF, AWQ, or FP8 for final deployment.

    Key Takeaways

    1. Quantization is controlled rounding plus runtime execution: Bit-width alone doesn't determine quality or speed.
    2. Pick the bottleneck first: weight memory, activation compute, KV cache, local runtime, fine-tuning memory, and image-pipeline memory lead to different choices.
    3. W4A16 is a memory tool: It helps fit weights and reduce decode traffic, but it doesn't automatically speed up compute-bound batches.
    4. FP8 is a serving tool: It works best when hardware and runtime kernels support low-precision tensor-core execution.
    5. Kernels can dominate benchmarks: Marlin, BitBLAS, and runtime-specific paths can matter as much as GPTQ or AWQ.
    6. KV cache is a first-class memory term: Long context and concurrency can make cache memory larger than quantized weights.
    7. Calibration data must look like production: Generic text can be useful for smoke tests, but not for final promotion.
    8. Diffusion quantization is different: Evaluate visual quality and component sensitivity, not only model size.
