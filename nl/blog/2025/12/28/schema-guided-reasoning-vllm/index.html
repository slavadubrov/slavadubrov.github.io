    Schema-gestuurd redeneren op vLLM: Gestructureerde uitvoer met xgrammar en Pydantic

    Een herprobeerloop is eigenlijk een bekentenis. Het zegt: ik verwachtte dat het model geldig JSON zou teruggeven, maar dat gebeurde niet, dus probeer ik het opnieuw. De meeste code voor LLM-agenten waarmee ik heb gewerkt, is sterk afhankelijk van herproberingen, waardoor de JSON-uitvoer nog steeds vaak corrupt is, zodat er soms alarms worden ingesteld voor dit probleem.

    Schema-Gestuurd Redeneren (SGR) omzeilt dit herprobeerproces door het schema op tokenniveau af te dwingen. Je definieert de logische structuur van het redeneren als een Pydantic-schema, en de inferentie-engine filtert al die tokens weg die hiermee in strijd zijn voordat er wordt gesampled. Hierdoor is de uitvoer van nature geldig, en niet alleen dankzij herproberingen.

    TL;DR. SGR maakt gebruik van beperkte decodering om de uitvoer van een LLM te binden aan een door u gecontroleerd Pydantic-schema. In combinatie met vLLM’s xgrammar-backend levert dit altijd geldig JSON op, met een verwaarloosbaar latency-overhead.

    Wat is schema-gestuurd redeneren?

    Schema-gestuurd redeneren is een techniek die Rinat Abdullin Opgesteld in 2024. In plaats van het model de vrije hand te geven om tekst af te maken (wat onconsistent of ambigu kan zijn), geeft u het een strikt template dat het volgende definiert:

    • welke stappen het moet doorlopen, zodat het de analyse niet kan overslaan
    • de volgorde van die stappen, zodat het redeneren van de gegevens naar de beslissing verloopt
    • waar het zijn aandacht op moet richten, zodat de diepte van het analyseren precies daar wordt toegepast waar het nodig is

    Zie het als een cognitieve controlelijst die het model moet volgen.

    Waarom SGR belangrijk is

    Wanneer je een schema definieert met velden zoals churn_analysis, margin_math, en max_discount_percent, het model moet ze in volgorde invullen. Het kan niet direct overgaan tot de beslissing over de korting zonder eerst de analyse op te stellen.

    Dit levert het volgende op:

    • reproduceerbaar redeneren bij herhaalde uitvoeringen
    • controleerbare uitvoerwaarden waarbij elke stap kan worden geïnspecteerd
    • tussentijdse velden die kunnen worden beoordeeld tegen een testdataset
    • kleinere modellen die bruikbaar worden, aangezien het schema bepaalt wat het model anders zou moeten leren
    • een verbetering van de nauwkeurigheid met 5-10% vaak gerapporteerd bij productiewerkloads

    Prompt engineering: semantische overtuigingskracht

    Please analyze the customer data and output your response as valid JSON
with the following structure: {"discount": <number>, "reason": <string>}
Be careful with the formatting!

    Je hoopt dat het begrip van het model voor “output JSON” zwaarder weegt dan zijn neiging om conversatief te zijn. Een modelupdate, een verandering in de temperature-ofactor of een ander few-shot voorbeeld kan je parser beschadigen.

    Chain of Thought: beter redeneren, maar nog steeds hetzelfde structuurprobleem

    Let's think step by step:
1. First, I'll analyze the customer's churn risk...
2. Then I'll calculate the margin...
3. Therefore, I recommend a 15% discount.

    CoT verbetert de nauwkeurigheid van het redeneren, maar verslechtert de structuur. Het resultaat is onvoorspelbaar proza dat bijna onmogelijk betrouwbaar te parsen is. Meestal moet je een tweede oproep aan de LLM maken om de gestructureerde gegevens te extraheren.

    SGR: gestructureerde keten van gedachten

    SGR behoudt het inzicht van CoT dat tussentijdse redenering de nauwkeurigheid verbetert. Het formaliseert slechts de stappen:

    class PricingLogic(BaseModel):
    # 1. Data Analysis (must complete before decision)
    churn_analysis: str = Field(..., description="Analyze churn_probability")
    financial_analysis: str = Field(..., description="Analyze cart_value and margin")

    # 2. Math Enforcement (explicit calculation)
    margin_math: str = Field(..., description="Calculate: 'Cart $X * Y% = $Z'")

    # 3. Decision Constraint (bounded by prior analysis)
    max_discount_percent: float = Field(..., description="Max allowed discount")

    # 4. Final Output
    offer_code: str
    customer_message: str

    Het model kan geen uitvoer genereren. max_discount_percent totdat churn_analysis, financial_analysis, en margin_math zijn gevuld. Het schema voert de volgorde van redenering af.

    SGR-patronen

    SGR beschikt over drie kernpatronen die samen grotere workflows vormen.

    1. Cascade: sequentiële redeneringsstappen

    Cascade voert een bepaalde volgorde voor het redeneren op. Elk veld moet eerst worden ingevuld voordat het volgende kan worden verwerkt.

    from pydantic import BaseModel
from typing import Literal, Annotated
from annotated_types import Ge, Le

class CandidateEvaluation(BaseModel):
    """Evaluate a job candidate with enforced reasoning order."""

    # Step 1: Summarize (forces context awareness)
    brief_candidate_summary: str

    # Step 2: Rate (bounded integer)
    rate_skill_match: Annotated[int, Ge(1), Le(10)]

    # Step 3: Decide (constrained choices)
    final_recommendation: Literal["hire", "reject", "hold"]

    Goede toepassingen: kandidaatbeoordeling, documentclassificatie, conformiteitsanalyse, medische diagnostiek. Het model moet tekst genereren. brief_candidate_summary Voordat het kan beoordelen, moet het eerst beoordelen, en voordat het kan aanbevelen, moet het eerst beoordelen. Er is geen kortere weg.

    2. Routing: een semantische switch-statement

    Routing zorgt ervoor dat het model zich voor één pad uit een reeks opties kiest, wat wordt gerealiseerd met Union typen.

    from pydantic import BaseModel
from typing import Literal, Union

class FeatureLookup(BaseModel):
    """Route to database lookup."""
    rationale: str
    tool_name: Literal["fetch_user_features"] = "fetch_user_features"
    user_id: str

class GeneralResponse(BaseModel):
    """Standard response for non-pricing queries."""
    tool_name: Literal["respond"] = "respond"
    content: str

class RouterSchema(BaseModel):
    """The model must pick exactly ONE branch."""
    action: Union[FeatureLookup, GeneralResponse]

    Goede toepassingen: intentclassificatie, keuze van hulpmiddelen, triaging van ondersteuning, dispatch van meerdere agenten.

    The Literal discriminator (tool_name) zorgt ervoor dat het model één specifieke tak kiest en alleen de velden invult die die tak nodig heeft.

    3. Cyclus: herhaald redeneren met lijsten

    Een cyclus dwingt het model om meerdere elementen te genereren, waarbij er grenzen zijn aan het aantal.

    from pydantic import BaseModel
from typing import List, Literal, Annotated
from annotated_types import MinLen, MaxLen

class RiskFactor(BaseModel):
    explanation: str
    severity: Literal["low", "medium", "high"]

class RiskAssessment(BaseModel):
    """Generate 2-4 risk factors."""
    factors: Annotated[List[RiskFactor], MinLen(2), MaxLen(4)]

    Goede toepassingen: risicobeoordeling, issue-extractie, gelijktijdige aanroepen van tools, meestappige planning. MinLen en MaxLen De grenzen vereisen ten minste 2 en ten hoogste 4 elementen. In combinatie met Routing is dit de manier om een batch met vaste breedte van toolaanroepen te verzenden.

    SGR in werking stellen: beperkte decodering

    De bovenstaande patronen zijn slechts Pydantic-schemas. Wat ze bindend maakt, is beperkte decodering (ook wel Structured Output genoemd).

    Bepaalde decodering wijzigt de stap van tokengeneratie. In plaats van het model vrij te laten kiezen uit zijn woordenschat, past de engine een grammaticamasker toe dat tokens blokkeert die het schema zouden schenden. Dit vindt plaats in de inference engine, niet in uw applicatiecode.

    [!TIP] SGR heeft geen “reasoning models” zoals o1 of DeepSeek-R1 nodig. Het werkt prima met instructietuned modellen, en vooral goed met modellen die zijn geïnd distilleerd uit reasoning-modellen.

    Waarom dit artikel zich richt op vLLM en xgrammar

    Enkele redenen:

    • vLLM is de meest wijdverspreide open-source inference-engine voor LLM’s, waardoor wat je hier bouwt gemakkelijk kan worden geport.
    • xgrammar is geïmplementeerd in C++ en zorgt voor een verwaarloosbare vertraging.
    • De API van vLLM is compatibel met OpenAI, waardoor de migratie van cloudproviders goedkoop blijft.
    • xgrammar kan complexe, genestelde schema’s, unions en recursieve structuren verwerken.

    In het volgende gedeelte wordt uitgelegd hoe xgrammar op tokenniveau daadwerkelijk een schema oplegt.

    Hoe xgrammar schema’s afdraagt

    Dit onderdeel is het belangrijk om nauwkeurig te begrijpen, aangezien het de manier waarop je SGR-workflows debugt en optimaliseert, beïnvloedt.

    Waar het maskeren plaatsvindt

    xgrammar wijzigt de uitvoerlogits na de forward pass van het model en vóór het sampling. Het verandert het model zelf niet; het filtreert welke tokens geselecteerd mogen worden.

    Een standaard inference-loop ziet er als volgt uit:

    1. Input tokens → GPU Forward Pass → Logits (probability scores for all ~128K tokens)
2. Logits → Sampling (temperature, top-p, etc.) → Next Token
3. Repeat until done

    xgrammar glipt weg tussen stap 1 en stap 2:

    1. Input tokens → GPU Forward Pass → Raw Logits
2. Raw Logits → xgrammar Logits Processor → Masked Logits
3. Masked Logits → Sampling → Next Token (guaranteed valid)
4. Repeat until done

    Het model berekent nog steeds zijn volledige waarschijnheidsverdeling op de GPU. Vervolgens draait xgrammar op de CPU en past er een bitmask op die logits toe voordat er wordt gesampled. Ongeldige tokens krijgen hun logits ingesteld op -∞, waardoor hun kans na de softmax precies 0 wordt.

    Twee fasen

    xgrammar verdeelt het werk in een compilatietijdfase en een uitvoeringsfase, en dat is wat ervoor zorgt dat het snel werkt.

    Fase 1: compilatie van de grammatica, één keer per schema

    # This happens once per schema
tokenizer_info = xgr.TokenizerInfo.from_huggingface(tokenizer)
grammar_compiler = xgr.GrammarCompiler(tokenizer_info)
compiled_grammar = grammar_compiler.compile_json_schema(schema_json)

    Tijdens de compilatie doet xgrammar het volgende:

    1. Zet het JSON Schema om in een contextvrije grammatica.
    2. Maakt een pushdownautomaat (PDA) aan, een toestandsmachine met een stapel, zodat deze geneste structuren kan verwerken. {"a": {"b": {"c": ...}}}.
    3. Er wordt van tevoren berekend welke tokens geldig zijn op elke grammaticale positie. Het resultaat vormt de “adaptive token mask cache”.
    4. Tokens worden gecategoriseerd als “context-onafhankelijk” (te cachen) of “context-afhankelijk” (moeten tijdens uitvoering worden gecontroleerd tegen de stackstatus).

    [!NOTE] Ongeveer 99% van de tokens blijkt context-onafhankelijk te zijn en wordt daardoor gecacheerd.Artikel over XGrammar). De meeste validatiecontroles tijdens de uitvoering zijn slechts cacheopvragen, waardoor xgrammar snel is.

    Fase 2: generatie van runtime-maskers, per token

    Bij elke generatiestap:

    1. De GrammarMatcher houdt de huidige positie in de grammatica bij.
    2. Het zoekt het van tevoren berekende masker op voor contextonafhankelijke tokens.
    3. Het voert de PDA uit om de overgebleven contextafhankelijke tokens te controleren.
    4. Het combineert deze tot een eindmasker en past dit toe op de logits.

    Waarom een pushdownautomaat en niet een regex?

    Vanwege nestering. Een reguliere expressie (een eindige staatsmachine) kan structuren zoals deze niet betrouwbaar matchen:

    { "user": { "profile": { "settings": { "theme": "dark" } } } }

    Het moeilijke deel zijn de sluitende haakjes. }}}: je moet onthouden hoeveel je hebt geopend. Een pushdownautomaat beschikt over een stapel die dit bijhoudt, zodat hij een willekeurige nestdiepte kan verwerken. Dat is ook de reden waarom xgrammar Unietypen, genest objecten en recursieve schema’s kan doorvoeren, terwijl regex-gebaseerde benaderingen hier niet toe in staat zijn.

    Een concreet voorbeeld: het genereren van een float-veld

    Wanneer het model aan het genereren is "max_discount_percent":, xgrammar weet uit het schema dat een float komt hierna. De masker:

    • staat toe (waarschijnlijkheid onveranderd): 0, 1, 2, ..., 9, ., -
    • blokken (waarschijnlijkheid ingesteld op 0): ", {, [, true, false, nullen de rest van het vocabulaire van 128K+

    De forward pass heeft mogelijk een hoge waarschijnlijkheid toegewezen aan het woord. "fifteen". Na het toepassen van de masker door xgrammar heeft dat token een waarschijnlijkheid van 0. Het model moet cijfers uitvoeren.

    Waarom “bijna-nul overhead”

    Drie redenen:

    1. Parallelle uitvoering. De berekening van het masker op de CPU overlapt met de volgende forward pass op de GPU. Terwijl de GPU de logits voor token N+1 berekent, berekent de CPU het masker voor token N.
    2. Caching. Het grootste deel van de validatiewerkzaamheden vindt plaats tijdens de compilatie. Tijdens de uitvoering gaat het voornamelijk om cacheopvragen.
    3. C++-implementatie. De belangrijkste uitvoeringsweg is in C++, niet in Python, en het masker wordt rechtstreeks toegepast op de logits.

    In benchmarks vertoont xgrammar een verwaarloosbare overhead, en gestructureerde generatie kan soms sneller zijn dan onbeperkte generatie, omdat het beperkte woordenschat het sampling goedkoper maakt.

    Stap 1: definieer de schema’s

    # sgr/models/schemas.py
from pydantic import BaseModel, Field
from typing import Literal, Union


# --- Phase 1: Routing (Union for branching) ---
class FeatureLookup(BaseModel):
    """Route to DB lookup if pricing context is needed."""
    rationale: str
    tool_name: Literal["fetch_user_features"] = "fetch_user_features"
    user_id: str


class GeneralResponse(BaseModel):
    """Standard response for non-pricing queries."""
    tool_name: Literal["respond"] = "respond"
    content: str


class RouterSchema(BaseModel):
    action: Union[FeatureLookup, GeneralResponse]


# --- Phase 2: Pricing Logic (Cascade for sequential reasoning) ---
class PricingLogic(BaseModel):
    """
    Strict reasoning topology for dynamic pricing.
    Fields are ordered to enforce the analysis→decision flow.
    """
    # 1. Data Analysis (Reflection)
    churn_analysis: str = Field(...,
        description="Analyze churn_probability (High > 0.7).")
    financial_analysis: str = Field(...,
        description="Analyze cart_value and profit_margin.")

    # 2. Hard Math Enforcement
    margin_math: str = Field(...,
        description="Calculate absolute profit: 'Cart $200 * 0.20 Margin = $40'.")

    # 3. The Decision Constraint
    max_discount_percent: float = Field(...,
        description="Max allowed discount %. NEVER exceed margin.")

    # 4. Final Output
    offer_code: str = Field(..., description="Generated code (e.g. SAVE20).")
    customer_message: str = Field(..., description="The final polite offer text.")

    Stap 2: een LLM-client die xgrammar inschakelt

    # sgr/utils/llm_client.py
from openai import OpenAI
from pydantic import BaseModel
from typing import TypeVar
import json

T = TypeVar("T", bound=BaseModel)


class LLMClient:
    """Wrapper for vLLM with xgrammar-enforced structured generation."""

    def __init__(self, base_url: str = "http://localhost:8000/v1"):
        self.client = OpenAI(base_url=base_url, api_key="EMPTY")
        self.model = self._get_available_model()

    def _get_available_model(self) -> str:
        """Auto-detect the model running on vLLM server."""
        try:
            models = self.client.models.list()
            if models.data:
                return models.data[0].id
        except Exception:
            pass
        return "Qwen/Qwen2.5-7B-Instruct"

    def run_sgr(self, messages: list[dict], schema_class: type[T]) -> T:
        """Run inference with Schema-Guided Response constraints.

        Uses vLLM's guided_json with xgrammar backend to enforce
        strict schema constraints at the token generation level.
        """
        schema_dict = schema_class.model_json_schema()

        # Enhance system message with schema for model guidance
        enhanced_messages = messages.copy()
        if enhanced_messages and enhanced_messages[0]["role"] == "system":
            schema_json = json.dumps(schema_dict, indent=2)
            enhanced_messages[0] = {
                "role": "system",
                "content": (
                    enhanced_messages[0]["content"]
                    + f"\n\nRespond with JSON matching this schema:\n{schema_json}"
                ),
            }

        # vLLM's guided_json with the xgrammar backend
        completion = self.client.chat.completions.create(
            model=self.model,
            messages=enhanced_messages,
            temperature=0.1,  # Low temp for deterministic reasoning
            extra_body={
                "guided_json": schema_dict,  # Pydantic schema as dict
                "guided_decoding_backend": "xgrammar",  # Hardware-enforced
            },
        )

        raw_response = completion.choices[0].message.content
        return schema_class.model_validate_json(raw_response)

    [!NOTE] > guided_json accepteert een JSON Schema-dictie. Met guided_decoding_backend: "xgrammar", de LLM kan alleen tokens genereren die geldig JSON vormen dat overeenkomt met uw schema.

    Stap 3: het agent organiseren

    # sgr/agent.py
from .models.schemas import PricingLogic, RouterSchema
from .prompts.routing import build_routing_prompt
from .prompts.pricing import build_pricing_context_prompt, ASSISTANT_FETCH_MESSAGE
from .store.hybrid_store import HybridFeatureStore
from .utils.llm_client import LLMClient


def pricing_agent(user_query: str, user_id: str) -> str:
    """Process a pricing query with three-phase SGR workflow."""

    llm = LLMClient()
    feature_store = HybridFeatureStore()

    # Build conversation history
    history = [
        {"role": "system", "content": build_routing_prompt(user_id)},
        {"role": "user", "content": user_query},
    ]

    # --- Phase 1: Routing (Uses RouterSchema) ---
    print(f"🤖 Processing: '{user_query}' for {user_id}")
    decision = llm.run_sgr(history, RouterSchema)
    print(f"📍 Routing decision: {decision.action.tool_name}")

    if decision.action.tool_name == "respond":
        return decision.action.content

    # --- Phase 2: Context Retrieval ---
    if decision.action.tool_name == "fetch_user_features":
        print(f"🔍 Fetching features for {user_id}...")
        context = feature_store.get_user_context(user_id)

        if not context:
            return "Error: User profile not found."

        print(f"   [Data] LTV: ${context.get('user_ltv')} | "
              f"Margin: {context.get('cart_profit_margin', 0) * 100}%")

        # Inject context into conversation
        history.append({"role": "assistant", "content": ASSISTANT_FETCH_MESSAGE})
        history.append({
            "role": "user",
            "content": build_pricing_context_prompt(
                churn_prob=context.get("churn_probability", 0.5),
                cart_val=context.get("current_cart_value", 100),
                margin=context.get("cart_profit_margin", 0.2),
                user_ltv=context.get("user_ltv", 0),
            ),
        })

        # --- Phase 3: SGR Logic Execution (Uses PricingLogic) ---
        print("🧠 Calculating Offer (Schema Enforced)...")
        offer = llm.run_sgr(history, PricingLogic)

        # Audit log — the SGR benefit: explicit reasoning traces
        print(f"   [Audit] Math: {offer.margin_math}")
        print(f"   [Audit] Max Allowed: {offer.max_discount_percent}%")

        return offer.customer_message

    return "I'm sorry, I couldn't process your request."


if __name__ == "__main__":
    response = pricing_agent("I want a discount or I'm leaving!", "user_102")
    print(f"\n💬 Final Reply: {response}")

    Beste praktijken

    Schemaontwerp

    1. Sorteer de velden op basis van de redeneringsstroom. Analysevelden gaan voor beslissingsvelden.
    2. Schrijf beschrijvende tekst. Field Beschrijvingen. Zij sturen de aandacht van het model net zo goed als de veldnaam dat doet.
    3. Beperk met Literal en Annotated. Gebruik Literal["a", "b"] voor enums en Annotated[int, Ge(1), Le(10)] voor de grenzen.
    4. Houd de schema’s gefocust. Eén schema per redeneringsfase, en combineer ze vervolgens met meerdere oproepen.

    vLLM-configuratie

    1. Gebruik een lage temperatuur (0,1-0,3) voor deterministisch redeneren.
    2. Laat xgrammar de structuur verzorgen. Probeer dit niet te ondermijnen met opmaak instructies in de prompt.
    3. Houd toezicht op het tokengebruik. SGR maakt doorgaans minder tokens gebruik dan CoT, omdat er geen uitgebreide beschrijvingen nodig zijn.

    Productieoverwegingen

    1. Versioneer je schema’s op dezelfde manier als je APIs versioneert.
    2. Zelfs met SGR vereisen netwerk- en serverfouten nog steeds een soepele afhandeling.
    3. Log de ruwe SGR-uitvoer voor conformiteit en debuggen.
    4. Test met randgevallen zodat het schema aan de grenzen blijft werken.

    Conclusie

    SGR zorgt ervoor dat je van “werkt in een demo” overgaat naar “werkt in productie”. Jij definieert de redeneringstopologie in Pydantic, laat xgrammar deze tijdens het decoderen handhaven, en het resultaat is:

    • altijd geldig, zonder herhalingscycli of parsingfouten
    • controleerbaar op veldniveau
    • bruikbaar met kleinere modellen, omdat ze zich niet langer zelf hoeven te richten op het formaat
    • goedkoper in uitvoering, omdat er minder tokens, herhalingen en kleinere modellen worden gebruikt

    The sgr-discount-beheerder De demo verbindt elk codevoorbeeld uit deze post met een echte vLLM-server. Kloon het project en begin de schema’s aan te passen aan je eigen werkwijze.

    Belangrijkste conclusies

    1. Schema-gestuurd redeneren maakt de topologie van het redeneringsproces expliciet, in plaats van te hopen dat het model de tekstuele instructies opvolgt.
    2. Beperkt decoderen voorkomt ongeldig JSON tijdens de generatie, wat efficiënter is dan eerst validatie uit te voeren en daarna opnieuw te proberen.
    3. Plaats analysevelden vóór beslissingsvelden wanneer het schema vereist dat er eerst wordt geredeneerd voordat er output wordt gegeven.
    4. Gebruik SGR wanneer de downstream-code afhankelijk is van een specifieke structuur, en niet wanneer het eindresultaat bestaat uit vrije tekst.
