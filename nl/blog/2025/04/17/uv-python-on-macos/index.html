UV op macOS: Beheer van Python-versies, projecten en tools

Snel starten

# Install uv
brew install uv

# For new projects (modern workflow)
uv init                # create project structure
uv add pandas numpy    # add dependencies
uv run train.py        # run your script

# For existing projects (legacy workflow)
uv venv                             # create virtual environment
uv pip install -r requirements.txt  # install dependencies
uv run train.py                     # run your script

# Run tools without installing them
uvx ruff check .       # run linter
uvx black .            # run formatter

# Run a single-file script with its own dependencies (no project needed)
uv run fetch.py        # uv reads the inline deps and runs it

Kort samengevat: Gebruik uv als standaard Python-project- en hulpprogrammabeheerder op macOS. Houd Homebrew voor systeempakketten, installeer Python-versies via uv python, pin afhankelijkheden vast in pyproject.toml, en gebruik uvx voor eenmalige CLI-hulpprogramma’s.

Waarom uv gebruiken voor Python op macOS?

Als je al een tijdje Python gebruikt, houd je waarschijnlijk meerdere zaken tegelijk in de gaten. pip, virtualenv, pip-tools, pyenv, en poetry afhankelijk van het project. uv Het doet precies hetzelfde als al die andere oplossingen, maar met slechts één binary. Het is geschreven in Rust en op mijn machine installeert het pakketten ongeveer 10-100 keer sneller dan de oude stack. De grootste voordelen voor mij zijn dat ik niet meer heen en weer hoef te schakelen tussen verschillende tools tijdens een taak.

Het vervangt pip en pip-tools voor pakketbeheer, pyenv voor het installeren van Python-versies, virtualenv/venv voor omgevingen, pipx voor het uitvoeren van tools, en poetry of pdm voor projectwerkflows.

uv installeren

De eenvoudigste manier om te installeren uv Op macOS gebeurt dit via Homebrew:

brew install uv

uv Detecteert automatisch de architectuur van je Mac (Apple Silicon of Intel), zodat er geen extra configuratie nodig is.

Om het bijgewerkt te houden:

brew upgrade uv
# OR
uv self update

Kernconcepten

uv omvat drie gebruiksscenario’s:

  1. Projecten — het bouwen van een applicatie of bibliotheek met afhankelijkheden.
  2. Scripts — het uitvoeren van een eenvoudig Python-script met ingebouwde afhankelijkheden.
  3. Hulpprogramma’s — het uitvoeren van command-line-hulpprogramma’s (zoals ruff of httpie) op wereldwijde schaal.

1. Projectbeheer

Voor nieuwe projecten, uv gebruikt de standaard pyproject.toml voor configuratie en cross-platform gebruik uv.lock voor reproduceerbare builds.

Een nieuw project starten:

uv init my-project
cd my-project

Dit genereert een pyproject.toml, een .gitignore, en een hello.pyVoeg afhankelijkheden toe:

# Runtime dependencies
uv add pandas requests

# Development dependencies
uv add pytest ruff --dev

Voer je code uit:

uv run hello.py

uv beheert de virtuele omgeving in .venv Voor u. U hoeft het niet handmatig te activeren.

2. Beheren van Python-versies

uv installeert en beheert Python-versies voor u, opgeslagen in ~/.cache/uv. Als je deze al gebruikt hebt pyenv Voor dit geval kunt u het gewoon weglaten.

Een specifieke versie installeren:

uv python install 3.12

Een versie vastzetten voor uw project:

uv python pin 3.11

Dit schrijft een .python-version bestand. Op de volgende uv run, het maakt gebruik van de gepinde versie en downloadt deze indien nodig. Zowel uw team als de CI werken vervolgens met dezelfde Python-versie, zonder extra coördinatie.

3. Het uitvoeren van tools met uvx en uv tool install

uvx (Een alias voor) uv tool run) voert Python-commandline-hulpprogramma’s uit zonder ze toe te voegen aan uw globale omgeving of projectafhankelijkheden.

# Run a linter
uvx ruff check .

# Run a formatter
uvx black .

# Start a temporary Jupyter server
uvx --from jupyterlab jupyter lab

Elk hulpprogramma draait in zijn eigen tijdelijke omgeving, waardoor er geen versieconflicten ontstaan met wat er al in uw project is geïnstalleerd.

Twee details die ik voortdurend raadpleeg:

  • Plaats de versie inline met @: uvx ruff@0.6.0 check ., of forceer het nieuwste met uvx ruff@latest.
  • Klopt de naam van het commando niet met die van het pakket? Gebruik dan --from. Het pakket jupyterlab stuurt de verzending op jupyter command, dus het is uvx --from jupyterlab jupyter lab. Hetzelfde geldt voor --from httpie http.
  • Hebt u een extra afhankelijkheid nodig voor een plugin? Voeg deze toe met --with: uvx --with mkdocs-material mkdocs serve.

Als je een tool elke dag gebruikt, installeer hem dan één keer in plaats van iedere keer een nieuw omgeving op te zetten:

uv tool install ruff      # now `ruff` is on your PATH
uv tool list              # see what's installed
uv tool upgrade --all     # update everything
uv tool uninstall ruff

De praktische richtlijn: uvx voor éénmalige of CI-executies, uv tool install voor uw dagelijkse scripts.

Legacy-projecten (requirements.txt)

requirements.txt Het heeft lang goed gewerkt. Het gaat om een eenvoudige lijst met pakketten zonder lockfile, zonder onderscheid tussen applicatie- en ontwikkelafhankelijkheden, en zonder enige registratie van waarom bepaalde versies zijn vastgezet. Dat is prima zolang je niet probeert een build uit acht maanden geleden na te maken en ontdekt dat ‘vastgezet’ simpelweg betekent wat PyPI op dat moment toevallig heeft geresolveerd.

Als u eigenaar bent van het project, migreer het dan. uv leest je bestaande lijst rechtstreeks in een pyproject.toml:

uv init --bare                        # minimal pyproject.toml, no scaffolding
uv add -r requirements.txt            # import runtime dependencies
uv add --dev -r requirements-dev.txt  # and dev ones, if you split them

Jullie hebben nu een pyproject.toml en een echte uv.lock dat specifieke, opgeloste versies vastzet. Controleer of alles correct is overgedragen. uv pip freeze, verwijder de oude requirements.txtEn kijk niet achterom.

Maar misschien voer je al die tijd hetzelfde uit. requirements.txt Sinds Python 3.6, en je bent niet geïnteresseerd in mijn mening daarover. Logisch. uv werkt nog steeds als directe vervanging voor pip en venv, er is geen migratie nodig:

# Create a virtual environment
uv venv

# Install dependencies
uv pip install -r requirements.txt

# Run it
uv run python app.py

Hetzelfde commando’s die je al kent, alleen sneller. Er verandert niets aan de oude werkwijze; je hoeft hem gewoon niet langer te gebruiken.

Scripten in één bestand met inline-dependenties

Dit is de functionaliteit die de manier waarop ik tijdelijke en ‘glue’-code schrijf heeft veranderd. Een Python-script kan zijn eigen afhankelijkheden binnen het bestand declareren, in een opmerkingenblok dat wordt gedefinieerd door PEP 723. Nee pyproject.toml, nee requirements.txt, er is geen virtuele omgeving om te maken — uv run Het leest de blokken in, maakt een gecacheerde omgeving aan en voert het bestand uit.

Het is belangrijk om duidelijk te zijn over wat hier de rol speelt: dit is geen functie van de Python-taal, en het is ook niet… uv Invinding. Het blok is slechts een opmerking, dus gewoon. python fetch.py hij negeert dit en faalt vanwege de ontbrekende imports. PEP 723 is een gedeelde standaard, zodat elke compatibele uitvoerder dezelfde blokken leest — pipx runHatch en PDM ondersteunen het eveneens. uv het is toevallig ook de snelste manier om het te gebruiken, en daarom wordt in de rest van dit gedeelte gebruikgemaakt van uv run.

Hier staat het hele geheel in één bestand. fetch.py:

# /// script
# requires-python = ">=3.12"
# dependencies = [
#     "httpx",
#     "rich",
# ]
# ///
import httpx
from rich import print

print(httpx.get("https://api.github.com/repos/astral-sh/uv").json())

Voer het uit:

uv run fetch.py

De eerste uitvoering lost het op en installeert het. httpx en rich in een gecacheerde omgeving; latere uitvoeringen maken hier gebruik van en starten onmiddellijk.

U hoeft de metadata-blok niet handmatig op te stellen. uv Ik zal een skelet maken en het voor u bewerken:

# Create a new script with the block pre-filled
uv init --script fetch.py --python 3.12

# Add or remove dependencies
uv add --script fetch.py httpx rich
uv remove --script fetch.py rich

Voor een snel experiment kun je de blok zelfs volledig overslaan en de afhankelijkheden via de commandoregel doorgeven:

uv run --with httpx --with rich fetch.py

Waarom dit uitstekend is voor scripten voor vaardigheden en automatisering

Ik maak gebruik van dit framework voor de kleine scripts die geen apart project verdienen: eenmalige correcties van gegevens, hulpprogramma’s voor CI-pipelines, of simpele automatiseringsoplossingen. Claude Code-vaardigheid script, een cron-job. Het script vormt de afzonderlijke eenheid. Je kunt het in een gist plaatsen, het committen naar een willekeurig repository, of het doorgeven aan een teamlid. uv run dit is het enige wat ze nodig hebben om het correct te laten draaien.

Maak het direct uitvoerbaar met een shebang:

#!/usr/bin/env -S uv run --script
# /// script
# dependencies = ["httpx"]
# ///
import httpx
...
chmod +x fetch.py
./fetch.py        # uv handles the environment behind the scenes

The -S splits de shebang op in afzonderlijke argumenten zodat env slaagt --script tot aan uv### Scripten vergrendelen en vastzetten

Voor scripts die maandenlang moeten blijven werken, heeft u twee opties.

Vergrendel de exacte resolutie naast het script:

uv lock --script fetch.py   # writes fetch.py.lock

Of pin een tijdpunt vast zodat uv houdt alleen rekening met pakketten die vóór een bepaalde datum zijn uitgebracht — handig voor reproduceerbaarheid zonder een lockfile:

# /// script
# dependencies = ["httpx"]
# [tool.uv]
# exclude-newer = "2025-04-17T00:00:00Z"
# ///

Tips en trucs die het waard zijn om te kennen

Nog een paar zaken die ik regelmatig gebruik en die niet meteen duidelijk zijn uit de snelle startgids.

Synchroniseer vanuit het lockfile in CI. uv sync installeert precies wat erin staat uv.lock. Voeg toe --frozen om luid te falen wanneer het lockfile verouderd is, in plaats van stilletjes opnieuw te proberen het te resoluwen — precies wat je wilt bij CI- en Docker-builds.

uv sync --frozen

Groep je ontwikkelingsafhankelijkheden. Daarnaast --dev, je kunt genummerde afhankelijkheidsgroepen definiëren in pyproject.toml (docs, lint, test) en synchroniseer alleen wat je nodig hebt:

uv add --group docs mkdocs-material
uv sync --only-group docs

Controleer de afhankelijkheidsboom. uv tree toont aan wat is opgehaald en wat — en --outdated Upgrade van vlaggen:

uv tree
uv tree --outdated

Exporteer naar requirements.txt wanneer een downstream-hulpprogramma er nog steeds eentje verwacht:

uv export --format requirements-txt > requirements.txt

Voer eenmalig een Python-script uit met extra pakketten, er is geen project vereist:

uv run --with pandas --with matplotlib python

Beheer de cache wanneer de schijfruimte beperkt is of een build mislukt:

uv cache clean      # wipe the whole cache
uv cache prune       # remove only unused entries

Waarom het snel is

Drie factoren spelen hier een rol. Het is geschreven in Rust, waardoor er bij elke oproep geen opstartkosten van Python zijn. De gebouwde versies worden wereldwijd gecacheerd — één keer numpy is geïnstalleerd in één project; het installeren ervan in een ander duurt in feite maar een ogenblik (op macOS worden copy-on-write-links gebruikt). Bovendien worden pakketten gelijktijdig gedownload en geïnstalleerd in plaats van één voor één.

Belangrijkste conclusies

  1. Gebruik uv voor Python-versies, virtuele omgevingen, projectafhankelijkheden en eenmalige hulpprogramma’s, in plaats van meerdere pakketbeheerders te combineren.
  2. Laat Homebrew verantwoordelijk blijven voor systeempakketten, en niet voor de isolatie van Python-projecten.
  3. Maak een commit. pyproject.toml en uv.lock Zodat de projectinrichting reproduceerbaar is op een nieuwe machine.
  4. Geef de voorkeur aan uvx voor af en toe gebruik van CLI’s wanneer je een globale omgeving niet wilt vervuilen.
