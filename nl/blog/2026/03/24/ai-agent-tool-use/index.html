    Gebruik van AI-agenthulpmiddelen in 2026: MCP, CLI, Skills en uitvoering van code

    Deel 3 van de serie ‘Engineering the Agentic Stack’

    Deel 1 Gecoverde redeneringslussen. Deel 2 Geblokkeerde geheugenruimte. Deze post gaat over het gebruik van AI-agent-hulpmiddelen: hoe agents omgaan met de wereld, en waarom het ontwerp van hulpmiddelen belangrijker is dan de meeste teams beseffen.

    De situatie rondom hulpmiddelen veranderde in 2025–2026. MCP werd de feitelijke standaard voor externe diensten, maar productieteams stuiten nog steeds op de beveiligingsproblemen en de hoge tokenkosten die ermee gepaard gaan. Een andere aanpak die steeds meer wordt gebruikt, zijn agents die code schrijven en uitvoeren in plaats van JSON-gedefinieerde hulpmiddelen te gebruiken. Anthropic constateerde een 98,7% lagere tokenverbruik bij een representatief werkproces, en het artikel van CodeAct rapporteert 20% hogere succespercentages bij taken.

    In deze post worden JSON-hulpmiddeloproepen, MCP, Skills, CLI-hulpmiddelen en codeuitvoering vergeleken, waarna ze worden gekoppeld aan de ontwerpprincipes van de Agent-Computer Interface (ACI) die ik toepas in Marktanalistagent.

    Kort samengevat: Er bestaan vijf modaliteiten voor AI-agenttools, waarbij elk type zijn eigen optimale toepassingsgebied heeft. De praktische stack voor 2026 omvat: expertisegerichte skills, CLI als standaard, MCP voor externe diensten, en codeuitvoering voor meerstappige orkestratie. Allemaal worden ze geleid door de ontwerpprincipes van ACI uit SWE-agent.

    In Deel 1, Ik heb uitgelegd hoe de redeneringslopen van AI-agenten bepalen hoe ze moeten denken. De toolmodaliteiten bepalen hoe ze moeten handelen. Er bestaan momenteel vijf verschillende benaderingen, waarbij elke benadering andere afwegingen kent op het gebied van tokenkosten, flexibiliteit en beveiliging.

    1. JSON-hulpprogramma-aanroepen — De basis

    Het oorspronkelijke patroon: je definieert schema’s voor hulpprogramma’s in JSON, de LLM genereert gestructureerde functieaanroepen, en jouw code voert deze uit. Dit is goed begrepen en werkt prima voor kleine sets hulpprogramma’s.

    # Traditional tool definition — each tool consumes ~550-1,400 tokens (Apideck benchmark)
tools = [
    {
        "name": "get_stock_price",
        "description": "Get the current stock price for a ticker symbol",
        "input_schema": {
            "type": "object",
            "properties": {
                "ticker": {"type": "string", "description": "Stock ticker (e.g., NVDA)"}
            },
            "required": ["ticker"]
        }
    }
]

    Met 5-10 tools is dit acceptabel. Het probleem ligt bij de schaal: elke tooldefinitie heeft kosten. 550-1.400 tokens. Met 20 tools besteed je al 15-25K tokens voordat de agent zelfs maar begint met redeneren.

    2. MCP — De universele standaard (met voorbehouden)

    Het Model Context Protocol is de standaard waarop de meeste aanbieders het eens zijn geworden. Anthropic heeft dit protocol in december 2025 gedoneerd aan de Linux Foundation onder de Agentic AI Foundation (AAIF), samen met OpenAI en Block, en wordt ondersteund door Google, Microsoft en AWS als platinumleden. OpenAI heeft MCP-ondersteuning toegevoegd aan zijn Responses API. Er zijn momenteel meer dan 10.000 actieve MCP-servers en meer dan 97 miljoen maandelijkse downloads van SDK’s.

    Waar MCP zijn voordeel heeft, zijn cross-aanbiederintegraties in SaaS-oplossingen (zoals Figma, Notion en Salesforce), bedrijfsbeheer met audittrail-functies, diensten zonder equivalenten in een CLI, en omgevingen die OAuth-orkestratie nodig hebben. Voor deze toepassingsgebieden is MCP duidelijk de beste keuze.

    De werkelijkheid in productieomgevingen is echter rommeliger dan de cijfers op het eerste gezicht doen vermoeden.

    De beveiligingsoppervlakte vormt het eerste probleem. De Vulnerable MCP-project Houdt 50 kwetsbaarheden in de MCP-servers in de gaten, waarvan 13 als kritiek worden geclassificeerd; deze zijn gemeld door 32 beveiligingsonderzoekers. De aanvalscategorieën omvatten prompt-injectie, fouten bij invoervalidatie, authenticatiefouten en netwerkbeveiligingsproblemen. Het eerste echte kwaadaardige MCP-server in de praktijk verscheen in september 2025: een pakket genaamd postmark-mcp Dat stuurde elke uitgaande e-mail naar het adres van de aanvaller via BCC, waardoor schattingen zijn dat ongeveer 300–500 organisaties werden getroffen voordat dit werd ontdekt.

    Tool poisoning is de type aanval waar ik me het meest zorgen over maak. Invariant Labs heeft dit gedemonstreerd De vergiftigde MCP-hulpmiddelen kunnen gegevens lekken, zelfs wanneer ze nooit worden opgeroepen. Het feit dat het model enkel de metadata van het hulpmiddel leest, is al voldoende om de aanval in gang te zetten. MCPTox-benchmarks Bij het testen van 20 LLM-agenten tegen 45 echte MCP-servers bleek dat het succespercentage bij aanvallen oploopt tot maar liefst 72,8%.

    Het token-overhead vormt het operationele probleem. Een team dat MCP-servers beheert voor GitHub, Slack en Sentry (in totaal ongeveer 40 tools) ontdekte dit 55.000 tokens van schema-definities die worden ingespoten voordat een gebruiker iets vraagt. Een andere gemelde 143.000 van de 200.000 beschikbare tokens (72%) Alleen verbruikt door de tooldefinities zelf.

    De mitigaties werken, maar ze bevestigen het onderliggende probleem. Van Anthropic’s De Tool Search Tool vermindert de overhead van het schema met 85%.. Dynamisch laden van tools (3-5 relevante tools per taak) zorgt voor een vermindering van 96%. Dit zijn workarounds voor een protocol dat niet is ontworpen met tokenbudgetten in gedachten.

    3. Vaardigheden (SKILL.md) — Expertise, niet uitvoering

    Eind 2025 werd de standaardisatie van agentvaardigheden als open formaat geïntroduceerd (gelanceerd in oktober 2025, gepubliceerd als open standaard in december 2025). Het onderscheid is belangrijk: tools bieden functionaliteiten (wat agenten kunnen doen), terwijl vaardigheden expertise leveren (wat agenten weten over hoe ze complexe taken kunnen uitvoeren).

    De SKILL.md-standaard definieert een vaardigheid als een markdown-bestand met YAML frontmatter:

    ---
name: deploy
description: Deploy the application to production
argument-hint: "[environment]"
user-invocable: true
---
Deploy the application to the $0 environment (default: staging).
Steps:
1. Run the test suite
2. Build the production bundle
3. Deploy using the deploy script
4. Verify the deployment health check

    Vaardigheden maken gebruik van progressieve onthulling: alleen de metadata (ongeveer 100 tokens voor het YAML frontmatter) wordt bij het opstarten geladen, terwijl de volledige instructies worden geladen wanneer de vaardigheid wordt geactiveerd. Vergelijk dit met MCP, waarbij ongeveer 40 hulpmiddelen uit typische diensten worden gebruikt ~55.000 tokens Vooraleer met enig redeneren te beginnen: de cross-platform adoptie per maart 2026 omvat Claude Code, OpenAI Codex CLI, Cursor, GitHub Copilot, Gemini CLI, Goose, Windsurf en Roo Code – een duidelijke trend. Victor Dibia heeft onderzocht in zijn analyse van de verschuiving van tools naar acties die worden aangestuurd door code.

    Wanneer vaardigheden doorslaggevend zijn: overdracht van domeinkennis, complexe meerdere-stappenprocedures, herhalende werkstromen zoals database-migraties of betalingsintegraties, en elke taak waarbij de agent moet weten hoe in plaats van alleen wat.

    4. CLI/Bash — De renaissance van Unix

    De Unix-terminal bleek de meest token-efficiënte interface voor agents te zijn. De cijfers spreken voor zich: een manifest voor een CLI-hulpmiddel kost ongeveer 100 tokens, terwijl de equivalentie in MCP-server-schemas 50.000+ tokens verbruikt. Dat is een 4-32x verschil gemeten met Scalekit in 75 benchmarkruns (32 keer voor taken zoals het opzoeken van de programmeertaal en licentie van een repository).

    Modellen zijn van nature bedreven in CLI-communicatie. Hun trainingsgegevens omvatten miljarden terminalinteracties uit Stack Overflow, GitHub-repos en documentatie. Zij begrijpen dit soort interacties van nature. git, docker, kubectl, gh, curl, en jq zonder enige schema-definities.

    De gids van Ugo Enyioha "CLI-tools schrijven die AI-agenten daadwerkelijk willen gebruiken" Codificeerde acht ontwerprules:

    1. Gestructureerde uitvoer is verplicht — ondersteuning --json
    2. Exitcodes vormen onderdeel van de controlestroom — gebruik aparte codes voor verschillende fouttypen.
    3. Commando’s moeten idempotent zijn.
    4. Zelfdocumenterend. --help met realistische voorbeelden
    5. Ontwerp voor componabiliteit — --quiet Voor eenvoudige waarden: ondersteuning voor stdin
    6. Geef --dry-run en --yes flags
    7. Steun voor versieniveau-introspectie
    8. Authenticatie afhandelen via omgevingsvariabelen

    De compromissen zijn reëel. De CLI mist de typensicherheid van MCP, ingebouwde OAuth-orkestratie, hulpprogrammaontdekking en een audittrail. Het patroon dat ik bij de meeste teams zie ontstaan, is: de CLI als standaard voor ontwikkeling en lokale operaties, en MCP voor integratie met externe diensten en bedrijfsbeheer.

    5. Codeuitvoering — De grote verschuiving

    Dit is de verandering in agent-hulpprogramma’s die ik het meest invloedrijk vind. In plaats van dat de LLM gestructureerd JSON uitzendt om vooraf gedefinieerde functies één voor één aan te roepen, schrijft de agent een compleet Python- of bash-script dat meerdere hulpprogramma’s oproept, resultaten verwerkt met lussen en conditionele instructies, en alleen de eindsamenvattingen teruggeeft aan de context van het model.

    Anthropic heeft dit geformaliseerd met Programmatische toolaanroeping (PTC), nu beschikbaar in de Claude API. De academische basis vormt... CodeAct-paper (Wang et al., ICML 2024), die 17 verschillende LLM’s hebben getest en ontdekten dat code‑acties tot 20% hogere succespercentages bij het uitvoeren van taken opleverden en 30% minder stappen vereisten vergeleken met JSON‑alternatieven.

    De productiebewijzen: Vercel heeft hun d0-text-naar-SQL-agent opnieuw gebouwd door 80% van de tools te verwijderen (van 15 naar 2): ExecuteCommand en ExecuteSQL). Het succespercentage van de taak steeg van 80% naar 100%, de uitvoeringstijd daalde met een factor 3,5 en het gebruik van tokens nam af met 40%. Hun formulering: “De beste agenten zijn mogelijk diegenen met het minste aantal hulpmiddelen.”

    • Cloudflare heeft “Code Mode” ontwikkeld. het mogelijk maken voor agents om TypeScript te gebruiken om hun API aan te roepen in plaats van tool-schemas te definiëren, waardoor de contextbelasting afneemt. Hun redenering: “LLMs beschikken over een enorme hoeveelheid echte TypeScript-code in hun trainingsset, maar slechts een klein aantal kunstmatig gemaakte voorbeelden van toolaanroepen.” De PTC-documentatie van Anthropic. Traditionele tools voor uitgavenanalyse vereisen 20+ afzonderlijke inferentie‑stappen, één per teamlid, waarbij alle tussentijdse gegevens via de context worden verwerkt. Een werkflow die ongeveer 150.000 tokens verbruikt bij directe tool‑aanroepen, is opnieuw geimplementeerd met slechts ongeveer 2.000 tokens, wat een 98,7% daling betekent. Met code‑uitvoering schrijft de agent slechts één script:
    # Agent generates this code, executes in sandbox
import json
members = get_team_members("engineering")
over_budget = []
for m in members:
    expenses = get_expenses(m["id"], "Q3")
    total = sum(e["amount"] for e in expenses)
    if total > 5000:
        custom = get_custom_budget(m["id"])
        limit = custom["limit"] if custom else 5000
        if total > limit:
            over_budget.append({"name": m["name"], "spent": total, "limit": limit})
# Only this final summary returns to the LLM context
print(json.dumps(over_budget))

    De LLM ziet alleen het eindresultaat in JSON, niet de duizenden uitgavenposten die in de sandbox zijn verwerkt. Token-efficiëntie, geïntegreerde compositibiliteit (lussen en conditionele constructies zijn standaard beschikbaar), en echte foutbehandeling.try/except In plaats van redeneren over fouten in natuurlijke taal verbeteren ook de privacy (gevoelige gegevens blijven binnen de sandbox) tegelijkertijd.

    Wanneer het nog zinvol is om JSON-hulpprogramma’s te gebruiken: voor enkelvoudige, atomaire operaties, omgevingen zonder sandboxinfrastructuur, kleinere modellen met zwakke codegeneratie, of wanneer er auditvereisten zijn die vereisen dat elke afzonderlijke oproep aan een hulpprogramma wordt geregistreerd.

    De Agent-Computer Interface (ACI) voor AI-agenthulpprogramma’s

    De term “Agent-Computer Interface” (ACI) werd bedacht door John Yang, Carlos E. Jimenez en hun collega’s aan Princeton in hun Artikel over de SWE-agent (NeurIPS 2024). Het concept is eenvoudig: net zoals mensen profiteren van goed ontworpen interfaces (HCI), zijn LM-agenten “een nieuwe categorie eindgebruikers met hun eigen behoeften en mogelijkheden, die ook baat zouden hebben bij speciaal ontworpen interfaces.” Resultaten van ablatietests Ik heb dit gereserveerd. De SWE-agent met zijn volledige ACI behaalde 18,0% op SWE-bench Lite, vergeleken met 7,3% met alleen een standaard Linux-shell – dat is een verbetering van 10,7 procentpunt alleen al door het interfaceontwerp. De linting-guardrail zorgde op zichzelf voor 3 procentpunt: 51,7% van de wijzigingen door de agent bevatte ten minste één fout die door de linter werd opgemerkt voordat deze zich kon verspreiden.

    Anthropic heeft ACI aangenomen als fundamenteel concept in hun Effectieve agents bouwen Gids, waarin dit wordt opgenomen als een van de drie kernprincipes: “Ontwerp uw agent-computerinterface zorgvuldig door middel van uitgebreide documentatie en testing van de tools.” Hun praktische aanwijzingen: “Een handige regel is om na te denken over de hoeveelheid inspanning die wordt gestoken in mens-computerinterfaces, en om evenveel inspanning te plannen voor het creëren van goede agent-computerinterfaces.”

    De vier principes in de praktijk

    1. Acties moeten eenvoudig en gemakkelijk te begrijpen zijn. De meest voorkomende fout is het één-op-één omhullen van API-eindpunten. In plaats daarvan list_users, list_events, create_event, implementeren schedule_event dat beschikbaarheid bepaalt en een planning maakt in één oproep. In plaats van read_logs, implementeren search_logs dat alleen de relevante regels teruggeeft, samen met de context.

    2. Acties moeten compact en efficiënt zijn. Consolideer belangrijke operaties tot zo weinig acties mogelijk. In de Marktanalistagent, Ik combineer het ophalen van prijzen met basismetrieken tot één geheel. get_stock_snapshot een tool in plaats van het maken van afzonderlijke aanroepen voor de prijs, het volume, de marktkapitalisatie en de PE-verhouding.

    3. Feedback over de omgeving moet informatief zijn, maar beknopt. Vermijd het retourneren van ruwe HTML-inhoud of volledige API-berichten. Zet cryptische ID’s om in semantische namen. De tests van Anthropic liet zien dat het toevoegen van een response_format een enum die agents in staat stelt om compacte (ongeveer 72 tokens) of gedetailleerde (ongeveer 206 tokens) antwoorden aan te vragen – met een drievoudig verschil in tokenkosten – heeft de prestaties in de echte wereld aanzienlijk verbeterd.

    4. Guardrails moeten foutverspreiding beperken. Automatische foutdetectie helpt agents om fouten snel te herkennen en te corrigeren. In SWE-agent, een aangepaste bestandbewerker met geïntegreerd linteningsmechanisme wijst syntaxisfouten automatisch af, waardoor 51,7% van de bewerkingss fouten van de agent al voordat ze zich verder kunnen verspreiden worden opgespoord. Ik toepas hetzelfde principe in de Market Analyst Agent door de argumenten van de tools vóór uitvoering te valideren met Pydantic-schemas:

    from pydantic import BaseModel, Field, field_validator

class StockQuery(BaseModel):
    """Validated input for stock queries.

    Pydantic catches malformed tickers before the API call,
    preventing error propagation through the reasoning loop.
    """
    ticker: str = Field(description="Stock ticker symbol (e.g., NVDA)")
    period: str = Field(default="1mo", description="Time period: 1d, 5d, 1mo, 3mo, 1y")

    @field_validator("ticker")
    @classmethod
    def validate_ticker(cls, v: str) -> str:
        v = v.upper().strip()
        if not v.isalpha() or len(v) > 5:
            raise ValueError(f"Invalid ticker format: {v}")
        return v

    @field_validator("period")
    @classmethod
    def validate_period(cls, v: str) -> str:
        valid = {"1d", "5d", "1mo", "3mo", "6mo", "1y", "5y"}
        if v not in valid:
            raise ValueError(f"Invalid period: {v}. Must be one of {valid}")
        return v

    Werkende ontwerppatronen voor AI-agenttoolen

    Anthropic’s "Het schrijven van effectieve hulpmiddelen voor agents." beschrijf guide frames-tools als “een nieuw type software dat een contract weerspiegelt tussen deterministische systemen en niet-deterministische agenten.” Hier zijn de patronen die uit productieervaring zijn geformuleerd.

    Beschouw toolbeschrijvingen als prompt engineering

    Beschrijvingen moeten functioneren als Er zijn minimaal 3 tot 4 zinnen nodig voor deze vertaling., inclusief wanneer het hulpmiddel moet worden gebruikt, vereiste versus optionele parameters, uitvoerformat en randgevallen. Naamruimtebeheer met prefixen (asana_search, jira_search) heeft "niet-triviale effecten" op de nauwkeurigheid van de keuze van hulpmiddelen. In de experimenten van Anthropic presteerden de door Claude geoptimaliseerde beschrijvingen van hulpmiddelen beter dan die door experts handgeschreven waren, volgens de evaluatiecriteria.

    # Bad: vague, no context for when to use
tools = [{
    "name": "search",
    "description": "Search for items",
}]

    Ontwerp van foutbeheer voor veerkracht

    Het patroon dat in productie heeft gewerkt, bestaat uit vier lagen fouttolerantie:

    1. Herproberen met exponentiële vertraging voor tijdelijke fouten
    2. Fallbackketens voor modellen bij storingen van providers
    3. Routing op basis van foutclassificatie — tijdelijke fouten worden herproberd, door LLM’s te herstellen fouten worden met context teruggezonden naar de agent, fouten die menselijke interventie vereisen worden verder geëscaleerd
    4. Herstel van checkpoints om crashes te overleven

    Volgens rapporten van teams daalt het aantal onherstelbare fouten met deze vielaagse aanpak van 23% tot minder dan 2%, zoals gedocumenteerd in Resilienstechnieken voor tools van Anthropic.

    Afwegingen en overwegingen

    Naast de hierboven genoemde specifieke beperkingen per modaliteit, spelen er nog enkele algemene factoren een rol bij de keuze:

    • De operationele kosten verschillen per dimensie. Het uitvoeren van code bespaart tokens, maar zorgt voor meer vertraging bij het opstarten van de sandbox. MCP bespaart ontwikkelingstijd voor SaaS-integraties, maar brengt extra overhead met zich mee bij het implementeren van servers. CLI is gratis om te starten, maar moeilijker te beheren op grote schaal. Optimaliseer voor je eigen bottleneck, of dat nu kosten, vertraging of operationele complexiteit zijn.

    • De vaardigheden van het team zijn belangrijk. Het uitvoeren van code gaat ervan uit dat je agents (en de modellen erachter) betrouwbare Python- of TypeScript-code kunnen genereren. CLI vereist kennis van Unix-conventies. MCP vereist inzicht in transportprotocollen en OAuth-workflows. Pas de modaliteit aan op de sterke punten van je team.

    • Het consolideren van tools kan te ver gaan. Als je alles samenvoegt tot één enorme tool, wordt de parameterset te complex om door de agent te worden begrepen. Voor de meeste agentsystemen is 5-10 goed geframeerde tools het ideale aantal.

    • Vaardigheden zijn gebaseerd op prompts, niet verplicht. Een vaardigheid zijn instructies die de agent moet opvolgen, geen regels die hij absoluut moet naleven. Voor kritieke werkprocessen moet je vaardigheden combineren met deterministische validatie.

    • De vereisten voor audits bepalen de keuze. Als u een volledig log nodig heeft van elke oproep aan een tool en het resulterende resultaat, bieden MCP en JSON-tooloproepen gestructureerde audittrails direct uit de doos. Het uitvoeren van code genereert een script en de bijbehorende uitvoer, wat nuttig is, maar moeilijker te ontleden in afzonderlijke acties voor rapportage op het gebied van compliance.

    Waar de ergonomie van tools naartoe gaat

    Er zijn drie richtingen die het waard zijn om in de gaten te houden.

    De eerste is RAG-tools voor schaalbaarheid. Naarmate toolsets groeien tot honderden of duizenden, daalt de nauwkeurigheid van een eenvoudige toolselectie tot 13,62%. De RAG-MCP paper Het bleek dat het toepassen van retrieval-augmented generation op de toolselectie – waarbij beschrijvingen van tools worden geïndexeerd in een vectordatabase en alleen relevante tools per query worden opgehaald – een nauwkeurigheid oplevert van 43,13%, oftewel een verbetering met 3,2 keer, terwijl er tegelijkertijd ongeveer 50% minder prompt-tokens nodig zijn.

    Het tweede voorbeeld betreft agents die hun eigen tools creëren. LATM-framework ("LLMs Als Hulpmiddelenontwikkelaars") heeft een tweefasig paradigma geïntroduceerd, waarbij een krachtige LLM herbruikbare Python-functies genereert en een lichtgewicht LLM deze functies gebruikt. ToolMaker (ACL 2025) transformeert op autonome manier GitHub-repo’s om tot tools die compatibel zijn met LLM’s, met een succesratio van 80%. De ontwikkeling beweegt zich van het gebruik van tools naar de creatie daarvan, en vervolgens naar het beheer van toolbibliotheken.

    De derde optie is de A2A + MCP-dubbelprotocolstack. Het Agent2Agent Protocol van Google (A2A) lost problemen op die MCP niet kan oplossen: communicatie tussen agenten. MCP zorgt voor integratie van agenten met tools; A2A regelt het ontdekken van agenten onderling, onderhandelingen en taaktoewijzing. De gecombineerde aanpak: bouw met je eigen framework, gebruik MCP voor integratie, en communiceer met A2A.

    Belangrijkste conclusies

    1. Het gebruik van tools kent vijf verschillende modaliteiten, waarbij elke modaliteit een optimale toepassing heeft. Gebruik niet automatisch MCP voor alles; kies de juiste modaliteit afhankelijk van de taak.

    2. Het uitvoeren van code in plaats van het oproepen van JSON-tools vormt de grootste verandering in agenttooling. Anthropic’s PTC, Vercel’s toolreductie en Cloudflare’s Code Mode wijzen allemaal naar dezelfde inzichten: laat agenten code schrijven in plaats van schema’s oproepen.

    3. Tokenoverhead vormt het operationele knelpunt. 55K+ tokens voor ongeveer 40 MCP-tools versus ongeveer 100 tokens voor CLI-manifesten is geen klein verschil. Het maakt namelijk het verschil of een agent 65% of 95% van zijn context beschikt voor redeneren.

    4. Ontwerpprincipes voor ACI zijn belangrijker dan het protocol zelf. SWE-agent liet zien dat alleen al het interfaceontwerp 10,7 procentpunten verschil oplevert op benchmarks. Eenvoudige acties, beknopt feedback en ingebouwde beperkingen zijn altijd van toepassing, ongeacht of je MCP, CLI of codeuitvoering gebruikt.

    5. Consolideer tools actief. Vijf goed ontworpen tools presteren beter dan twintig gedetailleerde tools. Elke tool is een keuze die je het model oplegt, dus verminder het aantal mogelijkheden.

    6. Beschouw toolbeschrijvingen als onderdeel van prompt engineering. Voorbeeldingen van invoer verbeteren de nauwkeurigheid aanzienlijk. Beschrijvingen die zijn geoptimaliseerd voor Claude presteren beter dan beschrijvingen die door experts handmatig zijn geschreven. Steken dezelfde inspanning in de UX van tools als u zou doen voor de UX van menselijke interfaces.

    7. De praktische stack voor 2026 bestaat uit Skills + CLI + MCP + Code Execution, gegroepeerd naar gebruikssituatie. Skills voor expertise, CLI voor lokale operaties, MCP voor externe diensten, en code uitvoering voor meerstappige orchestratie.
