Redeneringscycli van AI-agenten in 2026: ReAct versus ReWOO versus Plan-en-Uitvoeren

Deel 1 van de serie ‘Engineering the Agentic Stack’

Het ontwikkelen van bruikbare AI-agenten draait tegenwoordig voornamelijk om systeemontwerp, en niet om prompt-engineering. De belangrijkste beslissing is de agent-reasoning-loop: hoe het systeem plannen maakt, hulpmiddelen oproept, resultaten observeert en bepaalt wanneer het moet stoppen.

In deze post worden de drie belangrijkste patronen voor AI-agent-loops die je in 2026 moet kennen vergeleken: ReAct, ReWOO en Plan-and-Execute. Het voorbeeld dat wordt gebruikt, is een Market Analyst Agent die ik in LangGraph heb gebouwd; de volledige code is te vinden op GitHub.

Kort samengevat: ReAct is flexibel maar duur, ReWOO is snel wanneer het werkproces voorspelbaar is, en Plan-and-Execute is geschikt voor meerdere stappen tellende analyses. Een productie-AI-agent kan per verzoek overstappen tussen redeneringslopen, waarbij gebruik wordt gemaakt van gedeelde staat en gecontroleerde LangGraph-nodes, zodat elke taak de loop krijgt die past bij zijn structuur.

Waarom redeneringslopen van AI-agenten belangrijk zijn

Een jaar geleden draaide het er vooral om hoe je prompts formuleert om een LLM bruikbaar te maken. Tegenwoordig gaat het om grafontwerp: hoe redenering, het gebruik van tools en geheugen met elkaar worden verbonden.

De redeneringsloop bevindt zich in het midden van dat graf. Hij bepaalt wanneer het model nadacht, wanneer het een tool aanroept en wanneer het stopt. Kies je de verkeerde loop, dan besteed je extra tokens aan onnodige oproepen, neemt elke reactie seconden langer in beslag, of faalt je agent al bij de eerste keer dat een tool iets teruggeeft wat niet verwacht werd.

Drie redeneringspatronen voor AI-agenten

ReAct: denken, handelen, observeren, herhalen

ReAct (Yao et al., 2022) vormt het oorspronkelijke model voor interactieve agenten. Het werkt volgens een lus:

  1. Denkproces: de agent genereert een “denkproces” om het doel te analyseren en de volgende stap te plannen.
  2. Actie: op basis van dit denkproces roept de agent een hulpprogramma aan.
  3. Observatie: de agent leest het resultaat, waardoor zijn begrip wordt bijgewerkt voor het volgende denkproces.

Voordelen:

  • Een basis in echte observaties zorgt voor een verminderde hoeveelheid verzonnen feiten.
  • De agent kan op het moment zelf zijn strategie aanpassen op basis van wat hij zojuist heeft gezien.
  • Het “scratchpad” biedt een traceerbaar log van het redeneringsproces.

Nadelen:

  • De geschiedenis blijft zich opstapelen en wordt bij elke stap opnieuw verwerkt, waardoor de vertraging en de kosten toenemen naarmate de lus langer duurt.
  • Het is verspilling wanneer de toolaanroepen van tevoren hadden kunnen worden gepland, en dat is precies waar ReWOO zich op richt.
  • Zonder een stopconditie of limiet op het aantal stappen zal de lus oneindig doorlopen.

Ideaal voor: exploratieve taken, debugging, situaties waarin je niet kunt voorspellen wat er vervolgens gaat gebeuren.

ReWOO: plan alles van tevoren

ReWOO (Reasoning Without Observation) is een efficiëntere variant van ReAct. Het trucje zit hem in het scheiden van redeneren van de uitvoering van tools: in plaats van bij elke actie te stoppen om deze te observeren, plant ReWOO de hele reeks toolaanroepen in één keer.

  1. Plan: één oproep aan de LLM genereert het volledige plan voor de tooloproepen, waarbij variabelen als placeholders worden gebruikt#E1, #E2) voor uitvoer die nog niet bestaat.
  2. Worker: een niet-LLM uitvoerder voert de geplande hulpmiddelen sequentieel of parallel uit, waarbij de placeholders worden ingevuld.
  3. Solver: een laatste oproep aan de LLM gebruikt de verzamelde observaties om het antwoord op te stellen.

Voordelen:

  • Ongeveer 5 keer efficiënter in het gebruik van tokens dan ReAct, omdat de herhaalde Thought-Action-Observation-geschiedenis wordt weggelaten.
  • Lagere vertragingen. Er is geen noodzaak om bij elke stap de geschiedenis opnieuw in te dienen.
  • De planner kan apart worden gefine-tuned, zonder dat er een actieve omgeving nodig is.

Nadelen:

  • Gevoelig wanneer de tools zich abnormaal gedragen. Het plan gaat ervan uit dat alles correct werkt.
  • Vereist voorspelbare werkflows.
  • Zonder expliciete fallback-logica blijft een foutief plan doorlopen.

Ideaal voor: snelle snapshots, statuscontroles, dashboards. Alles waarbij de tools zich voorspelbaar gedragen.

Plan-and-Execute: een hybride aanpak

Plannen en uitvoeren Het bevindt zich tussen deze twee. In het artikel wordt een eenvoudige indeling beschreven die door de meeste moderne agents wordt gebruikt:

  1. Planningsfase: de agent genereert eerst een plan dat de taak opdelt in kleinere subtaken.
  2. Uitvoeringsfase: vervolgens voert de agent deze subtaken uit.

Het oorspronkelijke artikel richtte zich op zero-shot prompting. Moderne frameworks zoals LangGraph hebben dit verder ontwikkeld tot een volledig orchestratiepatroon met sequentiële uitvoering en keuze van een model per stap (een krachtig redeneringmodel voor het plannen, en een goedkoper model voor de uitvoering).

Voordelen:

  • Hiërarchisch redeneren dat de manier weerspiegelt waarop een menselijke expert een project opbouwt.
  • Herplanning is mogelijk. U kunt pauzeren en opnieuw evalueren als het resultaat van een stap onverwacht is.
  • Specialisatie van het model. De planner kan kostbaar zijn, terwijl de uitvoerder goedkoop kan zijn.
  • Begrensde complexiteit, met een duidelijke controlepunt na elke stap.

Nadelen:

  • Hogere vertraging dan bij ReWOO omdat de stappen sequentieel worden uitgevoerd.
  • Meer staat die beheerd moet worden.
  • Overbodig voor éénmalige aanvragen.

Ideaal voor: complexe meestapsanalyses, onderzoekstaakken, alles wat aan het einde synthese vereist.

1. ReAct: het “think-as-you-go”-patroon

Hoe het aanvoelt: alsof een mens een probleem oplost. “Ik probeer dit eens… oké, dat werkte niet, ik probeer dit in plaats daarvan.”

Sterkte: kan omgaan met onbekende onbekenden. Als een zoekresultaat een nieuw onderwerp onthult, kan de agent in de volgende stap van richting veranderen.

Nadelen: neigt tot herhalende uitvoering bij een mislukte actie. Het duurste patroon in termen van tokens.

2. Planen en uitvoeren: het op missie gerichte patroon

Hoe het aanvoelt: alsof je een projectmanager bent. “Hier is het 5-stappenplan. Laten we de stappen 1 tot en met 5 uitvoeren en daarna controleren of we klaar zijn.”

Sterkte: houdt de agent gefocust op het algemene doel. Hogere succespercentages bij lange, complexe taken.

Nadelen: als stap 1 op een manier faalt die stappen 2 tot en met 5 verstoort, kan de agent doorzetten volgens het gebroken plan voordat dit wordt opgemerkt.

3. ReWOO: het compilerpatroon

Hoe het aanvoelt: alsof je een klein programma schrijft. “Ik heb gegevens nodig van Tool A en Tool B, waarna ik ze in Tool C zal combineren.”

Sterkte: veel sneller en goedkoper. Het plan wordt één keer gecompileerd met placeholders (#E1 voor de uitvoer van het eerste hulpprogramma), waarna deze verder werd uitgevoerd zonder extra LLM-oproepen tot aan de finale synthese.

Nadelen: blind tijdens de uitvoering. Als het eerste hulpprogramma aangeeft “Ik kan die persoon niet vinden”, zal de agent toch de volgende stappen uitvoeren die afhankelijk waren van het bestaan van die persoon.

Welke kiezen

  • ReAct wanneer de agent met een gebruiker chatten heeft en onmiddellijk moet reageren.
  • Plan-and-Execute wanneer je een langdurige, meestapproceswerkzaamheid automatiseert, zoals het opstellen van een onderzoeksrapport.
  • ReWOO wanneer je een voorspelbare werkwijze hebt en je API-rekening met ongeveer 80% wilt verlagen.

Patroon 1: Implementatie volgens het ‘plan-en-uitvoeren’-model

Het ‘plan-en-uitvoeren’-model is zeer geschikt voor taken die een meervoudige synthese vereisen. Het belangrijkste is om planning en uitvoering van elkaar te scheiden: eerst een krachtig model dat de initiële strategie bepaalt, gevolgd door een ReAct-lus waarmee elke stap wordt uitgevoerd met de mogelijkheid om op de resultaten van de tools te reageren.

Hoe dit past bij het patroon:

  1. Eén fase voor planning. Eén oproep aan de LLM genereert het volledige plan als een lijst met beschrijvingen van de stappen.
  2. Gestructureerde uitvoer via Schema-Guided Reasoning, waardoor altijd geldig JSON wordt gegenereerd.
  3. Nog geen uitvoering van tools. De planner beslist alleen wat er moet gebeuren, niet hoe.
  4. Menselijk leesbare stappen. Elke stap is tekst die door de uitvoerder kan worden geïnterpreteerd.
# System prompt guides the LLM to think like a research analyst
# creating a strategic plan, not immediate tool calls
PLANNER_SYSTEM_PROMPT = """You are a senior investment research analyst.
Break down stock analysis requests into 4-6 research steps covering:
1. Current price and basic metrics
2. Recent news and announcements
3. Competitor analysis (if relevant)
4. Financial health assessment
5. Risk factors
6. Investment thesis synthesis

Output as JSON with step_number, description, and tool_hint."""

# Schema-Guided Reasoning: Enforce structure with Pydantic
class PlanOutput(BaseModel):
    """Structured output for the planner."""

    steps: list[PlanStep] = Field(description="Research steps to execute")
    ticker: str = Field(description="The stock ticker being analyzed")

def planner_node(state: AgentState) -> dict:
    """Generate a research plan from the user's request.

    This is Phase 1 of Plan-and-Execute: creating the high-level strategy.
    """

    # Use a powerful model for strategic planning
    llm = ChatAnthropic(model="claude-sonnet-4-5-20250929", temperature=0)

    # Apply Schema-Guided Reasoning to guarantee valid plan structure
    # This prevents common formatting errors that would break execution
    structured_llm = llm.with_structured_output(PlanOutput)

    # Context from long-term memory personalizes the plan
    profile_context = f"""
User Profile:
- Risk Tolerance: {state.user_profile.risk_tolerance}
- Investment Horizon: {state.user_profile.investment_horizon}
"""

    # Single LLM call creates the complete plan
    result: PlanOutput = structured_llm.invoke([
        SystemMessage(content=PLANNER_SYSTEM_PROMPT + profile_context),
        HumanMessage(content=f"Create a research plan for: {last_user_message}"),
    ])

    # State update: Store the plan and initialize tracking
    return {
        "plan": result.steps,           # The sequential steps to execute
        "current_step_index": 0,        # Start at step 0
        "research_data": ResearchData(ticker=result.ticker),  # Initialize data container
    }

Dat llm.with_structured_output(PlanOutput) Deze regel is Schema-Guided Reasoning (SGR), die ik al heb behandeld in een eerdere post. Het forceerden van de PlanOutput schema betekent dat de planner altijd een geldige lijst met stappen teruggeeft. LangGraph gebruikt deze gestructureerde uitvoer vervolgens om een deterministische controlestroom te realiseren via conditionele randen.

Patroon 2: ReAct-executie

Zodra het plan beschikbaar is, voert de executor elke stap uit als een aparte ReAct-lus. Dit is fase 2: elke stap is voldoende klein zodat de Thought-Action-Observation-cyclus gefocust blijft, en het agent kan reageren op wat de tool teruggeeft.

Hoe het ReAct-deel past:

  1. Iteratieve uitvoering. Eén stap tegelijk, met feedback van observaties.
  2. De Thought-Action-Observation-cyclus loopt binnenin create_react_agent3. De resultaten van de vorige stap worden gebruikt als context voor het huidige redeneren.
  3. Het agent kiest de benodigde hulpmiddelen op basis van de beschrijving van de stap.
  4. Het kan zijn aanpak halverwege een stap aanpassen, afhankelijk van de output van een hulpmiddel.
# Tools available for the ReAct agent to choose from
TOOLS = [
    get_stock_snapshot,
    get_price_history,
    search_news,
    search_competitors,
    get_financials,
]

def executor_node(state: AgentState) -> dict:
    """Execute the current step using a ReAct agent.

    This is Phase 2 of Plan-and-Execute: adaptive execution of each planned step.
    Each step runs as a mini ReAct loop until completion.
    """

    # Get the current step from the plan
    current_step = state.plan[state.current_step_index]

    # Build context from what we've learned so far
    # This matters: each step builds on previous observations
    previous_context = ""
    for step in state.plan[:state.current_step_index]:
        if step.result:
            previous_context += f"\nStep {step.step_number}: {step.result}\n"

    # Create a ReAct agent for this step
    # LangGraph's create_react_agent implements the full Thought-Action-Observation loop:
    # 1. Agent generates a "thought" about what tool to call
    # 2. Agent calls the tool ("action")
    # 3. Tool returns result ("observation")
    # 4. Agent decides: call another tool or finish
    react_agent = create_react_agent(
        model=ChatAnthropic(model="claude-sonnet-4-5-20250929"),
        tools=TOOLS,
    )

    # Invoke the ReAct loop for this single step
    # The agent will loop internally until it completes the step
    result = react_agent.invoke({
        "messages": [
            SystemMessage(content=EXECUTOR_SYSTEM_PROMPT),
            HumanMessage(content=f"""Execute Step {current_step.step_number}:
{current_step.description}

Ticker: {state.research_data.ticker}
Previous findings: {previous_context}"""),
        ]
    })

    # Extract the final answer from the ReAct agent's message history
    # The last message contains the synthesis after all tool calls
    updated_plan = list(state.plan)
    updated_plan[state.current_step_index] = PlanStep(
        step_number=current_step.step_number,
        description=current_step.description,
        completed=True,
        result=result["messages"][-1].content,  # Final observation
    )

    # State update: Mark step complete and advance to next
    return {
        "plan": updated_plan,
        "current_step_index": state.current_step_index + 1,
    }

Dit is de kern van het Plan-en-Execute-proces: een krachtig model stelt het plan op, waarna ReAct elke stap uitvoert met volledige aanpasbaarheid.

Patroon 3: ReWOO voor snelle snapshots

Voor snelle briefingen slaat ReWOO het gecombineerde redeneren over en voert de tools parallel uit. De planner genereert van tevoren een gecompileerd script met tooloproepen, en de worker voert dit script uit zonder verdere betrokkenheid van een LLM.

De structuur daarvan:

  1. Drie fasen (Planner → Worker → Solver), geen lussen.
  2. Toolaanroepen verwijzen naar #E1, #E2 Plaatshouders voor resultaten die nog niet beschikbaar zijn.
  3. Geen LLM tijdens de uitvoering. De worker gebruikt enkel tools.
  4. Onafhankelijke tools worden parallel uitgevoerd.
  5. Eén syntheseoproep aan het eind, waarbij alle gegevens tegelijk worden verwerkt.

Fase 1: ReWOO planner (creëert van tevoren de volledige uitvoeringsgraaf)

class ReWOOPlanStep(BaseModel):
    """A step in the ReWOO plan with variable placeholders.

    Key difference from Plan-and-Execute's PlanStep:
    - Contains actual tool_name and tool_args (not just description)
    - Uses variable references (#E1) for dependencies
    """
    step_id: str  # e.g., "#E1" - becomes a variable
    description: str
    tool_name: str     # Exact tool to call
    tool_args: dict    # May contain variable refs like {"price": "#E1"}
    depends_on: list[str] = []  # For dependency ordering
    result: str | None = None

class ReWOOPlanOutput(BaseModel):
    """Structured output for ReWOO planner."""
    steps: list[ReWOOPlanStep] = Field(description="Planned tool calls with variables")

def rewoo_planner_node(state: AgentState) -> dict:
    """Generate a complete plan of tool calls upfront.

    This is the key difference from Plan-and-Execute: instead of creating
    human-readable step descriptions, we create EXACT tool calls that
    the worker will execute blindly.
    """

    llm = ChatAnthropic(model="claude-sonnet-4-5-20250929", temperature=0)

    # Schema-Guided Reasoning ensures valid tool call specifications
    structured_llm = llm.with_structured_output(ReWOOPlanOutput)

    ticker = state.research_data.ticker if state.research_data else "UNKNOWN"

    # Single LLM call to plan ALL tool executions
    result: ReWOOPlanOutput = structured_llm.invoke([
        SystemMessage(content=REWOO_PLANNER_PROMPT),
        HumanMessage(content=f"""Create a ReWOO plan for: {query}

Ticker: {ticker}

Output tool calls with:
- step_id: Variable name (#E1, #E2, etc.)
- description: What this accomplishes
- tool_name: Exact tool from the list
- tool_args: Dictionary of arguments
- depends_on: List of step_ids this depends on"""),
    ])

    # State update: Store the complete execution plan
    # Worker will execute this without any LLM involvement
    return {"rewoo_plan": result.steps}

Fase 2: ReWOO worker (voert hulpprogramma’s uit zonder redenering door een LLM)

def rewoo_worker_node(state: AgentState) -> dict:
    """Execute all planned tools in parallel (no LLM calls).

    This is the key efficiency: Worker is "dumb" - it just runs tools
    according to the plan. No LLM calls = massive token savings.
    """

    results = {}  # Store results keyed by step_id (e.g., "#E1": "$150.23")

    # Execute ALL independent steps in parallel using ThreadPoolExecutor
    # This is where ReWOO gets its speed advantage
    with ThreadPoolExecutor(max_workers=5) as executor:
        futures = {
            executor.submit(execute_tool, step): step
            for step in state.rewoo_plan
            if not step.depends_on  # Only independent tools for parallel batch
        }

        # Collect results as they complete
        for future in as_completed(futures):
            step = futures[future]
            results[step.step_id] = future.result()
            # No LLM reasoning here - just store the raw tool output

    # State update: Store results for the Solver phase
    return {"rewoo_plan": updated_steps}

Fase 3: ReWOO solver (synthetiseert alle resultaten in één LLM-oproep)

def rewoo_solver_node(state: AgentState) -> dict:
    """Synthesize all tool results into a flash briefing.

    This is the second efficiency gain: Instead of interleaving
    LLM calls with tool execution (like ReAct), we make ONE
    final synthesis call with all gathered data.
    """

    # Build context from ALL tool results at once
    tool_results = []
    for step in state.rewoo_plan:
        if step.result:
            tool_results.append(f"### {step.description}\n{step.result}")

    context = "\n\n".join(tool_results)

    # Single LLM call to synthesize everything
    structured_llm = llm.with_structured_output(FlashBriefingOutput)
    result = structured_llm.invoke([
        SystemMessage(content=REWOO_SOLVER_PROMPT),
        HumanMessage(content=f"Create a flash briefing from this data:\n\n{context}"),
    ])

    return {"draft_report": result}

Het belangrijkste verschil: ReWOO plant elke toolaanroep van tevoren in met placeholders (#E1, #E2), voert ze parallel uit zonder tussentijdse oproepen aan een LLM, en synthetiseert de resultaten in één oproep aan het eind. Hierdoor is het goedkoop voor voorspelbare werkstromen.

plan = [

        step_number=2,

]

    Meerdere oproepen aan een LLM per stap, waarbij de strategie wordt aangepast op basis van waarnemingen. Het meest flexibel, maar ook het duurst.

        step_id="#E2",
        tool_name="search_news",
        tool_args={"query": "NVDA earnings", "limit": 5}
    ),
    # ... all tool calls planned upfront
]

    De worker draait blind, zonder enige betrokkenheid van een LLM. Alle redenering vindt plaats in de planner en solver. Dit is de goedkoopste optie van de drie.

Geheugen- en staatstroom:

  • Plan-en-uitvoeren: de toestand gaat door naar plan → current_step_index → research_data- ReAct: de toestand wordt opgebouwd in de messages array (het volledige gespreksverloop).
  • ReWOO: de toestand verandert doorheen rewoo_plan, met result Velden die door de gebruiker worden ingevuld.

Alles samenbrengen: het netwerk aansluiten

Hieronder zie je hoe de drie patronen samenbestaan binnen één LangGraph-systeem. Ze delen één AgentState en leven ze in één graaf. Een router kiest de route per verzoek, dus dit is één agent met drie uitvoeringsmodi, en niet drie agenten onder één hoedje.

LangGraph houdt de indeling declaratief:

def create_graph(checkpointer=None):
    builder = StateGraph(AgentState)

    # Add nodes
    builder.add_node("router", router_node)
    builder.add_node("planner", planner_node)
    builder.add_node("executor", executor_node)
    builder.add_node("reporter", reporter_node)
    builder.add_node("rewoo_planner", rewoo_planner_node)
    builder.add_node("rewoo_worker", rewoo_worker_node)
    builder.add_node("rewoo_solver", rewoo_solver_node)

    # Define edges
    builder.add_edge(START, "router")
    builder.add_conditional_edges("router", route_after_router, {
        "planner": "planner",
        "rewoo_planner": "rewoo_planner",
    })

    # Deep Research path
    builder.add_edge("planner", "executor")
    builder.add_conditional_edges("executor", route_after_executor, {
        "executor": "executor",  # Loop back for more steps
        "reporter": "reporter",  # Done with plan
    })
    builder.add_edge("reporter", END)

    # Flash Briefing path (ReWOO)
    builder.add_edge("rewoo_planner", "rewoo_worker")
    builder.add_edge("rewoo_worker", "rewoo_solver")
    builder.add_edge("rewoo_solver", END)

    return builder.compile(
        checkpointer=checkpointer,
        interrupt_before=["reporter"],  # HITL pause for approval
    )

Automatische patroonselectie met een router

Om voor elke verzoek het juiste patroon te kiezen, heb ik een router-classificator toegevoegd. Deze maakt gebruik van Schema-Guided Reasoning om de classificatie betrouwbaar te houden:

class ExecutionMode(str, Enum):
    """Execution mode for the agent."""

    DEEP_RESEARCH = "deep_research"  # Plan-and-Execute + ReAct (thorough)
    FLASH_BRIEFING = "flash_briefing"  # ReWOO (fast, token-efficient)

class RouterOutput(BaseModel):
    """Structured output for the router."""

    mode: ExecutionMode  # DEEP_RESEARCH or FLASH_BRIEFING
    ticker: str
    reasoning: str

ROUTER_SYSTEM_PROMPT = """Classify the user's request:

1. **deep_research**: Complex analysis requiring synthesis
   - Examples: "Analyze strategic risks", "investment thesis"

2. **flash_briefing**: Quick snapshots, simple data retrieval
   - Examples: "quick snapshot", "current price"

Default to deep_research if unclear."""

structured_llm = llm.with_structured_output(RouterOutput)

Met deze oplossing hoeven gebruikers geen modus te kiezen. De router stuurt automatisch “huidige prijs” naar ReWOO en “investeringsthese” naar Plan-and-Execute.

Belangrijkste conclusies

  1. ReAct is de standaard voor flexibiliteit, maar dit heeft als nadeel meer tokens en lagere snelheid.
  2. ReWOO is superieur wat betreft snelheid en kosten wanneer de tools betrouwbaar zijn en de resultaten voorspelbaar.
  3. Plan-and-Execute is de juiste keuze voor complexe analyses waarbij aan het eind synthese nodig is.
  4. Een router kan per verzoek kiezen tussen deze opties, zodat gebruikers dit niet zelf hoeven te doen.
  5. State management is cruciaal. Het checkpointing-mechanisme van LangGraph maakt onderbrekingen en herstel mogelijk.
